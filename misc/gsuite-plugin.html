用 React 开发 Google Workspace 插件：架构、限制与 Workaround

2026-03-08

背景

Google Workspace（原 GSuite）的插件生态有一个尴尬的现实：AppScript 是唯一的官方开发方式。你的后端逻辑必须用 AppScript 写，前端 UI 必须是 HtmlService 生成的 HTML，没有 npm，没有构建工具，没有模块系统。

但用户期待的是现代 Web 应用的交互体验——组件化、状态管理、类型安全。在 GSuite 的沙箱里写原生 HTML + jQuery，开发体验和产品质量都很痛苦。

这篇文章分享一个实际的方案：用 React + TypeScript + Webpack 开发 GSuite 插件的 sidebar 和 dialog，编译成符合 GSuite 安全要求的自包含 HTML 文件。

GSuite 插件的运行模型

首先要理解 GSuite 插件是怎么跑的：

┌──────────────────────────────────────────┐
│           Google Docs / Sheets / ...      │
│                                           │
│   ┌─────────────┐    ┌───────────────┐   │
│   │   Sidebar    │    │    Dialog     │   │
│   │  (HTML/JS)   │    │  (HTML/JS)   │   │
│   │              │    │              │   │
│   │  ← iframe →  │    │  ← iframe → │   │
│   └──────┬───────┘    └──────┬───────┘   │
│          │                    │           │
│          │  google.script.run │           │
│          ▼                    ▼           │
│   ┌──────────────────────────────────┐   │
│   │        AppScript (服务端)         │   │
│   │  - 访问 Drive / Gmail / Sheets   │   │
│   │  - 操作文档内容                    │   │
│   │  - 发送邮件                       │   │
│   └──────────────────────────────────┘   │
└──────────────────────────────────────────┘

关键约束：

  • Sidebar 和 Dialog 运行在 iframe 沙箱里，不能加载外部脚本（CSP 限制），所有 JS/CSS 必须内联在 HTML 里
  • 前端和 AppScript 之间通过 google.script.run 通信，这是一个异步 RPC 调用——前端调用服务端函数，通过 withSuccessHandler / withFailureHandler 拿结果
  • 一个插件可以运行在多个 GSuite 产品里（Docs、Sheets、Slides、Forms），但每个产品的 API 不同（DocumentApp vs SpreadsheetApp vs SlidesApp）

核心问题：CSS/JS 必须内联

GSuite 的 iframe 沙箱禁止加载外部资源。正常的 Webpack 构建会生成独立的 .js 和 .css 文件，通过 <script src> 引用——这在 GSuite 里不工作。

解决方案是 html-inline-script-webpack-plugin，它把 Webpack 输出的 JS bundle 直接内联到 HTML 的 <script> 标签里：

// webpack.config.js
const HtmlInlineScriptPlugin = require("html-inline-script-webpack-plugin");

module.exports = {
  plugins: [
    new HtmlWebpackPlugin({
      template: "./public/index.html",
      filename: "docs/sidebar.html",
      chunks: ["sidebar"],
      inject: "body",
    }),
    new HtmlInlineScriptPlugin(),  // 把 JS 内联进 HTML
  ],
};

CSS 用 style-loader 处理，它会把 CSS 注入到 <style> 标签里。最终每个 HTML 文件是完全自包含的——零外部依赖，直接丢给 HtmlService.createHtmlOutputFromFile() 就能用。

自动发现 Dialog

添加新 dialog 不应该需要改构建配置。Webpack 配置在启动时自动扫描 src/dialogs/ 目录：

function discoverDialogs() {
  const dialogsPath = path.resolve(__dirname, "src/dialogs");
  if (!fs.existsSync(dialogsPath)) return [];

  return fs.readdirSync(dialogsPath, { withFileTypes: true })
    .filter(item => item.isDirectory())
    .filter(dir => {
      // 只有包含 index.tsx 的目录才算
      return fs.existsSync(path.join(dialogsPath, dir.name, "index.tsx"));
    })
    .map(dir => dir.name);
}

// 动态生成 entry points
const entries = { sidebar: "./src/sidebar/index.tsx" };
discoverDialogs().forEach(name => {
  entries[name] = `./src/dialogs/${name}/index.tsx`;
});

然后为每个 GSuite 产品 × 每个 entry 生成一个 HTML：

const GSUITE_PRODUCTS = ["docs", "slides", "sheets", "forms"];

GSUITE_PRODUCTS.forEach(product => {
  // sidebar
  htmlPlugins.push(new HtmlWebpackPlugin({
    filename: `${product}/sidebar.html`,
    chunks: ["sidebar"],
  }));

  // 每个 dialog
  dialogs.forEach(name => {
    htmlPlugins.push(new HtmlWebpackPlugin({
      filename: `${product}/${name}.html`,
      chunks: [name],
    }));
  });
});

4 个产品 × 4 个组件 = 16 个 HTML 文件，全自动生成。加一个新 dialog？在 src/dialogs/ 下建个文件夹就行。

前端和 AppScript 的通信

google.script.run 是唯一的通信通道。它的 API 设计很原始——callback-based，没有 Promise，没有类型：

// AppScript 侧（服务端）
function getGoogleSheets() {
  var files = DriveApp.getFilesByType(MimeType.GOOGLE_SHEETS);
  var sheets = [];
  while (files.hasNext()) {
    var file = files.next();
    sheets.push({
      id: file.getId(),
      name: file.getName(),
      url: file.getUrl(),
    });
  }
  return { success: true, sheets: sheets };
}

// React 侧（客户端）
google.script.run
  .withSuccessHandler((result) => {
    if (result.success) {
      setGoogleSheets(result.sheets);
    }
  })
  .withFailureHandler((error) => {
    console.error("Error:", error);
  })
  .getGoogleSheets();

在 React 里，每个 AppScript 调用都封装成一个 useEffect：

// 组件挂载时加载 Google Sheets 列表
useEffect(() => {
  if (typeof google !== "undefined" && google.script) {
    google.script.run
      .withSuccessHandler((result) => {
        if (result.success) setGoogleSheets(result.sheets);
        setLoadingSheets(false);
      })
      .withFailureHandler((error) => {
        console.error(error);
        setLoadingSheets(false);
      })
      .getGoogleSheets();
  } else {
    // 开发模式——用 mock 数据
    setLoadingSheets(false);
  }
}, []);

注意 typeof google !== "undefined" 的检查——本地开发时 google.script.run 不存在，需要 fallback 到 mock 数据。这让你可以在浏览器里直接开发和调试 UI，不需要每次都部署到 GSuite。

跨产品适配

同一个插件要跑在 Docs、Sheets、Slides、Forms 里，但每个产品的 API 对象不一样。AppScript 没有"当前是哪个产品"的 API，只能用 try/catch 链来检测：

function showSidebar() {
  var html = HtmlService.createHtmlOutputFromFile("sidebar")
    .setTitle("Omni Assistant")
    .setWidth(320);

  try {
    DocumentApp.getUi().showSidebar(html);
  } catch (err) {
    try {
      SpreadsheetApp.getUi().showSidebar(html);
    } catch (err) {
      try {
        SlidesApp.getUi().showSidebar(html);
      } catch (err) {
        try {
          FormApp.getUi().showSidebar(html);
        } catch (err) {
          Logger.log("Error showing sidebar: " + err);
        }
      }
    }
  }
}

这段代码很丑，但这是 AppScript 的现实——没有 getActiveApp() 这样的统一 API。每个需要调用 UI 的函数都要写这个 try/catch 链。可以封装成 helper：

function getUi() {
  try { return DocumentApp.getUi(); } catch(e) {}
  try { return SpreadsheetApp.getUi(); } catch(e) {}
  try { return SlidesApp.getUi(); } catch(e) {}
  try { return FormApp.getUi(); } catch(e) {}
  return null;
}

function showSidebar() {
  var ui = getUi();
  if (!ui) return;
  var html = HtmlService.createHtmlOutputFromFile("sidebar")
    .setTitle("Omni Assistant").setWidth(320);
  ui.showSidebar(html);
}

实际案例：Mail Merge

用这套架构做一个 Mail Merge（邮件合并）功能——从 Google Sheets 读收件人列表，用 Google Docs 做邮件模板，批量发送个性化邮件：

用户操作流程（4 步向导）：

Step 1: 选数据源
  → 列出用户 Drive 里的 Google Sheets
  → 选择文件 → 选择 sheet tab
  → 预览 headers 和行数

Step 2: 选模板
  → 列出用户 Drive 里的 Google Docs
  → 预览模板内容（HTML 格式）

Step 3: 配置
  → 输入邮件主题（支持 {{Name}} 合并字段）
  → 可选 reply-to 地址
  → 显示可用的合并字段列表

Step 4: 执行
  → 汇总确认
  → Test mode（只发给自己）
  → 执行并显示结果

前端是标准的 React 多步表单，每一步的数据通过 google.script.run 从 AppScript 获取。AppScript 负责所有和 Google 服务的交互：

// 读 Sheets 数据
function getSelectedSheetData(spreadsheetId, worksheetName) {
  var spreadsheet = SpreadsheetApp.openById(spreadsheetId);
  var sheet = spreadsheet.getSheetByName(worksheetName);
  var values = sheet.getDataRange().getValues();
  var headers = values[0];
  return {
    success: true,
    headers: headers,
    dataRowCount: values.length - 1,
  };
}

// 读 Docs 内容（导出为 HTML，保留格式）
function getDocumentHtml(docId) {
  var url = "https://docs.google.com/feeds/download/documents/export/Export"
          + "?id=" + docId + "&exportFormat=html";
  var response = UrlFetchApp.fetch(url, {
    headers: { Authorization: "Bearer " + ScriptApp.getOAuthToken() }
  });
  return { success: true, html: response.getContentText() };
}

// 合并字段替换
function replaceMergeFields(text, data) {
  return text.replace(/\{\{([^}]+)\}\}/g, function(match, field) {
    return data[field.trim()] || match;
  });
}

// 发送邮件
function sendMergeEmail(recipient, subject, htmlBody, replyTo) {
  GmailApp.sendEmail(recipient, subject, "", {
    htmlBody: htmlBody,
    replyTo: replyTo || undefined,
  });
}

完整的 Mail Merge 流程：遍历 Sheets 每一行 → 用行数据替换模板里的 {{Field}} → 调用 Gmail API 发送。支持 test mode（所有邮件发给自己）、quota 检查（Gmail 有日发送量限制）、错误收集。

TypeScript 类型安全

google.script.run 没有类型定义。需要手动声明：

declare const google: {
  script: {
    run: {
      withSuccessHandler: (cb: (result: any) => void) => {
        withFailureHandler: (cb: (error: any) => void) => {
          getGoogleSheets: () => void;
          getWorksheets: (id: string) => void;
          getSelectedSheetData: (id: string, name: string) => void;
          getDocumentHtml: (id: string) => void;
          sendMailMerge: (
            spreadsheetId: string,
            worksheetName: string,
            docId: string,
            subject: string,
            replyTo: string,
            testMode: boolean
          ) => void;
        };
      };
    };
  };
};

这样至少能在 TypeScript 层面保证调用参数的类型正确。返回值还是 any——因为 AppScript 是动态类型的，没法在编译时知道返回什么。

部署流程

用 Google 的 clasp CLI 把构建产物推到 AppScript 项目：

# 1. 构建
npm run build    # → dist/docs/sidebar.html, dist/docs/about.html, ...

# 2. 推送到 AppScript
cd clasp/docs
clasp push       # 上传 HTML + .gs 文件到 Google Apps Script

# 3. 在 GSuite 里测试
# 打开任意 Google Doc → 扩展程序 → Omni Assistant → Start

.clasp.json 里配置了 AppScript 项目 ID 和 root 目录。clasp push 会把目录下的所有 .html 和 .gs 文件上传。

开发体验的痛点和解法

  • 本地开发：google.script.run 在浏览器里不存在。解法：检测 typeof google，不存在时用 mock 数据。这样可以在本地用 webpack-dev-server 开发 UI，不需要每次部署。
  • 调试：GSuite iframe 里的 console.log 很难看到。解法：AppScript 侧用 Logger.log()，在 Apps Script 编辑器的执行日志里查看。前端的错误可以通过 withFailureHandler 捕获。
  • Bundle 大小：每个 HTML 文件约 1.1MB（因为 React 被内联进去了）。4 个产品 × 4 个组件 = ~18MB 总产物。解法：用 TerserPlugin 压缩、drop_console 移除日志。如果还嫌大，可以用 Preact 替代 React。
  • AppScript 的 6 分钟执行限制：长时间运行的操作（如发送 1000 封邮件）会超时。解法：在循环里加 Utilities.sleep(100) 控制速率，检查 MailApp.getRemainingDailyQuota() 提前终止。
  • 没有 hot reload：每次改 AppScript 都要 clasp push + 刷新 GSuite 页面。这是最大的开发体验问题，目前没有好的解法。

这个方案的适用场景

  • 适合：需要在 GSuite sidebar/dialog 里放一个交互复杂的 UI（多步表单、实时预览、条件渲染）。用原生 HTML 写这些会很痛苦，React 的组件化和状态管理在这里有明显优势。
  • 不适合：简单的插件（一两个按钮、一个输入框）。引入 React + Webpack 的构建复杂度不值得。直接写 HTML + 几行 JavaScript 就够了。
  • 也不适合：需要大量和文档内容交互的插件（如实时协同编辑工具）。这类插件的核心逻辑在 AppScript 侧，前端只是触发器，React 的优势不明显。

总结

在 GSuite 的沙箱里开发现代前端应用，核心挑战是两个世界的碰撞：现代前端工具链（React、TypeScript、Webpack、npm）和AppScript 的原始运行环境（全局函数、callback-based RPC、no module system、try/catch 检测产品类型）。

这套方案的关键决策：

  • 用 HtmlInlineScriptPlugin 把 JS 内联到 HTML，满足 CSP 限制
  • 用文件系统扫描自动发现 dialog，消除构建配置的维护成本
  • 为每个 GSuite 产品生成独立的文件夹，用 clasp 分别部署
  • 用 typeof google 检测实现本地开发和 GSuite 环境的兼容

不完美，但可用。在 Google 提供更好的插件开发体验之前，这大概是能做到的最好了。
