PostgreSQL Storage Engine Internals (Part 2): Index Scans and MVCC

2026-03-08
PostgreSQL Storage Engine Internals
Part 1: Heap Pages and Buffer Pool · Part 2: Index Scans and MVCC · Part 3: WAL, Checkpoint, and Durability
What you'll learn from this post: The previous post covered Heap Page internal layout and how the Buffer Pool manages page caching. But a key question remains unanswered: when multiple transactions simultaneously read and write the same table, how is the data each transaction sees determined? This post traces two paths — how index scans jump from B-Tree leaf nodes to Heap Pages to retrieve data, and how MVCC snapshots determine whether a row is "visible or invisible to you." Understanding these will show you that transaction isolation levels are not an abstract concept, but a concrete data structure.

Index Scans: From B-Tree to Heap

The previous post covered sequential scans — traversing the entire Heap page by page. Index scans take a different path: first locate the TID in the index, then use the TID to fetch data from the Heap.

Indexes Are Also Tables

In PostgreSQL, an index is a Relation just like a regular table, with its own files and its own Page structure. The difference is that index Pages don't store user data, but rather index key + TID mappings.

  CREATE INDEX users_age_idx ON users(age);

  An index scan involves two Relations:

  indexRelation (users_age_idx)          heapRelation (users)
  ┌─────────────────────────┐           ┌──────────────────────────┐
  │  B-Tree Page            │           │  Heap Page               │
  │                         │           │                          │
  │  ┌───────────────────┐  │           │  ┌────────────────────┐  │
  │  │ age=20 → TID(1,3) │  │     ┌────►  │ ItemId[3] → Tuple  │  │
  │  │ age=25 → TID(5,1) │──┼─────┘    │  │  (id=42, name=     │  │
  │  │ age=25 → TID(8,7) │  │          │  │   'Alice', age=25) │  │
  │  │ age=30 → TID(2,5) │  │          │  └────────────────────┘  │
  │  └───────────────────┘  │           │                          │
  └─────────────────────────┘           └──────────────────────────┘

  IndexScanDesc holds references to both Relations

IndexScanDesc is the index scan descriptor with key fields:

  • heapRelation: The table being indexed
  • indexRelation: The index itself
  • xs_heaptid: The currently found TID, used for the next heap fetch step

Runtime Key is an important concept: values that can't be determined at query planning time. For example, the parameter in WHERE id = $1, or the subquery result in WHERE id = (SELECT max_id FROM others). These values are computed at executor startup, then passed to the index scan.

Inside B-Tree: btgettuple

B-Tree is PostgreSQL's most commonly used index type. The logic of btgettuple is straightforward:

  • First call: _bt_first() locates the starting position in the B-Tree — traverses from root to leaf node, finding the first index entry that satisfies the scan condition
  • Subsequent calls: _bt_next() moves forward/backward on the leaf node linked list, getting the next entry
  • After finding the position, _bt_returnitem() extracts the TID from the current position: scan->xs_heaptid = currItem->heapTid
  B-Tree structure (simplified):

          ┌──────────────┐
          │  Root Page   │
          │  [20] [50]   │
          └──┬──────┬────┘
             │      │
      ┌──────▼──┐  ┌▼──────────┐
      │ Internal │  │ Internal   │
      │ [10][20] │  │ [30][50]   │
      └──┬───┬──┘  └──┬────┬───┘
         │   │        │    │
    ┌────▼┐ ┌▼────┐ ┌─▼──┐ ┌▼────┐
    │Leaf │↔│Leaf │↔│Leaf│↔│Leaf │    ← Leaf node doubly linked list
    │5,8  │ │12,18│ │25  │ │35,42│
    │TIDs │ │TIDs │ │TIDs│ │TIDs │
    └─────┘ └─────┘ └────┘ └─────┘

  WHERE age = 25:
  1. _bt_first: Root → Internal → Locate 3rd Leaf
  2. Find age=25 entry in Leaf, extract TID
  3. _bt_next: Continue scanning on Leaf linked list until age ≠ 25

Heap Fetch: From TID to Tuple

After obtaining a TID, index_fetch_heap fetches data from the Heap. For heap tables, it actually calls heapam_index_fetch_tuple, with the core being heap_hot_search_buffer:

// Complete path for index heap fetch (simplified)

// 1. Index provides TID = (5, 1)
scan->xs_heaptid = {block: 5, offset: 1};

// 2. Read Heap Page 5 into Buffer Pool
buffer = ReadBuffer(heapRelation, 5);

// 3. Search HOT chain on Page 5
heap_hot_search_buffer(
    tid,        // Starting TID (5, 1)
    relation,   // users table
    buffer,     // Page 5's buffer
    snapshot,   // Current transaction's snapshot  ← MVCC kicks in here
    &found
);

// 4. Traverse HOT chain:
//    ItemId[1] → Tuple A (t_ctid points to ItemId[2])
//    ItemId[2] → Tuple B (t_ctid points to ItemId[3])
//    ItemId[3] → Tuple C (t_ctid points to itself = end of chain)
//
//    Call HeapTupleSatisfiesVisibility(snapshot) for each version
//    Return the version visible to the current snapshot

Note the snapshot parameter here — this is the entry point for MVCC. The index itself doesn't store version information; all version determination happens after the heap fetch. This means an index scan may get a TID from the index, find the row invisible to the current transaction after fetching from the heap, and then continue to get the next TID.

Lossy Indexes and Recheck

Certain indexes (like GIN, GiST) are "lossy" — the index tells you "this Tuple might match" but doesn't guarantee a match. After heap fetch, the original condition must be rechecked:

// Recheck logic in IndexNext
while (index_getnext_slot(scandesc, direction, slot)) {
    if (scandesc->xs_recheck) {
        // Index is lossy, need to re-verify with original condition
        if (!ExecQualAndReset(node->indexqualorig, econtext))
            continue;  // Doesn't match, skip
    }
    return slot;  // Matches, return
}

MVCC: Multi-Version Concurrency Control

Now we enter the core of this post. MVCC's goal is reads don't block writes, writes don't block reads. PG achieves this by having multiple versions of each row exist simultaneously — each transaction sees a "snapshot" from a certain point in time.

Transaction ID: The Logical Clock

PG uses Transaction IDs (TransactionId, or XID) as its "logical clock." Each write transaction is assigned a monotonically increasing XID at startup. The core of visibility determination is comparing XIDs:

Row header fieldMeaningVisibility check
t_xminTransaction that created this versionIf xmin's transaction is committed and before the snapshot → this version "was born"
t_xmaxTransaction that deleted/updated this versionIf xmax = 0 or xmax is uncommitted → this version "is still alive"

A Tuple is visible to you when: it has been born (xmin committed) and hasn't died yet (xmax invalid or uncommitted).

  Transaction timeline:

  XID:  100      101      102      103      104
         │        │        │        │        │
         │  INSERT │        │ UPDATE │        │
         │  xmin=101       │ new ver│        │
         │        │        │ xmin=103│        │
         │        │        │ old ver│        │
         │        │        │ xmax=103│        │

  Transaction 104's snapshot taken here ──────────────────►│

  For transaction 104:
  Old version: xmin=101(committed✓) xmax=103(committed✓) → dead → invisible
  New version: xmin=103(committed✓) xmax=0(still alive✓)  → visible ✓

SnapshotData: The Data Structure of a Snapshot

A snapshot is not a "point in time" but a concrete data structure:

typedef struct SnapshotData {
    TransactionId xmin;    // All XIDs < xmin are committed (definitely visible)
    TransactionId xmax;    // All XIDs >= xmax haven't started (definitely invisible)
    TransactionId *xip;    // List of running transactions between xmin and xmax
    uint32 xcnt;           // Length of xip array
} SnapshotData;
  Three intervals of a snapshot:

  XID:  95  96  97  98  99  100  101  102  103  104  105
        ────────────┬───────────────────────┬────────────
                    │                       │
               snapshot.xmin           snapshot.xmax

  ◄── Definitely  ──►│◄── Need to check  ──►│◄── Definitely   ──►
      visible         │    xip array         │    invisible

  xip[] = {99, 101}  ← Transactions still running when snapshot was taken

  Decision logic:
  XID 96: < xmin → committed → visible ✓
  XID 99: in xip → still running → invisible ✗
  XID 100: not in xip → committed → visible ✓
  XID 101: in xip → still running → invisible ✗
  XID 104: >= xmax → not started → invisible ✗

This design is elegant: xmin and xmax quickly eliminate most transactions, and only those falling in the middle interval need to check the xip array.

Isolation Levels: When Is the Snapshot Taken

The essential difference between transaction isolation levels isn't "what you see" but when the snapshot is taken:

Isolation LevelSnapshot StrategyEffect
Read CommittedTake a new snapshot at the start of each statementWithin the same transaction, two SELECTs may see different data
Repeatable ReadTake one snapshot at transaction start, reuse throughoutWithin the same transaction, multiple SELECTs see consistent data
SerializableSame as Repeatable Read + additional conflict detectionSimulates serial execution, rolls back on detected conflicts
 
-- Read Committed: new snapshot per statement
BEGIN;
SELECT count(*) FROM users;  -- Snapshot S1 → sees 100 rows
-- Meanwhile another transaction inserts 1 row and commits
SELECT count(*) FROM users;  -- Snapshot S2 (new!) → sees 101 rows
COMMIT;

-- Repeatable Read: reuse one snapshot for entire transaction
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM users;  -- Snapshot S1 → sees 100 rows
-- Meanwhile another transaction inserts 1 row and commits
SELECT count(*) FROM users;  -- Still snapshot S1 → still sees 100 rows
COMMIT;

At the code level, GetTransactionSnapshot() handles this decision:

  • Read Committed: Creates a new snapshot on every call
  • Repeatable Read / Serializable: Creates and caches a snapshot on first call, returns the cached snapshot on subsequent calls

When Is a Snapshot Needed

Not all SQL statements need a snapshot:

Statement TypeNeeds Snapshot?Reason
SELECT / INSERT / UPDATE / DELETEYesNeeds to access table data, must determine visibility
CREATE TABLE / DROP TABLE / ALTERNoDDL operates on metadata, doesn't involve row-level visibility
EXPLAINMaybeIf it's EXPLAIN ANALYZE that requires actual execution, then yes

PG uses the analyze_requires_snapshot function to determine whether the current statement needs a snapshot, avoiding the overhead of creating snapshots for operations that don't need them.

Complete Visibility Rules

Actual visibility determination is much more complex than "xmin committed and xmax invalid." HeapTupleSatisfiesMVCC must handle these scenarios:

Scenarioxmin statusxmax statusResult
Just inserted, transaction uncommittedIn progress—Visible only to the inserter
Inserted and committedCommitted, before snapshotInvalid (0)Visible
Deleted, delete transaction uncommittedCommittedIn progressVisible (delete hasn't taken effect)
Deleted and committedCommittedCommitted, before snapshotInvisible (dead)
Updated, new version existsCommittedCommittedInvisible (see new version)
Insert transaction rolled backRolled back—Invisible (never existed)

PG also has an optimization: hint bits in t_infomask. The first time a row's xmin/xmax status is checked, it requires querying the CLOG (transaction commit log). Once confirmed, "committed" or "rolled back" markers are set in t_infomask, and subsequent checks read the markers directly, skipping the CLOG query.

EPQ: Conflict Handling for Concurrent Updates

When a scan discovers that a Tuple is being modified by another transaction, the EvalPlanQual (EPQ) mechanism is triggered:

  • During normal scanning, a target Tuple is found to be concurrently modified
  • Wait for the modifying transaction to complete (commit or rollback)
  • Get the new version of the Tuple after modification
  • Re-evaluate the new version with the current query's WHERE conditions
  • If the new version still satisfies the conditions → return the new version; otherwise skip

This is a necessary mechanism for UPDATE and DELETE under Read Committed isolation level — you can't skip a row being modified, nor can you make decisions based on the old version.

Scan Method Overview

PG provides multiple scan methods, each with a different accessMtd function pointer:

Scan MethodaccessMtdPrincipleUse Case
Sequential ScanSeqNextTraverse entire Heap page by pageFull table scan, no index
Index ScanIndexNextB-Tree locate → heap fetch TupleHighly selective queries
Index Only ScanIndexOnlyNextRead index only, no heap fetchQuery columns all in the index
Bitmap Heap ScanBitmapHeapNextCollect all TIDs first, sort, then batch heap fetchMedium selectivity, reduces random I/O
TID ScanTidNextLocate directly by TIDCURRENT OF cursor

All scan methods are ultimately dispatched through ExecScanExtended:

// Unified scan framework
ExecScanExtended(
    ScanState *node,
    ExecScanAccessMtd accessMtd,    // SeqNext / IndexNext / BitmapHeapNext / ...
    ExecScanRecheckMtd recheckMtd,  // Corresponding recheck function
    EPQState *epqstate,             // Concurrency control
    ExprState *qual,                // WHERE conditions
    ProjectionInfo *projInfo        // SELECT projection
);

This design makes storage access methods fully pluggable — heapam_methods is the default heap table implementation, but PG's Table Access Method API allows you to replace the entire storage engine:

// heapam_methods: function table for heap tables
static const TableAmRoutine heapam_methods = {
    .scan_begin          = heap_beginscan,
    .scan_getnextslot    = heap_getnextslot,
    .index_fetch_tuple   = heapam_index_fetch_tuple,
    .tuple_insert        = heapam_tuple_insert,
    .tuple_delete        = heapam_tuple_delete,
    .tuple_update        = heapam_tuple_update,
    // ... 50+ function pointers
};

This means theoretically you could implement a columnar storage engine, an LSM-Tree engine, etc., and plug them into PG's execution framework. Heap is just the default choice, not the only one.

Summary

Key intuitions worth remembering:

  • Indexes don't store version information: Indexes only store key → TID mappings; all MVCC determination happens after the heap fetch. This means an index scan may retrieve a TID, find it invisible after fetching from the heap, and then get the next TID. Index Only Scan is the sole exception — it avoids heap fetches through the Visibility Map
  • A snapshot is a data structure, not a point in time: SnapshotData's xmin/xmax quickly eliminate most transactions, and the xip array precisely handles the middle interval. The difference between isolation levels is simply when the snapshot is taken
  • Visibility = xmin committed + xmax invalid: This is the most essential intuition. All complex scenarios (rollback, in-progress, HOT chains) are variations of this rule
  • Hint bits avoid repeated CLOG lookups: After the first visibility check, markers are set in the row header, and subsequent checks read the markers directly
  • The storage engine is pluggable: TableAmRoutine defines 50+ function pointers, and heapam is just the default implementation. IndexAmRoutine works the same way — B-Tree, Hash, GIN, GiST all share the same call framework
