PostgreSQL Storage Engine Internals (Part 3): WAL, Checkpoint, and Durability

2026-03-08
PostgreSQL Storage Engine Internals
Part 1: Heap Pages and Buffer Pool · Part 2: Index Scans and MVCC · Part 3: WAL, Checkpoint, and Durability
What you'll learn from this post: Databases promise you "data won't be lost after a transaction commits." How is this promise kept? The answer is WAL (Write-Ahead Log) — before modifying a data page, first write a change record to the log. This post traces an INSERT statement's complete path from generating a WAL record, to WAL Writer flushing to disk, to Checkpoint creating a recovery point, and explains how this mechanism simultaneously supports crash recovery and streaming replication.

The Core Idea of WAL

Write-Ahead Logging has just one rule: before modifying a data page, you must first write a log record describing the modification to persistent storage.

Why? Because writing modified data pages to disk is expensive (random I/O, 8KB at a time), while appending log records to the end of a file is cheap (sequential I/O, typically tens to hundreds of bytes). WAL converts random writes to sequential writes while guaranteeing durability.

  World without WAL (direct data page writes):

  Transaction commit → must flush all modified Pages to disk → random I/O → slow
  Crash → some Pages half-written → data corruption → unrecoverable

  World with WAL:

  Transaction commit → only need to flush WAL records to disk → sequential I/O → fast
  Data pages can be written lazily later (async)
  Crash → replay WAL log → data restored to consistent state

XLogRecord: The Structure of a WAL Record

Every database write operation (INSERT, UPDATE, DELETE, and various internal operations) produces one or more XLogRecords:

typedef struct XLogRecord {
    uint32      xl_tot_len;   // Total length of this record
    TransactionId xl_xid;     // Transaction ID that produced this record
    XLogRecPtr  xl_prev;      // Position of the previous WAL record (forms a linked list)
    uint8       xl_info;      // Operation type (e.g., XLOG_HEAP_INSERT)
    RmgrId      xl_rmid;      // Resource manager ID (which subsystem produced it)
    pg_crc32c   xl_crc;       // CRC checksum
} XLogRecord;

A WAL record produced by an INSERT looks roughly like this:

XLogRecord {
    xl_tot_len = 156,              // Entire record is 156 bytes
    xl_xid = 12345,                // Transaction ID
    xl_prev = 0/1A2B3C4D,          // Previous record's LSN
    xl_info = XLOG_HEAP_INSERT,    // Heap insert operation
    xl_rmid = RM_HEAP_ID,          // Heap table resource manager
    xl_crc = 0x1A2B3C4D            // CRC checksum
}
// Followed by: block reference of the modified Page + actual inserted data

LSN: WAL's Timeline

LSN (Log Sequence Number) is the globally unique position identifier of each WAL record, in the format segment/offset (e.g., 0/1A2B3C4D). It's another representation of "time" in PostgreSQL:

  • Each Page's pd_lsn records the LSN of the last WAL record that modified that Page
  • During recovery, if a Page's pd_lsn is already >= the WAL record's LSN, that WAL has already been applied — skip it
  • During replication, the standby tells the primary "I've replayed up to LSN X," and the primary uses this to decide which WAL files can be recycled

Resource Managers: Who Produced This Log Entry

Different subsystems produce different types of WAL records. Each subsystem registers a "Resource Manager" providing its own redo and decode functions:

RmgrIdSubsystemExample Operation Types
RM_HEAP_IDHeap tablesINSERT, DELETE, UPDATE, HOT_UPDATE
RM_BTREE_IDB-Tree indexesIndex page splits, inserting index entries
RM_XACT_IDTransaction managementCOMMIT, ABORT
RM_CLOG_IDTransaction commit logCLOG page updates
RM_XLOG_IDWAL itselfCheckpoint records

During recovery, each WAL record invokes the corresponding resource manager's rm_redo function. During logical replication, rm_decode converts physical changes to logical changes (e.g., DecodeInsert parses XLOG_HEAP_INSERT into understandable row change events).

WAL Write Path: From Memory to Disk

        │ XLogInsert()            │ XLogInsert()           │
        ▼                         ▼                        │

  └─────────────────────────────────┬─────────────────┘    │
                                    │                       │
                                    │  XLogWrite()          │
                                    ▼                       │

                          │  16MB each        │  XLogBackgroundFlush()
                          └──────────────────┘

Two key inter-process communication mechanisms:

  • Shared memory: Backends write WAL records to the shared buffer, WAL Writer reads from the shared buffer and writes to disk
  • Latch: Backends use SetLatch() to wake up WAL Writer, WAL Writer uses WaitLatch() to wait for wake-up signals. Latch is PG's lightweight synchronization primitive, with much lower overhead than semaphores

Write vs Flush: Two-Phase Write

WAL writing to disk has two phases, and understanding this distinction is crucial:

PhaseOperationData reachesPerformanceDurability
WriteMemory → OS file cacheOS kernel bufferFast (memory copy)Lost on power failure
FlushOS file cache → diskPersistent storageSlow (real I/O)Survives power failure

Transaction COMMIT must ensure WAL Flush is complete — this is the key to the durability guarantee. The synchronous_commit parameter controls this behavior:

  • on (default): COMMIT waits for WAL Flush to complete before returning
  • off: COMMIT doesn't wait for Flush, may lose the last few hundred milliseconds of transactions. Better performance but sacrifices durability

XLogWrite: The Actual Disk Write

XLogWrite is the lowest-level write function. Its core logic:

  • Enter critical section (START_CRIT_SECTION) — cannot be interrupted during writing, preventing WAL file corruption from a half-finished write
  • Wait for all in-progress WAL insertions to complete (WaitXLogInsertionsToFinish) — multiple Backends may be concurrently writing to the WAL buffer
  • Calculate the range of Pages to write, batch write via pg_pwrite
  • Handle crossing WAL Segment boundaries — when the current file fills 16MB, switch to the next file
// Core write loop of XLogWrite (simplified)
do {
    written = pg_pwrite(openLogFile, from, nleft, startoffset);
    if (written <= 0) {
        if (errno == EINTR) continue;  // Interrupted by signal, retry
        ereport(PANIC, ...);           // Write failed → PANIC, database stops
    }
    nleft -= written;
    from += written;
    startoffset += written;
} while (nleft > 0);

Note that write failure triggers PANIC — WAL write failure means the durability guarantee is broken, and the database must immediately stop and wait for manual intervention.

Checkpoint: Consistent Recovery Points

Without Checkpoint, crash recovery would need to replay from the very first WAL record — as runtime grows, recovery time grows without bound. Checkpoint solves this: periodically flush all dirty pages to disk, creating a "consistency point" so recovery only needs to replay from the most recent Checkpoint.

  Time ──────────────────────────────────────────────────────►

  WAL: ═══════════════════════════════════════════════════════
       │              │                    │          │
    Checkpoint 1   Checkpoint 2         Checkpoint 3  Crash!
       │              │                    │          │
       │              │                    │◄────────►│
       │              │                    Recovery only
       │              │                    needs to replay
       │              │                    this WAL segment
       │              │
       │              After Checkpoint 2,
       │              WAL before Checkpoint 1 can be recycled

Checkpoint's entry point is CheckpointerMain (B_CHECKPOINTER process), triggered by:

  • Timeout trigger: checkpoint_timeout parameter (default 5 minutes)
  • WAL volume trigger: When WAL produced exceeds max_wal_size (default 1GB)
  • Manual trigger: Executing the CHECKPOINT command
  • Shutdown trigger: Normal database shutdown

Key constraint: Dirty page flushing during Checkpoint is gradual (checkpoint_completion_target parameter), avoiding a sudden burst of I/O that would impact normal queries. The default setting evenly distributes the flushing over 90% of the interval between two Checkpoints.

Crash Recovery: Replaying WAL

After a database crash and restart, StartupProcessMain (B_STARTUP process) handles recovery:

  • Read the pg_control file, find the most recent Checkpoint position
  • Start reading WAL from the Checkpoint's redo LSN
  • For each WAL record, check the corresponding Page's pd_lsn: if pd_lsn >= record_lsn, the modification is already persisted — skip; otherwise call rm_redo to replay
  • After replaying to the end of WAL, recovery is complete and the database begins normal service
// Core logic of crash recovery (pseudocode)
checkpoint = ReadControlFile();       // Read pg_control
lsn = checkpoint.redo;                // Recovery starting point

while (record = ReadWALRecord(lsn)) {
    page = GetPage(record.target);    // Find target Page

    if (page.pd_lsn >= record.lsn)
        continue;                     // Already applied, skip

    rmgr = GetRmgr(record.rmid);
    rmgr.rm_redo(record);             // Replay this WAL record
    lsn = NextRecordLSN(record);
}
// Recovery complete

Streaming Replication: Another Consumer of WAL

WAL isn't just for crash recovery — it's also the foundation of streaming replication. The B_WAL_SENDER process sends WAL to standbys in real time:

  │  Backend Process     │                  │  WAL Receiver        │

Physical Replication vs Logical Replication

Physical ReplicationLogical Replication
What's sentRaw WAL byte streamDecoded logical changes (INSERT row X)
Send functionXLogSendPhysicalXLogSendLogical
Standby requirementsExact same PG version and platformCan be different versions, even different databases
GranularityEntire clusterTable-level, selectively replicated
Use caseHigh availability, read replicasData integration, cross-version migration

The logical replication decoding process: XLogSendLogical → XLogReadRecord reads WAL records → LogicalDecodingProcessRecord calls the corresponding resource manager's rm_decode function. For example, DecodeInsert parses XLOG_HEAP_INSERT type WAL records into row change events that logical replication can understand.

Replication Slots and LSN

Replication Slots track each subscriber's consumption progress:

  • restart_lsn: The starting position from which this slot needs WAL retained — WAL older than this cannot be recycled
  • confirmed_flush_lsn: The position the subscriber has received and processed

If a standby is disconnected for too long, restart_lsn will prevent WAL recycling, causing the primary's disk to fill up. This is a common failure scenario in production environments.

Timeline: The Database's Branch History

Timeline is PG's mechanism for handling database "branches." Each Timeline has a unique numeric ID, and new Timelines are created in these scenarios:

  • PITR (Point-in-Time Recovery): Recovering from a specific historical point in time
  • Standby Promotion: A standby is promoted to primary
  • Restoring from backup
  Timeline 1: ════════════════════════╗
                                      ║ Standby promotion
  Timeline 2:                         ╚═══════════════════
                                            │
                                    00000002.history file

Timeline history files (e.g., 00000002.history) record branch points, helping the recovery process correctly select WAL files.

PG Process Overview

Finally, let's summarize all background processes covered in this series:

  postmaster (main process)
  │
  ├── Backend (B_BACKEND)           Handles client SQL queries
  │   └── WAL Sender (B_WAL_SENDER) Upgraded from replication connections
  │
  ├── Background Writer (B_BG_WRITER)  Periodically flushes dirty pages, reducing Checkpoint pressure
  ├── Checkpointer (B_CHECKPOINTER)    Periodically executes Checkpoints
  ├── WAL Writer (B_WAL_WRITER)        Periodically flushes WAL buffer to disk
  ├── Autovacuum Launcher              Schedules VACUUM work
  │   └── Autovacuum Worker            Executes actual VACUUM
  ├── WAL Receiver (B_WAL_RECEIVER)    Receives WAL on Standby
  ├── WAL Summarizer                   WAL summaries (for incremental backups)
  ├── Startup (B_STARTUP)              Crash recovery / Standby continuous replay
  ├── Archiver (B_ARCHIVER)            Archives old WAL files
  ├── IO Worker (B_IO_WORKER)          Async I/O worker threads
  └── Syslogger (B_LOGGER)            Log collection

PG uses a multi-process architecture where each background process runs independently, communicating through shared memory and Latches. The advantage is good isolation (one process crashing won't bring down others) and debug-friendly; the cost is that inter-process communication is heavier than inter-thread communication.

Summary

Key intuitions worth remembering:

  • WAL = sequential log writes + deferred data page writes: Converts expensive random I/O to cheap sequential I/O. Transaction commit only needs to ensure WAL is flushed; data pages can be written asynchronously
  • Write ≠ Flush: Write only reaches the OS cache, Flush reaches disk. COMMIT needs Flush to guarantee durability
  • WAL Writer and Backends communicate via shared memory + Latch: Multiple Backends concurrently write to the WAL buffer, WAL Writer handles flushing to disk. Write failure → PANIC
  • Checkpoint = flush all dirty pages + create recovery point: Crash recovery only needs to replay from the most recent Checkpoint, not from the beginning. WAL before the Checkpoint can be recycled
  • WAL serves both recovery and replication: The same WAL records are replayed by the Startup process during crash recovery, and sent to Standbys by WAL Sender during streaming replication
  • Physical vs Logical: Physical replication sends raw byte streams (simple and efficient), logical replication decodes through resource managers into row change events (flexible but more complex)

With this, the PostgreSQL storage engine trilogy is complete. From the 8KB internal layout of Heap Pages, to Buffer Pool page cache management, to index scans and MVCC visibility determination, to WAL's durability guarantees and Checkpoint's recovery mechanism — these layers of nested abstractions together form the core skeleton of a relational database.
