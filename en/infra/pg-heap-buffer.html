PostgreSQL Storage Engine Internals (Part 1): Heap Pages and Buffer Pool

2026-03-08
PostgreSQL Storage Engine Internals
Part 1: Heap Pages and Buffer Pool · Part 2: Index Scans and MVCC · Part 3: WAL, Checkpoint, and Durability
What you'll learn from this post: When you execute SELECT * FROM users WHERE id = 123, how does the data travel from disk to your screen? This post traces the lowest layer of that path: what the inside of an 8KB Page looks like, how the Buffer Pool manages in-memory page caching, how sequential scans read data page by page, and how HOT Chains optimize update operations. Understanding these will help you truly grasp why VACUUM is necessary, why cross-host container communication is slower than same-host, and the fundamental storage-layer differences between PG and MySQL.

The Big Picture: From Files to Tuples

PostgreSQL's storage is a four-layer structure. Understanding the nesting of these four layers is the foundation for everything that follows.

  ┌─────────────────────────────────────────────────────────┐
  │ Relation (table/index)                                    │
  │                                                         │
  │  Data directory: $PGDATA/base/{dbOid}/{relFileNumber}   │
  │                                                         │
  │  ┌──────────┐ ┌──────────┐ ┌──────────┐                │
  │  │ Segment 0 │ │ Segment 1 │ │ Segment 2 │  ...         │
  │  │  (1 GB)   │ │  (1 GB)   │ │  (1 GB)   │              │
  │  └─────┬────┘ └──────────┘ └──────────┘                │

  │        ▼                                                │
  │  ┌────────┬────────┬────────┬────────┬─────┐           │
  │  │ Page 0 │ Page 1 │ Page 2 │ Page 3 │ ... │  8KB each  │
  │  └────┬───┘ └────────┘ └────────┘ └────────┘           │
  │       │                                                 │

How to calculate single-table capacity limit? Page size (8KB) × max Block count (2^32) = 32 TB. This is a hard-coded upper limit.

Each Relation also has several "fork files" on disk:

Inside a Page: An 8KB Micro-World

This is the most important diagram in this entire post. Each Page is a fixed-size 8192-byte block with the following internal structure:

  0 bytes                                                  8192 bytes
  ┌──────────────────────────────────────────────────────────────┐

  │  pd_lsn │ pd_checksum │ pd_flags │ pd_lower │ pd_upper │... │
  ├──────────────────────────────────────────────────────────────┤
  │ ItemId[1] │ ItemId[2] │ ItemId[3] │ ... │ ItemId[N] │       │
  │ (4 bytes)  (4 bytes)    (4 bytes)         (4 bytes)          │
  │                                          ↓ pd_lower          │
  ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤
  │                                                              │
  │                     Free  Space                               │
  │                                                              │
  ├ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─ ─┤
  │                                          ↑ pd_upper          │
  │  Tuple N  │  ...  │  Tuple 3  │  Tuple 2  │  Tuple 1        │
  │                                                              │
  ├──────────────────────────────────────────────────────────────┤
  │                   Special Space                     ↑pd_special│
  └──────────────────────────────────────────────────────────────┘

  ← ItemId array grows left to right    Tuple data grows right to left →

  The two grow toward each other, with free space in the middle.

This "growing toward each other from both ends" design is a classic — the same idea as heap and stack growing toward each other in operating systems. The advantage: you don't need to know in advance how much space ItemIds and Tuples each require; the free space is shared between them.

PageHeaderData: Page Metadata

FieldSizeMeaning
pd_lsn8 bytesLSN of the last WAL record that modified this Page. Determines whether replay is needed during recovery
pd_checksum2 bytesPage checksum, detects storage-level data corruption
pd_flags2 bytesFlag bits (has free space, all-visible, etc.)
pd_lower2 bytesPoints to the start of free space (end of ItemId array)
pd_upper2 bytesPoints to the end of free space (start of Tuple data)
pd_special2 bytesStart offset of Special Space (used by index pages)

pd_upper - pd_lower = available space in the current Page. When INSERTing a new row, PG checks if this value is large enough to fit a new Tuple + a new ItemId.

ItemId: The Indirection Layer

An ItemId is a 4-byte structure containing three pieces of information:

  • offset (15 bits): Byte offset of the Tuple within the Page
  • len (15 bits): Length of the Tuple
  • flags (2 bits): Status (normal / dead / redirect)

Why this extra layer of indirection? Because VACUUM needs to defragment within Pages. After defragmentation, Tuples' physical positions may change, but ItemId slot numbers (OffsetNumber) remain stable. External references (index entries, foreign keys from other rows) always locate a row via (BlockNumber, OffsetNumber) — this is the TID (Tuple Identifier).

  TID = (BlockNumber, OffsetNumber)
        (  page #   ,   slot #     )

  e.g., TID = (5, 3) means:
  The Tuple pointed to by ItemId[3] in Block 5

  Index entry:  age=25 → TID(5, 3)
                         │
                         ▼
  Page 5:   ItemId[3] = {offset: 6800, len: 120, flags: NORMAL}
                            │
                            ▼
            Offset 6800:  HeapTupleHeader + user data

This indirection layer lets VACUUM freely move Tuples' physical positions — it just needs to update the corresponding ItemId, without modifying any indexes.

TID, ItemId, Tuple: The Relationship Between the Three

These three concepts are often confused, so let's clarify their relationship thoroughly. In one sentence: TID is the address, ItemId is the mapping from door number to room number, and Tuple is the person in the room.

ConceptWhat it isWhere it livesIs it stable?
TIDLogical address (BlockNumber, OffsetNumber)In index entries, in references from other rowsStable — VACUUM won't change it
ItemIdIndirect pointer within the Page (offset, len, flags)In the ItemId array after the Page headerSlot number is stable, but internal offset can change
TupleActual row data (row header + user data)In the Tuple area at the end of the PageNot stable — physical position can change after VACUUM defragmentation

Key insight: The OffsetNumber in a TID is not a byte offset, but an index into the ItemId array (starting from 1). The actual byte offset is hidden inside the ItemId. This indirection layer is the essence of the entire design.

  A complete row lookup process:

  Externally held: TID = (5, 3)
                   │  │
                   │  └─ OffsetNumber = 3 (3rd slot in ItemId array)
                   └──── BlockNumber = 5  (5th Page)

  Step 1: Find the Page using BlockNumber
  ┌─────────────────────────────────────────────────────────────┐
  │ Page 5                                                       │
  │                                                              │
  │  Step 2: Look up OffsetNumber in the ItemId array             │
  │                                                              │
  │  ItemId[1]    ItemId[2]    ItemId[3]                         │
  │  off:7100     off:6900     off:6700  ◄── this one            │
  │  len:120      len:80       len:150                           │
  │  flags:NORM   flags:DEAD   flags:NORM                        │
  │                               │                              │
  │                               │ Step 3: Jump to Tuple via    │
  │                               │ offset                       │
  │                               ▼                              │
  │                    ┌──────────────────────┐                  │
  │  Offset 6700 ──►  │ HeapTupleHeader      │                  │
  │                    │  t_xmin = 100        │                  │
  │                    │  t_xmax = 0          │                  │
  │                    │  t_ctid = (5, 3)     │  ← points to     │
  │                    │                      │    self = latest  │
  │                    ├──────────────────────┤                  │
  │                    │ User data            │                  │
  │                    │  id=42               │                  │
  │                    │  name='Alice'        │                  │
  │                    │  age=25              │                  │
  │                    └──────────────────────┘                  │
  │                    ◄──── 150 bytes ────►                     │
  └─────────────────────────────────────────────────────────────┘

Understanding these three layers explains the following behaviors:

  • VACUUM defragmentation: Moves a Tuple's physical position (e.g., from offset 6700 to 7000), only needs to update ItemId[3]'s offset field. TID (5, 3) stays unchanged, no index entries need modification
  • DELETE a row: Doesn't immediately delete the Tuple data, but marks ItemId's flags as DEAD. Space is only truly reclaimed during VACUUM
  • HOT UPDATE: Old and new versions stay in the same Page, linked via the Tuple header's t_ctid. Index entries still point to the old version's TID, traversing the chain to find the new version
  • Index scan heap fetch: Index provides TID → read Page → use OffsetNumber to find ItemId → use ItemId's offset to find Tuple → perform MVCC visibility check

A common misconception is that "TID is the physical address of a row." It's not — TID is a logical address, with the ItemId indirection layer in between. It's precisely this indirection that allows PG to move rows' physical positions without modifying any external references.

HeapTupleHeader: The Row Header

The first 23 bytes of each Tuple are the row header, recording all information needed for MVCC:

FieldMeaning
t_xminTransaction ID that inserted this row — "who created me"
t_xmaxTransaction ID that deleted/updated this row — "who killed me" (0 means still alive)
t_cidCommand sequence number, operation order within the same transaction
t_ctidTID of the current version (if updated, points to new version → forms a version chain)
t_infomaskFlag bits (committed, has NULL columns, has TOAST, etc.)
t_hoffStart offset of user data (skips past NULL bitmap)

Key point: PG's MVCC information is stored directly in the row header, on the same Page as user data. This means old versions of rows don't disappear — they pile up in the Heap until VACUUM cleans them up. This is one of the biggest differences between PG and MySQL InnoDB, which we'll compare in detail later.

Buffer Pool: In-Memory Page Cache

Blocks on disk are never accessed directly. All reads and writes go through the Buffer Pool — a page cache maintained by PostgreSQL in shared memory.

                       ┌──────────────────────────────┐
                       │       Backend Process         │
                       │    (your SQL query process)   │
                       └──────────┬───────────────────┘
                                  │ Request Page (5, 0)
                                  │ = Block 5, main data file
                                  ▼
  ┌───────────────────────────────────────────────────────────┐
  │                   Shared Buffer Pool                       │
  │                  (shared_buffers parameter)                 │
  │                                                            │
  │  ┌──────────────────────────────────────────────────┐      │
  │  │            Buffer Table (hash table)              │      │
  │  │                                                   │      │
  │  │  BufferTag → Buffer ID                            │      │
  │  │  {spc, db, rel, fork, block} → slot number        │      │
  │  └──────────────────────────────────────────────────┘      │
  │                         │                                   │
  │                         ▼                                   │
  │  ┌────────┬────────┬────────┬────────┬────────┐            │
  │  │ Slot 0 │ Slot 1 │ Slot 2 │ Slot 3 │  ...   │  Buffer   │
  │  │ 8KB    │ 8KB    │ 8KB    │ 8KB    │        │  array     │
  │  └────────┘ └────────┘ └────────┘ └────────┘            │
  │                                                            │
  │  ┌──────────────────────────────────────────────────┐      │
  │  │         Buffer Descriptors (descriptor array)     │      │
  │  │  Each slot has a corresponding descriptor:        │      │
  │  │  - tag: which page this slot caches               │      │
  │  │  - state: pin count + usage count + dirty flag    │      │
  │  │  - buf_id: corresponding slot number              │      │
  │  └──────────────────────────────────────────────────┘      │
  └───────────────────────────────────────────────────────────┘
                                  │
                                  │ Cache miss → read from disk
                                  ▼

                       │   base/12345/67890    │
                       └──────────────────────┘

BufferTag: Five-Tuple Identification

A single BufferTag can pinpoint the exact location of data on disk, without consulting system tables:

typedef struct buftag {
    Oid         spcOid;     // Tablespace OID
    Oid         dbOid;      // Database OID
    RelFileNumber relNumber; // File number
    ForkNumber  forkNum;    // Fork (0=main data, 1=FSM, 2=VM)
    BlockNumber blockNum;   // Block number
} BufferTag;

// e.g.: {spcOid=1663, dbOid=12345, relNumber=67890, forkNum=0, blockNum=5}
// → The 5th 8KB block of $PGDATA/base/12345/67890

The self-contained nature of this design is important: the background dirty page flushing process may start working before the corresponding transaction has committed, and it must not depend on system tables to locate data.

In production, shared_buffers is typically set to 25% of physical memory. PG relies on the OS Page Cache as a second-level cache — data might not be in the Buffer Pool, but still in the OS cache, making reads still fast.

Clock Sweep Algorithm

When the Buffer Pool is full and pages need to be evicted, PG uses the Clock Sweep algorithm. Each buffer descriptor maintains a usage_count:

  • When a page is accessed, usage_count++ (capped at 5)
  • When the clock hand reaches a buffer: if usage_count > 0, decrement by 1 and skip; if usage_count == 0, evict the page
  • If the page is dirty, it must be flushed to disk before eviction
  • If the page is pinned (a process is currently using it), skip
  Clock hand →

  ┌───┐  ┌───┐  ┌───┐  ┌───┐  ┌───┐  ┌───┐  ┌───┐  ┌───┐
  │ 3 │  │ 1 │  │ 0 │  │ 2 │  │ 0 │  │ 1 │  │ 0 │  │ 4 │
  └───┘  └───┘  └─┬─┘  └───┘  └─┬─┘  └───┘  └─┬─┘  └───┘
                   │              │              │
                  Decremented    Decremented    Decremented
                  to 0 → Evict!  to 0 → Evict!  to 0 → Evict!

  Higher usage_count → "hotter" → harder to evict

This is better suited for databases than LRU: a single full-table scan won't flush all hot data from the cache (because the scan only gives each page +1, while hot pages may have a usage_count of 5).

The Complete I/O Path of a Sequential Scan

When you execute SELECT * FROM users, the executor calls ExecSeqScan, which ultimately reaches the storage layer's heap_getnextslot. Let's trace the complete path:

  ExecSeqScan
      │
      ▼
  ExecScanExtended(SeqNext, SeqRecheck)
      │
      ▼
  SeqNext
      │ First call: table_beginscan() initializes scan descriptor
      │ Subsequent calls:
      ▼
  table_scan_getnextslot
      │ → heapam_methods.scan_getnextslot
      ▼
  heap_getnextslot
      │
      ├─── Page Mode (default):
      │    heapgettup_pagemode
      │    1. heap_fetch_next_buffer → read next Page into Buffer Pool
      │    2. heap_prepare_pagescan → check visibility of all Tuples on the page at once
      │    3. Build rs_vistuples[] array (list of visible Tuple OffsetNumbers)
      │    4. Iterate through rs_vistuples[], returning Tuples one by one
      │
      └─── Tuple Mode:
           heapgettup
           Read Tuples one by one, each with individual visibility check

Page Mode is the default strategy with better performance: it gets the visibility results for the entire page at once, avoiding repeated locking.

heap_fetch_next_buffer: Read-Ahead Mechanism

Sequential scans don't read one page at a time — PG has a streaming read-ahead (Read Stream) mechanism:

  • Maintains a distance parameter to control read-ahead depth
  • Consecutive blocks are merged into a single large I/O request (I/O coalescing)
  • Fast path: if there's only one pinned buffer and read-ahead distance is 1, read the next block directly, skipping complex logic
  • Checks for interrupt requests (CHECK_FOR_INTERRUPTS) after each Page read, preventing long scans from being uncancellable

Traversal Within a Page

After obtaining a Page, here's how the Tuples inside are read:

// Core loop of heapgettup_pagemode (simplified)

page = BufferGetPage(scan->rs_cbuf);      // Get Page pointer

for (i = 0; i < scan->rs_ntuples; i++) {
    OffsetNumber lineoff = scan->rs_vistuples[i];     // Visible Tuple's slot number
    ItemId lpp = PageGetItemId(page, lineoff);         // Get pointer from ItemId array

    tuple->t_data = (HeapTupleHeader) PageGetItem(page, lpp);  // Locate Tuple data
    tuple->t_len = ItemIdGetLength(lpp);                        // Tuple length
    ItemPointerSetOffsetNumber(&tuple->t_self, lineoff);        // Set TID

    // Check scan conditions (if there's a scan key)
    if (key != NULL && !HeapKeyTest(tuple, ...))
        continue;

    return;  // Return this Tuple to upper layer
}

Note that the TID is "reconstructed" into tuple->t_self during scanning — it's a (BlockNumber, OffsetNumber) combination, determined at write time and derived from the current position during scanning for use by upper layers.

HOT Chain: Updates Within a Page

PG's UPDATE is actually INSERT new version + DELETE old version. This means every UPDATE: creates a new Tuple, allocates a new ItemId, updates all indexes pointing to that row. The cost is significant.

HOT (Heap Only Tuple) is a key optimization: if an UPDATE doesn't modify any indexed columns, and the new Tuple fits on the same Page, PG creates a HOT chain and doesn't update indexes.

  UPDATE users SET name = 'Alice2' WHERE id = 123;
  UPDATE users SET name = 'Alice3' WHERE id = 123;

  Index entry: (id=123) → TID(5, 1)    Only one index entry!
                          │
                          ▼
  Page 5:
  ┌────────────────────────────────────────────────────────┐
  │ ItemId[1]          ItemId[2]         ItemId[3]         │
  │ offset→A           offset→B          offset→C          │
  │ flags: REDIRECT    flags: NORMAL     flags: NORMAL     │
  ├────────────────────────────────────────────────────────┤
  │                                                        │
  │  Tuple C (latest)  ←  Tuple B (old)  ←  Tuple A (oldest)│
  │  name='Alice3'      name='Alice2'     name='Alice'      │
  │  t_xmax=0           t_xmax=103        t_xmax=102        │
  │  t_ctid=(5,3)       t_ctid=(5,3)      t_ctid=(5,2)      │
  │                                                        │
  └────────────────────────────────────────────────────────┘

  During index scan:
  1. Index provides TID(5, 1)
  2. Read ItemId[1], find REDIRECT → follow it
  3. Traverse t_ctid chain: A → B → C
  4. Perform MVCC visibility check on each version
  5. Return the version visible to the current transaction

Three conditions for HOT:

  • UPDATE doesn't modify any indexed columns (otherwise index entries must be updated)
  • New Tuple fits on the same Page (otherwise can't link via in-page pointers)
  • HOT chain for the same row cannot span Pages

Benefits of HOT:

  • Reduces index bloat: One index entry can cover multiple updates
  • Improves UPDATE performance: Saves I/O from index updates
  • Simplifies VACUUM: HOT chains within a Page can be quickly cleaned up at the Page level (Pruning)

The logic for index scans following HOT chains is in the heap_hot_search_buffer function — starting from the TID given by the index, it traverses along t_ctid to the end of the chain, performing visibility checks on each version.

PG vs InnoDB: Two Storage Philosophies

Finally, let's compare the fundamental storage-layer design differences between PostgreSQL and MySQL InnoDB:

PostgreSQL (Heap)MySQL InnoDB (Clustered Index)
Data organizationHeap: rows are piled up in insertion order, unorderedClustered index: rows stored in primary key order in B+ Tree leaf nodes
UPDATE methodAppend new version to Heap (multiple versions coexist)In-place update, old version moved to Undo Log
Where are old versionsMixed with new versions in the same Heap PageSeparate Undo Log space
Bloat issueSevere — old versions keep occupying space if not cleanedLighter — Undo Log can be independently reclaimed
VACUUM requirementEssential — otherwise table and indexes keep bloatingNo equivalent needed (Purge thread automatically cleans Undo)
Primary key lookupRequires index scan + heap fetch (TID → Heap Page)Data found directly on B+ Tree (index is data)
Secondary index lookupSecondary index → TID → one Heap I/OSecondary index → primary key → another clustered index traversal
Write amplificationHOT optimization reduces index writesChange Buffer defers secondary index updates

The core trade-offs of the two philosophies:

  • PG chose a simpler write path: Appending is always simpler and safer than in-place updates. The cost is needing VACUUM to reclaim space, with higher operational complexity
  • InnoDB chose compact storage: In-place updates + Undo Log avoids bloat. The cost is more complex Undo Log management logic, and secondary index lookups require two B+ Tree traversals

Summary

Key intuitions worth remembering:

  • Pages are the foundation of everything: 8KB fixed size, ItemId array and Tuple data grow toward each other. Understanding Page internal layout explains why PG designs TID this way and why VACUUM can defragment at the Page level
  • ItemId is the indirection layer: External references use (BlockNumber, OffsetNumber) to reference a row, and VACUUM can move Tuples without modifying indexes
  • Buffer Pool is the intermediary for all I/O: BufferTag five-tuple for self-contained location, Clock Sweep algorithm better suited for database workloads than LRU, read-ahead mechanism reduces random I/O
  • HOT Chain is PG's compensation for the cost of "append-style updates": Updates that don't modify indexed columns can be completed within a Page without triggering index updates
  • MVCC info lives in the row header: This is the most fundamental difference between PG and InnoDB — old versions aren't in Undo Log but in the Heap itself, directly causing bloat issues and the necessity of VACUUM
