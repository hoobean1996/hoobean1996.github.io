Scaling Stateful Services: From Methodology to Real-World Redis, Memcached, and MySQL

2026-03-08
What you'll learn from this post: Scaling stateless services is simple — add machines and put them behind a load balancer. But scaling stateful services (databases, caches, message queues) is one of the hardest problems in distributed systems: which machine has the data? How do you move it? What happens to requests during the move? This post first presents a reusable five-step methodology, then uses Redis Cluster, Memcached, and MySQL to show how each step is implemented in practice.

Why Stateful Services Are Hard to Scale

Scaling model for stateless services:

  Request ──► Load Balancer ──┬──► Server A
                              ├──► Server B    Any server can handle it
                              └──► Server C    Add machines to scale

Scaling model for stateful services:

  Request (user_id=42)
    │
    │  This request can only go to the machine storing user_id=42's data
    │  Can't send it anywhere
    ▼
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Node A   │  │ Node B   │  │ Node C   │
  │ id: 1-100│  │id:101-200│  │id:201-300│
  └──────────┘  └──────────┘  └──────────┘

  Adding Node D:
  - Whose data needs to move?
  - How are reads/writes routed during the move?
  - What if it crashes halfway through?

The fundamental problem is that data has ownership. Scaling stateful services = reassigning data ownership without downtime.

Methodology: The Five-Step Scaling Model

Step 1: Sharding Strategy — Mapping Keys to Nodes

The most intuitive approach is hash(key) % N to directly map to N machines. But after adding a machine, N becomes N+1, and almost all key mappings change — meaning a full data migration.

The solution is adding a layer of virtual slots:

  Bad design: Key → Node (direct mapping)
  hash(key) % 3 = Node A
  hash(key) % 4 = Node B  ← Add one machine, almost all keys must move

  Good design: Key → Slot → Node (two-layer mapping)
  hash(key) % 16384 = Slot 5742    ← This layer never changes
  Slot 5742 → Node A               ← Only need to move Slot ownership

  ┌─────────────────────────────────────────────────────────┐
  │  Key Space                                               │
  │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐     │
  │  │Slot │Slot │Slot │Slot │Slot │Slot │Slot │ ... │     │
  │  │  0  │  1  │  2  │  3  │  4  │  5  │  6  │     │     │
  │  └──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴─────┘     │
  │     │     │     │     │     │     │                      │
  │     ▼     ▼     ▼     ▼     ▼     ▼                      │
  │   Node A        Node B        Node C                     │
  │  Slot 0,1,2    Slot 3,4      Slot 5,6                   │
  └─────────────────────────────────────────────────────────┘

  Scaling: Move Slot 2 from Node A to Node D
  - Key-to-Slot mapping doesn't change
  - Only change Slot-to-Node mapping
  - Only move data in Slot 2

Choosing Slot count: typically 100x+ the number of machines. Too few and Slots are too coarse for even distribution; too many and metadata overhead grows. Redis Cluster chose 16384, a battle-tested balance point.

Step 2: Control Plane — Where Metadata Lives

The routing table "Slot X is on Node A" needs to be stored somewhere, and all participants need to see a consistent version.

ApproachPrincipleAdvantageDisadvantageTypical System
GossipNodes propagate to each other, eventually consistentNo single point, self-healingSlow convergence, split-brain risk, hard to do complex schedulingRedis Cluster
Centralized storeetcd/ZooKeeper stores authoritative metadataStrong consistency, easy orchestrationOne more component to maintainTiDB PD, Vitess
Config serviceMcrouter reads routing table from config filesExtremely simple, no runtime dependencyChanges require pushes, not suitable for dynamic scalingMemcached + Mcrouter

The 2026 trend: centralized control plane + K8s Operator is mainstream. etcd provides strongly consistent metadata storage, Operators automate scaling, failover, and data migration. Gossip protocols are still used for failure detection (quickly discovering downed nodes), but no longer for routing decisions.

Large-scale production environments lean toward the Proxy model. When you have dozens of programming languages for clients, it's impossible to maintain a complex smart client for each one. Proxy centralizes routing logic — clients just need to speak the protocol.

Step 4: Replicas and Availability — Data Can't Exist in Just One Copy

Sharding solves the capacity problem, but each shard is still a single point of failure. Replicas solve availability:

  Slot 0-5461              Slot 5462-10922          Slot 10923-16383
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ Master A     │         │ Master B     │         │ Master C     │
  │ (read/write) │         │ (read/write) │         │ (read/write) │
  └──────┬───────┘         └──────┬───────┘         └──────┬───────┘
         │ Async replication      │ Async replication      │ Async replication
         ▼                        ▼                        ▼
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ Replica A'   │         │ Replica B'   │         │ Replica C'   │
  │ (read-only)  │         │ (read-only)  │         │ (read-only)  │
  └──────────────┘         └──────────────┘         └──────────────┘

Standard failover procedure:

  1. Detect that Master A is down (heartbeat timeout / Gossip voting / control plane probe)
  2. Select a new Master from replicas (choose the one closest in replication progress)
  3. Key step: Update control plane metadata — Owner of Slot 0-5461 changes from A to A'
  4. Notify routing layer (Proxy updates routing table / Client refreshes Slot Map)
  5. New Master A' starts accepting reads and writes

Replication mode selection:

ModeConsistencyLatencyAvailabilityTypical System
Async replicationMay lose dataLowestHighestRedis, Memcached
Semi-sync replicationAt least one replica confirmsMediumHighMySQL Semi-sync
Raft/PaxosMajority confirms, strong consistencyHigherMediumTiKV, CockroachDB

Step 5: Data Migration — Changing Engines Mid-Flight

This is the hardest step. After adding a new machine, some Slots must be moved from old nodes to new ones, while not interrupting live service.

  Migrating Slot 3 from Node A → Node D:

  Phase 1: Mark migration state
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Slot 3:  │                    │ Slot 3:  │
  │ MIGRATING│                    │ IMPORTING│
  └──────────┘                    └──────────┘

  Phase 2: Full snapshot transfer
  ┌──────────┐  ══ Snapshot data ══►  ┌──────────┐
  │ Node A   │                        │ Node D   │
  │ Continues│                        │ Receiving │
  │ serving  │                        │ data      │
  └──────────┘                        └──────────┘

  Phase 3: Incremental catch-up
  ┌──────────┐  ── Incremental log ──► ┌──────────┐
  │ Node A   │  (new writes during     │ Node D   │
  │ Continues│   migration)            │ Catching  │
  │ serving  │                         │ up        │
  └──────────┘                         └──────────┘

  Phase 4: Atomic switchover (critical moment)
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Pause    │  ← Brief write     │ Caught   │
  │ Slot 3   │  block (typically  │ up!      │
  │ writes   │  milliseconds)     │          │
  └──────────┘                    └──────────┘
       │
       │  Control plane update: Slot 3 → Node D
       │  Routing layer refresh
       ▼
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Delete   │                    │ Slot 3:  │
  │ Slot 3   │                    │ Normal   │
  │ data     │                    │ service  │
  └──────────┘                    └──────────┘

Phase 4 is the most dangerous moment in the entire process. If the incremental log can't keep up (writes are too fast), the blocking time extends and impacts live service. Two coping strategies:

  • Throttle writes: Rate-limit the Slot during catch-up phase, reducing write speed so incremental catch-up can keep up
  • Dual-write mode: Write to both nodes during switchover, no blocking, but more complex logic requiring conflict handling

In Practice: Redis Cluster Scaling

StepRedis Cluster Implementation
Sharding16384 Slots, CRC16(key) % 16384
Control planeDecentralized Gossip. Each node maintains the full Slot Map, synchronized via Gossip protocol
RoutingSmart Client. Client caches Slot Map, connects directly to target node. If sent to wrong node, it returns MOVED redirect
ReplicasEach Master has 1-N Replicas, async replication. Failover through Gossip voting to elect new Master
MigrationCLUSTER SETSLOT + MIGRATE commands, key-by-key migration

Redis Migration Details

# 1. Mark migration state
CLUSTER SETSLOT 3 MIGRATING target-node-id   # Mark on source node
CLUSTER SETSLOT 3 IMPORTING source-node-id   # Mark on target node

# 2. Key-by-key migration (Redis has no snapshot migration, moves one at a time)
CLUSTER GETKEYSINSLOT 3 100                  # Get 100 keys from Slot 3
MIGRATE target-host 6379 "" 0 5000 KEYS key1 key2 key3 ...

# 3. Complete switchover
CLUSTER SETSLOT 3 NODE target-node-id        # Execute on all nodes

Redis migration characteristics and limitations:

  • Key-by-key migration, not snapshot. Large keys (e.g., a Hash with millions of elements) block the source node during migration
  • During migration, source node handling of MIGRATING Slot: if key is still local → handle normally; if key has been moved → return ASK redirect to target node
  • Gossip convergence is slow — in large clusters (200+ nodes), a Slot change may take several seconds to propagate to all nodes
  • No incremental log: Key migration is atomic (MIGRATE = DEL + RESTORE), so there's no "key modified during migration" problem, but the cost is that large key migration blocks

Redis Scale-Out in Practice

  Before scaling: 3 Masters, each ~5461 Slots

  Master A: Slot 0-5460
  Master B: Slot 5461-10922
  Master C: Slot 10923-16383

  Adding Master D, need to move some Slots from A/B/C:

  Master A: Slot 0-4095          (moved out 1366 Slots)
  Master B: Slot 5461-9556       (moved out 1366 Slots)
  Master C: Slot 10923-15017     (moved out 1366 Slots)
  Master D: Slot 4096-5460,      (received 4098 Slots)
            9557-10922,
            15018-16383

  redis-cli --cluster reshard automates this process

In Practice: Memcached Scaling

StepMemcached Implementation
ShardingClient-side consistent hashing. No Slot concept, virtual nodes on a hash ring
Control planeNone. Memcached servers don't communicate at all, don't know about each other
RoutingPure client-side routing. Mcrouter (Proxy mode) or libmemcached (Smart Client mode)
ReplicasNone by default. Mcrouter can configure shadow/replica pools for read replicas
MigrationNo migration. Cache misses are rebuilt from the source of truth

Memcached's scaling philosophy is completely different from Redis — it positions itself as a pure cache, not a data store:

Consistent Hashing: Adding Machines Only Affects Adjacent Nodes

  Hash ring (0 ~ 2^32)

        Node A                    Node A
       ╱      ╲                  ╱      ╲
      ╱   key1  ╲               ╱   key1  ╲
  Node D ─── Node B    →    Node D ─── Node B
      ╲        ╱               ╲    ↑   ╱
       ╲      ╱                 ╲ Node E╱    ← Newly joined
        Node C                   Node C

  Adding Node E:
  - Only keys between Node B and Node E need remapping
  - All other keys completely unaffected
  - No data migration needed — cache misses naturally rebuild to Node E

Mcrouter: Memcached's "Brain"

Memcached itself is brainless; all intelligence is in Mcrouter (Meta's open-source Proxy):

  • Consistent hash routing: Computes which Memcached server a key should go to
  • Failure handling: Automatically removes downed nodes from the ring, keys auto-remap to the next node
  • Canary switchover: Config file updates routing rules, gradually shifting traffic to a new cluster
  • Connection pooling: Thousands of frontend client connections multiplexed into a few long-lived connections to Memcached
  • Cold start warm-up: When a new cluster comes online, read misses automatically fall back to the old cluster, simultaneously writing to the new cluster
// Mcrouter routing config example
{
  "pools": {
    "A": { "servers": ["mc1:11211", "mc2:11211", "mc3:11211"] },
    "B": { "servers": ["mc4:11211", "mc5:11211", "mc6:11211"] }
  },
  "route": {
    "type": "WarmUpRoute",          // Cold start warm-up
    "cold": "PoolRoute|B",          // New cluster
    "warm": "PoolRoute|A",          // Old cluster (fallback on miss)
    "exptime": 180
  }
}

Memcached scaling core philosophy: don't move data, rely on cache self-healing. Add machines → update Mcrouter config → new keys auto-route to new nodes → old key misses naturally rebuild from source. The cost is temporarily lower cache hit rates after scaling (cold start), but this is acceptable for cache use cases.

MySQL Shard Migration: Binlog-Based

MySQL data migration is essentially the same pattern as PG's streaming replication — full snapshot + incremental log catch-up:

  1. Full snapshot: Take a mysqldump or Xtrabackup physical snapshot of the source Shard, transfer to the new node
  2. Incremental catch-up: Starting from the Binlog position at snapshot time, continuously consume Binlog and apply to the new node. This step mirrors PG's WAL replay
  3. Verification: Compare data consistency between source and target (VDiff in Vitess)
  4. Atomic switchover: Brief write block → wait for Binlog to catch up → update topology → resume writes

Vitess's Reshard command automates this process. The key is the VReplication engine — it doesn't simply replay Binlog, but filters by sharding rules: only row changes belonging to the target Shard are applied.

MySQL Scaling Special Challenges

  • Cross-Shard JOINs: SELECT * FROM orders JOIN users ON orders.user_id = users.id — if orders and users are on different Shards, the Proxy must query each separately and merge. Performance drops dramatically
  • Cross-Shard transactions: Requires two-phase commit (2PC); Vitess coordinates via VTGate, but performance and reliability are inferior to single-Shard transactions
  • Shard key selection: Once chosen (e.g., user_id), it's nearly impossible to change. A wrong choice (e.g., sharding by order_id but frequently querying by user_id) leads to massive cross-Shard queries
  • Hotspot problem: If one large customer (user_id=1) has far more data than others, their Shard becomes a hotspot. Solution: further split that Shard, or handle it specially at the application layer

Summary

The five-step model for scaling stateful services:

  1. Sharding: Key → Slot → Node two-layer mapping, Slot count = machines x 100+
  2. Control plane: Where the routing table lives and who is authoritative. Trend is etcd + Operator
  3. Routing: Smart Client (low latency) vs Proxy (easy ops). Multi-language environments choose Proxy
  4. Replicas: Async (fast but may lose data) vs semi-sync (balanced) vs Raft (strong consistency but slower)
  5. Migration: Full snapshot + incremental log + atomic switchover. The critical part is the strategy for the switchover moment

Three levels of practice:

  • Memcached: Simplest — don't move data, cache self-heals on miss. Suitable for pure cache scenarios
  • Redis: Medium complexity — key-by-key migration, Gossip coordination. Suitable for both cache and lightweight storage
  • MySQL: Most complex — snapshot + Binlog incremental, plus handling cross-Shard JOINs and transactions. Suitable for persistent relational data

Regardless of the specific system, the five-step model is universal. Next time you encounter a new stateful system that needs scaling, ask these five questions: how to shard, where metadata lives, how requests are routed, how replicas work, and how data is moved.
