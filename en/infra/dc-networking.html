The Journey of a Request: A Panoramic View of Data Center Networking

2026-03-08
What you'll learn from this post: Most backend engineers know that a request goes from the browser to the server and back. But what happens in between? PoP nodes, backbone fiber optics, Clos switching fabrics, eBPF L4 load balancing, VXLAN container networking — this infrastructure determines what your system can and cannot do. This post traces a request's complete journey from a user's phone to a data center server, dissecting the key technologies at each layer along the way.

The Big Picture: Three-Layer Architecture

The network infrastructure of any large-scale internet company follows a three-layer structure:

LayerResponsibilityExample
Edge / PoPUser traffic ingress, TLS termination, geographic routingTokyo PoP receives requests from Japanese users
BackboneHigh-speed private fiber between data centers, running SR/MPLSSingapore DC ↔ US DC, ~120ms RTT
DC InternalClos switching fabric connecting all servers within a DCLeaf-Spine-Core three-layer switching

A Tokyo user opens an app: Phone DNS resolves to the nearest Tokyo PoP IP → PoP terminates TLS, static content returned directly from CDN → Dynamic requests travel over the backbone (SR/MPLS) to the Singapore DC → Enter the DC through Katran (L4 LB) → Proxygen (L7 LB) → Application server → If data from the US DC is needed, another cross-ocean backbone hop → Response returns directly to the user via DSR, bypassing the L4 LB.

Let's dissect each layer along this path.

DC Internal: Clos Architecture

A single large switch cannot scale to hundreds of thousands of servers. Clos topology combines many cheap small switches into a non-blocking, horizontally scalable network.

Traffic path hop counts:

  • Same rack: Leaf local switching, 0 hops
  • Same Pod, different rack: Leaf → Spine → Leaf, 2 hops
  • Cross Pod: Leaf → Spine → Core → Spine → Leaf, 4 hops

Pod: The Repeatable Building Block

A Pod is a group of Leafs + corresponding Spines, the basic unit for DC scaling. Need more capacity? Add a Pod and connect it to Core.

Scale calculation using 64-port switches:

Leaf:  32 ports facing down to servers → 32 servers per rack
Pod:   32 Leafs                        → 32 × 32 = 1,024 servers
DC:    32 Pods                         → 32 × 1,024 = 32,768 servers

Need more? Add another layer (5-stage Clos / Super Spine), supporting hundreds of thousands of servers. Large internet companies' F16 Fabric uses this multi-stage Clos design.

ECMP: Equal-Cost Multi-Path

The core design principle of Clos architecture. When Leaf 1 sends a packet to Leaf 4, there are multiple equal-cost paths through different Spines. Switches hash the five-tuple (source IP, destination IP, source port, destination port, protocol) to evenly distribute across all paths.

  Leaf 1 ──sends packet──►  To reach Leaf 4, there are 4 equal-cost paths:

  Leaf 1 ─── Spine A ─── Leaf 4    ← hash(five-tuple) = 0
  Leaf 1 ─── Spine B ─── Leaf 4    ← hash(five-tuple) = 1
  Leaf 1 ─── Spine C ─── Leaf 4    ← hash(five-tuple) = 2
  Leaf 1 ─── Spine D ─── Leaf 4    ← hash(five-tuple) = 3

  Traffic evenly distributed → No bottleneck → If a Spine fails, auto-redistribute

Why ECMP matters:

  • No bottleneck: Traffic is evenly distributed, no single link overloaded
  • Fault tolerance: If one Spine goes down, traffic automatically redistributes to remaining Spines
  • Linear scaling: Need more bandwidth? Just add Spines

Oversubscription

The ratio between a Leaf switch's downstream bandwidth (facing servers) and upstream bandwidth (facing Spines).

RatioMeaningUse Case
1:1Non-blocking, all servers can communicate at full speed simultaneouslyLatency-sensitive (DB clusters, ML training)
3:1Only 1/3 of server bandwidth can go upstream simultaneouslyGeneral compute (statistically sufficient)
4:1+Higher oversubscription, cheaper but riskierCold storage, batch processing

Practical impact: When designing services, place frequently communicating services in the same rack or Pod to avoid oversubscription bottlenecks. This is why HDFS places one replica in the same rack and another cross-rack.

Backbone: Segment Routing

Data centers are connected via private backbone networks. Traditional MPLS requires each intermediate router to maintain forwarding state (via LDP/RSVP). Segment Routing (SR) is its evolution: encode the complete path in the packet header, intermediate routers just read the next instruction.

MPLS analogy: Taking a taxi, asking for directions at every intersection
SR analogy:   Getting full GPS navigation before departure, just follow the next instruction at each turn
  MPLS: Every router maintains forwarding state
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ A │───►│ B │───►│ C │───►│ D │     B,C each lookup LDP table to decide next hop
  └───┘    └───┘    └───┘    └───┘

  SR: Path encoded in packet header, intermediate routers just execute instructions
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ A │───►│ B │───►│ C │───►│ D │
  └───┘    └───┘    └───┘    └───┘
  Header: [B, C, D]   [C, D]    [D]       Pop one SID per hop

Two types of Segment IDs:

  • Node SID: Take the shortest path to a specific router — "go to router D"
  • Adjacency SID: Take a specific physical link — "use the B→C link"

SRv6 is the direction being pushed by major hyperscalers. The 128-bit SID can simultaneously encode destination and operation (e.g., "decapsulate", "lookup VPN table"), far more flexible than SR-MPLS's 20-bit labels.

Three key design decisions:

  • XDP/eBPF: Packets processed at the NIC driver layer, bypassing the kernel network stack. A single machine can achieve millions of PPS
  • Consistent hashing: The same TCP connection always maps to the same backend. When backends change, only a small fraction of connections need remapping
  • DSR (Direct Server Return): Responses go from backend servers directly to the user, bypassing Katran. Since responses are much larger than requests (think an HTTP request vs. a page with images), DSR prevents the LB from becoming a bandwidth bottleneck

Overlay Networks: How Containers Communicate

Your code runs in containers, containers have virtual IPs, and the physical network doesn't recognize these IPs. Overlay networks bridge this gap.

Underlay vs Overlay

LayerWhat it isIPDevices
UnderlayClos physical switching fabric192.168.x.x (host IP)Leaf, Spine, Core
OverlayContainer virtual network10.244.x.x (Pod IP)veth, bridge, VXLAN

Linux Virtual Network Devices

  • veth pair: A virtual cable connecting two network namespaces. Docker/K8s creates a pair: one end goes inside the container (eth0), the other stays on the host connected to a bridge
  • bridge: A virtual L2 switch on the host. Containers on the same host communicate through it without touching the physical network
  • VXLAN device (VTEP): When a container needs to reach a container on another host, the VXLAN device encapsulates the original frame in a UDP packet (port 4789), using host IPs as the outer addresses. The physical network routes normally using ECMP, and the destination host's VXLAN device decapsulates and delivers to the target container

Why VXLAN instead of VLAN? VLAN supports only 4,096 IDs (12 bits). VXLAN uses 24-bit VNI, supporting 16 million virtual networks — essential for multi-tenant DC environments.

DC Internal vs Public Internet: Two Completely Different Worlds

This is the most counterintuitive part. Textbook TCP/IP is designed for the public internet — high latency, high packet loss, unknown bandwidth. The DC interior is a completely different environment that requires different optimizations at every layer.

RTT10-200ms10-100μs (1000x lower)
Packet loss1-5%<0.001%
BandwidthUnknown, variableKnown (25G/100G/400G)

Incast: A DC-Specific Congestion Pattern

A request fans out to N servers (e.g., distributed query), N servers respond simultaneously, instantly overwhelming the requester's Leaf switch buffer. Result: packet loss, TCP retransmissions, latency spikes.

Solutions:

  • ECN: Switches mark packets when buffer reaches threshold (e.g., 30%), senders slow down before packet loss occurs
  • DCTCP: Proportionally reduces speed based on the fraction of ECN-marked packets, rather than a binary "halve"
  • Application-level staggering: Add jitter to response times to avoid synchronized bursts

From the moment a user sends a request to receiving the response, it passes through DNS, CDN, PoP, backbone, border router, ECMP, eBPF L4 LB, L7 LB, container overlay network — each layer solves a specific problem, and each layer's design is constrained by the laws of physics (speed of light, bandwidth, buffer size).

Summary

Key intuitions worth remembering:

  • Clos + ECMP: Combine cheap small switches into a large network, distributing traffic evenly via hashing. The physical foundation of DC networking
  • Katran + DSR: eBPF does load balancing at the NIC layer, responses bypass the LB and go directly to users. Solves the bandwidth bottleneck and single point of failure of traditional LBs
  • DC RTT is 1000x lower than the public internet: This order-of-magnitude difference changes all assumptions. Public internet TCP congestion control, connection management, and security models don't apply inside DCs — specialized solutions are needed (DCTCP, RDMA, Zero Trust)
  • Overlay networks are the glue: Containers' virtual IPs are mapped to the physical network via VXLAN/BGP. Understanding this encapsulation/decapsulation explains why cross-host container communication is slower than same-host
  • Oversubscription determines service deployment strategy: Placing frequently communicating services in the same rack or Pod isn't "optimization" — it's a necessary measure to avoid oversubscription bottlenecks
