From 0.98 to 267 GFLOPS: A Complete Record of Matrix Multiplication Optimization

2026-02-16

Introduction

Matrix multiplication (GEMM, General Matrix Multiply) is one of the most fundamental operations in high-performance computing. Neural network fully connected layers, convolutional layers, and attention mechanisms all boil down to matrix multiplication. The sgemm function in BLAS libraries is one of the most aggressively optimized functions in existence, and the core competitive advantage of libraries like Intel MKL, OpenBLAS, and BLIS lies here.

I wanted to understand what exactly happens between a textbook triple loop and an implementation approaching the theoretical hardware peak. So I started from scratch in C++, optimizing step by step, documenting the reasoning, results, and pitfalls at each stage.

Problem Definition

Compute C = A × B, where A is an M×K matrix, B is a K×N matrix, and C is an M×N matrix. All matrices are row-major float32.

Hardware:

  • CPU: Intel i7-14700KF (Raptor Lake, 14 cores / 20 threads)
  • L1d: 48 KB/core, L2: 2 MB/core, L3: 33 MB shared
  • Supports AVX2 + FMA, no AVX-512

How is the theoretical peak calculated?

Let's break this down. The i7-14700KF has a single-core turbo frequency of approximately 5.5 GHz, meaning 5.5 billion clock cycles per second. The key question is: how many floating-point operations can be done per cycle?

This CPU supports the AVX2 instruction set. AVX2 registers are 256 bits (32 bytes) wide, fitting 8 float32 values simultaneously. It also supports FMA (Fused Multiply-Add) instructions — one instruction completes both a multiply and an add, i.e., a * b + c, counting as 2 floating-point operations. And Raptor Lake has 2 FMA execution units per core, capable of simultaneously executing 2 FMA instructions in the same clock cycle.

So single-core FLOP per cycle = 2 FMA units × 8 floats/FMA × 2 FLOP/FMA = 32 FLOP/cycle. Theoretical single-core peak:

5.5 GHz × 32 FLOP/cycle ≈ 176 GFLOPS (single-precision, single-core)

Multi-core theoretical peak is higher, but limited by memory bandwidth and shared cache, actual achievable performance depends on computational density. As we'll see, single-thread optimization to 144 GFLOPS (82% of peak) is already an excellent result.

Benchmark matrix size is N=1024, i.e., 1024×1024 square matrix multiplication. The computation formula for matrix multiplication is 2×M×N×K (each output element requires K multiplications and K additions), so total computation is 2×1024³ ≈ 2.15 billion floating-point operations.

Step 0: Textbook Version (i-j-k) — 0.98 GFLOPS

The first version everyone writes when learning linear algebra:

void matmul_baseline(const float* A, const float* B, float* C,
                     size_t M, size_t N, size_t K) {
  std::memset(C, 0, M * N * sizeof(float));
  for (size_t i = 0; i < M; ++i) {
    for (size_t j = 0; j < N; ++j) {
      for (size_t k = 0; k < K; ++k) {
        C[i * N + j] += A[i * K + k] * B[k * N + j];
      }
    }
  }
}

0.98 GFLOPS, about 1% of theoretical peak. Absurdly slow.

The problem lies in the inner loop's memory access pattern. Our matrices are stored in row-major order, meaning B[0][0], B[0][1], B[0][2], ... are contiguous in memory. But the inner loop iterates over k, accessing B[k * N + j] each time — every increment of k jumps N floats (4096 bytes) in memory, effectively jumping row by row.

Modern CPUs have an important design: when you read one piece of data from memory, the CPU doesn't just read that one value — it loads the entire cache line (typically 64 bytes) containing it. For float32, one cache line holds 16 consecutive floats. The CPU assumes "if you read this value, you'll probably read its neighbors next" — this is called spatial locality.

But the i-j-k loop completely violates this assumption. Each cache line load (64 bytes = 16 floats) uses just 1 float before jumping away. Cache line utilization is only 1/16 = 6.25%. Cachegrind measurements show D1 miss rate as high as 32.8% — one in every 3 memory accesses is a cache miss.

Step 1: Loop Reordering (i-k-j) — 35.3 GFLOPS

Move the k loop to the middle, making the inner loop iterate over j:

void matmul_naive(const float* A, const float* B, float* C,
                  size_t M, size_t N, size_t K) {
  std::memset(C, 0, M * N * sizeof(float));
  for (size_t i = 0; i < M; ++i) {
    for (size_t k = 0; k < K; ++k) {
      float a_ik = A[i * K + k];
      for (size_t j = 0; j < N; ++j) {
        C[i * N + j] += a_ik * B[k * N + j];
      }
    }
  }
}

36x speedup, just by swapping two loop lines.

Now in the inner loop, both B[k*N+j] and C[i*N+j] are accessed sequentially along the row (stride-1). Each 64-byte cache line loaded has all 16 floats fully utilized, raising cache line utilization from 6.25% to 100%. Meanwhile A[i*K+k] is constant within the inner loop, hoisted into a local variable a_ik — this is called scalar replacement, which the compiler typically does automatically.

Cachegrind measurements: D1 miss rate drops from 32.8% to 4.8%.

This is the single largest improvement in the entire project. The lesson is simple: memory access patterns matter more than algorithmic complexity. Both versions perform exactly the same number of arithmetic operations (both 2N³), but the difference in memory access patterns causes a 36x performance gap.

Step 2: Cache Tiling — 29.3 GFLOPS

The classic textbook optimization: divide the large matrix into small blocks (64×64×64), so each block's data (64×64×4B = 16KB) fits in the 48KB L1d cache. This way, while processing a block, data can be reused repeatedly without being evicted from cache.

void matmul_tiled(const float* A, const float* B, float* C,
                  size_t M, size_t N, size_t K) {
  constexpr size_t BM = 64, BN = 64, BK = 64;
  std::memset(C, 0, M * N * sizeof(float));

  for (size_t i0 = 0; i0 < M; i0 += BM) {
    const size_t imax = std::min(i0 + BM, M);
    for (size_t k0 = 0; k0 < K; k0 += BK) {
      const size_t kmax = std::min(k0 + BK, K);
      for (size_t j0 = 0; j0 < N; j0 += BN) {
        const size_t jmax = std::min(j0 + BN, N);
        for (size_t i = i0; i < imax; ++i) {
          for (size_t k = k0; k < kmax; ++k) {
            float a_ik = A[i * K + k];
            for (size_t j = j0; j < jmax; ++j) {
              C[i * N + j] += a_ik * B[k * N + j];
            }
          }
        }
      }
    }
  }
}

Result: actually slower. From 35.3 down to 29.3 GFLOPS.

The reason is the compiler (GCC/Clang -O3) already optimizes the simple i-k-j loop well. Tiling introduces extra loop control overhead and std::min boundary calculations, which don't pay for themselves. Moreover, i-k-j already has very cache-friendly access patterns — both B and C are stride-1 access, so tiling offers little marginal benefit.

Lesson: Tiling's value only shows when combined with register tiling and packing. Used alone, the modern compiler may do better than you.

Step 3: Hand-Written SIMD (AVX2 FMA) — 29.3 GFLOPS

First, let me explain what SIMD is. SIMD (Single Instruction, Multiple Data) is a form of parallel computation — one instruction processes multiple data elements simultaneously. A regular scalar instruction can only add 1 float at a time; AVX2 SIMD instructions can add 8 floats simultaneously (because YMM registers are 256 bits wide, 256 / 32 = 8 floats).

FMA (Fused Multiply-Add) goes further: one instruction completes a * b + c, doing both multiply and add, with only one rounding error (more precise than separate multiply then add). For matrix multiplication, the core operation is C[i][j] += A[i][k] * B[k][j], naturally suited for FMA.

In C/C++, you can call these instructions directly via intrinsics functions, without writing assembly:

// Inner loop core
__m256 va = _mm256_set1_ps(A[i * K + k]);     // Broadcast 1 float to 8 lanes
size_t j = j0;
for (; j + 8 <= jmax; j += 8) {
  __m256 vb = _mm256_loadu_ps(&B[k * N + j]); // Load 8 consecutive floats
  __m256 vc = _mm256_loadu_ps(&C[i * N + j]); // Load 8 consecutive floats
  vc = _mm256_fmadd_ps(va, vb, vc);            // 8-way parallel: C += A * B
  _mm256_storeu_ps(&C[i * N + j], vc);         // Write back 8 floats
}

Each _mm256 function corresponds to an AVX2 instruction. _mm256_set1_ps broadcasts a scalar to all 8 positions of a 256-bit register; _mm256_loadu_ps loads 8 consecutive floats from memory to a register; _mm256_fmadd_ps performs FMA on 8 pairs simultaneously.

Result: exactly the same as the tiled version.

With -O3 -march=native -ffast-math, the compiler already auto-vectorized the inner loop. The compiler recognized the simple for j loop and automatically converted it to SIMD instructions — doing exactly what our hand-written intrinsics do. You can confirm auto-vectorization with -fopt-info-vec (GCC) or -Rpass=loop-vectorize (Clang).

Lesson: With modern compilers, hand-written SIMD for simple loops usually can't beat auto-vectorization. Real gains come from things compilers can't do — like the next step's register tiling, which requires the programmer to precisely plan each register's purpose.

Step 4: Register Tiling — 125.8 GFLOPS (Key Breakthrough)

All previous versions share a fundamental problem: every multiply-add reads and writes C from memory (or cache). In the i-k-j inner loop, C[i*N+j] += a_ik * B[k*N+j] means reading C, doing FMA, and writing back C every iteration. Even when C is in L1 cache, read/write latency still exists.

Register tiling's approach: keep a small sub-block of C entirely in CPU registers. Registers are the fastest storage in the CPU — zero access latency, no cache hierarchy involved. If we can keep C's sub-block in registers throughout the entire k loop, loading once at the start and storing once at the end, we dramatically reduce memory accesses.

The key question is: how large can the sub-block be? That depends on how many registers the CPU has.

              NR=16 (2 × __m256)
           +----------------------+
     MR=6  | c00 c01              |  12 YMM registers hold C's 6×16 sub-block
    rows   | c10 c11              |
     of C  | c20 c21              |  In k loop:
           | c30 c31              |    Load 2 B vectors (b0, b1)
           | c40 c41              |    Broadcast 6 A scalars
           | c50 c51              |    12 FMAs
           +----------------------+

AVX2 provides 16 YMM registers (ymm0-ymm15), each 256 bits wide (8 floats). We need to allocate them carefully:

  • C sub-block: 6 rows × 2 __m256 (16 floats per row) = 12 registers
  • B vectors: 2 __m256 per k step = 2 registers
  • A broadcast: broadcast 1 A element per k step = 1 register
  • Temporary: 1 register
  • Total: 12 + 2 + 1 + 1 = 16, exactly full

Why MR=6, NR=16 and not other combinations? Because this combination has the highest computational density. Each k step does 12 FMAs (6 rows × 2 vectors), needing only 3 memory reads (2 B vectors + 1 A scalar), giving a density of 12/3 = 4 FMA/load. With 4×24 (also 12 C registers), you'd need 3 B + 1 A = 4 reads, density = 12/4 = 3 FMA/load, which is lower.

Core implementation (micro-kernel):

// 1. Load C's 6×16 sub-block into 12 registers (done once)
__m256 c00 = _mm256_loadu_ps(&C[0 * N + 0]);
__m256 c01 = _mm256_loadu_ps(&C[0 * N + 8]);
__m256 c10 = _mm256_loadu_ps(&C[1 * N + 0]);
__m256 c11 = _mm256_loadu_ps(&C[1 * N + 8]);
// ... c20, c21, c30, c31, c40, c41, c50, c51

// 2. Accumulate entirely in registers during k loop
for (size_t k = 0; k < K; ++k) {
  __m256 b0 = _mm256_loadu_ps(&B[k * N + 0]);  // Load 16 floats of B
  __m256 b1 = _mm256_loadu_ps(&B[k * N + 8]);

  __m256 a;
  a = _mm256_set1_ps(A[0 * lda + k]);  // Broadcast A[0][k]
  c00 = _mm256_fmadd_ps(a, b0, c00);   // c00 += a * b0
  c01 = _mm256_fmadd_ps(a, b1, c01);   // c01 += a * b1

  a = _mm256_set1_ps(A[1 * lda + k]);  // Broadcast A[1][k]
  c10 = _mm256_fmadd_ps(a, b0, c10);
  c11 = _mm256_fmadd_ps(a, b1, c11);

  // ... same for rows 2-5, 12 FMAs total
}

// 3. After k loop, write C back to memory (done once)
_mm256_storeu_ps(&C[0 * N + 0], c00);
_mm256_storeu_ps(&C[0 * N + 8], c01);
// ...

Cachegrind results: D refs (total data accesses) dropped from 177M to 21M (8.4x reduction). Because C is no longer read/written repeatedly in the k loop — previously each k step read/wrote C's 6×16 = 96 floats, now the entire k loop only does it once at the beginning and once at the end.

4.3x speedup, from 29 GFLOPS to 126 GFLOPS. This is the most important single-step optimization after loop reordering. The compiler can't do this level of register orchestration — it won't proactively keep a large chunk of C in registers across an entire loop. This requires the programmer to have precise understanding of hardware resources.

Step 5: Goto-style Packing — 144.4 GFLOPS

The register tiling micro-kernel still accesses A with stride=K and B with stride=N. With large matrices, between adjacent micro-kernel calls, A and B data is non-contiguous in memory — separated by lots of unneeded data, wasting cache line space.

The solution comes from Kazushige Goto's classic 2008 paper (the author of GotoBLAS): before executing the micro-kernel, rearrange (pack) A and B data into contiguous memory regions, so every memory read by the micro-kernel is completely sequential.

Original A (stride = K):              After pack_A (stride = MR):
+------------------+              +---------+
| a00 a01 a02 ...  |  MR=6 rows  | a00     |
| a10 a11 a12 ...  |  -------->  | a10     |  Every MR values contiguous
| ...              |              | a20     |  Micro-kernel reads sequentially
| a50 a51 a52 ...  |              | ...     |
+------------------+              | a01     |
                                  | a11     |
                                  +---------+

But packing alone isn't enough — the loop structure must also be carefully designed so each cache level is fully utilized. This is the Goto-style three-level loop nesting:

for (size_t jc = 0; jc < N; jc += NC) {       // Layer 3: B panel resides in L3
  for (size_t kc = 0; kc < K; kc += KC) {     // Layer 2: B slice resides in L2
    pack_B(&B[kc * N + jc], packed_B, kk, nc, N);

    for (size_t ic = 0; ic < M; ic += MC) {   // Layer 1: A slice resides in L1
      pack_A(&A[ic * K + kc], packed_A, mc, kk, K);

      // Macro-kernel: iterate over MR×NR micro-tiles.
      for (size_t ir = 0; ir < mc; ir += MR) {
        for (size_t jr = 0; jr < nc; jr += NR) {
          microkernel_packed_6x16(
              &packed_A[ir * kk],
              &packed_B[jr * kk],
              &C[(ic + ir) * N + (jc + jr)],
              kk, N);
        }
      }
    }
  }
}

Each tile dimension precisely corresponds to a cache level:

  • MC×KC = 72×256 = 72KB: packed_A size, resides in L1d cache (48KB) + minor spill to L2
  • KC×NC = 256×4096 = 4MB: packed_B size, resides in L2 (2MB) + L3
  • MC must be a multiple of MR(6), NC must be a multiple of NR(16), ensuring the micro-kernel doesn't need boundary handling

The core design idea: packed_B is packed once in the outermost loop, then shared and reused by all MC-strips. packed_A is packed once per MC-strip, then shared and reused by all NR-strips in the inner loop. This way, packing overhead is amortized across many micro-kernel computations.

Single-thread from 126 to 144 GFLOPS, 15% improvement. More significant on larger matrices, since the problem of non-contiguous data in cache is more severe with larger matrices.

Step 6: Multi-threaded Parallelism (OpenMP) — 267.2 GFLOPS

All previous optimizations were single-threaded. The i7-14700KF has 14 cores and 20 threads — we were only using one core. Time to put the rest to work.

OpenMP is the most common shared-memory parallel programming interface for C/C++/Fortran. It uses compiler directives (pragmas) to control parallelization instead of manual thread management. Just add #pragma omp parallel for before a loop, and the compiler automatically distributes loop iterations across multiple threads.

#pragma omp parallel
{
  // Per-thread private packed_A buffer
  float* packed_A = static_cast<float*>(
      std::aligned_alloc(32, MC * KC * sizeof(float)));

  #pragma omp for schedule(dynamic)
  for (size_t ic = 0; ic < M; ic += MC) {
    const size_t mc = std::min(MC, M - ic);
    pack_A(&A[ic * K + kc], packed_A, mc, kk, K);

    for (size_t ir = 0; ir < mc; ir += MR) {
      for (size_t jr = 0; jr < nc; jr += NR) {
        microkernel_packed_6x16(
            &packed_A[ir * kk],
            &packed_B[jr * kk],
            &C[(ic + ir) * N + (jc + jr)],
            kk, N);
      }
    }
  }

  std::free(packed_A);
}

Key parallelization design decisions:

  • packed_B shared: B panel is packed once outside the parallel region, read-only shared by all threads, no locking needed
  • packed_A private: Each thread allocates its own packed_A buffer. If multiple threads shared one buffer, false sharing would occur — different threads writing the same cache line, causing cache lines to bounce between cores (cache ping-pong), severely degrading performance
  • Parallelization granularity: Parallel at the MC-strip (ic loop) level. Each MC-strip processes 72 independent rows of the C matrix, with no write conflicts between strips, requiring no synchronization
  • schedule(dynamic): Dynamic scheduling means idle threads automatically pick up the next unprocessed strip, suitable for uneven workloads (e.g., when matrix rows aren't evenly divisible by MC, the last strip is smaller)

From 144 to 267 GFLOPS, 1.85x speedup. Didn't achieve ideal linear scaling (14 cores should be 14x), due to: memory bandwidth becoming a bottleneck, L3 cache contention, thread scheduling overhead, and pack_B still executing serially on a single thread. But at the N=1024 scale, 267 GFLOPS is quite good.

Step 7: Micro-Optimizations (Unroll + Prefetch + Huge Pages) — 266.3 GFLOPS

Finally tried three micro-optimizations:

1) k-unroll by 4

Unroll the micro-kernel's k loop 4 times. Without unrolling, each iteration ends with compare, increment, and jump instructions; after unrolling, each group of 4 iterations only needs one set of control instructions, reducing loop overhead. More importantly, unrolling gives the CPU more instructions to execute simultaneously — modern CPUs execute out-of-order, executing later independent instructions while an earlier one waits for data. Unrolling exposes more such parallelism opportunities.

for (; k + 4 <= kc; k += 4) {
  _mm_prefetch(packed_A + 4 * MR, _MM_HINT_T0);
  _mm_prefetch(packed_B + 4 * NR, _MM_HINT_T0);

  MICRO_ITER(packed_A + 0 * MR, packed_B + 0 * NR);
  MICRO_ITER(packed_A + 1 * MR, packed_B + 1 * NR);
  MICRO_ITER(packed_A + 2 * MR, packed_B + 2 * NR);
  MICRO_ITER(packed_A + 3 * MR, packed_B + 3 * NR);

  packed_A += 4 * MR;
  packed_B += 4 * NR;
}

2) Software prefetch

CPUs have hardware prefetchers that automatically detect sequential memory access patterns and preload data. But hardware prefetchers aren't omniscient — they can only recognize simple linear patterns (e.g., consecutive increments) and may mispredict for strided or post-branch access patterns.

Software prefetch via _mm_prefetch explicitly tells the CPU: "I'll need this address's data in a few steps, start loading it from memory to cache now." By the time the code actually needs the data, it's already in cache, avoiding memory wait latency (typically 50-100ns).

_MM_HINT_T0 means load to L1 cache (closest, fastest), _MM_HINT_T1 means load to L2 cache. We use _MM_HINT_T1 for the next B panel since it's larger and better suited for L2.

3) Transparent Huge Pages

This involves a key component of virtual memory: TLB (Translation Lookaside Buffer).

Modern OSes use virtual memory — the memory addresses programs see aren't real physical addresses and need translation via page tables. This translation is slow (may require multi-level page table walks), so CPUs use TLB to cache recently used address translations. TLB capacity is very small; a typical L1 dTLB has only 64-128 entries.

Default memory page size is 4KB. With 128 TLB entries, TLB can only cover 128 × 4KB = 512KB of memory. Our packed_B is 4MB — far exceeding TLB coverage. When the micro-kernel traverses packed_B, frequent TLB misses trigger page table walks.

Huge pages increase page size from 4KB to 2MB. The same 128 TLB entries now cover 128 × 2MB = 256MB, easily fitting the 4MB packed_B.

// packed_B (~4MB) uses 2MB huge pages to reduce TLB misses
void* ptr = std::aligned_alloc(2 * 1024 * 1024, aligned_bytes);
madvise(ptr, aligned_bytes, MADV_HUGEPAGE);

madvise is a Linux system call, MADV_HUGEPAGE tells the kernel this memory region should use transparent huge pages. Note packed_A (72KB per thread) is too small — using 2MB pages would be wasteful, so only packed_B uses them.

Actual results: at N=1024, 267 → 266 GFLOPS, essentially no improvement. At this scale, the hardware prefetcher already works well, and packed data access patterns are already very regular. These micro-optimizations yield more benefit on larger matrices (N=2048+) — the larger the data, the more TLB misses and prefetch impact become noticeable.

Cachegrind Cache Analysis

Valgrind is a program analysis tool framework, and its cachegrind module specializes in analyzing a program's cache behavior. It simulates the CPU's cache hierarchy, counting whether each memory access hits L1/L2/L3 cache.

Usage is simple:

# First compile an optimized executable
bazel build -c opt //bench:cache_profile

# Run under cachegrind, specifying version and matrix size
valgrind --tool=cachegrind ./bazel-bin/bench/cache_profile naive 512

Cachegrind outputs detailed cache statistics. Key metrics:

  • D refs: Total data access count (reads and writes). Reflects the program's memory access frequency
  • D1 misses: L1 data cache miss count. Each miss means data isn't in the fastest L1 cache and must be fetched from slower L2/L3 or main memory
  • D1 miss rate: D1 misses / D refs. Lower is better

Note cachegrind slows programs 10-50x (because it simulates the cache), so N=512 is used instead of N=1024:

Version D refs D1 misses D1 miss rate
Baseline (i-j-k) 177M 58.1M 32.8%
Naive (i-k-j) 177M 8.5M 4.8%
Tiled 179M 1.6M 0.9%
Register Tiling 21M 1.6M 7.8%

This table is worth studying carefully, as it precisely explains the mechanism behind each optimization:

  • Baseline → Naive: D refs identical (177M), but D1 misses drop from 58.1M to 8.5M. Same number of accesses, but dramatically fewer misses — this is the power of loop reordering. Same computation, different access order, cache hit rate improved from 67% to 95%.
  • Naive → Tiled: D refs essentially unchanged (177M → 179M), but D1 misses drop from 8.5M to 1.6M, miss rate from 4.8% to 0.9%. Tiling lets data in cache be reused more times (data loaded into cache gets used repeatedly before being evicted).
  • Tiled → Register Tiling: The key change is D refs — from 179M down to 21M, an 8.5x reduction! Absolute D1 misses stay the same (1.6M), but because the denominator (D refs) shrinks dramatically, miss rate actually rises to 7.8%. This is the essence of register tiling: not fewer misses, but fundamentally fewer memory accesses — C's reads and writes are entirely done in registers.

Summary and Reflections

A few things from this project that impressed me most:

1. Memory access patterns are the #1 performance killer. Loop reordering alone yielded a 36x speedup. On modern hardware, cache-friendly access patterns matter more than any advanced optimization. Two versions with identical arithmetic operation counts differ in performance by 36x.

2. Not all "classic optimizations" are effective. Cache tiling and hand-written SIMD yielded no improvement in this scenario. Modern compilers are very powerful — don't underestimate -O3 -march=native -ffast-math. Before optimizing, check what the compiler has already done.

3. Register tiling is the key breakthrough. This is an optimization the compiler can't do — it requires the programmer to precisely understand how many registers the hardware has, what each register should hold, and how to maximize computational density. 12 YMM registers keeping C's sub-block eliminated repeated reads and writes to C in the k loop, reducing D refs by 8.5x.

4. Packing solves the data layout problem. Rearranging A and B into micro-kernel-friendly contiguous memory ensures every byte of each cache line is fully utilized. Goto-style three-layer loops precisely correspond to L1/L2/L3 cache levels, a textbook example of combining engineering and theory.

5. Diminishing marginal returns from micro-optimizations. Unroll, prefetch, and huge pages had virtually no effect at N=1024. Performance optimization follows the Pareto principle — 80% of gains come from 20% of key optimizations.

From 0.98 to 267 GFLOPS, 273x speedup. This journey gave me a true understanding of the first half of Knuth's famous quote (people often only remember the second half):

"We should forget about small efficiencies, say about 97% of the time: premature optimization is the root of all evil. Yet we should not pass up our opportunities in that critical 3%."

The key is finding that 3%.
