From Zero to Generation: Building an LLM Inference Engine in Pure C++

2026-02-19

Introduction

I use LLMs every day, but how exactly does one turn a sequence of tokens into coherent text, step by step? Frameworks like PyTorch, vLLM, and llama.cpp hide all the details — you call model.generate() and text comes out. But I wanted to understand what actually happens in between.

So I built an LLM inference engine from scratch in pure C++20, with no ML framework dependencies: vvllm. It can run Qwen2.5-0.5B, taking a prompt as input and producing coherent text output. The entire process — from mmap-loading weights, to BPE encoding, to Transformer forward pass, to nucleus sampling — is hand-written.

This post follows the actual code flow, documenting the construction and optimization of this engine step by step.

Problem Definition

Goal: Given a text prompt, have the model generate coherent continuation.

Model: Qwen2.5-0.5B (0.5B parameter version)

  • hidden_size: 896
  • num_hidden_layers: 24
  • num_attention_heads: 14
  • num_key_value_heads: 2 (GQA, Grouped Query Attention)
  • intermediate_size: 4864
  • vocab_size: 151936
  • Weight format: SafeTensors, BF16 storage, ~1GB file / 1.9GB float32

Constraints: Pure C++20, no PyTorch / TensorFlow / ONNX Runtime. The only allowed external libraries are OpenBLAS (optional matrix multiplication acceleration) and nlohmann/json (config file parsing).

Build system: Bazel 7.4.1, cross-platform support for aarch64 / x86_64.

Overview

Step 0: Infrastructure — Backend + Tensor + SafeTensors

Before writing any inference logic, three foundational problems need solving: how to manage memory, how to abstract computation, and how to load model weights.

Backend Abstraction

Define an abstract compute backend interface. All operators (matmul, add, softmax, RoPE, etc.) are virtual functions with concrete implementations provided by subclasses. This allows seamlessly switching between naive CPU and OpenBLAS backends later:

class Backend {
public:
    virtual void matmul(float* out, const float* A, const float* B,
                        std::size_t M, std::size_t N, std::size_t K) = 0;
    virtual void rms_norm(float* out, const float* x, const float* weight,
                          std::size_t n, float eps) = 0;
    virtual void softmax(float* out, const float* x, std::size_t n) = 0;
    virtual void rope(float* q, float* k, std::size_t seq_len,
                      std::size_t num_heads, std::size_t num_kv_heads,
                      std::size_t head_dim, std::size_t pos, float theta) = 0;
    virtual void linear(float* out, const float* inp, const float* weight,
                        const float* bias, std::size_t M, std::size_t N,
                        std::size_t K) = 0;
    // ... add, mul, silu, embedding
};

The first implementation is a naive CPU backend — all operations are straightforward for loops. For example, linear projection (the most critical computation in inference):

void BackendCPU::linear(float* out, const float* inp, const float* weight,
                        const float* bias, std::size_t M, std::size_t N,
                        std::size_t K) {
    // out[M, N] = inp[M, K] @ weight^T[K, N] + bias[N]
    for (std::size_t i = 0; i < M; i++) {
        for (std::size_t j = 0; j < N; j++) {
            float sum = bias ? bias[j] : 0.0f;
            for (std::size_t k = 0; k < K; k++) {
                sum += inp[i * K + k] * weight[j * K + k];
            }
            out[i * N + j] = sum;
        }
    }
}

Triple loop, O(M×N×K) scalar multiplications. Slow, but correct. Get it running first, then optimize.

SafeTensors Loader

HuggingFace model weights are stored in SafeTensors format. The format is simple:

+------------------+
| 8 bytes          |  header_size (uint64)
+------------------+
| JSON header      |  tensor names, shapes, dtypes, offsets
+------------------+
| raw tensor data  |  binary floating-point data (BF16 or F32)
+------------------+

The key design decision is using mmap instead of read to load the file. mmap maps the file into virtual address space, and the OS loads data pages from disk on demand — rather than reading the entire file into memory at once. For 1GB+ weight files, this reduces startup time and peak memory usage.

mapped_ = mmap(nullptr, file_size_, PROT_READ, MAP_PRIVATE, fd_, 0);

Qwen2.5-0.5B weights are stored in BF16 (Brain Float 16) format. BF16 and float32 share the same 8-bit exponent, just truncating the mantissa from 23 bits to 7 bits. Conversion is simple — left shift by 16 bits:

float bf16_to_f32(uint16_t bf16) {
    uint32_t bits = static_cast<uint32_t>(bf16) << 16;
    float result;
    std::memcpy(&result, &bits, sizeof(float));
    return result;
}

A model has 290 tensors (each layer has attention Q/K/V/O weights, MLP gate/up/down weights, two layernorm weights, plus embedding and final norm), occupying ~1.9GB after loading.

Step 1: BPE Tokenizer — Turning Text into Numbers

LLM input isn't raw text, but a token ID sequence. The Tokenizer handles conversion between text and token IDs. Qwen2.5 uses BPE (Byte Pair Encoding) tokenization.

Encoding has three steps:

"Hello world"

Step 1: Pre-tokenize (split on spaces, spaces become Ġ prefix)
  → ["Hello", "Ġworld"]

Step 2: BPE merge (repeatedly merge the highest-priority adjacent pair)
  "Ġworld" → ['Ġ', 'w', 'o', 'r', 'l', 'd']
          → ['Ġ', 'wo', 'r', 'l', 'd']     // merge('w','o'), rank=300
          → ['Ġ', 'wor', 'l', 'd']          // merge('wo','r')
          → ['Ġ', 'world']                   // ...
          → ['Ġworld']                        // final merge

Step 3: Lookup (vocabulary, token → ID)
  ['Hello', 'Ġworld'] → [9707, 1879]

BPE's core loop: each round scans all adjacent token pairs, finds the one with the lowest merge rank (highest priority), and merges it into a new token. Repeat until no mergeable pairs remain.

// Step 3: Apply BPE merges
while (tokens.size() > 1) {
    int best_rank = -1;
    size_t best_idx = 0;
    for (size_t j = 0; j < tokens.size() - 1; j++) {
        std::string key = tokens[j] + " " + tokens[j + 1];
        auto it = merge_ranks_.find(key);
        if (it != merge_ranks_.end()) {
            if (best_rank < 0 || it->second < best_rank) {
                best_rank = it->second;
                best_idx = j;
            }
        }
    }
    if (best_rank < 0) break;
    tokens[best_idx] = tokens[best_idx] + tokens[best_idx + 1];
    tokens.erase(tokens.begin() + best_idx + 1);
}

Decoding is the reverse: token ID looks up the vocabulary to get the token string, concatenate, and replace Ġ (UTF-8: 0xC4 0xA0) back to spaces.

Qwen2.5's vocabulary has 151,936 tokens — meaning the final logits output is also a 151,936-dimensional vector.

Step 2: Transformer Forward Pass — The Engine's Core

This is the most important part of the entire engine. What does a complete forward pass do?

token_ids: [t0, t1, t2, ..., tn]
                |
        Embedding Lookup
                |
          x: [seq_len, 896]
                |
     +----------+----------+
     |     Transformer      |
     |      Layer × 24       |
     |                      |
     |  RMS Norm            |
     |  Q, K, V Projection  |
     |  RoPE                |
     |  Causal Attention    |
     |  Output Projection   |
     |  + Residual          |
     |  RMS Norm            |
     |  MLP (SwiGLU)        |
     |  + Residual          |
     +----------+-----------+
                |
          Final RMS Norm
                |
        Logits Projection
                |
        logits: [151936]

1. Embedding Lookup

The simplest step: map a token ID to a 896-dimensional vector. The embedding table is a [151936, 896] matrix, and lookup is just memcpy of one row:

void embedding(float* out, const float* table,
               std::size_t token_id, std::size_t hidden_size) {
    std::memcpy(out, table + token_id * hidden_size,
                hidden_size * sizeof(float));
}

2. RMS Norm

Each layer's input is first RMS-normalized — similar to LayerNorm but without subtracting the mean, only dividing by the RMS (Root Mean Square):

void rms_norm(float* out, const float* x, const float* weight,
              std::size_t n, float eps) {
    float sum = std::accumulate(x, x + n, 0.0f,
        [](float acc, float x) { return acc + x * x; });
    float rms = sqrt(sum / n + eps);
    for (size_t i = 0; i < n; i++) {
        out[i] = x[i] / rms * weight[i];
    }
}

Formula: out[i] = (x[i] / √(mean(x²) + ε)) × weight[i]. RMS Norm saves one mean calculation and one subtraction compared to Layer Norm, making it faster during inference, and experiments show comparable performance.

3. Q, K, V Linear Projections

The first step of the attention mechanism: project the 896-dimensional hidden state into Q, K, V spaces. This is matrix multiplication — the most time-consuming computation in LLM inference.

// For each token, compute Q, K, V
for (std::size_t s = 0; s < seq_len; s++) {
    const float* inp = norm_out.data() + s * hidden;
    linear(q.data() + s * num_heads * head_dim,
           inp, attn.q_proj_weight, q_bias(attn),
           num_heads * head_dim, hidden, backend);   // Q: [14×64, 896]
    linear(k_new.data() + s * kv_dim,
           inp, attn.k_proj_weight, k_bias(attn),
           kv_dim, hidden, backend);                  // K: [2×64, 896]
    linear(v_new.data() + s * kv_dim,
           inp, attn.v_proj_weight, v_bias(attn),
           kv_dim, hidden, backend);                  // V: [2×64, 896]
}

Note a key detail: Qwen2 uses Grouped Query Attention (GQA) — 14 query heads share 2 KV heads. This means K and V dimensions are 2×64=128, much smaller than Q's 14×64=896. This dramatically reduces KV cache memory usage with minimal impact on model quality.

4. RoPE (Rotary Position Encoding)

Transformers are inherently position-invariant — scramble the input order, and the output doesn't change. RoPE applies position-dependent rotations to Q and K, letting the model know each token's position in the sequence.

Core idea: for each pair of elements (xi, xi+half) in the head_dim vector, perform a 2D rotation based on position pos and frequency θ:

for (std::size_t i = 0; i < half; i++) {
    float freq = 1.0f / std::pow(theta, (float)(2 * i) / head_dim);
    float angle = position * freq;
    float x0 = head[i];
    float x1 = head[i + half];
    head[i]        = x0 * cos(angle) - x1 * sin(angle);
    head[i + half] = x0 * sin(angle) + x1 * cos(angle);
}

Frequencies range from low to high: lower dimensions encode high frequencies (local positional relationships), higher dimensions encode low frequencies (long-range dependencies). This is similar in concept to the Fourier transform.

5. Causal Attention

This is the Transformer's core. For each query position, compute attention scores with all preceding tokens, then take a weighted sum of values:

void causal_attention(float* out, const float* q, std::size_t q_idx,
                      const float* k, const float* v,
                      std::size_t attend_len, std::size_t num_heads,
                      std::size_t num_kv_heads, std::size_t head_dim,
                      float scale, Backend& backend) {
    std::size_t groups = num_heads / num_kv_heads;  // 14/2 = 7

    for (std::size_t h = 0; h < num_heads; h++) {
        std::size_t kv_h = h / groups;  // GQA: 7 Q heads share 1 KV head

        // 1. scores = Q · K^T / sqrt(head_dim)
        for (std::size_t t = 0; t < attend_len; t++) {
            float dot = 0.0f;
            for (std::size_t d = 0; d < head_dim; d++) {
                dot += q_head[d] * k_head[d];
            }
            scores[t] = dot * scale;  // scale = 1/√64 = 0.125
        }

        // 2. attention weights = softmax(scores)
        backend.softmax(scores.data(), scores.data(), attend_len);

        // 3. output = weighted sum of V
        for (std::size_t t = 0; t < attend_len; t++) {
            for (std::size_t d = 0; d < head_dim; d++) {
                head_out[d] += scores[t] * v_head[d];
            }
        }
    }
}

"Causal" means: each token can only see itself and preceding tokens, not the future. This is implemented via attend_len = pos + s + 1 — the s-th token only attends to s+1 positions.

6. MLP (SwiGLU)

After attention comes a two-layer MLP. Qwen2 uses the SwiGLU activation function:

// gate = linear(x, gate_proj)    // [896] → [4864]
// up   = linear(x, up_proj)      // [896] → [4864]
// gate = silu(gate) * up          // element-wise
// out  = linear(gate, down_proj)  // [4864] → [896]

backend.silu(gate.data(), gate.data(), seq_len * intermediate);
backend.mul(gate.data(), gate.data(), up.data(), seq_len * intermediate);

SiLU (also called Swish): silu(x) = x × σ(x) = x / (1 + e-x). SwiGLU's "gating" mechanism lets the network selectively retain or filter information — the gate path controls the switch, the up path provides content.

7. Residual Connection

After each attention block and MLP block is a residual connection: x = x + block_output. It solves the vanishing gradient problem in deep networks — gradients can flow directly along the skip connection without gradually decaying to zero across 24 layers.

8. Final Prediction

After processing through 24 Transformer layers, take the last token's hidden state, apply RMS Norm, then project to vocab dimension to get logits:

// Final norm (only last token)
backend.rms_norm(final_out.data(),
                 x.data() + (seq_len - 1) * hidden,
                 final_norm_weight, hidden, eps);

// Logits = final_out @ embed_tokens^T  (tied weights)
linear(logits.data(), final_out.data(), embed_tokens, nullptr,
       config.vocab_size, hidden, backend);

Note tied weights: the logits projection matrix and the embedding table are the same matrix. This is a common parameter sharing trick — input (token→vector) and output (vector→token probability) use the same parameters, saving 151936×896 ≈ 136 million parameters.

Step 3: Bug Fix — RoPE Rotary Position Encoding Pairing Convention

After finishing the forward pass, I ran the model — the output was all garbage. After much debugging, I found the problem was in RoPE's pairing convention.

RoPE needs to pair up elements of the head_dim vector for rotation. But there are two pairing methods:

head_dim = 8 vector: [x0, x1, x2, x3, x4, x5, x6, x7]

Interleaved (GPT-NeoX convention):
  Pairs (x0,x1), (x2,x3), (x4,x5), (x6,x7)
  Adjacent elements paired

Split-half (Qwen2 / Llama convention):
  Pairs (x0,x4), (x1,x5), (x2,x6), (x3,x7)
  First half paired with second half

I initially used the interleaved convention (since many early GPT implementations did this), but Qwen2 uses split-half. The two conventions are mathematically identical — both are 2D rotations — but applied to different element pairs, the results are completely different.

Debug process:

  1. First ran the same input in Python, comparing HuggingFace's logits output
  2. Layer-by-layer comparison revealed the first layer's Q projection was correct, but after RoPE it diverged
  3. Wrote numpy scripts to compute RoPE with both conventions, comparing against HuggingFace
  4. Confirmed split-half matches HuggingFace results, fixed the C++ implementation

Lesson: The hardest bugs to debug in LLM inference aren't algorithm errors, but convention differences. RoPE pairing method, attention mask implementation, weight layout ([N,K] vs [K,N]) — these details aren't covered in papers, only found by reading source code.

Step 4: KV Cache — 56x Decode Speedup

After fixing RoPE, the model could correctly generate text. But the speed was atrocious — generating each token required re-running the complete forward pass.

The problem lies in the nature of autoregressive generation: generating token N requires the attention context of all N-1 preceding tokens. The naive approach re-feeds all tokens into the model every time — but the K, V for the preceding N-1 tokens were already computed in previous steps!

Naive (no cache):
  Step 1: forward([t0, t1, t2, t3, t4])        ← 5 tokens, compute 5 K,V sets
  Step 2: forward([t0, t1, t2, t3, t4, t5])     ← 6 tokens, recompute 6 K,V sets
  Step 3: forward([t0, t1, t2, t3, t4, t5, t6]) ← 7 tokens, recompute 7 K,V sets
  → Complexity O(n²): recompute all history each step

KV Cache:
  Prefill:  forward([t0, t1, t2, t3, t4])  ← Cache 5 K,V sets
  Decode 1: forward([t5], pos=5)             ← Only compute t5's Q, use cached K,V
  Decode 2: forward([t6], pos=6)             ← Only compute t6's Q, use cached K,V
  → Complexity O(n): only compute new token each step

KV Cache implementation is straightforward — each layer maintains a growing K and V vector:

struct KVCache {
    std::vector<std::vector<float>> k_cache;  // [num_layers][seq_len * kv_dim]
    std::vector<std::vector<float>> v_cache;
    std::size_t seq_len = 0;

    void append(std::size_t layer_idx, const float* new_k,
                const float* new_v, std::size_t num_new_tokens,
                std::size_t kv_dim) {
        std::size_t count = num_new_tokens * kv_dim;
        auto& kc = k_cache[layer_idx];
        kc.insert(kc.end(), new_k, new_k + count);
        auto& vc = v_cache[layer_idx];
        vc.insert(vc.end(), new_v, new_v + count);
    }
};

In the forward pass, new token K, V are appended to cache, and attention uses the complete cache (not just the current token):

// Append new K/V to cache
kv_cache.append(layer_idx, k_new.data(), v_new.data(), seq_len, kv_dim);

// Attention uses all K/V from cache
causal_attention(attn_out, q.data(), s,
                 kv_cache.k_data(layer_idx),   // All historical K
                 kv_cache.v_data(layer_idx),   // All historical V
                 pos + s + 1,                   // Attend length
                 num_heads, num_kv_heads, head_dim, scale, backend);

The generation loop becomes two phases:

// Prefill: process entire prompt at once, fill KV Cache
auto logits = forward(model, token_ids, 0, kv_cache);

// Decode: only feed new token each time
for (int step = 0; step < max_tokens; step++) {
    int next_token = sampler.sample(logits);
    std::size_t pos = token_ids.size() - 1;
    logits = forward(model, {next_token}, pos, kv_cache);  // Only 1 token
}

Benchmark results (Google Benchmark, synthetic tiny model):

Method Context=64 Comparison
DecodeNoCache (full recompute each step) ~20 ms 1×
DecodeWithCache (only compute new token) ~362 μs 56×

56x speedup. And the gap grows with context length — without cache, each step recomputes all tokens' K/V projections at O(n) cost; with cache, only 1 token is computed at O(1) cost (attention itself is still O(n), but the projection savings are enormous).

GQA plays a role here too: each layer only needs to cache 2 KV heads (128 dimensions) instead of 14 (896 dimensions). For a 24-layer model, KV cache memory usage is reduced by 7x.

Step 5: OpenBLAS — 8.6x Speedup for Linear Projections

Profiling revealed the vast majority of inference time is spent in linear() — unsurprisingly, a Transformer layer has 7 linear projections (Q, K, V, O, gate, up, down), each being matrix multiplication. For Qwen2.5-0.5B, the largest projection is [896, 4864], i.e., 896×4864 ≈ 4.36 million multiply-add operations. Across 24 layers, that's over 100 million.

The naive backend's triple-loop matmul is extremely inefficient — same as the baseline in the previous matmul optimization post, with no SIMD, no tiling, no cache optimization.

The most direct solution: swap the linear operation for OpenBLAS's cblas_sgemm:

void BackendBLAS::linear(float* out, const float* inp, const float* weight,
                         const float* bias, std::size_t M, std::size_t N,
                         std::size_t K) {
    // out[M, N] = inp[M, K] @ weight^T[K, N]
    // weight is [N, K] row-major, so we use CblasTrans on B.
    cblas_sgemm(CblasRowMajor, CblasNoTrans, CblasTrans,
                M, N, K, 1.0f, inp, K, weight, K, 0.0f, out, N);

    if (bias) {
        for (std::size_t i = 0; i < M; i++)
            for (std::size_t j = 0; j < N; j++)
                out[i * N + j] += bias[j];
    }
}

Note the weight storage format: PyTorch's convention is weight shape [out_features, in_features], i.e., [N, K] row-major. So inp @ weight^T requires transposing the weight, corresponding to the CblasTrans parameter.

Behind this single cblas_sgemm call is years of OpenBLAS optimization — SIMD vectorization, cache-friendly tile layout, register-level micro-kernels — exactly the optimizations that took 7 steps to achieve in the previous post.

Benchmark results (aarch64, -c opt):

Dimension CPU (naive) BLAS (OpenBLAS) Speedup
64 2,227 ns 884 ns 2.5×
128 12,148 ns 3,804 ns 3.2×
256 58,317 ns 16,672 ns 3.5×
576 366,660 ns 42,761 ns 8.6×

Larger dimensions yield higher speedup ratios. For small matrices (dim=64), OpenBLAS's thread scheduling and packing overhead is relatively large, limiting gains. But at actual model dimensions (hidden_size=896, intermediate_size=4864), the speedup is very significant.

Switching backends just requires a startup parameter: --backend blas. The Backend abstraction makes this switch zero-invasive — forward pass code needs no changes at all.

Step 6: Sampler — From Logits to Text

The forward pass outputs a 151,936-dimensional logits vector — each value represents a token's "likelihood score." How do we select the next token?

Simplest method: Greedy

Take the token corresponding to the maximum logit. Deterministic, but prone to repetition.

Temperature Scaling

Divide logits by the temperature parameter, then apply softmax to get a probability distribution. Temperature < 1 makes the distribution sharper (more deterministic), temperature > 1 makes it flatter (more random).

Top-p (Nucleus Sampling)

Sort tokens by probability from high to low, accumulate probabilities until exceeding the top_p threshold, and only sample from these tokens. This preserves randomness (unlike greedy's rigidity) while excluding low-probability "noise" tokens (unlike pure random's unpredictability).

int Sampler::sample(const std::vector<float>& logits) {
    if (temperature_ == 0.0f) {
        return argmax(logits);  // greedy
    }

    // 1. Temperature scaling + softmax
    for (std::size_t i = 0; i < n; i++)
        scaled[i] = logits[i] / temperature_;
    softmax(scaled);

    // 2. Top-p: sort by probability, keep cumsum <= top_p
    std::sort(indices.begin(), indices.end(),
              [&](int a, int b) { return scaled[a] > scaled[b]; });

    float cumsum = 0.0f;
    for (std::size_t i = 0; i < n; i++) {
        cumsum += scaled[indices[i]];
        if (cumsum >= top_p_) { cutoff = i + 1; break; }
    }

    // 3. Sample from the truncated distribution
    std::uniform_real_distribution<float> dist(0.0f, kept_sum);
    float r = dist(rng_);
    // ... weighted random selection
}

Default parameters are temperature=0.7, top_p=0.9 — a balance point between diversity and coherence.

Putting It All Together

The final inference flow:

// 1. Load config, weights, tokenizer
auto config = load_config("config.json");
auto weights = SafeTensorsLoader("model.safetensors").load_all(backend);
auto model = create_model(config, backend);
load_weights(model, weights);

// 2. Tokenize prompt
auto token_ids = tokenizer.encode("The capital of France is");  // [5 tokens]

// 3. Prefill: process entire prompt
KVCache kv_cache(config.num_hidden_layers);
auto logits = forward(model, token_ids, 0, kv_cache);

// 4. Decode: generate token by token
for (int step = 0; step < max_tokens; step++) {
    int next_token = sampler.sample(logits);
    if (next_token == config.eos_token_id) break;

    std::cout << tokenizer.decode({next_token}) << std::flush;

    std::size_t pos = token_ids.size() - 1;
    logits = forward(model, {next_token}, pos, kv_cache);
    token_ids.push_back(next_token);
}

Summary and Reflections

Building this inference engine from scratch, a few profound takeaways:

1. LLM inference = massive matrix multiplication. One forward pass has 24 layers × 7 linear projections = 168 matmuls. This is why GPUs are so important for LLM inference — their core strength is massively parallel matmul. And why a single line of OpenBLAS brought an 8.6x speedup.

2. KV Cache is the #1 inference optimization priority. 56x speedup, and simple to implement — essentially an append-only array. But it changes the generation's complexity class: from O(n²) to O(n). LLM inference without KV Cache is practically unusable.

3. Convention differences are harder to find than algorithm bugs. RoPE split-half vs interleaved, weight [N,K] vs [K,N], BPE space prefix — papers gloss over these, but each one can cost you half a day debugging. Comparing intermediate results with a reference implementation (HuggingFace) is the only reliable debugging method.

4. Abstraction layer design matters. The Backend interface let me first ensure correctness with naive loops, then swap in OpenBLAS for performance, without changing a single line of forward pass code. This simple strategy avoided simultaneously debugging correctness and performance.

5. The biggest reward is understanding. After using so many LLM frameworks, only by hand-writing one did I truly understand: why KV Cache is critical for long contexts, why GQA dramatically reduces memory, why prefill and decode have completely different performance characteristics, and why tied embedding saves memory without sacrificing quality.

The entire project is about 1500 lines of C++ (excluding tests and benchmarks), running a real 0.5B model.

Update: The follow-up CUDA optimization is complete, see Part 2: From CPU to GPU — The CUDA Optimization Journey. From 33 tok/s to 504 tok/s.

The current engine works, but is far from "production-ready." Planned next steps by priority:

  1. Instruct model support — Currently only supports base model continuation. Instruct models need chat template formatting (<|im_start|>user\n...<|im_end|>) and handling of multiple stop tokens. This is the prerequisite for making the model truly "conversational."
  2. Multi-threaded parallelism — Currently entirely single-threaded. The most direct optimization is parallelizing linear projections (multiple heads' QKV projections can be computed simultaneously) and the independent computation of multiple heads in attention. Expected 3-5x further improvement on CPU.
  3. CUDA GPU backend — The CPU inference bottleneck is matmul throughput. GPU Tensor Cores are naturally suited for large-scale matrix multiplication. Goal is to write a minimal CUDA backend, replacing OpenBLAS with cuBLAS, reducing decode latency from milliseconds to microseconds.
  4. Quantized inference (INT8/INT4) — Currently all weights are float32, and a 0.5B model already takes 1.9GB. INT8 quantization halves memory, INT4 halves it again, while SIMD/GPU integer operation throughput is higher. The key challenge is controlling precision loss after quantization.
  5. Continuous Batching — Currently can only process a single sequence. Production inference engines (vLLM, TensorRT-LLM) can simultaneously process multiple requests with dynamic KV cache memory scheduling. This involves more complex memory management strategies like PagedAttention.
