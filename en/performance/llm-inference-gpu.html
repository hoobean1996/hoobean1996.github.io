From CPU to GPU: CUDA Optimization for an LLM Inference Engine

2026-03-18
Series: LLM Inference Engine
Part 1: From Zero to Generation — Building an Inference Engine in Pure C++
Part 2: From CPU to GPU — The CUDA Optimization Journey (this post)

Introduction

In the previous post, we built an LLM inference engine from scratch in pure C++, capable of running Qwen2.5-0.5B. On CPU (OpenBLAS + INT8 quantization) it achieved 33 tok/s.

This post documents the subsequent optimization journey: moving the engine onto GPU, step by step from 33 tok/s to 504 tok/s. Every optimization step is driven by clear profiling data, no guessing.

Optimization path overview:

Stage Decode Throughput Improvement
CPU BLAS + INT8 33 tok/s baseline
CUDA + cuBLAS + INT8 110 tok/s 3.3×
+ FP16 inference 153 tok/s 4.6×
+ Flash Attention 161 tok/s 4.9×
+ GPU-resident sampling 161 tok/s -
+ CUDA caching allocator 504 tok/s 15.3×

The last step (caching allocator) brought the largest improvement, but had nsys profiling not revealed the true bottleneck, I never would have thought to optimize memory allocation. This is the key message of this post: profile first, optimize second.

Step 1: CUDA Backend — Moving Computation to GPU

The Backend abstraction from the previous post paid off here. Create a new BackendCUDA subclass implementing the same interface, backed by CUDA kernels:

class BackendCUDA : public BackendCPU {
public:
    void linear(float* out, const float* inp, const float* weight,
                const float* bias, std::size_t M, std::size_t N,
                std::size_t K) override;
    void rms_norm(...) override;
    void rope(...) override;
    void causal_attention(...) override;
    // ... all operators have GPU implementations
};

Core design:

  • Weight caching: Weights are uploaded to GPU on first use (cudaMemcpy H2D), then device pointers are reused thereafter. An unordered_map<host_ptr, device_ptr> handles this.
  • cuBLAS GEMM: Linear projections use cublasGemmEx (FP16 input × FP16 weight → FP32 output), leveraging Tensor Core acceleration.
  • GPU-resident KV Cache: KV cache is allocated and grows directly on GPU, attention reads/writes don't cross PCIe. Each decode step saves 96 memcpys (2 tensors × 2 KV × 24 layers).
  • INT8 → FP16 dequant: INT8 weights are uploaded to GPU, then a kernel dequantizes to FP16 for FP16 GEMM.

Result: from CPU BLAS's 33 tok/s to CUDA's 110 tok/s, 3.3x.

Step 2: FP16 Full-Pipeline Inference

The first CUDA backend's data flow was: FP32 intermediate results → convert to FP16 → cuBLAS GEMM → FP32 output. Lots of FP32↔FP16 conversions in between.

Optimization idea: make all intermediate Tensors FP16. After embedding output converts to FP16, the entire pipeline (RMSNorm, RoPE, Attention, MLP) runs on FP16, only converting back to FP32 for the final logits output.

Wrote FP16 versions of every kernel. Taking RMSNorm as an example, internally using FP32 for accumulation to avoid precision loss:

__global__ void rms_norm_fp16_kernel(half* out, const half* x,
                                      const half* weight, size_t n,
                                      float eps) {
    // Accumulate in FP32 to avoid precision loss
    float local_sum = 0.0f;
    for (size_t i = tid; i < n; i += stride) {
        float v = __half2float(x[i]);
        local_sum += v * v;
    }
    // ... reduction ...
    float rms = sqrtf(sdata[0] / n + eps);

    for (size_t i = tid; i < n; i += stride) {
        out[i] = __float2half(
            __half2float(x[i]) / rms * __half2float(weight[i]));
    }
}

There was a gotcha here: the Sampler also needs to know the logits data type. We convert logits from FP16 to FP32 before passing to the sampler, but the sampler had an fp16 flag telling the GPU sampling kernel what precision to read. Since logits are already FP32, this flag should always be 0 — but the code erroneously set it based on backend.is_fp16().

Result: FP16 mode produced all garbage. INT8 FP32 worked fine, FP16 produced garbage — because the sampling kernel read FP32 data as FP16, every pair of bytes was misinterpreted as a half float.

One-line fix:

// Before (bug):
int fp16 = backend_.is_fp16() ? 1 : 0;
return cuda_sample(d_logits_, logits.size(), ..., fp16);

// After (fix):
// Logits are always FP32 (converted in transformer_forward)
return cuda_sample(d_logits_, logits.size(), ..., /*fp16=*/0);

Lesson: data types must be aligned at every stage in the pipeline. If one stage converts to FP32 but downstream still reads as FP16, it's a disaster.

After getting the full FP16 pipeline working: 110 tok/s → 153 tok/s, +39%. Halved bandwidth + improved Tensor Core utilization.

Step 3: Flash Attention

The old attention kernel stored the full scores[attend_len] array in shared memory. This had two problems:

  1. Shared memory grows linearly with sequence length, ~12K tokens would OOM (48KB limit)
  2. Must compute all QKT scores before softmax, then compute V weighting — three serial steps

Flash Attention's core insight: softmax can be computed incrementally. Process only a small tile (128 KV positions) at a time, using two scalars running_max and running_sum to correct historical accumulated values:

// Main loop: process 128 KV positions at a time
for (tile_start = 0; tile_start < attend_len; tile_start += 128) {

    // 1. Compute only these 128 scores → shared memory (512 bytes)
    for (t in tile)
        scores[t] = dot(q, k[t]) * scale;

    // 2. Find this tile's max
    tile_max = reduce_max(scores);

    // 3. Correct history (the essence of Flash Attention)
    new_max = max(running_max, tile_max);
    correction = exp(running_max - new_max);
    running_sum *= correction;
    out_acc *= correction;       // Correct all previous tiles' accumulated values

    // 4. Accumulate current tile
    for (t in tile) {
        e = exp(scores[t] - new_max);
        running_sum += e;
        out_acc += e * v[t];     // Weighted V accumulation
    }
    running_max = new_max;
}

// Final normalization
out = out_acc / running_sum;

Why does correction correctly fix things? Mathematically:

exp(score - old_max) × exp(old_max - new_max) = exp(score - new_max)

So multiplying all historical accumulated values by correction is equivalent to recomputing with new_max.

Shared memory went from (attend_len + 256) × 4 to a constant 1536 bytes, regardless of sequence length.

attend_len Old kernel smem/head Flash Attention smem/head
541.2 KB1.5 KB
4482.8 KB1.5 KB
409617.0 KB1.5 KB
1228849.0 KB (OOM)1.5 KB

Short sequence performance is roughly the same (attention isn't the bottleneck), but long context capability is unlocked. Decode throughput stays stable at ~161 tok/s across 50–750 token attend_len range.

Step 4: GPU-Resident Sampling

After Flash Attention, I used nsys to inspect the data flow and discovered an unnecessary CPU roundtrip:

GPU: FP16 logits → fp16_to_fp32 → FP32 logits
                                          ↓ cudaMemcpy D2H (593KB)
CPU: std::vector<float> logits
                                          ↓ cudaMemcpy H2D (593KB)
GPU: sample_kernel → token ID

Qwen2.5-0.5B's vocab_size=151936, each decode step transfers 1.2MB back and forth over PCIe.

Fix: have forward() return a Tensor (GPU-resident) instead of std::vector<float>. Sampler samples directly from the GPU Tensor:

// Before: forward() downloads to CPU, sampler re-uploads to GPU
std::vector<float> forward(model, token_ids, pos, kv_cache);
sampler->sample(logits, step);  // internally: H2D upload + sample kernel

// After: logits stay on GPU, sampler uses device pointer directly
Tensor forward(model, token_ids, pos, kv_cache);
sampler->sample(logits, step);  // directly: sample kernel on device ptr

This change didn't significantly impact decode throughput (PCIe transfer time is short relative to kernel computation time), but the code is cleaner — and it paved the way for the next optimization.

Step 5: nsys Profile → Finding the Real Bottleneck → CUDA Caching Allocator

This is the most important step in the entire post. Not optimizing an algorithm, but using a profiling tool to discover a completely unexpected bottleneck.

Running nsys profile:

nsys profile --trace=cuda,osrt --stats=true \
  bazel-bin/bin/qwen/qwen \
    --model models/Qwen2.5-0.5B \
    --prompt "Once upon a time" \
    --max_tokens 50 \
    --backend cuda --quantize int8 --fp16

CUDA API time distribution:

 Time (%)  Total Time (ns)  Num Calls   Name
 --------  ---------------  ---------   ----
     36.0%       216913486       1341   cudaFree           ← 36%!!
     33.6%       202353991       2963   cudaMemcpy
     17.3%       104017852      17773   cudaLaunchKernel   ← actual computation
      8.2%        49292731       1341   cudaMalloc         ← 8%

Seeing this data I was shocked: cudaMalloc + cudaFree accounted for 44% of CUDA API time (266ms), more than 2.5x the actual GPU kernel computation (104ms)!

Why? Because cudaMalloc and cudaFree are both synchronous operations — they block the CPU thread, waiting for all pending kernels on the GPU to complete. This means every allocation/deallocation forces a GPU pipeline sync.

1341 mallocs + 1341 frees. Where from? Each transformer_forward() creates ~12 temporary Tensors:

Tensor x({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor norm_out({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor q({seq_len * num_heads * head_dim}, DType::Float16, Device::CUDA);
Tensor k_new({seq_len * kv_dim}, DType::Float16, Device::CUDA);
Tensor v_new({seq_len * kv_dim}, DType::Float16, Device::CUDA);
Tensor attn_out({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor gate_up({seq_len * 2 * intermediate}, DType::Float16, Device::CUDA);
Tensor gate({seq_len * intermediate}, DType::Float16, Device::CUDA);
Tensor final_out({hidden}, DType::Float16, Device::CUDA);
Tensor logits({vocab_size}, DType::Float16, Device::CUDA);
// ... plus logits_fp32, x_fp32 for conversions

When the function returns, these Tensors' destructors call cudaFree. 54 forwards × ~25 allocs/frees ≈ 1350 times. Perfectly matching nsys's reported 1341.

Key insight: during the decode phase, every forward call creates Tensors with exactly the same shapes (seq_len=1). Mallocing the same size buffer every time, freeing it, then mallocing the same size again — pure waste.

Fix: CUDA caching allocator

The core idea is dead simple: freed GPU buffers aren't actually released, but placed in a free list (indexed by size). Next time a buffer of the same size is requested, it's taken from the free list, O(1) lookup, zero GPU synchronization:

// Free list: byte_size → [ptr, ptr, ...] (exact size matching)
static std::multimap<std::size_t, void*> cuda_free_list_;

// Live allocations: ptr → byte_size (for returning to correct bucket)
static std::unordered_map<void*, std::size_t> cuda_alloc_sizes_;

static void* cuda_pool_alloc(std::size_t bytes) {
    auto it = cuda_free_list_.find(bytes);
    if (it != cuda_free_list_.end()) {
        void* ptr = it->second;
        cuda_free_list_.erase(it);
        cuda_alloc_sizes_.emplace(ptr, bytes);
        return ptr;    // ← Hit! Zero GPU sync
    }
    void* ptr = cuda_malloc(bytes);  // miss: actual allocation
    cuda_alloc_sizes_.emplace(ptr, bytes);
    return ptr;
}

static void cuda_pool_free(void* ptr) {
    auto it = cuda_alloc_sizes_.find(ptr);
    if (it != cuda_alloc_sizes_.end()) {
        cuda_free_list_.emplace(it->second, ptr);  // Return to free list
        cuda_alloc_sizes_.erase(it);
    } else {
        cuda_free(ptr);  // Not pool-managed, actually release
    }
}

30 lines of code. Inserted into device_allocate() and device_deallocate(), completely transparent to upper layers.

On the first forward, all Tensors miss (cold start), going through cudaMalloc. When the function returns, buffers enter the free list. From the second forward onward, all same-size Tensors hit, directly reused — zero cudaMalloc, zero cudaFree, zero GPU synchronization.

Running nsys again after the fix:

 Time (%)  Total Time (ns)  Num Calls   Name
 --------  ---------------  ---------   ----
      7.5%        45833654        751   cudaMalloc    ← -44% calls
      7.3%        44586600        729   cudaFree      ← -79% time!
     16.6%       101269184      17773   cudaLaunchKernel  ← unchanged

cudaFree dropped from 217ms to 45ms (-79%). 612 allocations hit the pool, completely avoiding GPU synchronization.

Performance results:

Metric Before (no pool) After (caching) Improvement
Decode throughput 153 tok/s 504 tok/s +3.3×
Decode avg latency 6.5 ms 2.0 ms -69%
Decode P50 latency 6.0 ms 1.6 ms -73%
cudaFree time 217 ms 45 ms -79%

3.3x speedup from 30 lines of memory pool code. This is the value of profiling — not guessing the bottleneck is in operators, attention, or GEMM, but letting the data tell you: the biggest bottleneck is actually cudaFree.

Summary

From CPU's 33 tok/s to GPU's 504 tok/s, 15x speedup. Key lessons from the entire journey:

1. Profile first, optimize second. Without running nsys, I might have spent days optimizing CUDA kernel occupancy or shared memory layout — but the actual bottleneck was cudaFree's GPU synchronization. Data doesn't lie.

2. The biggest improvements often come from the simplest fixes. Flash Attention implementation took hours, yielding +5%. The caching allocator took 30 lines, yielding +230%. Complexity and payoff are not proportional.

3. End-to-end thinking. The FP16 sampler bug was an end-to-end type alignment issue. GPU-resident sampling was a data transfer issue. These aren't kernel-level optimizations but system-level ones. You need to see the entire pipeline, not just stare at individual operators.

4. The value of abstraction layers extends beyond code reuse. The Backend abstraction made CPU/BLAS/CUDA switching zero-invasive. The Tensor's device_allocate/device_deallocate abstraction let the caching allocator be transparently inserted without changing any upper-layer code. Good abstractions make optimization local.

The entire project is now 6200 lines of C++/CUDA, running Qwen2.5-0.5B on an RTX 4060 at 504 tok/s decode. There's still much to do — INT4 quantization, prefill Q tiling, multi-request batching — but the core profiling-driven optimization loop is established.

"Premature optimization is the root of all evil. But so is premature pessimism — profile, then decide."
