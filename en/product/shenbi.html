Shenbi: Building a Programming Language for Kids in the Browser

2026-03-08
The value of this article: Scratch is powerful but too complex for young children. Apple Swift Playgrounds is elegant but disconnected from real programming languages. This article shares a different approach вҖ” building a bilingual (Chinese/English) Python-like programming language from scratch in the browser, with a complete Lexer вҶ’ Parser вҶ’ Compiler вҶ’ VM pipeline, bidirectional block/code synchronization, and a World abstraction that lets the same language drive different types of games. If you're interested in "how to build a language" or "how to design a kids' programming product," this has the full engineering details.

Why Build Our Own

Existing kids' programming tools fall roughly into two categories:

  • Scratch-like: Pure block dragging, extremely powerful (event system, concurrency, message broadcasting), but too many concepts for 5-8 year olds. A "get the character to the goal" task in Scratch requires understanding coordinate systems, event loops, broadcast messages вҖ” these aren't the essence of programming, they're Scratch's own complexity
  • Apple Swift Playgrounds-like: Beautiful 3D interface, guided teaching, great experience. But it teaches Swift syntax, which has nothing to do with Python/JavaScript that kids will eventually use. After finishing Playgrounds, switching to another language means starting from scratch

What I wanted was:

  1. Simple enough: 5-year-olds can understand blocks, 8-year-olds can read code
  2. Aligned with real languages: Syntax highly consistent with Python, so learning transfers directly to real Python
  3. Bilingual Chinese/English: if and its Chinese equivalent are the same thing вҖ” kids can start in Chinese and gradually switch to English
  4. Bidirectional block-code switching: Not a one-way "blocks generate code" relationship, but blocks and code that can be converted at any time

Couldn't find a tool meeting these criteria, so I built one. Called Shenbi (Magic Brush) вҖ” writing code is like drawing, and what you draw comes alive.

Part 1: Building a Language

Lexer: Bilingual Implementation Here

The core bilingual design is simple: at the Lexer layer, Chinese keywords and English keywords map to the same Token type. The Parser and all subsequent stages have no idea whether the user wrote in Chinese or English.

const KEYWORDS: Record<string, TokenType> = {
  // Chinese and English map to the same Token
  'еҰӮжһң': 'IF',      'if': 'IF',
  'еҗҰеҲҷеҰӮжһң': 'ELIF', 'elif': 'ELIF',
  'еҗҰеҲҷ': 'ELSE',    'else': 'ELSE',
  'йҮҚеӨҚ': 'REPEAT',  'repeat': 'REPEAT',
  'ж¬Ў': 'TIMES',     'times': 'TIMES',
  'еҪ“': 'WHILE',     'while': 'WHILE',
  'еҜ№дәҺ': 'FOR',     'for': 'FOR',
  'еңЁ': 'IN',        'in': 'IN',
  'е®ҡд№ү': 'DEF',     'def': 'DEF',
  'иҝ”еӣһ': 'RETURN',  'return': 'RETURN',
  'зңҹ': 'TRUE',      'True': 'TRUE',
  'еҒҮ': 'FALSE',     'False': 'FALSE',
  'е’Ң': 'AND',       'and': 'AND',
  'жҲ–': 'OR',        'or': 'OR',
  'дёҚ': 'NOT',       'not': 'NOT',
  'зұ»': 'CLASS',     'class': 'CLASS',
};

Identifiers also support Chinese characters (Unicode CJK range), so variable names can be in Chinese:

// These two code snippets are completely equivalent after lexing
йҮҚеӨҚ 3 ж¬Ў:
    еүҚиҝӣ()
    е·ҰиҪ¬()

repeat 3 times:
    forward()
    turnLeft()

The Lexer also needs to handle Python-style indentation вҖ” using INDENT/DEDENT tokens to represent code block beginnings and endings, rather than curly braces. This is the most complex part of the entire Lexer.

Parser вҶ’ Compiler вҶ’ IR

The Parser is a hand-written recursive descent parser that generates an AST. The Compiler does multi-pass compilation: first collecting function and class definitions, then compiling the main body, and finally compiling function bodies and class methods. Output is a set of linear stack-based instructions:

// IR instruction set (~35 opcodes)
type OpCode =
  // Stack operations
  | 'PUSH' | 'POP' | 'DUP'
  // Variables
  | 'LOAD' | 'STORE'
  // Arithmetic
  | 'ADD' | 'SUB' | 'MUL' | 'DIV' | 'MOD' | 'NEG'
  // Comparison
  | 'EQ' | 'NEQ' | 'LT' | 'GT' | 'LTE' | 'GTE'
  // Control flow
  | 'JUMP' | 'JUMP_IF' | 'JUMP_IF_NOT'
  // Functions
  | 'CALL' | 'CALL_USER' | 'RETURN'
  // Data structures
  | 'ARRAY_CREATE' | 'ARRAY_GET' | 'ARRAY_SET'
  | 'OBJECT_CREATE' | 'OBJECT_GET' | 'OBJECT_SET'
  // Classes
  | 'CLASS_DEF' | 'NEW' | 'METHOD_CALL'
  | 'HALT' | 'NOP';

A simple example's compilation result:

// MiniPython source code
repeat 3 times:
    forward()

// Compiled IR
PUSH 0          // Loop counter = 0
STORE $i        // Save to variable
PUSH 3          // Loop upper bound
STORE $limit
LOAD $i         // вҶҗ Loop start
LOAD $limit
LT              // i < limit ?
JUMP_IF_NOT end // Jump out if false
CALL forward:0  // Call forward(), 0 arguments
POP             // Discard return value
LOAD $i
PUSH 1
ADD
STORE $i        // i++
JUMP loop       // Jump back to loop start
HALT            // вҶҗ end

VM: A Pausable Virtual Machine

The VM is a stack-based virtual machine: program counter (PC), operand stack, global variable table, call stack. But it doesn't run all instructions at once вҖ” it executes only one step at a time, then returns control to the caller.

This is the core requirement for kids' programming: when a child writes forward(), you need to see the character walk one step with animation, then execute the next instruction. If the VM ran everything at once, users would only see the final state.

// VM.step() returns the execution result instead of running to completion
step(): StepResult {
  const inst = this.program[this.pc];

  switch (inst.opcode) {
    case 'CALL': {
      const { name, argCount } = parseCallArg(inst.arg);
      const args = this.popN(argCount);

      // Game commands: _forward, _turnLeft, ...
      if (this.commandHandlers.has(name)) {
        this.commandHandlers.get(name)!(args);
        this.stack.push(null);
        this.pc++;
        return { action: name, actionArgs: args };
        //       вҶ‘ Returns to caller: "just executed forward"
        //         Caller can play animation, then call step() again
      }

      // Sensors: frontBlocked, atGoal, ...
      if (this.sensorHandlers.has(name)) {
        const result = this.sensorHandlers.get(name)!(args);
        this.stack.push(result);
        this.pc++;
        return { sensorQuery: name };
      }

      // Built-in functions: len, print, random, ...
      // ...
    }
    // ... other opcodes
  }
}

This design also naturally supports step-by-step debugging and breakpoints: the caller controls when to call step(), and can inspect variables and highlight the current line between each step.

Furthermore, the VM supports reverse debugging вҖ” saving a complete VM state snapshot before each step, allowing users to "rewind" program execution. This is particularly helpful for kids understanding code flow.

Part 2: Bidirectional Block-Code Synchronization

Most block programming platforms are one-directional: blocks generate code, but code can't be converted back to blocks. This means once a child switches to code mode and makes changes, they can never go back to block mode.

Shenbi implements bidirectional synchronization:

Blocks вҶ’ Code

Each block type (command, repeat, while, if, function definition...) has corresponding code generation rules, while maintaining a LineToBlockMap вҖ” recording which block ID each generated line of code corresponds to, for highlighting the corresponding block during execution:

function generateBlockCode(block: Block, indent: number,
                           lineMap: LineToBlockMap) {
  const spaces = '    '.repeat(indent);

  switch (block.type) {
    case 'command':
      lines.push(`${spaces}${commandName}(${args})`);
      lineMap.set(currentLine, block.id);
      break;

    case 'repeat':
      lines.push(`${spaces}repeat ${block.repeatCount} times:`);
      lineMap.set(currentLine, block.id);
      // Recursively process child blocks
      for (const child of block.children) {
        generateBlockCode(child, indent + 1, lineMap);
      }
      break;

    case 'if':
      lines.push(`${spaces}if ${conditionToCode(block)}:`);
      // ...
  }
}

Code вҶ’ Blocks

This is the more interesting direction. Code first goes through the actual Lexer/Parser to compile into an AST, then AST nodes are reverse-converted into blocks:

function parseCodeToBlocks(code: string): Block[] | null {
  try {
    const ast = compile(code);  // Use the real compiler to parse
    return ast.body
      .map(stmt => statementToBlock(stmt))
      .filter(b => b !== null);
  } catch {
    return null;  // Syntax error, can't convert
  }
}

This reuses MiniPython's compiler вҖ” not a separate simplified parser, but actually compiling the code, getting the AST, then converting to blocks. This ensures the codeвҶ’blocks and blocksвҶ’code conversions are perfectly symmetric.

The difficulty is handling Chinese function names, English function names, and legacy naming mappings. For example, forward, its Chinese equivalent, and move all need to map to the same command type block:

const COMMAND_MAP: Record<string, CommandId> = {
  move: 'move', turn: 'turn',
  з§»еҠЁ: 'move', иҪ¬еҗ‘: 'turn',
  forward: 'move', backward: 'move',  // legacy
  еүҚиҝӣ: 'move', еҗҺйҖҖ: 'move',
  е·ҰиҪ¬: 'turn', еҸіиҪ¬: 'turn',
  setColor: 'setColor', и®ҫзҪ®йўңиүІ: 'setColor',
};

The result: kids assemble logic in block mode, switch to code mode to see corresponding Python code; edit a few lines in code mode, switch back to block mode to see corresponding block changes. The two modes stay synchronized at all times.

Part 3: The World Model

This is the part I think is best designed. Shenbi doesn't just have maze games вҖ” it also has Turtle Graphics, with potential for more game types. The question is: how do you let the same language and VM drive completely different types of games?

The answer is the World abstraction:

Architecture pattern:
- World: Data model (state + commands + sensors)
- VM: Code executor (maps code to World methods)
- Canvas: Renderer (displays World state)

Each game type implements its own World, its own VM bindings, and its own Canvas, but shares the same MiniPython language and VM core.

MazeWorld

class MazeWorld {
  // State
  grid: Cell[][];
  x: number; y: number;
  direction: 'up' | 'down' | 'left' | 'right';
  collected: number;
  steps: number;

  // Commands
  moveForward(): boolean;
  turnLeft(): void;
  turnRight(): void;
  collect(): boolean;

  // Sensors
  isFrontBlocked(): boolean;
  isAtGoal(): boolean;
  hasStarHere(): boolean;

  // Serialization (for VM)
  toSharedState(): SharedMazeState;
  syncFromSharedState(state: SharedMazeState): void;
}

TurtleWorld

class TurtleWorld {
  // State
  x: number; y: number;
  angle: number;
  penDown: boolean;
  penColor: string;
  lines: LineSegment[];

  // Commands
  forward(distance: number): void;
  turnLeft(degrees: number): void;
  penUp(): void;
  setColor(color: string): void;

  toSharedState(): SharedTurtleState;
  syncFromSharedState(state: SharedTurtleState): void;

Key Design: SharedState Bridge

World and VM connect through SharedState. When the VM starts, it serializes World state into a plain JavaScript object and injects it into MiniPython's global scope as world:

// MazeVM.loadWithSource()
loadWithSource(userCode: string): void {
  const fullCode = MAZE_STDLIB + '\n' + userCode;
  const program = compileToIR(compile(fullCode));
  this.sharedState = this.world.toSharedState();
  this.vm.load(program);
  this.vm.setGlobal('world', this.sharedState);
  //                  вҶ‘ MiniPython code can directly read/write world.x, world.y, ...
}

Then the MiniPython standard library code directly operates on this world object:

# MiniPython standard library (maze)
def forward():
    offset = DIRECTION_OFFSETS[world.direction]
    newX = world.x + offset[0]
    newY = world.y + offset[1]
    if getCell(newX, newY) == 'wall':
        return False
    _forward()    # вҶҗ The only native call, notifies Canvas to play animation
    return True

def frontBlocked():
    offset = DIRECTION_OFFSETS[world.direction]
    newX = world.x + offset[0]
    newY = world.y + offset[1]
    return getCell(newX, newY) == 'wall'

Note: forward()'s collision detection and coordinate calculation are all done by MiniPython itself. Only _forward() is a native command вҖ” it tells Canvas "the character moved one step," and Canvas plays the animation. The standard library is written in its own language.

The turtle graphics standard library follows the same pattern:

# MiniPython standard library (turtle)
def forward(distance):
    radians = world.angle * 3.14159265359 / 180
    dx = _cos(radians) * distance * world.stepSize
    dy = _sin(radians) * distance * world.stepSize * -1
    oldX = world.x
    oldY = world.y
    world.x = world.x + dx
    world.y = world.y + dy
    if world.penDown:
        _drawLine(oldX, oldY, world.x, world.y,
                  world.penColor, world.penWidth)

Only _cos, _sin, and _drawLine are native вҖ” because trigonometric functions and line drawing are things MiniPython can't do. All other coordinate calculations happen in MiniPython.

What's Needed to Add a New Game Type

The World model makes extending new games very clear:

  1. Define a new World class (state + commands + sensors + toSharedState)
  2. Write a corresponding VM binding class, registering a few native commands
  3. Write the standard library in MiniPython вҖ” most game logic goes here
  4. Write a Canvas component to render World state

The language, compiler, VM core, and block editor вҖ” all reused, zero modifications.

Level System: Conditions Are Also Expressions

Each level is a JSON file with the map defined using ASCII art:

{
  "id": "maze-01",
  "name": "First Steps",
  "grid": [
    "########",
    "#>...*G#",    // > = player facing right, * = star, G = goal
    "########"
  ],
  "availableBlocks": ["command"],
  "winCondition": "collectedCount() >= 1 and atGoal()",
  "failCondition": "stepCount() > 10",
  "hints": [
    "Use the Forward block to move the robot",
    "Keep going until you reach the flag!"
  ]
}

winCondition and failCondition aren't hardcoded enums вҖ” they're MiniPython expressions, evaluated by the VM. This means level designers (teachers) can combine arbitrary conditions without changing code:

// Various win conditions
"atGoal()"                                // Reach the goal
"collectedCount() >= 3 and atGoal()"      // Collect 3 stars + reach goal
"stepCount() <= 8 and atGoal()"           // Reach goal within 8 steps
"collectedCount() == getRemainingStars()" // Collect all stars

Summary

Back to the original goals:

  • Simple enough вҶ’ Block mode lets 5-year-olds get started, code mode syntax is consistent with Python
  • Aligned with real languages вҶ’ MiniPython's syntax is a subset of Python, learning transfers directly to real Python
  • Bilingual Chinese/English вҶ’ Implemented via keyword mapping at the Lexer layer, with the rest of the pipeline completely unaware
  • Bidirectional block-code switching вҶ’ Reverse conversion from codeвҶ’blocks achieved by reusing the compiler's AST

The three most satisfying engineering designs:

  1. Bilingual handling only at the Lexer layer: One keyword mapping table solves the problem. Parser/Compiler/VM don't need to care about language. This decision prevented the entire system's complexity from doubling due to "bilingual support"
  2. World abstraction: Game worlds are pure data models, the VM bridges through SharedState, and standard libraries are written in the language itself. Adding new games doesn't require changing the language or VM
  3. VM's step() design: Executing only one step at a time instead of running to completion makes animation playback, step debugging, breakpoints, and reverse debugging all feel natural
