Building a Webpack Scaffold for Google Workspace Add-ons

2026-03-08
The value of this article: The development experience for Google Workspace add-ons is quite primitive — no npm, no build tools, and all resources must be inlined in a single HTML file. This article shares a Webpack scaffold solution that lets you develop GSuite add-ons with React + TypeScript, with automatic entry discovery, automatic JS/CSS inlining, and a single build covering all GSuite products. If you need complex UI interactions in GSuite add-ons, this scaffold can save you a lot of repetitive work.

What Are Google Workspace Add-ons

Google Workspace (formerly GSuite) allows developers to build add-ons for Docs, Sheets, Slides, Forms, and other products. Add-on UI comes in two forms: Sidebar (side panel, persistent) and Dialog (popup, temporary).

The add-on architecture has two layers:

┌──────────────────────────────────────────┐
│           Google Docs / Sheets / ...      │
│                                           │
│   ┌─────────────┐    ┌───────────────┐   │
│   │   Sidebar    │    │    Dialog     │   │
│   │  (HTML/JS)   │    │  (HTML/JS)   │   │
│   │  ← iframe →  │    │  ← iframe → │   │
│   └──────┬───────┘    └──────┬───────┘   │
│          │                    │           │
│          │  google.script.run │           │
│          ▼                    ▼           │
│   ┌──────────────────────────────────┐   │
│   │      AppScript (Server-side)     │   │
│   │  - Access Drive / Gmail / Sheets │   │
│   │  - Manipulate document content   │   │
│   │  - Send emails                   │   │
│   └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
  • Frontend: HTML pages running in an iframe sandbox, hosted by AppScript's HtmlService
  • Backend: AppScript code running on Google's servers, with access to Google APIs (Drive, Gmail, Sheets, etc.)
  • Communication: Frontend calls backend functions asynchronously via google.script.run

This model has three hard constraints that directly determine our build approach:

  1. CSP restrictions: The iframe sandbox prohibits loading external scripts and stylesheets. All JS and CSS must be inlined in the HTML file
  2. Single-file delivery: HtmlService.createHtmlOutputFromFile("sidebar") only accepts a single HTML filename. Cannot be split into multiple files
  3. Cross-product deployment: The same add-on must be deployed separately to Docs, Sheets, Slides, and Forms, each being an independent AppScript project

The officially recommended development approach is writing HTML + JavaScript directly in the Apps Script editor. This works for simple add-ons, but once the UI requires state management, multi-step forms, or conditional rendering, the development experience without componentization and build tools becomes painful.

Below is the Webpack scaffold I built to solve these problems.

Scaffold Goals

I wanted this scaffold to achieve:

  1. React + TypeScript development: Build GSuite add-on UI using modern frontend practices
  2. Self-contained HTML output: Build artifacts are single HTML files with all JS/CSS inlined, directly satisfying GSuite's CSP restrictions
  3. Auto-discovery: Adding a new dialog only requires creating a folder in the conventional directory, without changing any build configuration
  4. Multi-product matrix build: One build generates independent deployment directories for each GSuite product
  5. Local development: Debug UI in the browser without deploying to GSuite

Project Structure Convention

project/
├── src/
│   ├── sidebar/           # Main sidebar (must exist)
│   │   ├── App.tsx
│   │   └── index.tsx
│   └── dialogs/           # Dialog directory (auto-discovered)
│       ├── about/
│       │   ├── App.tsx
│       │   └── index.tsx
│       └── settings/
│           ├── App.tsx
│           └── index.tsx
├── appscript/
│   └── shared.gs          # AppScript server-side code
├── clasp/
│   ├── docs/
│   │   ├── .clasp.json    # Docs AppScript project ID
│   │   └── appsscript.json
│   ├── sheets/
│   │   └── .clasp.json    # Sheets AppScript project ID
│   └── ...
├── webpack.config.js
└── package.json

The core convention of this structure is: src/sidebar/ is the fixed entry, and every subdirectory under src/dialogs/ containing an index.tsx automatically becomes a dialog entry.

Step 1: Auto-Discover Entry Points

Webpack's entry is usually configured manually. But we want adding dialogs without changing configuration. So we scan the filesystem at Webpack startup:

const path = require("path");
const fs = require("fs");

function discoverDialogs() {
  const dialogsPath = path.resolve(__dirname, "src/dialogs");
  if (!fs.existsSync(dialogsPath)) return [];

  return fs.readdirSync(dialogsPath, { withFileTypes: true })
    .filter(item => item.isDirectory())
    .filter(dir => {
      return fs.existsSync(
        path.join(dialogsPath, dir.name, "index.tsx")
      );
    })
    .map(dir => dir.name);
}

// Dynamically generate entry points
const dialogs = discoverDialogs();
const entry = { sidebar: "./src/sidebar/index.tsx" };
dialogs.forEach(name => {
  entry[name] = `./src/dialogs/${name}/index.tsx`;
});

This makes entry become { sidebar: "...", about: "...", settings: "..." }. Want to add a new dialog? Create a folder under src/dialogs/, add an index.tsx, and it takes effect automatically on the next build.

Step 2: Inline JS and CSS

Normal Webpack build artifacts are separate .js and .css files, with HTML referencing them via <script src> and <link rel="stylesheet">. This doesn't work in GSuite — the iframe sandbox blocks loading these external resources.

The solution is a combination of two plugins:

const HtmlWebpackPlugin = require("html-webpack-plugin");
const HtmlInlineScriptPlugin =
  require("html-inline-script-webpack-plugin");

// JS inlining: HtmlInlineScriptPlugin puts the Webpack JS bundle
//              directly inside the HTML's <script> tag
plugins: [
  new HtmlWebpackPlugin({
    template: "./public/index.html",
    chunks: ["sidebar"],
    inject: "body",
  }),
  new HtmlInlineScriptPlugin(),
]

// CSS inlining: use style-loader instead of mini-css-extract-plugin
module: {
  rules: [
    {
      test: /\.css$/,
      use: ["style-loader", "css-loader"],
      // style-loader injects CSS into <style> tags at runtime
      // instead of generating separate .css files
    },
  ],
}

After building, each HTML file is completely self-contained — zero external dependencies. HtmlService.createHtmlOutputFromFile("sidebar") works directly.

Step 3: Multi-Product Matrix Generation

The same add-on needs to be deployed to Docs, Sheets, Slides, and Forms. Each product is an independent AppScript project requiring its own deployment directory. The Webpack config dynamically generates all HtmlWebpackPlugin instances:

const GSUITE_PRODUCTS = ["docs", "slides", "sheets", "forms"];
const htmlPlugins = [];

GSUITE_PRODUCTS.forEach(product => {
  // sidebar
  htmlPlugins.push(new HtmlWebpackPlugin({
    template: "./public/index.html",
    filename: `${product}/sidebar.html`,
    chunks: ["sidebar"],
    inject: "body",
  }));

  // Each auto-discovered dialog
  dialogs.forEach(name => {
    htmlPlugins.push(new HtmlWebpackPlugin({
      template: "./public/index.html",
      filename: `${product}/${name}.html`,
      chunks: [name],
      inject: "body",
    }));
  });
});

Build artifact structure:

dist/
├── docs/
│   ├── sidebar.html       # Self-contained HTML
│   ├── about.html
│   ├── settings.html
│   └── appscript/
│       └── shared.gs      # Copied AppScript code
├── sheets/
│   ├── sidebar.html
│   ├── about.html
│   └── settings.html
├── slides/
│   └── ...
└── forms/
    └── ...

4 products x 3 entries = 12 HTML files, all auto-generated. JS code is compiled once and distributed to the corresponding HTML files via the chunks configuration.

Step 4: AppScript File Copying

AppScript .gs files don't need compilation but need to be copied to each product's output directory. Using CopyWebpackPlugin:

const CopyWebpackPlugin = require("copy-webpack-plugin");

GSUITE_PRODUCTS.forEach(product => {
  plugins.push(new CopyWebpackPlugin({
    patterns: [{
      from: "appscript/",
      to: `${product}/appscript/`,
    }],
  }));
});

This way, when running clasp push, each product directory has both HTML and .gs files — a complete AppScript project.

Step 5: Local Development Support

google.script.run only exists in the GSuite environment. When developing locally in the browser, it's undefined. The approach is simple: detect the environment, and use mock data when it's not present.

// Wrap AppScript calls
function callAppScript<T>(
  methodName: string,
  ...args: any[]
): Promise<T> {
  return new Promise((resolve, reject) => {
    if (typeof google !== "undefined" && google.script) {
      const runner = google.script.run
        .withSuccessHandler(resolve)
        .withFailureHandler(reject);
      runner[methodName](...args);
    } else {
      // Local development: return mock data
      resolve(getMockData(methodName, args) as T);
    }
  });
}

// Usage
const sheets = await callAppScript<Sheet[]>("getGoogleSheets");

This has two benefits:

  1. Local development doesn't depend on the GSuite environment: Use webpack-dev-server to start a local server, debugging UI and interaction logic in the browser
  2. Wraps callback-style google.script.run as Promises: React components can use async/await, making code cleaner

Step 6: TypeScript Type Declarations

google.script.run has no official type definitions. Manual declarations are needed so TypeScript knows which AppScript functions are callable:

// types/google.d.ts
interface AppScriptRunner {
  getGoogleSheets: () => void;
  getWorksheets: (spreadsheetId: string) => void;
  getSelectedSheetData: (
    spreadsheetId: string,
    worksheetName: string
  ) => void;
  getDocumentHtml: (docId: string) => void;
  sendMailMerge: (
    spreadsheetId: string,
    worksheetName: string,
    docId: string,
    subject: string,
    replyTo: string,
    testMode: boolean
  ) => void;
}

declare const google: {
  script: {
    run: {
      withSuccessHandler: (cb: (result: any) => void) => {
        withFailureHandler: (
          cb: (error: any) => void
        ) => AppScriptRunner;
      };
    };
  };
};

For each new AppScript function, add a declaration here. While return value types are still any (AppScript is dynamically typed), at least call parameters have type checking.

Step 7: Deployment Pipeline

Use Google's clasp CLI to push build artifacts to AppScript projects. Each GSuite product is an independent AppScript project with its own .clasp.json:

# clasp/docs/.clasp.json
{
  "scriptId": "1a2b3c...",        # Docs AppScript project ID
  "rootDir": "../../dist/docs"    # Points to build artifacts
}

# Deployment flow
npm run build                     # Build HTML for all products
cd clasp/docs && clasp push       # Push to Docs AppScript project
cd clasp/sheets && clasp push     # Push to Sheets AppScript project
# ... push separately for each product

clasp push uploads all .html and .gs files under rootDir to Google Apps Script. You can write a script to chain all product pushes:

# deploy.sh
npm run build
for product in docs sheets slides forms; do
  echo "Deploying to $product..."
  (cd clasp/$product && clasp push)
done

Complete webpack.config.js

Combining all the steps above:

const path = require("path");
const fs = require("fs");
const HtmlWebpackPlugin = require("html-webpack-plugin");
const HtmlInlineScriptPlugin =
  require("html-inline-script-webpack-plugin");
const CopyWebpackPlugin = require("copy-webpack-plugin");
const TerserPlugin = require("terser-webpack-plugin");

// ---- Auto-discovery ----
function discoverDialogs() {
  const dialogsPath = path.resolve(__dirname, "src/dialogs");
  if (!fs.existsSync(dialogsPath)) return [];
  return fs.readdirSync(dialogsPath, { withFileTypes: true })
    .filter(d => d.isDirectory())
    .filter(d =>
      fs.existsSync(path.join(dialogsPath, d.name, "index.tsx"))
    )
    .map(d => d.name);
}

const dialogs = discoverDialogs();
const PRODUCTS = ["docs", "slides", "sheets", "forms"];

// ---- Entry ----
const entry = { sidebar: "./src/sidebar/index.tsx" };
dialogs.forEach(name => {
  entry[name] = `./src/dialogs/${name}/index.tsx`;
});

// ---- HTML + Copy plugins ----
const plugins = [];

PRODUCTS.forEach(product => {
  // sidebar HTML
  plugins.push(new HtmlWebpackPlugin({
    template: "./public/index.html",
    filename: `${product}/sidebar.html`,
    chunks: ["sidebar"],
    inject: "body",
  }));
  // dialog HTMLs
  dialogs.forEach(name => {
    plugins.push(new HtmlWebpackPlugin({
      template: "./public/index.html",
      filename: `${product}/${name}.html`,
      chunks: [name],
      inject: "body",
    }));
  });
  // AppScript file copy
  plugins.push(new CopyWebpackPlugin({
    patterns: [{
      from: "appscript/",
      to: `${product}/appscript/`,
    }],
  }));
});

plugins.push(new HtmlInlineScriptPlugin());

// ---- Export ----
module.exports = {
  mode: "production",
  entry,
  output: {
    path: path.resolve(__dirname, "dist"),
    clean: true,
  },
  resolve: {
    extensions: [".ts", ".tsx", ".js", ".jsx"],
  },
  module: {
    rules: [
      {
        test: /\.tsx?$/,
        use: "ts-loader",
        exclude: /node_modules/,
      },
      {
        test: /\.css$/,
        use: ["style-loader", "css-loader"],
      },
    ],
  },
  plugins,
  optimization: {
    minimizer: [
      new TerserPlugin({
        terserOptions: { compress: { drop_console: true } },
      }),
    ],
  },
};

GSuite AppScript-Side Patterns

The scaffold solves the frontend build problems, but there are some patterns worth documenting on the AppScript side.

Cross-product adaptation: The same add-on runs in Docs/Sheets/Slides/Forms, but AppScript has no "which product am I in" API. The only option is try/catch detection:

function getUi() {
  try { return DocumentApp.getUi(); } catch(e) {}
  try { return SpreadsheetApp.getUi(); } catch(e) {}
  try { return SlidesApp.getUi(); } catch(e) {}
  try { return FormApp.getUi(); } catch(e) {}
  return null;
}

function showSidebar() {
  var ui = getUi();
  if (!ui) return;
  var html = HtmlService.createHtmlOutputFromFile("sidebar")
    .setTitle("My Plugin").setWidth(320);
  ui.showSidebar(html);
}

How google.script.run works: Frontend calls backend functions and gets results via callbacks. Every global function in AppScript can be called from the frontend:

// AppScript side — define global functions
function getGoogleSheets() {
  var files = DriveApp.getFilesByType(MimeType.GOOGLE_SHEETS);
  var sheets = [];
  while (files.hasNext()) {
    var file = files.next();
    sheets.push({
      id: file.getId(),
      name: file.getName(),
      url: file.getUrl(),
    });
  }
  return { success: true, sheets: sheets };
}

// React side — call it
google.script.run
  .withSuccessHandler((result) => {
    setSheets(result.sheets);
  })
  .withFailureHandler((error) => {
    console.error(error);
  })
  .getGoogleSheets();

Useful APIs: AppScript can export Google Docs as HTML, read Sheets data, send Gmail, etc. These are the core capabilities of GSuite add-ons:

// Read Sheets data
function getSheetData(spreadsheetId, worksheetName) {
  var spreadsheet = SpreadsheetApp.openById(spreadsheetId);
  var sheet = spreadsheet.getSheetByName(worksheetName);
  var values = sheet.getDataRange().getValues();
  return { headers: values[0], rows: values.slice(1) };
}

// Read Docs content (export as HTML, preserving formatting)
function getDocumentHtml(docId) {
  var url = "https://docs.google.com/feeds/download/"
    + "documents/export/Export?id=" + docId
    + "&exportFormat=html";
  var response = UrlFetchApp.fetch(url, {
    headers: {
      Authorization: "Bearer " + ScriptApp.getOAuthToken()
    }
  });
  return response.getContentText();
}

// Send email
function sendEmail(to, subject, htmlBody) {
  GmailApp.sendEmail(to, subject, "", {
    htmlBody: htmlBody,
  });
}

Real-World Case: Mail Merge

Using this scaffold, I built a Mail Merge feature: read a recipient list from Google Sheets, use a Google Doc as a template, and send personalized emails in bulk.

User workflow (4-step wizard in the sidebar):

Step 1: Select data source → List Sheets in Drive → Select sheet tab → Preview headers
Step 2: Select template   → List Docs in Drive → Preview HTML content
Step 3: Configure         → Email subject (supports {{Name}} merge fields) → reply-to address
Step 4: Execute           → Summary confirmation → Test mode → Send and show results

The frontend is a standard React multi-step form (using state to control steps), with each step's data fetched from the backend via callAppScript. The backend does field replacement and sends emails:

function sendMailMerge(spreadsheetId, worksheetName,
                       docId, subject, replyTo, testMode) {
  var data = getSheetData(spreadsheetId, worksheetName);
  var template = getDocumentHtml(docId);
  var results = [];

  for (var i = 0; i < data.rows.length; i++) {
    var row = {};
    data.headers.forEach(function(h, j) { row[h] = data.rows[i][j]; });

    var body = template.replace(/\{\{([^}]+)\}\}/g,
      function(match, field) {
        return row[field.trim()] || match;
      });
    var emailSubject = subject.replace(/\{\{([^}]+)\}\}/g,
      function(match, field) {
        return row[field.trim()] || match;
      });

    var recipient = testMode
      ? Session.getActiveUser().getEmail()
      : row["Email"];

    try {
      GmailApp.sendEmail(recipient, emailSubject, "", {
        htmlBody: body,
        replyTo: replyTo || undefined,
      });
      results.push({ row: i + 1, status: "sent" });
    } catch (e) {
      results.push({ row: i + 1, status: "error", error: e.message });
    }

    Utilities.sleep(100); // Rate limiting
  }

  return { success: true, results: results };
}

This Mail Merge demonstrates the scaffold's value: the frontend has complex multi-step interactions (selecting files, previewing data, configuring merge fields, displaying send results) that would be painful to write with raw HTML. React's componentization and state management have clear advantages here.

Development Experience Pain Points

This approach isn't perfect. A few unavoidable issues:

  • No hot reload: Changing AppScript code requires clasp push + refreshing the GSuite page to see the effect. The frontend portion can be debugged locally with webpack-dev-server, but logic involving AppScript interaction can only be tested in GSuite
  • Bundle size: React inlined into HTML makes each file about 1MB. 4 products x N entries = total artifacts can be large. Consider using Preact instead of React, or using TerserPlugin for compression
  • AppScript's 6-minute execution limit: Long tasks (like sending 1000 emails) may time out. Need to control rate in loops, or process in batches
  • Debugging difficulty: console.log inside iframes may not always be visible. On the AppScript side, use Logger.log(), viewable in the Apps Script editor's execution logs

Summary

The core idea of this scaffold: use Webpack to bridge "modern frontend development" and "GSuite sandbox constraints."

  • HtmlInlineScriptPlugin + style-loader solves CSP restrictions — generating self-contained HTML
  • Filesystem scanning implements dialog auto-discovery — zero-config new entries
  • Product x entry matrix loop generates HtmlWebpackPlugin instances — one build covers all products
  • clasp + rootDir configuration enables per-product deployment — each product pushed independently
  • Environment detection + mock data enables local development — debug UI without depending on the GSuite environment

If you need complex UI interactions in GSuite add-ons, this scaffold can save a lot of repetitive work. If it's just a simple add-on with a button or two, writing raw HTML is sufficient — you don't need this setup.
