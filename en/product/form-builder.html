Schema-Driven Forms (Part 1): Data Model and Renderer

2026-03-08
The value of this article: B2B product form requirements are never-ending — registration, approval, surveys, configuration, each scenario is different. Rather than writing a component for every form, design a JSON Schema to describe the form structure and let the frontend render automatically. This article shares a production-tested data model: the FormSpec → Question → QuestionResponse three-layer structure, supporting 15+ field types, pagination, validation, and enabling non-developers to create forms by editing JSON.

The Starting Point

Almost every B2B product encounters the same problem: there are too many forms.

Registration forms, approval forms, surveys, configuration pages, data entry... every business scenario needs a set of forms, and each set has different fields, validation rules, and layouts. If every form requires a handwritten React component, the codebase would explode, and changing a single field would require a new release.

Even worse, many form creators aren't developers — they're product managers, operations staff, or HR. They need a "drag and drop to build forms" tool, not a spot on the sprint queue.

This is where Schema-driven forms come in: describe the form structure with a JSON document, and the frontend automatically renders the complete form from that JSON. Change a field? Change the JSON. Add validation? Change the JSON. Change the layout? Still change the JSON. No code changes, no releases needed.

Core Idea

The core of Schema-driven forms is a simple separation:

Form definition (What)  →  JSON Schema  →  Stored in database
Form rendering (How)    →  Generic renderer  →  One set of code for all forms
Form data (Data)        →  JSON response  →  Structured storage and querying

Data Model Design

A complete Schema-driven form system needs a three-layer data model:

Layer 1: FormSpec (Form Definition)

// FormSpec — metadata for a form
type FormSpec struct {
    ID          int
    Name        string       // "Employee Onboarding"
    Description string       // Form description
    Cover       string       // Cover image URL
    Enabled     bool         // Whether enabled
    CreatedBy   int          // Creator
    Groups      []QuestionGroup  // Question groups
}

// QuestionGroup — question grouping (for form pagination or sectioning)
type QuestionGroup struct {
    ID        int
    Label     string       // "Basic Information" / "Work Experience"
    Order     int          // Sort order
    Questions []Question
}

Layer 2: Question (Question Definition)

type QuestionType string

const (
    ShortText   QuestionType = "short_text"
    Paragraph   QuestionType = "paragraph"
    MultiChoice QuestionType = "multi_choice"
    Checkboxes  QuestionType = "checkboxes"
    DropDown    QuestionType = "drop_down"
    FileUpload  QuestionType = "file"
    LinearScale QuestionType = "linear_scale"
    Date        QuestionType = "date"
    Time        QuestionType = "time"
)

type Question struct {
    ID       int
    Label    string         // "Your name"
    Type     QuestionType   // "short_text"
    Required bool           // Whether required
    Options  []string       // Options for choice questions
    Config   QuestionConfig // Additional configuration
}

type QuestionConfig struct {
    Placeholder  string  // Input hint
    MinLength    *int    // Minimum length
    MaxLength    *int    // Maximum length
    MinValue     *int    // linear_scale minimum
    MaxValue     *int    // linear_scale maximum
    FileTypes    []string // Allowed file types
}

Layer 3: QuestionResponse (User Response)

type QuestionResponse struct {
    ID             int
    QuestionID     int      // Links to Question
    FormInstanceID int      // Links to a specific submission
    Label          string   // Redundant storage of question label (for querying)
    Value          string   // User's answer (uniformly stored as string)
}

type FormInstance struct {
    ID         int
    FormSpecID int          // Links to FormSpec
    SubmittedBy int         // Submitter
    SubmittedAt time.Time
    Responses  []QuestionResponse
}

There's a key design decision here: QuestionResponse.Value is uniformly stored as string. Whether it's a date, number, or multi-select, everything is serialized to a string. The advantage is a simple schema without needing different columns for each type. The downside is that queries require deserialization, but for form scenarios, most queries are "export all responses for a submission" rather than "find all responses where age > 25," so this trade-off is worthwhile.

Frontend Renderer

The renderer's core is a switch statement — rendering different UI components based on the Question's type field:

// Question.tsx
function Question({ question, value, onChange }) {
  switch (question.type) {
    case "short_text":
      return (
        <div className="form-field">
          <label>{question.label}{question.required && " *"}</label>
          <input
            type="text"
            value={value}
            onChange={e => onChange(e.target.value)}
            placeholder={question.config?.placeholder}
            required={question.required}
          />
        </div>
      );

    case "paragraph":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <textarea
            value={value}
            onChange={e => onChange(e.target.value)}
            rows={4}
          />
        </div>
      );

    case "multi_choice":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          {question.options.map(opt => (
            <label key={opt} className="radio-option">
              <input
                type="radio"
                name={`q-${question.id}`}
                value={opt}
                checked={value === opt}
                onChange={() => onChange(opt)}
              />
              {opt}
            </label>
          ))}
        </div>
      );

    case "checkboxes":
      const selected = value ? JSON.parse(value) : [];
      return (
        <div className="form-field">
          <label>{question.label}</label>
          {question.options.map(opt => (
            <label key={opt} className="checkbox-option">
              <input
                type="checkbox"
                checked={selected.includes(opt)}
                onChange={() => {
                  const next = selected.includes(opt)
                    ? selected.filter(s => s !== opt)
                    : [...selected, opt];
                  onChange(JSON.stringify(next));
                }}
              />
              {opt}
            </label>
          ))}
        </div>
      );

    case "drop_down":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <select value={value} onChange={e => onChange(e.target.value)}>
            <option value="">Please select</option>
            {question.options.map(opt => (
              <option key={opt} value={opt}>{opt}</option>
            ))}
          </select>
        </div>
      );

    case "date":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <input type="date" value={value} onChange={e => onChange(e.target.value)} />
        </div>
      );

    case "file":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <input
            type="file"
            accept={question.config?.fileTypes?.join(",")}
            onChange={e => {
              const file = e.target.files?.[0];
              if (file) uploadAndGetUrl(file).then(onChange);
            }}
          />
        </div>
      );

    default:
      return <div>Unsupported question type: {question.type}</div>;
  }
}

This switch statement looks "simple," but that's precisely its advantage: each type's rendering logic is completely independent, with no interference. Adding a new type? Add a case. Changing a type's UI? Only modify that case. There's no "modifying radio buttons affects checkboxes" problem.

Complete Form Rendering

Composing Question components to render the entire form:

function FormRenderer({ formSpec }) {
  const [responses, setResponses] = useState({});

  const handleChange = (questionId, value) => {
    setResponses(prev => ({ ...prev, [questionId]: value }));
  };

  const handleSubmit = async () => {
    // Validate required fields
    for (const group of formSpec.groups) {
      for (const q of group.questions) {
        if (q.required && !responses[q.id]) {
          alert(`Please fill in: ${q.label}`);
          return;
        }
      }
    }

    // Construct submission data
    const payload = Object.entries(responses).map(([qId, value]) => ({
      questionId: parseInt(qId),
      value: String(value),
    }));

    await submitFormResponse(formSpec.id, payload);
  };

  return (
    <form onSubmit={e => { e.preventDefault(); handleSubmit(); }}>
      <h2>{formSpec.name}</h2>
      <p>{formSpec.description}</p>

      {formSpec.groups.map(group => (
        <fieldset key={group.id}>
          <legend>{group.label}</legend>
          {group.questions.map(q => (
            <Question
              key={q.id}
              question={q}
              value={responses[q.id] || ""}
              onChange={v => handleChange(q.id, v)}
            />
          ))}
        </fieldset>
      ))}

      <button type="submit">Submit</button>
    </form>
  );
}

Backend API Design

Using GraphQL for API design is particularly natural because the form's data structure is inherently nested:

type FormSpec {
  id: ID!
  name: String!
  description: String
  enabled: Boolean!
  groups: [QuestionGroup!]!
}

type QuestionGroup {
  id: ID!
  label: String!
  order: Int!
  questions: [Question!]!
}

type Question {
  id: ID!
  label: String!
  type: QuestionType!
  required: Boolean!
  options: [String!]
  config: QuestionConfig
}

enum QuestionType {
  SHORT_TEXT
  PARAGRAPH
  MULTI_CHOICE
  CHECKBOXES
  DROP_DOWN
  FILE
  LINEAR_SCALE
  DATE
  TIME
}

# Query
type Query {
  formSpec(id: ID!): FormSpec
  formInstances(formSpecId: ID!): [FormInstance!]!
}

# Submission
input QuestionResponseInput {
  questionId: ID!
  value: String!
}

type Mutation {
  submitForm(formSpecId: ID!, responses: [QuestionResponseInput!]!): FormInstance!
}

GraphQL's nested queries let the frontend fetch a complete form definition (FormSpec → Groups → Questions) in a single request, without multiple round trips. Combined with Relay or Apollo's fragment mechanism, Question components can declare what data they need, and the framework automatically merges queries.

Simplifying the Backend with ORM Code Generation

If you use Go + Ent ORM, you can directly generate CRUD code and GraphQL resolvers from schema definitions:

// ent/schema/formspec.go
func (FormSpec) Fields() []ent.Field {
    return []ent.Field{
        field.String("name").NotEmpty(),
        field.String("description").Optional(),
        field.String("cover").Optional(),
        field.Bool("enabled").Default(true),
        field.Time("created_at").Default(time.Now),
        field.Int("created_by"),
    }
}

func (FormSpec) Edges() []ent.Edge {
    return []ent.Edge{
        edge.To("question_groups", QuestionGroup.Type),
    }
}

// ent/schema/question.go
func (Question) Fields() []ent.Field {
    return []ent.Field{
        field.String("label").NotEmpty(),
        field.Enum("type").Values(
            "short_text", "paragraph", "multi_choice",
            "checkboxes", "drop_down", "file",
            "linear_scale", "date", "time",
        ),
        field.Bool("required").Default(false),
        field.JSON("options", []string{}).Optional(),
    }
}

Ent automatically generates from these schema definitions: database migrations, Go type definitions, CRUD methods, and GraphQL types and resolvers (with entgql). From schema to usable API, almost no glue code is needed.

Validation Strategy

Form validation is split into two layers:

Frontend validation (instant feedback):

function validate(question, value) {
  if (question.required && !value) {
    return `${question.label} is required`;
  }

  const config = question.config;
  if (!config) return null;

  switch (question.type) {
    case "short_text":
    case "paragraph":
      if (config.minLength && value.length < config.minLength)
        return `Minimum ${config.minLength} characters required`;
      if (config.maxLength && value.length > config.maxLength)
        return `Maximum ${config.maxLength} characters allowed`;
      break;

    case "file":
      if (config.fileTypes && !config.fileTypes.some(t => value.endsWith(t)))
        return `Only ${config.fileTypes.join(", ")} formats are supported`;
      break;
  }
  return null;
}

Backend validation (security guarantee):

func ValidateResponse(spec *ent.FormSpec, responses []ResponseInput) error {
    questionMap := buildQuestionMap(spec)

    for _, resp := range responses {
        q, ok := questionMap[resp.QuestionID]
        if !ok {
            return fmt.Errorf("unknown question ID: %d", resp.QuestionID)
        }
        if q.Required && resp.Value == "" {
            return fmt.Errorf("question %q is required", q.Label)
        }
        if err := validateType(q.Type, resp.Value); err != nil {
            return fmt.Errorf("question %q: %w", q.Label, err)
        }
    }

    // Check that all required questions have responses
    for id, q := range questionMap {
        if q.Required && !hasResponse(responses, id) {
            return fmt.Errorf("missing required question: %q", q.Label)
        }
    }
    return nil
}

Frontend validation is for user experience (see errors without waiting for a server round trip), backend validation is for data integrity (frontend validation can be bypassed). Neither layer can be omitted.

Issues You'll Encounter in Practice

  • Schema versioning: After publishing a form, if the Schema is modified (e.g., deleting a question), what happens to already-submitted Responses? The common approach is to create a new version rather than modifying the original, or soft-delete questions while preserving historical responses.
  • Conditional logic: Real-world forms often have logic like "if A is selected, show question X." This requires adding a visibility_rules field to the Schema, with the renderer dynamically showing/hiding questions based on current responses. This is the main source of complexity.
  • Performance: When rendering large forms (100+ questions), re-rendering each Question component becomes a bottleneck. Solution: React.memo + wrapping onChange callbacks with useCallback, ensuring only Questions with changed values re-render.
  • File upload: File-type questions need special handling — first upload to OSS to get a URL, then store the URL as the value. This means file upload and form submission are two independent processes.
  • Export and analysis: Form data ultimately needs to be exported to Excel or analyzed statistically. Since Value is a string type, formatting during export is needed based on Question Type (converting dates to Date format, multi-select to comma-separated, etc.).

Comparison with Existing Solutions

Solution Schema Storage Rendering Suitable For
Google Forms Proprietary format Google-hosted Simple surveys
JSON Schema + react-jsonschema-form Standard JSON Schema Auto-generated Developer tools, config forms
Formily / XRender Custom DSL Visual editor + renderer Enterprise complex forms
Custom FormSpec model Database + ORM Custom renderer Products needing deep customization

react-jsonschema-form uses standard JSON Schema (Draft 7). Its advantage is a good ecosystem with ready-made validation libraries (ajv). But standard JSON Schema is designed to describe data structures, not form UI — it lacks concepts like "grouping," "placeholder," and "conditional display," which need to be supplemented via uiSchema.

The custom FormSpec model's advantage is complete control over the data structure, enabling deep customization for business needs (e.g., adding approval workflows, permission controls, template marketplaces). The cost is needing to write your own renderer and validation logic.

Architectural Reflections

  • Schema as contract: FormSpec is the sole contract between frontend and backend. The backend doesn't need to know how the frontend renders, and the frontend doesn't need to know how the backend stores. As long as the Schema format doesn't change, both sides can evolve independently.
  • Code generation vs handwriting: With code generation tools like Ent + gqlgen, the path from Schema definition to API endpoint is almost fully automated. Handwriting CRUD code has zero value — let machines generate it and focus your energy on business logic.
  • The aesthetics of the switch statement: The big switch in the renderer looks "primitive," but it's the right level of abstraction. Don't over-abstract into a "registry pattern" or "plugin system" — unless you truly need third-party developers to extend question types. Most of the time, 9 types are enough.
  • About string storage for Value: This is a classic "flexibility vs. query capability" trade-off. If your scenario requires complex queries on responses ("find all users who selected option A"), consider PostgreSQL's JSONB type, or write to a wide table for analysis at submission time.

Summary

Schema-driven forms are essentially a concrete application of an old software engineering idea: separation of data and behavior. The form's structure (data) is described by the Schema, and rendering and validation (behavior) are handled by generic code. This lets non-developers create and modify forms, and lets developers serve all form scenarios with a single codebase.

This pattern isn't limited to forms. Configuration pages, approval workflows, data dashboards — any scenario where "structure varies but behavior is fixed" can be implemented with a Schema-driven approach. The core is always: define a good Schema, write a good renderer.
