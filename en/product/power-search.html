PowerSearch: Building a Composable Frontend Advanced Search Bar

2026-03-08
The value of this article: Almost every B2B product needs advanced search, but most implementations are either crude piles of input boxes or half-baked solutions that can't compose or nest. This article shares a recursive condition tree data model that naturally supports AND/OR composition and infinite nesting — when the data structure is right, UI interactions and backend mapping both become simple.

The Problem

List pages in B2B products almost always need search and filtering. The simplest approach is placing a few input boxes and dropdowns, but this gets ugly with many fields and can't express complex logic — like "status is completed AND (creator is Alice OR creator is Bob)."

Products like Jira, Linear, and Notion have search bars that support dynamically adding filter conditions, selecting fields, operators, and values, and combining conditions with AND/OR and nested groups. This interaction is called PowerSearch (or Advanced Filter, Structured Search).

It looks complex, but the data structure is actually quite clear — it's essentially a condition expression tree.

Data Model

The entire system's type definitions are just a few:

// Field configuration — defines which fields can be searched
type FieldConfig = {
  id: string;          // "status"
  label: string;       // "Status"
  type: "text" | "number" | "date" | "select" | "boolean";
  operators: OperatorOption[];     // Operators supported by this field
  options?: { value: string; label: string }[];  // Options for select type
};

// Operator
type OperatorOption = {
  id: string;      // "eq"
  label: string;   // "equals"
  symbol: string;  // "="
};

// Single filter condition
type FilterCondition = {
  id: string;
  type: "condition";
  fieldId: string;     // Which field
  operatorId: string;  // Which operator
  value: string;       // Comparison value
};

// Condition group (recursive structure, supports nesting)
type ConditionGroup = {
  id: string;
  type: "group";
  operator: "AND" | "OR";
  conditions: (FilterCondition | ConditionGroup)[];  // Can nest
};

The key is the last type: ConditionGroup's conditions array can contain FilterCondition (leaf nodes) or ConditionGroup (subtrees). This is a recursive expression tree.

An actual query looks like this:

// "status = done AND (creator = Alice OR creator = Bob)"
{
  id: "root",
  type: "group",
  operator: "AND",
  conditions: [
    {
      id: "c1", type: "condition",
      fieldId: "status", operatorId: "eq", value: "done"
    },
    {
      id: "g1", type: "group",
      operator: "OR",
      conditions: [
        {
          id: "c2", type: "condition",
          fieldId: "creator", operatorId: "eq", value: "alice"
        },
        {
          id: "c3", type: "condition",
          fieldId: "creator", operatorId: "eq", value: "bob"
        }
      ]
    }
  ]
}

The component internally maintains a ConditionGroup tree as state. Each time a user adds/modifies/deletes a condition, the tree is updated and emitted via the onSearch callback. The caller takes this tree and converts it to a GraphQL where clause or REST API query string to send to the backend.

Interaction Flow

Adding a filter condition is a three-step guided flow:

1. Select field → Show field list (Field Dropdown)
                ┌─────────────────┐
                │ Select Field    │
                ├─────────────────┤
                │ Name      text  │
                │ Status  select  │
                │ Created   date  │
                │ Priority number │
                └─────────────────┘

2. Select operator → Show operators based on field type
                ┌─────────────────┐
                │ Select Operator │
                ├─────────────────┤
                │ equals        = │
                │ not equals    ≠ │
                └─────────────────┘

3. Enter value → Show different input methods based on field type
   text    → Text input
   number  → Number input
   date    → Date picker
   select  → Option list
   boolean → true/false selection

Each step automatically advances to the next upon completion, and the condition is automatically submitted when all three steps are done. This progressive guidance is a much better experience than showing three side-by-side dropdowns — users only need to make one decision at a time.

State Management

The component has two core states:

// 1. Condition tree (complete query structure)
const [rootGroup, setRootGroup] = useState<ConditionGroup>({
  id: "root",
  type: "group",
  operator: defaultOperator,
  conditions: [],
});

// 2. Currently editing condition (intermediate state of the three-step guide)
const [editingCondition, setEditingCondition] = useState<{
  condition: Partial<FilterCondition>;  // May only have the field, operator not yet selected
  groupId: string;   // Which group it belongs to
  index?: number;    // Index when editing an existing condition
} | null>(null);

Condition tree modifications are done via deep copy (immutable update), ensuring React correctly detects changes:

// Deep clone condition tree
const deepCloneGroup = (group: ConditionGroup): ConditionGroup => ({
  ...group,
  conditions: group.conditions.map((c) =>
    c.type === "group" ? deepCloneGroup(c) : { ...c }
  ),
});

// Find target group in nested tree
const findGroupById = (
  group: ConditionGroup, id: string
): ConditionGroup | null => {
  if (group.id === id) return group;
  for (const c of group.conditions) {
    if (c.type === "group") {
      const found = findGroupById(c, id);
      if (found) return found;
    }
  }
  return null;
};

Each modification: deep clone tree → find target group → modify → set new state → trigger onSearch callback.

Value Input Type Adaptation

Step 3 "enter value" renders different input components based on field type — a switch dispatch:

const renderValueInput = () => {
  const field = getField(editingCondition.condition.fieldId);

  switch (field.type) {
    case "select":
      // Render option list, click to submit directly
      return <OptionsList options={field.options} onSelect={handleSubmit} />;

    case "boolean":
      // Only two options: true/false
      return <BooleanPicker onSelect={handleSubmit} />;

    case "date":
      // Date picker
      return <input type="date" onChange={handleValueChange} />;

    case "number":
      // Number input
      return <input type="number" onChange={handleValueChange} />;

    default:  // text
      // Text input, Enter to submit
      return <input type="text" onChange={handleValueChange}
                     onKeyDown={e => e.key === "Enter" && handleSubmit()} />;
  }
};

Note the special handling for select and boolean types: clicking an option directly submits the condition, without needing an extra "confirm" button. This eliminates one click, making the interaction smoother.

AND/OR Toggling and Condition Grouping

The logical relationship between conditions is controlled by ConditionGroup.operator. In the UI, it's a clickable small label that toggles between AND and OR on each click:

const handleToggleOperator = (groupId: string) => {
  setRootGroup((prev) => {
    const updated = deepCloneGroup(prev);
    const group = findGroupById(updated, groupId);
    if (group) {
      group.operator = group.operator === "AND" ? "OR" : "AND";
    }
    onSearch(updated);
    return updated;
  });
};

Nested grouping lets users express more complex logic. For example, "(A AND B) OR (C AND D)" — top level is OR, two sub-groups each are AND. The implementation limits maximum nesting depth to 3 levels, preventing overly complex condition trees:

// Only show "Add Group" button when depth < 3
{depth < 3 && (
  <button onClick={() => handleAddGroup(group.id)}>
    Add Group
  </button>
)}

Condition Rendering

Condition tree rendering is also recursive. Each ConditionGroup renders its own operator label and conditions list, recursively calling itself for sub-groups:

const renderGroup = (group: ConditionGroup, depth = 0) => (
  <div className={depth > 0 ? "nested-group" : ""}>
    {/* AND/OR toggle button */}
    {group.conditions.length > 0 && (
      <button onClick={() => handleToggleOperator(group.id)}>
        {group.operator}
      </button>
    )}

    {/* Render conditions one by one */}
    {group.conditions.map((condition, index) => {
      if (condition.type === "group") {
        // Recursively render sub-group
        return renderGroup(condition, depth + 1);
      } else {
        // Render single condition pill
        const field = getField(condition.fieldId);
        const operator = getOperator(condition.fieldId, condition.operatorId);
        return (
          <div className="condition-pill">
            <span className="field">{field?.label}</span>
            <span className="operator">{operator?.symbol}</span>
            <span className="value">{condition.value}</span>
            <button onClick={() => handleRemove(group.id, index)}>×</button>
          </div>
        );
      }
    })}

    {/* Add condition/group buttons */}
    <button onClick={() => handleAddCondition(group.id)}>+ Add Filter</button>
    {depth < 3 && <button onClick={() => handleAddGroup(group.id)}>() Add Group</button>}
  </div>
);

Nested sub-groups are visually indented with a left border (border-left + padding-left), clearly expressing the hierarchy.

Backend Integration

The onSearch callback returns a ConditionGroup tree. The backend typically needs to convert it to a database query. This conversion is recursive:

// Convert to SQL WHERE clause (pseudocode)
function toSQL(group: ConditionGroup): string {
  const parts = group.conditions.map((c) => {
    if (c.type === "group") {
      return `(${toSQL(c)})`;
    }
    const op = operatorToSQL(c.operatorId);  // "eq" → "=", "contains" → "LIKE"
    const val = escapeSQL(c.value);
    return `${c.fieldId} ${op} ${val}`;
  });
  return parts.join(` ${group.operator} `);
}

// Output: status = 'done' AND (creator = 'alice' OR creator = 'bob')

If using GraphQL, you can directly map the tree structure to a nested where input:

// GraphQL where input
{
  and: [
    { status: { eq: "done" } },
    {
      or: [
        { creator: { eq: "alice" } },
        { creator: { eq: "bob" } }
      ]
    }
  ]
}

The data structures are isomorphic — the frontend's ConditionGroup and GraphQL's where input are just different representations of the same tree. The conversion is almost a one-to-one mapping.

Design Reflections

  • Why a tree instead of a flat array: A flat { field, op, value }[] array can only express all-AND or all-OR relationships. Once you need mixed logic ("A AND (B OR C)"), you must use a tree. A tree is the most general structure; a flat array is just a degenerate special case of a tree.
  • Why FieldConfig includes operators: Different field types support different operators — text fields support "contains," number fields support "greater than/less than," boolean fields only support "equals." Binding operators to fields rather than defining them globally avoids meaningless combinations like "date field selected contains."
  • Why three separate steps instead of three side-by-side dropdowns: Progressive guidance reduces cognitive load. Users only need to focus on one selection at a time, instead of facing three dropdowns simultaneously without knowing where to start.
  • Limiting nesting depth: Theoretically, recursive structures can nest infinitely, but in practice, search conditions nested beyond 3 levels are nearly unmaintainable. Limiting depth protects the user.
  • Conditions as data: The entire search state is a serializable JSON tree. This means it can be saved to a URL query string (share search results), persisted to user settings (save common filters), or saved as a "view" (similar to Notion's database views).

Extension Directions

  • Save as view: Save the ConditionGroup JSON to the database, loading it directly when the user returns. Like filter views in Notion.
  • URL serialization: Encode the condition tree into the URL (base64 or custom syntax), so search result pages can be shared with others.
  • Inter-field dependencies: After selecting "country = China," the "city" field's options automatically filter to Chinese cities. Requires adding dependsOn and dynamic option loading to FieldConfig.
  • Custom operators: Such as "in list" (IN), "is empty" (IS NULL), "in range" (BETWEEN). Just add new OperatorOptions to FieldConfig.operators, and the renderer adapts automatically.
  • Keyboard navigation: Support arrow keys to move between conditions, Tab to switch steps, Delete to remove conditions. For power users, keyboard operations are much faster than mouse.

Summary

The core of PowerSearch is two things: a recursive condition tree data structure, and a three-step progressive condition editing interaction. The data structure determines how complex a query it can express (arbitrary AND/OR nesting), and the interaction design determines whether complex queries are easy to build (the field → operator → value guided flow).

From an engineering perspective, this component's core code is small — type definitions 30 lines, state management 50 lines, recursive rendering 50 lines. Most of the code is in UI details (dropdown styles, hover effects, keyboard events). This is typical for frontend components: the core logic is simple, but delivering good interaction experience requires extensive detail polishing.
