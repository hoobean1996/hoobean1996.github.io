Building a Distributed Async Job Queue from Scratch: Lessons from Shopee

2026-03-08
The value of this article: Every backend system of a certain scale needs async tasks вҖ” deferred execution, scheduled jobs, failure retries. Open-source solutions (Celery, Bull) are often too heavy or not flexible enough. This article shares the complete experience of building a distributed async job queue from scratch at Shopee: the storage model, scheduling strategies, failure retries, priority queues, and the pitfalls encountered in production.

Why Every Company Ends Up Building Their Own Async Task System

Almost every backend system that reaches a certain scale encounters the same problem: some things can't be done synchronously within a user request.

After a user places an order, you need to check 30 minutes later whether they've paid вҖ” if not, cancel it. After a user registers, you need to asynchronously send a welcome email. Every day at midnight, you need to run a reconciliation job. Every hour, you need to sync external data. These "do it later" tasks are async tasks.

The problem is, "do it later" sounds simple but is anything but:

  • Reliability: If the service restarts, are the tasks still there? If there's a network hiccup, will tasks be lost?
  • Delay control: "Execute in 30 minutes" вҖ” who keeps track of the time? What's the precision?
  • Concurrency control: With 10 instances running simultaneously, how do you ensure the same task isn't executed twice?
  • Failure handling: If a task fails, how many retries? What's the retry interval? When do you give up?
  • Observability: How many tasks are currently queued? What's the average latency? Which tasks have failed?

This is why projects like Celery, Sidekiq, Bull, and Temporal exist вҖ” they all solve the same problem. But in large companies, generic solutions often aren't enough: you need to integrate with internal RPC frameworks, adapt to the company's deployment platform, and connect with existing monitoring systems. So almost every company of a certain scale ends up building their own.

The Situation at Shopee

In 2019, I joined Shopee's platform team, responsible for middleware services targeting business development teams. At the time, each business line had its own "async task" solution вҖ” some used crontab + scripts, some wrote infinite loops polling the database, and some just spawned goroutines in business code and prayed they wouldn't lose tasks.

The problems were obvious: no unified task management, no monitoring, no retries, and lost tasks went unnoticed. Even worse, every team was repeatedly stepping on the same pitfalls вҖ” retries without idempotency leading to duplicate charges, crontab machines crashing unnoticed causing reconciliation to break for a week. The platform team decided to build a distributed async job queue system from scratch to provide standardized async task capabilities.

This article documents the complete process from design to production. Over three years, it went from 0 to 10+ business integrations, QPS grew from 2K to 25K+, and handled 50K peak during Double 11.

Requirements Analysis

After talking with various business teams, async task requirements fell into roughly three categories:

Type Example Scenarios Latency Requirement
Short-delay tasks Order timeout cancellation, coupon expiration, push notifications Seconds to minutes
Long-delay tasks T+1 reconciliation, weekly report generation, data archiving Hours to days
Cron scheduled tasks Daily midnight batch jobs, hourly data sync, heartbeat checks Per cron expression

Beyond functional requirements, there were several hard constraints:

  • At-least-once delivery: Tasks can be executed multiple times (business handles idempotency), but they must never be lost
  • Multi-language support: Business teams use Go and Python, so SDKs for both languages were required
  • Integration with internal RPC platform: Task triggering calls business services through the company's internal RPC framework, without introducing additional network protocols
  • Observability: The status, duration, and retry count of every task must be queryable

Why Kafka + MySQL, Not Pure Kafka or Pure Redis

This was the most debated decision during the design phase.

The pure Kafka approach had problems: Kafka doesn't support arbitrary delayed delivery. While you can simulate different delay levels with multiple topics (1s/5s/30s/1min/...), the granularity is too coarse and the number of topics explodes. For "trigger in 3 days" long delays, Kafka is completely unsuitable вҖ” messages pile up on brokers, wasting resources with rebalance risks.

The pure Redis approach (Sorted Set + ZRANGEBYSCORE): store trigger time as the score for delayed tasks, periodically scan for due tasks. The problem is Redis persistence isn't reliable enough вҖ” RDB has time windows, and AOF performance degrades under high QPS. For the hard requirement of "at-least-once delivery," Redis as the sole storage is too risky.

The final approach: Kafka handles immediate and short-delay tasks (вүӨ 10 minutes), MySQL handles long-delay and Cron tasks.

  • Kafka: High throughput, native support for consumer groups and partition-level parallelism. Short-delay tasks are consumed and placed into an in-memory ETA priority queue (min-heap), triggered when due.
  • MySQL: ACID guarantees that tasks aren't lost. Long-delay tasks are written with a trigger time, and the Dispatcher scans for due tasks every second. Cron tasks have their next trigger time pre-calculated by the Scheduler.

Short-Delay Tasks: ETA Priority Queue

After consuming short-delay tasks from Kafka, instead of simply sleeping to wait, they're placed into a min-heap-based priority queue sorted by ETA (Expected Time of Arrival):

// ETA Priority Queue вҖ” based on container/heap
type Task struct {
    ID        string
    Type      string
    Payload   string
    ETA       time.Time  // Expected trigger time
    Priority  int        // Business priority (tiebreak for same ETA)
    index     int        // Internal heap index
}

type ETAQueue []*Task

func (q ETAQueue) Len() int { return len(q) }
func (q ETAQueue) Less(i, j int) bool {
    if q[i].ETA.Equal(q[j].ETA) {
        return q[i].Priority > q[j].Priority  // Same ETA, higher priority first
    }
    return q[i].ETA.Before(q[j].ETA)
}
func (q ETAQueue) Swap(i, j int) {
    q[i], q[j] = q[j], q[i]
    q[i].index = i
    q[j].index = j
}
func (q *ETAQueue) Push(x any) {
    t := x.(*Task)
    t.index = len(*q)
    *q = append(*q, t)
}
func (q *ETAQueue) Pop() any {
    old := *q
    n := len(old)
    t := old[n-1]
    old[n-1] = nil
    t.index = -1
    *q = old[:n-1]
    return t
}

The Dispatcher's scheduling loop:

func (d *Dispatcher) Run() {
    pq := &ETAQueue{}
    heap.Init(pq)

    for {
        // 1. Consume new tasks from Kafka, push onto heap
        for _, task := range d.consumeBatch() {
            heap.Push(pq, task)
        }

        // 2. Pop all due tasks
        now := time.Now()
        for pq.Len() > 0 && (*pq)[0].ETA.Before(now) {
            task := heap.Pop(pq).(*Task)
            go d.dispatch(task)  // Async execution
        }

        // 3. Calculate next wake time: heap top's ETA or fallback 100ms
        var sleepUntil time.Duration
        if pq.Len() > 0 {
            sleepUntil = time.Until((*pq)[0].ETA)
            if sleepUntil < 0 {
                sleepUntil = 0
            }
        } else {
            sleepUntil = 100 * time.Millisecond
        }
        time.Sleep(sleepUntil)
    }
}

Why a heap instead of a Timer Wheel:

  • Intuitive semantics: Each task has an ETA, and the heap top is always the next task to trigger. The code logic is clear вҖ” no need to understand concepts like cascade or tick.
  • Flexible precision: Timer Wheel precision is limited by tick granularity (e.g., 100ms per slot), while the heap has no such limitation вҖ” ETA is precise to nanoseconds.
  • Natural priority support: The Less function can compare both ETA and business priority simultaneously. Among tasks due within the same second, higher priority ones execute first.
  • O(log n) is sufficient: Heap insertion and extraction are O(log n). For our scenario (typically thousands to tens of thousands of tasks in the heap), performance is not a bottleneck. Timer Wheel's O(1) advantage only shows at the millions-of-tasks scale.
  • Go standard library built-in: container/heap already provides a complete interface, no extra dependencies needed.

Long-Delay Tasks: MySQL Poller

Long-delay tasks are stored in MySQL with this core table structure:

CREATE TABLE async_tasks (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    task_type   VARCHAR(128) NOT NULL,
    payload     TEXT NOT NULL,
    trigger_at  DATETIME NOT NULL,    -- Trigger time
    status      TINYINT DEFAULT 0,    -- 0=pending, 1=dispatched, 2=success, 3=failed
    retry_count INT DEFAULT 0,
    max_retries INT DEFAULT 3,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_trigger (status, trigger_at)
);

The Dispatcher's poller executes every second:

SELECT id, task_type, payload
FROM async_tasks
WHERE status = 0 AND trigger_at <= NOW()
ORDER BY trigger_at
LIMIT 1000
FOR UPDATE SKIP LOCKED;

FOR UPDATE SKIP LOCKED is the key вҖ” when multiple Dispatcher instances scan in parallel, rows already locked by other instances are skipped, avoiding the complexity of distributed locks. This is a feature introduced in MySQL 8.0, perfectly suited for this scenario.

Scanned tasks are immediately updated to dispatched status, then the business service's executor is called via RPC. If the RPC fails or times out, the task status reverts to pending, retry_count is incremented, and trigger_at is set to the current time plus a backoff delay (exponential backoff + jitter).

Cron Scheduled Tasks

Cron task implementation has two layers:

  1. Cron Scheduler: Parses cron expressions, calculates the next trigger time, and writes to MySQL's async_tasks table. Essentially, it converts Cron tasks into long-delay tasks.
  2. Cron Registry: Stores cron rule metadata (cron expression, associated task_type, owner, etc.).
// Cron Scheduler core loop
func (s *CronScheduler) Run() {
    for {
        rules := s.loadActiveRules()  // Load from cron_registry
        for _, rule := range rules {
            next := rule.NextTriggerTime()
            if !s.taskExistsForTime(rule.TaskType, next) {
                s.createTask(rule.TaskType, rule.Payload, next)
            }
        }
        time.Sleep(10 * time.Second)  // Check every 10 seconds
    }
}

The beauty of this design is that the Cron Scheduler is stateless вҖ” it only ensures "the next trigger task exists in MySQL." Even if the Scheduler crashes and restarts, no tasks are lost because it's idempotent (checks existence before creating).

SDK Design

The Go SDK and Python SDK are the interfaces directly used by business teams. Design principle: be as simple as possible, hiding underlying complexity.

// Go SDK - Submit task
client := asyncd.NewClient(asyncd.WithServiceName("order-service"))

// Immediate task
err := client.Submit(ctx, &asyncd.Task{
    Type:    "cancel_expired_order",
    Payload: `{"order_id": 12345}`,
})

// Delayed task
err := client.SubmitDelayed(ctx, &asyncd.Task{
    Type:    "send_review_reminder",
    Payload: `{"user_id": 67890}`,
}, 3 * 24 * time.Hour)  // 3 days later

// Register executor
client.RegisterExecutor("cancel_expired_order", func(ctx context.Context, payload string) error {
    var req CancelOrderRequest
    json.Unmarshal([]byte(payload), &req)
    return orderService.Cancel(ctx, req.OrderID)
})

The SDK handles several things internally:

  • Serialization: Payload is uniformly JSON strings, avoiding cross-language serialization compatibility issues
  • RPC integration: Services are registered through the company's internal RPC framework. When the Dispatcher calls executors, it goes through the standard RPC path (service discovery, load balancing, and timeout control are all handled by the framework)
  • Idempotency reminders: SDK documentation and code reviews repeatedly emphasize вҖ” executors must be idempotent, because "at-least-once delivery" means the same task may be executed multiple times

Kubernetes Operator

The system is deployed on Kubernetes. Initially we used standard Deployment + ConfigMap, but operational pain points quickly emerged:

  • Onboarding new business required manually creating Kafka topics, updating routing configs, and adjusting consumer counts
  • Dispatcher partition assignment and MySQL poller concurrency needed dynamic adjustment based on QPS
  • Cron rule changes required restarting the Scheduler

So we developed a Custom Operator, introducing a CRD:

apiVersion: asyncd.example.io/v1
kind: AsyncJobConfig
metadata:
  name: order-cancel-job
spec:
  taskType: cancel_expired_order
  executor:
    service: order-service
    method: CancelExpiredOrder
    timeout: 30s
  retry:
    maxRetries: 3
    backoff: exponential
  kafka:
    topic: asyncd-order-cancel
    partitions: 6
    consumerGroup: asyncd-order-cancel-cg

The Operator watches CRD changes and automatically:

  • Creates/updates Kafka topics and consumer groups
  • Updates the Dispatcher's routing table (via ConfigMap hot reload)
  • Automatically adjusts consumer replica count based on QPS metrics (HPA integration)

This dramatically lowered the barrier for business onboarding вҖ” from "file a ticket for the platform team to configure" to "submit a YAML."

Monitoring and Alerting

Every component exposes Prometheus metrics:

// Task submission
asyncd_task_submitted_total{task_type, source_service}
asyncd_task_submit_latency_seconds{task_type}

// Task scheduling
asyncd_task_dispatched_total{task_type}
asyncd_task_dispatch_delay_seconds{task_type}  // Actual trigger time - expected trigger time

// Task execution
asyncd_task_executed_total{task_type, status}   // status: success/failed/timeout
asyncd_task_execution_duration_seconds{task_type}
asyncd_task_retry_total{task_type}

// Queue health
asyncd_kafka_consumer_lag{topic, partition}
asyncd_mysql_pending_tasks{task_type}
asyncd_mysql_poll_duration_seconds

The most critical alerting rules:

  • dispatch_delay > 5s: Task scheduling latency exceeds 5 seconds, indicating insufficient consumption capacity or a stuck MySQL poller
  • kafka_consumer_lag continuously growing: Consumers can't keep up with producers, need to scale out
  • mysql_pending_tasks continuously growing: Due tasks are piling up, poller may have issues
  • task retry rate > 10%: Downstream service may be having problems

Double 11: From 25K to 50K QPS

The 2021 Double 11 was the biggest test for this system. Daily QPS was around 25K, with an estimated peak requirement of 50K+.

Several things were done in advance:

  • Kafka partition expansion: Core topics went from 12 to 24 partitions, with corresponding consumer scaling
  • MySQL read-write splitting: Poller queries go to replicas, status updates go to the primary, preventing scan queries from impacting writes
  • Batch submission optimization: SDK added a SubmitBatch interface, submitting multiple tasks in a single RPC call, reducing network round trips
  • Pre-warming: The day before Double 11, all Cron tasks' next trigger records were pre-created to avoid write spikes at midnight

The actual peak reached approximately 48K QPS. The system held up with zero task loss. Key metrics:

Pitfalls Encountered

1. MySQL datetime Precision

MySQL's DATETIME defaults to second-level precision. When multiple tasks have trigger_at within the same second, the order of ORDER BY trigger_at is non-deterministic. For second-level delay tasks, this caused out-of-order execution. Solution: switch to DATETIME(3) (millisecond precision) and add id as a tiebreaker in ORDER BY.

2. Kafka Rebalance Storms

Early on, consumers didn't have reasonable session.timeout.ms and heartbeat.interval.ms settings. Occasional slow requests from business executors caused consumers to be deemed dead, triggering rebalances. During rebalance, all consumers stopped consuming, causing latency spikes. Finally adjusted to session.timeout=30s, heartbeat.interval=10s, max.poll.interval=5min, and executor calls were made asynchronous вҖ” messages are committed immediately after consumption, with execution results reported via callbacks.

3. Poller Thundering Herd Effect

Multiple Dispatcher instances scanning MySQL simultaneously вҖ” while FOR UPDATE SKIP LOCKED prevented duplicate execution, heavy lock contention caused MySQL CPU to spike. Solution: each Dispatcher instance handles specific task_type ranges (assigned via consistent hashing), preventing all instances from scanning the same data.

4. Timezone Issues

Shopee's business covers multiple Southeast Asian countries with different timezones. The cron expression 0 0 * * * (daily at midnight) вҖ” which midnight exactly? Final solution: all internal times are stored and calculated in UTC. Cron rules must explicitly specify timezone CRON_TZ=Asia/Singapore 0 0 * * *. The SDK validates whether cron expressions include timezone information and rejects those that don't.

5. The Difficulty of Promoting Idempotency

Technically the simplest but hardest to promote. "At-least-once delivery" requires business executors to be idempotent, but many business developers didn't understand or didn't take it seriously. After launch, there were several incidents of "duplicate charges" and "duplicate coupon issuance," all because executors weren't idempotent. Eventually, the SDK layer was updated to require an idempotency_key, along with a Redis-based deduplication helper, reducing error probability at the tooling level.

Retrospective and Reflections

This system started design in mid-2019, had its first business integration in early 2020, and survived 50K QPS during the 2021 Double 11. A few insights from those three years:

  • Storage choice determines the system's ceiling: The Kafka + MySQL combination isn't glamorous, but it wins on maturity and reliability. In middleware systems, "never lose data" is far more important than "low latency." Kafka's persistence and at-least-once semantics are the foundation of this system вҖ” any alternative must pass this bar first.
  • SDK usability determines the system's adoption speed: No matter how good the system is, if the SDK is cumbersome to use, business teams won't adopt it. Time spent on SDK design is never wasted.
  • Kubernetes Operator is leverage for middleware teams: Encoding operational procedures as CRD + Controller allows business teams to self-serve onboarding, and the platform team is no longer the bottleneck. This pattern was later reused by other middleware (config center, message queue).
  • Monitoring isn't an afterthought вҖ” it should be there from day one: The system had a complete Prometheus + Grafana dashboard from the first day of launch. Many early issues were discovered proactively through metrics, rather than waiting for business teams to report problems.
  • Idempotency is a social problem, not a technical problem: You can provide tools, but ultimately business developers need to understand and take it seriously. Documentation, training, code reviews, and incident retrospectives are all essential.

If I Were to Do It Again

The biggest lesson: model use cases clearly first, then make technology choices.

At the time, we treated "async tasks" as a unified concept for system design, but in reality, different tasks vary enormously in key attributes. If I were to start over, I would first spend enough time with business teams to map out each task type's core attributes:

Attribute Possible Values Design Decisions Affected
Delay type Immediate / Short delay / Long delay / Cron Storage choice (Kafka vs MySQL), scheduling strategy
Priority High (order cancellation) / Medium / Low (reports) Queue isolation strategy, resource allocation
Uniqueness Allow duplicates / Unique at a given time / Globally unique Deduplication mechanism, idempotency requirements
Delivery semantics At-least-once / At-most-once / Exactly-once Acknowledgment mechanism, retry strategy, message persistence
Execution duration Milliseconds / Seconds / Minutes+ Timeout settings, heartbeat detection, resource isolation
Failure handling Retry / Discard / Manual intervention Retry strategy, dead letter queue, alerting rules
Dependencies Independent / Has prerequisites / DAG Task orchestration, state machine

Different combinations of these attributes mean completely different design trade-offs. A few examples:

  • Order timeout cancellation (short delay + high priority + globally unique + at-least-once): Needs a dedicated high-priority queue, the deduplication mechanism must be reliable, and messages absolutely cannot be lost вҖ” Kafka is the right choice, Redis won't do.
  • Statistics report generation (Cron + low priority + allow duplicates + at-most-once): Running twice doesn't matter, but it shouldn't block high-priority tasks вҖ” needs queue isolation rather than sharing with order tasks.
  • Data export (immediate + long execution time + has prerequisites): Needs heartbeat detection to prevent false timeout detection, needs DAG orchestration rather than manual chaining.

Our mistake was cramming these vastly different tasks into a single model. The result: high-priority tasks blocked by low-priority tasks (no queue isolation), tasks requiring uniqueness executed repeatedly (no deduplication mechanism), tasks needing orchestration manually chained by business teams (no DAG support).

Other specific improvements:

  • Introduce a dead letter queue: Tasks that exhaust retries should enter a dead letter queue instead of just being marked as failed. Without this mechanism, some failed tasks were forgotten.
  • Isolate queues by priority: High-priority tasks should use dedicated Kafka topics and consumer groups to avoid being slowed by low-priority task backlogs. This was added later, but if identified during the use case modeling phase, the implementation would have been much cleaner.
  • Task uniqueness constraints: Provide declarative uniqueness configuration (e.g., unique_key = "order:{order_id}"), with the system guaranteeing that tasks with the same key won't be executed repeatedly, rather than relying entirely on business-side idempotency.
  • DAG-style task orchestration: Business frequently needs "trigger task B after task A completes." At the time, business teams manually submitted the next task in their executors, which was error-prone. Declarative DAG orchestration capabilities should have been provided.

In one sentence: technology choices serve use cases, not the other way around. Spend time first to clearly map out the task attribute matrix, think through the trade-offs for each combination, then start writing code. This is far more valuable than immediately agonizing over "Kafka or Redis."
