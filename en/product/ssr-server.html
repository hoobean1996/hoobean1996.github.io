Hand-Rolling an SSR Server: Building Custom Server-Side Rendering for GraphQL Apps

2026-03-08
The value of this article: Next.js is great, but if your tech stack is Relay + GraphQL, its SSR model conflicts with Relay's EntryPoint data preloading philosophy. This article documents the process of hand-rolling an SSR Server from scratch: route matching, data preloading, streaming rendering, Hydration, and how to make Relay's EntryPoint pattern work correctly on the server side. Suitable for teams that need deeply customized SSR solutions.

Background

Next.js is a great tool, but its integration with Relay (Meta's GraphQL client) has always been a bit rough. Relay has its own data preloading philosophy — the EntryPoint pattern. It wants to start fetching data at route matching time, not wait until components render to issue requests. This has a subtle conflict with Next.js's getServerSideProps / Server Components model.

So I decided to write my own SSR server, tailored specifically for the Relay + Fluent UI tech stack. This article documents the key steps and pitfalls encountered in this process.

Overall Architecture

Browser request GET /pages/admin/42?sort=name
    ↓
Express Server
    ↓
Route matching (ssrrouter.lookup)
    ↓ Returns entrypoint name + parameter bindings
EntryPoint resolution
    ↓ Find EntryPoint config by name
GraphQL data preloading (Relay loadQuery)
    ↓
React renderToPipeableStream (streaming SSR)
    ↓
HTML streaming response → Browser
    ↓
hydrateRoot (client takeover)

Step 1: Define the EntryPoint Interface

EntryPoint is the core abstraction of the entire architecture. Each "page" is not a React component, but an EntryPoint object that describes: what component to render, what parameters are needed, and what data to preload.

// EntryPoint.tsx
interface EntryPoint<TParams> {
  root: ComponentType<PreloadedProps>;
  getParams: (path: string) => TParams;
  getPreloadedProps: (params: TParams) => PreloadedProps;
}

interface PreloadedProps {
  queries: {
    [key: string]: {
      query: GraphQLTaggedNode;
      variables: Variables;
    };
  };
}

A concrete EntryPoint looks like this:

// App.entrypoint.tsx
const AppEntryPoint: EntryPoint<Params> = {
  root: App,
  getParams: (path: string): Params => {
    return { limit: 10 };
  },
  getPreloadedProps: (params: Params): PreloadedProps => {
    return {
      queries: {
        preloadedQuery: {
          query: AppQuery,        // Relay GraphQL query
          variables: {},
        },
      },
    };
  },
};

This design comes from the Relay team's EntryPoint pattern. The key idea is: data dependencies are declared at the routing layer, not the component layer. The moment a route matches, you know what data needs to be fetched, without waiting for the React render tree to be built.

Step 2: Route Matching

Route configuration maps URL patterns to EntryPoints:

// routes.ts
export const ssrRouterConfig = [
  "route /      AppEntrypoint",
  "route /admin AdminEntrypoint",
  "route /admin/:id(int) AppEntrypoint",
];

export const ssrEntrypointConfig = new Map<string, EntryPoint<any>>([
  ["AppEntrypoint", AppEntryPoint],
  ["AdminEntrypoint", AdminEntryPoint],
]);

When a request comes in, route matching happens first, then the corresponding EntryPoint is looked up:

const path = req.path.substring(6);  // Strip /pages prefix
const entrypointRoute = ssrrouter.lookup(path);
const entrypoint =
  ssrEntrypointConfig.get(entrypointRoute?.entrypoint ?? "") ??
  FallbackEntryPoint;

Step 3: GraphQL Data Preloading

This is the most critical step of SSR. After obtaining the EntryPoint, call getPreloadedProps to get all required GraphQL queries, then use Relay's loadQuery to issue server-side requests:

const { root: Root, getPreloadedProps, getParams } = entrypoint;
const params = getParams(path);
const { queries } = getPreloadedProps(params);

const preloadedData = Object.keys(queries).reduce((acc, key) => {
  acc[key] = loadQuery(
    RelayEnvironment,    // Server-side Relay Environment
    queries[key].query,
    queries[key].variables
  );
  return acc;
}, {});

loadQuery immediately starts fetching and returns a reference. When React renders to a component that needs this data, if the data has arrived it's used directly, if not it suspends (with Suspense).

Step 4: Streaming SSR Rendering

React 18 introduced renderToPipeableStream, supporting streaming HTML output. Compared to the old renderToString, it has two core advantages:

  • Shorter Time to First Byte: Start sending as soon as the shell is ready, without waiting for all data
  • Suspense support: Parts where data isn't ready send a fallback first, with data streamed in later
const { pipe, abort } = renderToPipeableStream(
  <RelayEnvironmentProvider environment={RelayEnvironment}>
    <Suspense fallback={<div>...</div>}>
      <RendererProvider renderer={renderer}>
        <SSRProvider>
          <FluentProvider theme={webLightTheme}>
            <Root queries={preloadedData} />
          </FluentProvider>
        </SSRProvider>
      </RendererProvider>
    </Suspense>
  </RelayEnvironmentProvider>,
  {
    bootstrapScripts: ["/entrypoint.client.js"],
    onShellReady() {
      res.setHeader("content-type", "text/html");
      res.write(`<!DOCTYPE html>
<html>
<head>...styles...</head>
<body>
  <div id="root">`);
      pipe(res);        // Pipe React render stream to HTTP response
      res.write(`</div>
</body>
</html>`);
    },
    onShellError(error) {
      res.statusCode = 500;
      res.send("Something went wrong");
    },
  }
);

Note the onShellReady callback — it fires when React finishes rendering the synchronous part (shell). At this point, write the HTML header first, then pipe(res) connects the render stream to the HTTP response stream. Async data inside Suspense boundaries is injected later as <script> tags in a streaming fashion.

Step 5: Fluent UI SSR Style Handling

Fluent UI (Microsoft's React component library) uses CSS-in-JS, which requires special handling during SSR to avoid page flashing (FOUC):

const renderer = createDOMRenderer();
const style = renderToStaticMarkup(
  <>{renderToStyleElements(renderer)}</>
);

First create a DOM renderer, collect all used styles during rendering, then inline the styles into HTML's <head>. During client-side hydration, these styles are reused to avoid duplicate injection.

Step 6: Client-Side Hydration

The client needs to do the exact same route matching and data preloading as the server, but with one key difference — fetchPolicy: "store-only":

// Entrypoint.client.tsx
ssrrouter.createRouter(ssrRouterConfig);  // Same routes

const path = window.location.pathname.substring(6);
const entrypointRoute = ssrrouter.lookup(path);
const entrypoint = ssrEntrypointConfig.get(entrypointRoute?.entrypoint ?? "");
const { root: Root, getPreloadedProps, getParams } = entrypoint;
const params = getParams(path);
const { queries } = getPreloadedProps(params);

const preloadedData = Object.keys(queries).reduce((acc, key) => {
  acc[key] = loadQuery(
    RelayEnvironment,
    queries[key].query,
    queries[key].variables,
    { fetchPolicy: "store-only" }  // Only read from store, don't re-fetch
  );
  return acc;
}, {});

hydrateRoot(
  document.getElementById("root"),
  <RelayEnvironmentProvider environment={RelayEnvironment}>
    <Root queries={preloadedData} />
  </RelayEnvironmentProvider>
);

store-only means the client won't re-issue GraphQL requests, instead using data the server already fetched and transmitted through HTML. This avoids duplicate requests during hydration.

Major Challenges

1. Relay Environment Server/Client Differences

Relay's Environment encapsulates the network layer. The server can call the GraphQL server directly (or even call functions directly), while the client needs to go through HTTP. The two sides have completely different Environment configurations, but use the same queries. How to seamlessly transfer server-fetched data to the client was the biggest challenge.

Solution: After the server fetches data, Relay's store is serialized into the HTML (React 18's streaming injection mechanism handles this). The client uses store-only during hydration to read it.

2. Streaming Rendering and Suspense Timing Issues

renderToPipeableStream is asynchronous. When onShellReady fires, only the synchronous portion has been rendered. If all data is behind Suspense boundaries, the shell might be nearly empty. Careful design is needed for what goes outside Suspense (rendered immediately) and what goes inside (can be delayed).

3. CSS-in-JS and Streaming Rendering Conflicts

Fluent UI's styles are dynamically generated during rendering. But streaming rendering means <head> needs to be sent before rendering starts. The solution is to do a "pre-render" first to collect styles, then do the actual streaming render. This means the component tree is actually rendered twice — once for styles, once for HTML.

4. Isomorphic Routing Logic

Server and client must use exactly the same route matching logic, otherwise hydration will fail (client render result doesn't match the server's). This is why the router is compiled to JS — ensuring the same code runs on both sides.

5. Request Abort Handling

Users might leave before the page finishes loading. During streaming rendering, you must listen for request close events and promptly abort rendering:

req.on("close", () => {
  abort();
});

Without handling this, the server would continue rendering a page nobody wants, wasting resources.

Looking Back

This project gave me deep understanding of every SSR step, but honestly, I don't recommend doing this in production. Reasons:

  • Extremely high maintenance cost: React SSR's internal behavior can change even in minor version updates. You need to continuously keep up. The Next.js team works closely with the React team to ensure these details are correct — you can't keep up alone.
  • Too many missing features: Code splitting, prefetching, ISR, static generation, middleware, error boundaries... a production-grade SSR framework needs far more than just "rendering components to HTML."
  • Relay's SSR support is itself incomplete: The Relay team has their own SSR infrastructure internally at Meta. The open-source version has very little SSR documentation and examples. You'll spend a lot of time searching for answers in source code.

But as a learning project, the value is significant:

  • You'll truly understand what renderToPipeableStream does, not just "it's faster than renderToString"
  • You'll understand why hydration mismatches happen, because you hand-wrote the rendering logic for both sides
  • You'll understand why the EntryPoint pattern is more suitable for large applications than getServerSideProps — data dependencies are hoisted to the routing layer, decoupling data fetching from component rendering
  • You'll understand the real pain points of CSS-in-JS in SSR scenarios

If You Want to Try This Too

Suggested steps:

  1. Start with renderToString + Express to build the simplest SSR (no GraphQL, pure static components)
  2. Switch to renderToPipeableStream, understand streaming rendering and onShellReady
  3. Add Suspense, letting some data load asynchronously
  4. Integrate a GraphQL client (Apollo is easier to start with than Relay), handle server-side data preloading
  5. Implement client-side hydration, ensuring both sides render identical results
  6. Handle CSS (start with vanilla CSS, then try CSS-in-JS)

Each step deepens your understanding of SSR. Don't try to do everything at once — that will only leave you lost in hydration mismatch error messages.
