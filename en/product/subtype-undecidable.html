Why Is Subtype Checking Undecidable with Union + Intersection Types?

2026-03-13
The value of this article: If you've used TypeScript, you may have encountered the type checker getting stuck, producing inexplicable errors, or the IDE just spinning. This isn't a bug — it's a fundamental limitation of the type system. When a type system simultaneously supports union (|), intersection (&), function types, and recursive types, subtype checking is theoretically undecidable. This article derives this conclusion in a way engineers can understand: starting from the simplest subtype system, progressively adding union, intersection, function contravariance, and recursive types, showing how each step makes the problem harder, ultimately exceeding what algorithms can handle. Understanding this "why" can help you understand why TypeScript makes certain design compromises, and how to avoid writing types that make the compiler explode.

Problem Definition: Subtype Checking Is a Decision Problem

Let's be clear about the problem. Subtype checking means answering a yes/no question: given types A and B, does A <: B hold?

You do this every day. When you write TypeScript:

function greet(name: string) { ... }
greet(42)  // Compile error: number is not a subtype of string

What the compiler does behind the scenes is subtype checking: number <: string? Doesn't hold, error.

Decidable means there exists an algorithm that, for any input A and B, can give the correct answer in finite steps. Undecidable means no such algorithm exists — for certain inputs, any correct algorithm will either never finish computing or give a wrong answer.

The core question this article answers: why does subtype checking become undecidable after adding union and intersection types?

Baseline: Simple Subtype Systems Are Decidable

First, the simplest case. Types are only base types and function types:

Type ::= number | string | A → B

Subtype rules are straightforward:

number <: number                                // Reflexivity
string <: string                                // Reflexivity
A → B <: C → D   if and only if   C <: A and B <: D   // Function type

The function type rule is worth pausing on: parameters are contravariant, return values are covariant. The intuition is — if you need a function that "eats Cat and produces Animal," then a function that "eats Animal and produces Cat" can substitute, because it can eat more (accepts broader input) and produces something more precise (returns more specific output).

Subtype checking for this system is obviously decidable: each rule application makes types smaller, there are no loops, no branches, solved in linear time. The key is that this process is syntax-directed — the type's structure directly tells you which rule to use, no choices needed.

First Complication: Union and Intersection Introduce Search

Union Type: Disjunction / OR

A | B means "is A or is B." It's everywhere in TypeScript:

type Result = Success | Error
type ID = string | number

Subtype rules:

A <: A | B          // Left injection: A is obviously "A or B"
B <: A | B          // Right injection
A | B <: C  requires  A <: C and B <: C   // Union elimination: both branches must satisfy

But the reverse is tricky — when determining C <: A | B, you can't simply check C <: A or C <: B. Look at this example:

(string | number) <: (string | number)   // ✅ holds
(string | number) <: string              // ❌ doesn't hold
(string | number) <: number              // ❌ also doesn't hold

You need to break apart C's "members" and check that each belongs to either A or B. This is no longer a single-step decision — you need search.

Intersection Type: Conjunction / AND

A & B means "is both A and B simultaneously." Common in TypeScript's mixin pattern:

type Serializable = { serialize(): string }
type Loggable = { log(): void }
type Entity = Serializable & Loggable  // Satisfies both interfaces

Subtype rules:

A & B <: A          // Left projection
A & B <: B          // Right projection
C <: A & B  requires  C <: A and C <: B   // Intersection introduction

The reverse direction has the same problem: A & B <: C can't be simply split into A <: C or B <: C.

First Key Change: Subtype Checking Becomes a Search Problem

In the simple system before, each derivation step's rule was uniquely determined. Now with union and intersection, choices must be made:

  • When determining C <: A | B, you may need to "guess" which part of C goes to A and which to B
  • When determining A & B <: C, you may need to "choose" whether to use A's information or B's

Subtype checking goes from "deterministic computation" to "search problem." But mere search isn't enough — NP problems are also search problems but still decidable (just slow). Reaching undecidability requires more.

The Critical Step: Function Contravariance Flips Union and Intersection

This is the most critical step in the entire derivation chain. Function type parameters are contravariant, and when this property collides with union/intersection, it produces an important set of equivalences (distributive laws):

  Distributive laws (function parameter position = contravariant position):

  (A | B) → C   ≡   (A → C) & (B → C)     ① union through parameter → becomes intersection
  (A & B) → C   ≡   (A → C) | (B → C)     ② intersection through parameter → becomes union

  Distributive laws (function return position = covariant position):

  A → (B | C)   ≡   (A → B) | (A → C)     ③ union stays union
  A → (B & C)   ≡   (A → B) & (A → C)     ④ intersection stays intersection

Equations 1 and 2 are the killer. They mean: each time you pass through a function type's parameter position, union and intersection flip once.

Engineering intuition for equation 1: A function accepting string | number is essentially a function that can handle both string and number — it's equivalent to "being both string → C and number → C." So union in the parameter position becomes intersection overall.

This causes the search nature of subtype determination to keep flipping:

  Level 0: Need to verify an AND condition (intersection)
     │
     ▼  Pass through one layer of function parameter
  Level 1: Becomes an OR choice (union)
     │
     ▼  Pass through another layer of function parameter
  Level 2: Flips back to AND condition (intersection)
     │
     ▼
    ...alternating

This is no longer simple SAT-style search (pure OR branches), but an AND-OR alternating search — similar to a game tree (I make a move, opponent makes a move, I make another move...). Game tree determination is much harder than SAT.

Adding Fuel: Recursive Types Provide Infinite Expansion

So far, if all types are finite-sized (no recursion), the search tree is complex but bounded in depth — still decidable (possibly EXPTIME or higher, but will always terminate).

Recursive types break this limitation. They allow types to reference themselves:

// Recursive types in TypeScript
type Json = string | number | boolean | null | Json[] | { [key: string]: Json }

// Formal notation
type List = μX. Nil | Cons(number, X)    // Fixed point of X = Nil | Cons(number, X)

Under equi-recursive semantics (the most common interpretation), a recursive type equals its one-step expansion:

μX. F(X)  =  F(μX. F(X))    // Can expand infinitely

This means types can grow infinitely. Combined with the earlier conclusion:

  1. Each pass through a function parameter position flips union/intersection once
  2. Recursive types can expand infinitely, continuously producing new function layers
  3. Each expansion introduces new union/intersection branches
  4. The AND/OR nature of branches alternates with expansion depth
  5. There is no natural termination condition

A concrete example:

type T = μX. (X → X) | number

// Expand once:
T = (T → T) | number

// Expand T → T's T:
T = (((T → T) | number) → ((T → T) | number)) | number

// Each expansion produces new unions, passing through → flips to intersection, continues...

A subtype checking algorithm facing this type can infinitely expand, flip, and branch. Any algorithm attempting to decide A <: B may run forever on such recursive structures.

The Ultimate Question: Why Can It Encode a Turing Machine

The previous four steps gave us all the parts. Now let's see how they assemble into Turing-complete computation.

What a Turing Machine Needs vs. What the Type System Provides

Encoding Intuition

The core idea is constructing two recursive types A and B such that A <: B holds if and only if a particular Turing machine halts.

  • Encode current symbol with union branches: Symbol = Zero | One, different branches represent different values on the tape
  • Encode tape continuation with recursion: Tape = μX. Symbol & (Unit → X), recursively representing an infinite tape
  • Encode "simultaneously satisfying multiple conditions" with intersection: State transitions need to check both current symbol and current state
  • Encode read/write with function contravariance: Reading is in the contravariant position (input), writing is in the covariant position (output)

Each time subtype checking expands one step of a recursive type, it's equivalent to the Turing machine executing one step. If the Turing machine doesn't halt, subtype checking never finishes.

Standard Reduction Path

Classic proofs typically use a two-step reduction:

  Halting problem (known undecidable)
     │
     │  Reduces to
     ▼
  Post Correspondence Problem (PCP)
  Given string pairs (u₁,v₁), ..., (uₙ,vₙ)
  Does there exist a sequence i₁,...,iₖ such that
  u_{i₁}...u_{iₖ} = v_{i₁}...v_{iₖ} ?
     │
     │  Reduces to
     ▼
  Subtype checking problem
  A <: B ?

PCP to subtype checking encoding:

  • Each PCP string pair → a pair of type constructors
  • String concatenation → type nesting/composition (using recursion)
  • String "equality" → subtype relationship
  • "There exists a sequence" → union type choice

Validating the Intuition: What Happens If You Remove Any Component?

If the analysis above is correct, then removing any key component should make the problem decidable again. Indeed it does:

ConfigurationDecidable?Reason
Only Union, no IntersectionDecidableWithout intersection, function parameter position doesn't flip union into intersection, AND-OR alternation doesn't occur
Only Intersection, no UnionGenerally decidableEasier to control, though some combinations (like bounded quantification) may still cause problems
Union + Intersection, no recursionDecidableAll types are finite, search tree has bounded depth. May be EXPTIME, but always terminates
Union + Intersection + recursion, no function typesDecidableNo contravariant positions, union and intersection don't flip each other, degenerates to set containment checking

Conclusion: Undecidability requires all four components working together — remove any one and it breaks.

  Union (provides OR branches)
    + Intersection (provides AND branches)
    + Function type contravariance (makes OR ↔ AND flip)
    + Recursive types (makes this process infinite)
    ────────────────────────────────
    = Can encode Turing machines → Undecidable

Complete Causal Chain

  WHY-0: The halting problem is undecidable
            ↓
  WHY-1: Any decision problem in a formal system that can encode Turing machines is undecidable
            ↓
  WHY-2: Union + Intersection + Function types + Recursive types can encode Turing machines
            ↓ because
  WHY-3: Union encodes OR choice, Intersection encodes AND conjunction
            ↓ plus
  WHY-4: Function parameter contravariance makes union ↔ intersection flip
            ↓ causing
  WHY-5: Subtype checking becomes AND-OR alternating search
            ↓ plus
  WHY-6: Recursive types make the expansion process have no termination point
            ↓ ultimately
  WHY-7: Subtype checking can encode arbitrary Turing machines → Undecidable

How Do Real Type Systems Cope?

Theoretically undecidable, but TypeScript, Flow, and Scala all work just fine in daily use. How do they manage? Essentially by giving up "completeness" or "full recursion" in exchange for decidability.

Strategy 1: Incomplete but Sound (TypeScript's Approach)

TypeScript's subtype checking algorithm is sound but incomplete — it won't accept incorrect subtype relationships (sound), but may reject some subtype relationships that actually hold (incomplete).

// TypeScript's subtype checking can handle some cases
type A = { x: string } & { y: number }
type B = { x: string, y: number }
// A <: B and B <: A, TypeScript handles this

// But for deeply nested recursive + union + intersection combinations
// the compiler may just error out or time out

In practice: if you've done complex TypeScript type gymnastics (conditional types + infer + recursion), you may have seen Type instantiation is excessively deep and possibly infinite. This is TypeScript's depth limit in action — when recursive expansion exceeds a certain depth (typically 50 levels), it simply stops.

Strategy 2: Semantic Subtyping (CDuce, Ballerina's Approach)

Interpret types as sets of values, with subtyping being set containment. Decidability is recovered by restricting the form of recursive types:

  • Contractive condition: The recursive body can't be just the variable itself (μX. X is illegal), it must pass through at least one type constructor
  • This restriction blocks "meaningless infinite expansion" while preserving useful recursive types (lists, trees, etc.)

Strategy 3: Restrict Where Union/Intersection Can Appear

Some type systems only allow union/intersection in specific positions — for example, only at the top level, not nested inside function parameters. This directly blocks the contravariance flipping chain.

Strategy 4: Iso-recursive Instead of Equi-recursive

Iso-recursive types require explicit fold/unfold operations to expand recursion. Each expansion is actively triggered by the programmer, rather than the type checker automatically expanding infinitely. This limits expansion depth.

Strategy 5: MLsub's Clever Restriction

Dolan and Mycroft's 2017 MLsub system demonstrated a clever approach: by restricting union and intersection to only appear in specific "polarity" positions — union only in covariant positions, intersection only in contravariant positions — completely eliminates the possibility of AND-OR alternation, yielding a decidable and efficient system.

Implications for Engineering Practice

Understanding the root cause of undecidability can guide us to write more "friendly" types:

  • Avoid nesting union + intersection + recursion in function parameter positions. This is the precise condition that triggers AND-OR alternating search. Extract complex types into named type aliases to reduce nesting depth
  • Keep recursive types simple. type Tree = Leaf | Node(Tree, Tree) is safe. type T = (T → T) | number is dangerous — because T in the function parameter position triggers contravariance flipping
  • When you see "excessively deep" errors, don't just add // @ts-ignore. Re-examine your type definitions, looking for whether union/intersection is constantly flipping through function parameter positions within recursion
  • Type gymnastics in moderation. TypeScript's conditional type + infer is essentially performing recursive search over union/intersection. Each additional nesting level can exponentially expand the search space
