Schema-Driven Forms (Part 2): Expression Engine and Generic UI Rendering

2026-03-08
The value of this article: The previous article solved the problem of "describing forms with JSON," but forms are only a subset of UI. This article extends Schema capabilities to describe entire interfaces — layout, conditional rendering, expression engine, action system. The end result is a generic SchemaRenderer: give it a JSON, and it can render a complete page with conditional logic and interactive behavior. This approach is suitable for scenarios that need "low-code page configuration."

From Forms to Interfaces

The previous article discussed using JSON Schema to drive form rendering — data model, renderer, validation strategy. But if you think about it, forms are just a subset of UI. A form contains more than just input fields — there are layouts (columns, cards, tab switching), display components (text, images, badges), and interaction logic (clicking a button triggers an action, showing/hiding fields based on selections).

If we extend the Schema's capability from "describing form fields" to "describing the entire UI tree," we get a more powerful system: Schema-driven UI. A JSON document defines not just data input, but also layout, conditional rendering, validation rules, and interaction actions — the frontend needs only a generic SchemaRenderer to render complete pages.

Core Type Design

The entire system's type definitions:

// Complete description of a component
interface ComponentSchema {
  type: string;              // Component type: "Input" | "Card" | "Row" | ...
  name?: string;             // Field name (only for form components)
  props?: Record<string, unknown>;  // Props passed to the component
  validation?: ValidationSchema;     // Validation rules
  conditional?: ConditionalSchema;   // Conditional rendering
  actions?: Record<string, ActionSchema>;  // Event handlers
  children?: ComponentSchema[];      // Child components (for layout containers)
}

// Top-level Schema
interface FormSchema {
  id: string;
  title?: string;
  description?: string;
  form?: {
    initialValues?: Record<string, unknown>;  // Default values
    values?: Record<string, unknown>;         // Pre-filled values (highest priority)
    validateOn?: "change" | "blur" | "submit";
  };
  components: ComponentSchema[];  // Component tree
}

Key design decision: ComponentSchema is recursive. A Card's children can contain Row, Row can contain Input and Button. This means the Schema can describe UI trees of arbitrary depth, not limited to flat field lists.

This JSON describes: layout (Card wrapper, Row columns), form fields (Input, DatePicker, Selector, RadioGroup), validation rules (name length >= 2), conditional rendering (show company name only when "Business" is selected), and interaction actions (button submit). The frontend doesn't need to write any component code, just a generic renderer.

Expression Engine

The core of conditional rendering and validation rules is an expression engine. String expressions in the Schema (like @accountType == 'business') need to be evaluated at runtime.

The @ prefix references form field current values. Expressions support comparison, logical operations, and built-in functions:

// Validation rule examples
"LEN(@name) >= 2"                              // String length
"@age >= 18 AND @age <= 120"                   // Range validation
"LEN(@email) > 5 AND FIND(\"@\", @email) > 0" // Email format

// Conditional rendering examples
"@accountType == 'business'"                    // Equality check
"@role == 'admin' OR @role == 'super_admin'"   // Multiple conditions
"@items > 0"                                    // Numeric comparison

The expression engine implementation doesn't need to be very complex. A simple recursive descent parser is sufficient — tokenize the expression string, parse into an AST, then recursively evaluate. The key is: the expression evaluation context must stay synchronized with form state. Every time form values change, the expression engine can access the latest values for re-evaluation.

The implementation uses React Context to inject form values into the expression engine:

function FormContent({ schema, registry }) {
  const formContext = useFormSchema();
  const evalContext = useLibExprEvaluation();

  // Sync to expression engine when form values change
  useEffect(() => {
    evalContext.setValues(formContext.values);
  }, [formContext.values]);

  return (
    <form>
      {schema.components.map((comp, i) => (
        <SchemaField key={comp.name || i} schema={comp} registry={registry} />
      ))}
    </form>
  );
}

Conditional Rendering

Conditional rendering Schema definition:

interface ConditionalSchema {
  when: string;                     // Expression, determines visibility
  then?: Partial<ComponentSchema>;  // Props to merge when condition is true
  else?: Partial<ComponentSchema>;  // Props to merge when condition is false
}

Not just show/hide control — it can also modify component props based on conditions. For example:

{
  "type": "Input",
  "name": "budget",
  "props": { "label": "Budget" },
  "conditional": {
    "when": "@role == 'manager'",
    "then": { "props": { "label": "Department Budget (Admin)" } },
    "else": { "props": { "disabled": true } }
  }
}

The renderer implementation is straightforward:

function SchemaField({ schema, registry }) {
  const evalContext = useLibExprEvaluation();
  const { conditional } = schema;

  const isVisible = useMemo(() => {
    if (!conditional?.when) return true;
    try {
      return evalContext.validate(conditional.when);
    } catch {
      return true;  // Default to showing on expression error
    }
  }, [conditional?.when, evalContext]);

  if (!isVisible) return null;

  // ... render component
}

Component Registry

How does type: "Input" in the Schema become an actual React component? Through a component registry:

interface ComponentRegistry {
  [key: string]: React.ComponentType<any>;
}

const defaultRegistry: ComponentRegistry = {
  // Form inputs
  Input: XDSInput,
  TextArea: XDSTextArea,
  NumberInput: XDSNumberInput,
  Checkbox: XDSCheckbox,
  RadioGroup: XDSRadioGroup,
  Selector: XDSSelector,
  Switch: XDSSwitch,
  DatePicker: XDSDatePicker,
  TimePicker: XDSTimePicker,
  FileUpload: XDSFileUpload,
  Slider: XDSSlider,
  Rating: XDSRating,
  ColorPicker: XDSColorPicker,

  // Layout
  Card: XDSCard,
  Row: XDSFlexbox,     // direction: row
  Column: XDSFlexbox,  // direction: column
  Tabs: XDSTabs,
  Accordion: XDSAccordion,
  Divider: XDSDivider,

  // Display
  Text: XDSText,
  Badge: XDSBadge,
  Alert: XDSAlert,
  Image: XDSImage,
  Table: XDSTable,

  // Interactive
  Button: XDSButton,
  Dropdown: XDSDropdown,
};

Users can inject custom components to override defaults or extend with new types:

<SchemaRenderer
  schema={schema}
  components={{
    RichTextEditor: MyRichTextEditor,  // New addition
    Input: MyCustomInput,              // Override default
  }}
/>

The beauty of this design: Schema and component implementation are completely decoupled. The same Schema can be rendered with different component libraries — desktop with Ant Design's registry, mobile with React Native's registry, or even rendered as Flutter Widgets.

Action System

Forms don't just collect data — they have interactions. Action Schema describes user-triggered behaviors:

type ActionType = "submit" | "setValue" | "navigate" | "toast" | "api" | "custom";

interface ActionSchema {
  type: ActionType;
  field?: string;    // setValue: target field
  value?: unknown;   // setValue: new value
  to?: string;       // navigate: target path
  message?: string;  // toast: message content
  url?: string;      // api: request URL
  method?: string;   // api: HTTP method
  handler?: string;  // custom: handler function name
}

Used in Schema like this:

// Click button to submit form
{
  "type": "Button",
  "props": { "label": "Submit" },
  "actions": {
    "onClick": { "type": "submit" }
  }
}

// Click button to set another field's value
{
  "type": "Button",
  "props": { "label": "Fill Default Address" },
  "actions": {
    "onClick": {
      "type": "setValue",
      "field": "address",
      "value": "123 Main Street"
    }
  }
}

// Call custom handler function
{
  "type": "Button",
  "props": { "label": "Export PDF" },
  "actions": {
    "onClick": {
      "type": "custom",
      "handler": "exportPDF"
    }
  }
}

The renderer dispatches to corresponding handler logic based on Action type:

const buildActionHandler = (action: ActionSchema) => () => {
  switch (action.type) {
    case "setValue":
      formContext.setValue(action.field, action.value);
      break;
    case "submit":
      formContext.submit();
      break;
    case "custom":
      actionHandlers?.[action.handler]?.(action);
      break;
  }
};

State Management and Persistence

Form state management is implemented with React Context, providing a set of hooks:

// Complete form context
const { values, setValue, reset, isDirty, submit, getSchema } = useFormSchema();

// Single field binding
const [email, setEmail, { touched, setTouched }] = useFormField<string>("email");

// Read-only values
const values = useFormValues();

// Watch specific field changes
const { name, age } = useWatchFields(["name", "age"]);

State persistence is a clever design: the getSchema() method returns the current Schema with user-filled values embedded in form.values. Save this JSON, and next time the form automatically restores to its previous state:

// Save current progress
const schemaWithValues = getSchema();
localStorage.setItem("draft", JSON.stringify(schemaWithValues));

// Restore progress — just render the saved Schema, form auto-fills
const saved = JSON.parse(localStorage.getItem("draft"));
<SchemaRenderer schema={saved} />

The value priority design is also deliberate: form.values (pre-fill/restore) > props.initialValues (externally provided) > form.initialValues (Schema defaults). This ensures persistence restoration always overrides defaults.

Dynamic Expressions in Props

Props in the Schema can contain not just static values but also expression interpolation:

{
  "type": "Text",
  "props": {
    "content": "${@name ? 'Hello, ' + @name + '!' : 'Please enter your name'}"
  }
}

The renderer recursively traverses props, calling the expression engine when encountering ${...}:

function evaluateProps(props, evaluate) {
  const result = {};
  for (const [key, value] of Object.entries(props)) {
    if (typeof value === "string" && value.includes("${")) {
      result[key] = safeEvaluate(value, evaluate);
    } else if (typeof value === "object" && value !== null) {
      result[key] = evaluateProps(value, evaluate);  // Recursive
    } else {
      result[key] = value;
    }
  }
  return result;
}

This gives the Schema "data binding" capability — display component content can change in real-time following form input, without writing any JavaScript.

The Renderer's Switch Strategy

Different component types need different props binding approaches. This is the "dirtiest" but most critical part of the renderer:

// Input uses onValueChange
case "Input":
  componentProps.onValueChange = handleChange;
  break;

// TextArea uses event-based onChange
case "TextArea":
  componentProps.onChange = (e) => handleChange(e.target.value);
  break;

// Checkbox/Switch uses checked + event
case "Checkbox":
case "Switch":
  componentProps.checked = Boolean(value);
  componentProps.onChange = (e) => handleChange(e.target.checked);
  break;

// DatePicker needs string ↔ Date conversion
case "DatePicker":
  componentProps.value = toDate(value);
  componentProps.onChange = handleChange;
  break;

// Layout components don't need value/name
case "Card":
case "Row":
case "Divider":
  delete componentProps.value;
  delete componentProps.name;
  break;

This switch is an unavoidable "adaptation layer." Every UI component library has different onChange conventions — some pass value, some pass event, some pass Date objects. The Schema renderer's job is to smooth over these differences, so Schema authors don't need to worry about underlying component API details.

Selection Mode: Paving the Way for Visual Editors

If you want to build a drag-and-drop form builder, you need users to be able to click on components in the page to select and edit their properties. This requires a "selection mode":

<SchemaRenderer
  schema={schema}
  selectionMode={true}
  onComponentSelect={(info) => {
    console.log(info.path);  // "components[0].children[2]"
    console.log(info.type);  // "Input"
    console.log(info.name);  // "email"
  }}
/>

In selection mode, each component is wrapped with a hover overlay. Clicking returns the component's path in the Schema tree. The editor can use this path to locate the Schema JSON, modify props, and re-render.

This forms a complete loop:

Visual editor
    ↓ Drag / Configure
FormSchema (JSON)
    ↓ SchemaRenderer
Rendered UI
    ↓ User interaction
FormResponse (JSON)
    ↓ getSchema()
FormSchema with state (persistable)

Boundaries of This Approach

  • Suitable for: Configuration-driven forms, survey systems, low-code platforms, approval flow forms, CMS content editors — these scenarios where UI structure is "determined at runtime," not hardcoded at compile time.
  • Not suitable for: Highly custom interactions (drag-and-drop sorting, canvas editing, complex animations), performance-sensitive scenarios (100+ components with real-time updates), pages requiring extensive custom logic. In these scenarios, Schema abstraction becomes a limitation — you'll constantly add escape hatches to the Schema, until the Schema becomes more complex than writing code directly.
  • Complexity ceiling: When conditional rendering rules exceed 10 or expression nesting exceeds 3 levels, JSON Schema readability degrades sharply. At this point, a good visual editor is needed for assistance, rather than having people hand-write JSON.

This article's approach sits between "form Schema" and "low-code platform." It's more powerful than react-jsonschema-form (supports layout, conditional rendering, actions) but lighter than Retool (doesn't involve data source connections, API orchestration, state machines). This positioning suits scenarios that "need non-developers to configure UI, but don't need a full low-code platform."

Summary

The core formula of Schema-driven UI:

ComponentSchema (recursive component tree description)
  + ComponentRegistry (type → component mapping)
  + Expression Engine (dynamic evaluation)
  + FormContext (state management + persistence)
  = SchemaRenderer (generic renderer)

The four parts are independently modular, and combined they can render UI of arbitrary complexity. Schema is responsible for "what," Registry for "what it looks like," Expression Engine for "when," and FormContext for "what the current state is."

This isn't a new idea — from WinForms' XML descriptions to SwiftUI's declarative syntax, "describing UI with data" is a recurring theme. JSON Schema-driven UI is just a concrete instance of this idea in web frontend, particularly B2B products. Its value isn't in how technically advanced it is, but in: it transfers UI creation authority from developers to business users.
