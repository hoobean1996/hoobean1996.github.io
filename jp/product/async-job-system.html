ゼロから構築する分散非同期タスクキュー：Shopee での実践経験

2026-03-08
この記事の価値：一定規模に達したすべてのバックエンドシステムは非同期タスクを必要とする——遅延実行、定時スケジューリング、失敗時のリトライ。オープンソースソリューション（Celery、Bull）は往々にして重すぎるか柔軟性に欠ける。この記事では、Shopee でゼロから分散非同期タスクキューを構築した完全な経験を共有する：ストレージモデル、スケジューリング戦略、失敗リトライ、優先度キュー、そして本番環境で踏んだ落とし穴について。

なぜどの企業も非同期タスクの車輪を再発明するのか

ほぼすべての一定規模に達したバックエンドシステムは同じ問題に直面する：ユーザーリクエスト内で同期的に完了できないことがある。

ユーザーが注文した後、30 分後に支払い済みかチェックし、未払いならキャンセルする必要がある。ユーザー登録後、非同期でウェルカムメールを送る必要がある。毎日深夜に照合処理を実行する必要がある。毎時間、外部データを同期する必要がある。これらの「後でやる」タスクが、非同期タスクである。

問題は、「後でやる」は聞こえは簡単だが、実装は全く簡単ではないということ：

  • 信頼性：サービスが再起動したら、タスクはまだあるか？ネットワークが揺れたら、タスクは失われるか？
  • 遅延制御：「30 分後に実行」——誰がタイマーを管理する？精度はどれくらい？
  • 並行制御：10 インスタンスが同時に実行される場合、同一タスクが 2 回実行されないようにするには？
  • 失敗処理：タスク実行が失敗したら、何回リトライする？リトライ間隔は？いつ諦める？
  • 可観測性：現在キューに何個のタスクがある？平均遅延は？どのタスクが失敗した？

これが Celery、Sidekiq、Bull、Temporal などのプロジェクトが存在する理由——すべて同じ問題を解決している。しかし大企業では、汎用ソリューションは往々にして不十分：社内 RPC フレームワークとの統合、会社のデプロイプラットフォームへの適応、既存の監視体系との連携が必要。だからほぼすべての一定規模の企業が自前で作ることになる。

Shopee の状況

2019 年に Shopee のプラットフォームチームに加わり、業務開発チーム向けのミドルウェアサービスを担当した。当時、各事業部門はそれぞれの「非同期タスク」ソリューションを持っていた——crontab + スクリプトを使う人、無限ループでデータベースをポーリングする人、ビジネスコード内で goroutine を起動して失われないよう祈る人。

問題は明白：統一的なタスク管理なし、監視なし、リトライなし、タスクが失われても誰も気づかない。さらに致命的なことに、各チームが同じ落とし穴を繰り返し踏んでいた——リトライに冪等性がなく二重課金、crontab マシンがダウンしても誰も気づかず照合が 1 週間中断。プラットフォームチームはゼロから分散非同期タスクキューシステムを構築し、標準化された非同期タスク機能を提供することを決定した。

この記事ではこのシステムの設計から本番投入までの完全なプロセスを記録する。3 年間で 0 から 10 以上の事業部門が接続、QPS は 2K から 25K 以上に成長、ダブルイレブンのピーク時は 50K に達した。

要件分析

各事業チームとの議論の結果、非同期タスクの要件は大きく 3 つに分類される：

タイプ シナリオ例 遅延要件
短遅延タスク 注文タイムアウトキャンセル、クーポン期限切れ、プッシュ通知 秒レベル～分レベル
長遅延タスク T+1 照合、週次レポート生成、データアーカイブ 時間レベル～日レベル
Cron 定時タスク 毎日深夜のバッチ処理、毎時間のデータ同期、ヘルスチェック cron 式に従う

機能要件に加えて、いくつかの必須制約がある：

  • 少なくとも 1 回の配信：タスクは重複実行可能（業務側で冪等性を担保）だが、絶対に失われてはならない
  • 多言語対応：事業チームは Go と Python を使用しており、2 つの SDK を提供する必要がある
  • 社内 RPC プラットフォームとの統合：タスクトリガーは社内 RPC フレームワークを通じて業務サービスを呼び出し、追加のネットワークプロトコルを導入しない
  • 可観測性：各タスクのステータス、所要時間、リトライ回数がすべて確認可能であること

なぜ Kafka + MySQL であって、純粋な Kafka や純粋な Redis ではないのか

これは設計段階で最も議論された決定である。

純粋な Kafka 方式の問題：Kafka は任意の遅延配信をサポートしない。複数の topic で異なる遅延レベルを模擬できる（1s/5s/30s/1min/...）が、粒度が粗く、topic 数が爆発する。「3 日後にトリガー」のような長遅延には Kafka は全く適さない——メッセージが broker に蓄積され、リソースの無駄であり rebalance 時のリスクがある。

純粋な Redis 方式（Sorted Set + ZRANGEBYSCORE）：遅延タスクの score にトリガー時刻を格納し、定期的に到期タスクをスキャン。問題は Redis の永続化が十分に信頼できないこと——RDB には時間窓があり、AOF は高 QPS 時にパフォーマンスが低下する。「少なくとも 1 回の配信」という必須要件に対して、Redis を唯一のストレージとするのはリスクが高すぎる。

最終方式：Kafka が即時および短遅延タスク（10 分以内）を処理し、MySQL が長遅延と Cron タスクを処理する。

  • Kafka：高スループットで、コンシューマーグループと partition レベルの並列処理を自然にサポート。短遅延タスクは消費後にメモリ内の ETA 優先キュー（最小ヒープ）に入れ、到期後にトリガーする。
  • MySQL：ACID がタスクの紛失を防ぐ。長遅延タスクはトリガー時刻を付けて書き込み、Dispatcher が毎秒到期タスクをスキャン。Cron タスクは Scheduler が事前に次回トリガー時刻を計算して書き込む。

短遅延タスク：ETA Priority Queue

短遅延タスクは Kafka から消費後、直接 sleep で待つのではなく、最小ヒープベースの優先キューに入れ、ETA（Expected Time of Arrival、予定トリガー時刻）でソートする：

// ETA Priority Queue —— container/heap ベース
type Task struct {
    ID        string
    Type      string
    Payload   string
    ETA       time.Time  // 予定トリガー時刻
    Priority  int        // 業務優先度（同一 ETA の tiebreak に使用）
    index     int        // heap 内部インデックス
}

type ETAQueue []*Task

func (q ETAQueue) Len() int { return len(q) }
func (q ETAQueue) Less(i, j int) bool {
    if q[i].ETA.Equal(q[j].ETA) {
        return q[i].Priority > q[j].Priority  // 同一 ETA なら高優先度が先
    }
    return q[i].ETA.Before(q[j].ETA)
}
func (q ETAQueue) Swap(i, j int) {
    q[i], q[j] = q[j], q[i]
    q[i].index = i
    q[j].index = j
}
func (q *ETAQueue) Push(x any) {
    t := x.(*Task)
    t.index = len(*q)
    *q = append(*q, t)
}
func (q *ETAQueue) Pop() any {
    old := *q
    n := len(old)
    t := old[n-1]
    old[n-1] = nil
    t.index = -1
    *q = old[:n-1]
    return t
}

Dispatcher のスケジューリングループ：

func (d *Dispatcher) Run() {
    pq := &ETAQueue{}
    heap.Init(pq)

    for {
        // 1. Kafka から新しいタスクを消費し、ヒープに追加
        for _, task := range d.consumeBatch() {
            heap.Push(pq, task)
        }

        // 2. 到期したすべてのタスクをポップ
        now := time.Now()
        for pq.Len() > 0 && (*pq)[0].ETA.Before(now) {
            task := heap.Pop(pq).(*Task)
            go d.dispatch(task)  // 非同期実行
        }

        // 3. 次の起動時刻を計算：ヒープ先頭の ETA またはフォールバック 100ms
        var sleepUntil time.Duration
        if pq.Len() > 0 {
            sleepUntil = time.Until((*pq)[0].ETA)
            if sleepUntil < 0 {
                sleepUntil = 0
            }
        } else {
            sleepUntil = 100 * time.Millisecond
        }
        time.Sleep(sleepUntil)
    }
}

なぜ Timer Wheel ではなくヒープを選んだか：

  • 意味が直感的：各タスクに ETA があり、ヒープの先頭は常に最も近くトリガーされるタスク。コードロジックが明確で、cascade や tick の概念を理解する必要がない。
  • 精度が柔軟：Timer Wheel の精度は tick 粒度（例えば 100ms 1 目盛り）に制限されるが、ヒープにはその制限がない——ETA はナノ秒精度。
  • 優先度を自然にサポート：Less 関数で ETA と業務優先度を同時に比較できる。同じ秒内に到期するタスクは、高優先度のものが先に実行される。
  • O(log n) で十分：ヒープの挿入と取り出しは O(log n) で、我々のシナリオ（ヒープ内には通常数千～数万のタスク）ではパフォーマンスは全くボトルネックにならない。Timer Wheel の O(1) の優位性は百万レベルのタスクでのみ現れる。
  • Go 標準ライブラリに含まれる：container/heap が完全なインターフェースを提供しており、追加の依存関係は不要。

長遅延タスク：MySQL Poller

長遅延タスクは MySQL に保存される。コアテーブル構造：

CREATE TABLE async_tasks (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    task_type   VARCHAR(128) NOT NULL,
    payload     TEXT NOT NULL,
    trigger_at  DATETIME NOT NULL,    -- トリガー時刻
    status      TINYINT DEFAULT 0,    -- 0=pending, 1=dispatched, 2=success, 3=failed
    retry_count INT DEFAULT 0,
    max_retries INT DEFAULT 3,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_trigger (status, trigger_at)
);

Dispatcher の poller は毎秒実行する：

SELECT id, task_type, payload
FROM async_tasks
WHERE status = 0 AND trigger_at <= NOW()
ORDER BY trigger_at
LIMIT 1000
FOR UPDATE SKIP LOCKED;

FOR UPDATE SKIP LOCKED がポイント——複数の Dispatcher インスタンスが並行してスキャンする際、他のインスタンスがロック中の行はスキップされ、分散ロックの複雑さを回避する。これは MySQL 8.0 で導入された機能で、このシナリオに完璧に適合する。

スキャンされたタスクは直ちにステータスを dispatched に更新し、RPC 経由で業務サービスのエグゼキュータを呼び出す。RPC が失敗またはタイムアウトした場合、タスクステータスは pending に戻り、retry_count が 1 増加し、trigger_at は現在時刻にバックオフ遅延（指数バックオフ + ジッター）を加えた値に設定される。

Cron 定時タスク

Cron タスクの実装は 2 層に分かれる：

  1. Cron Scheduler：cron 式を解析し、次のトリガー時刻を計算し、MySQL の async_tasks テーブルに書き込む。本質的には Cron タスクを長遅延タスクに変換する。
  2. Cron Registry：cron ルールのメタ情報（cron 式、関連する task_type、担当者など）を保存する。
// Cron Scheduler コアループ
func (s *CronScheduler) Run() {
    for {
        rules := s.loadActiveRules()  // cron_registry から読み込み
        for _, rule := range rules {
            next := rule.NextTriggerTime()
            if !s.taskExistsForTime(rule.TaskType, next) {
                s.createTask(rule.TaskType, rule.Payload, next)
            }
        }
        time.Sleep(10 * time.Second)  // 10 秒ごとにチェック
    }
}

この設計の利点は Cron Scheduler がステートレスであること——「次のトリガーのタスクが MySQL に存在すること」を保証するだけ。Scheduler がダウンして再起動しても、冪等であるため（存在チェック後に作成）、タスクが失われることはない。

SDK 設計

Go SDK と Python SDK は事業チームが直接使用するインターフェース。設計原則：可能な限りシンプルに、内部の複雑さを隠す。

// Go SDK - タスク送信
client := asyncd.NewClient(asyncd.WithServiceName("order-service"))

// 即時タスク
err := client.Submit(ctx, &asyncd.Task{
    Type:    "cancel_expired_order",
    Payload: `{"order_id": 12345}`,
})

// 遅延タスク
err := client.SubmitDelayed(ctx, &asyncd.Task{
    Type:    "send_review_reminder",
    Payload: `{"user_id": 67890}`,
}, 3 * 24 * time.Hour)  // 3 日後

// エグゼキュータ登録
client.RegisterExecutor("cancel_expired_order", func(ctx context.Context, payload string) error {
    var req CancelOrderRequest
    json.Unmarshal([]byte(payload), &req)
    return orderService.Cancel(ctx, req.OrderID)
})

SDK 内部で行っていること：

  • シリアライズ：payload は統一的に JSON 文字列を使用し、クロス言語のシリアライズ互換性問題を回避
  • RPC 統合：社内 RPC フレームワーク経由でサービスを登録し、Dispatcher がエグゼキュータを呼び出す際は標準 RPC パス（サービスディスカバリ、ロードバランシング、タイムアウト制御はすべてフレームワークが処理）を通る
  • 冪等性の注意喚起：SDK ドキュメントとコードレビューで繰り返し強調——エグゼキュータは冪等でなければならない。「少なくとも 1 回の配信」は同じタスクが複数回実行される可能性があることを意味する

Kubernetes Operator

システムは Kubernetes 上にデプロイされている。最初は標準的な Deployment + ConfigMap を使用したが、運用上の痛みがすぐに露呈した：

  • 新しい事業の接続には手動で Kafka topic を作成し、ルーティング設定を更新し、consumer 数を調整する必要がある
  • Dispatcher の partition 割り当てと MySQL poller の並行度は QPS に応じて動的に調整する必要がある
  • Cron ルールの変更には Scheduler の再起動が必要

そこで Custom Operator を開発し、CRD を導入した：

apiVersion: asyncd.example.io/v1
kind: AsyncJobConfig
metadata:
  name: order-cancel-job
spec:
  taskType: cancel_expired_order
  executor:
    service: order-service
    method: CancelExpiredOrder
    timeout: 30s
  retry:
    maxRetries: 3
    backoff: exponential
  kafka:
    topic: asyncd-order-cancel
    partitions: 6
    consumerGroup: asyncd-order-cancel-cg

Operator は CRD の変更を監視し、自動的に以下を完了する：

  • Kafka topic と consumer group の作成/更新
  • Dispatcher のルーティングテーブルの更新（ConfigMap のホットアップデート経由）
  • QPS メトリクスに基づく consumer レプリカ数の自動調整（HPA 連携）

これにより事業接続のハードルが大幅に下がった——「チケットを出してプラットフォームチームに設定を依頼」から「YAML を提出」に変わった。

監視とアラート

各段階で Prometheus metrics を公開している：

// タスク送信
asyncd_task_submitted_total{task_type, source_service}
asyncd_task_submit_latency_seconds{task_type}

// タスクスケジューリング
asyncd_task_dispatched_total{task_type}
asyncd_task_dispatch_delay_seconds{task_type}  // 実際のトリガー時刻 - 予定トリガー時刻

// タスク実行
asyncd_task_executed_total{task_type, status}   // status: success/failed/timeout
asyncd_task_execution_duration_seconds{task_type}
asyncd_task_retry_total{task_type}

// キューヘルス
asyncd_kafka_consumer_lag{topic, partition}
asyncd_mysql_pending_tasks{task_type}
asyncd_mysql_poll_duration_seconds

最も重要なアラートルール：

  • dispatch_delay > 5s：タスクスケジューリング遅延が 5 秒を超えている。消費能力が不足しているか MySQL poller が詰まっている
  • kafka_consumer_lag が継続的に増加：コンシューマーがプロデューサーに追いつけていない。スケールアウトが必要
  • mysql_pending_tasks が継続的に増加：到期タスクが蓄積している。poller に問題がある可能性
  • タスクリトライ率 > 10%：下流サービスに問題がある可能性

ダブルイレブン：25K から 50K QPS へ

2021 年のダブルイレブンはこのシステム最大の試練だった。日常の QPS は約 25K で、ピーク時は 50K 以上をサポートする見積もりだった。

事前に行った対策：

  • Kafka partition の拡張：コア topic を 12 partition から 24 に拡張し、consumer も相応に拡張
  • MySQL の読み書き分離：poller クエリはレプリカを使用し、ステータス更新はマスターを使用して、スキャンクエリが書き込みに影響しないようにした
  • バッチ送信の最適化：SDK に SubmitBatch インターフェースを追加し、1 回の RPC で複数タスクを送信してネットワークの往復を削減
  • ウォームアップ：ダブルイレブンの前日に、すべての Cron タスクの次回トリガーレコードを事前に作成し、深夜の書き込みスパイクを回避

実際のピークは約 48K QPS に達し、システムは耐え抜き、タスクの紛失はなかった。主要指標：

指標 日常 ダブルイレブンピーク
QPS ~25K ~48K
P99 スケジューリング遅延 180ms 520ms
タスク紛失 0 0

踏んだ落とし穴

1. MySQL の datetime 精度

MySQL の DATETIME はデフォルト精度が秒。複数のタスクの trigger_at が同一秒内の場合、ORDER BY trigger_at の順序が不定。秒レベルの遅延タスクでは、タスクが順序通りに実行されない原因となった。解決策：DATETIME(3)（ミリ秒精度）に変更し、ORDER BY に id を tiebreaker として追加。

2. Kafka rebalance ストーム

初期の consumer は session.timeout.ms と heartbeat.interval.ms を適切に設定しておらず、業務エグゼキュータの偶発的な遅延リクエストにより consumer がダウンと判定され、rebalance がトリガーされた。rebalance 中はすべての consumer が消費を停止し、遅延が急上昇した。最終的に session.timeout=30s、heartbeat.interval=10s、max.poll.interval=5min に調整し、同時にエグゼキュータ呼び出しを非同期化——メッセージ消費後に即座に commit し、実行結果はコールバックで報告するようにした。

3. Poller のサンダリングハード効果

複数の Dispatcher インスタンスが同時に MySQL をスキャンすると、FOR UPDATE SKIP LOCKED で重複実行は回避されるが、大量のロック競合により MySQL の CPU が急上昇した。解決策：各 Dispatcher インスタンスが特定の task_type 範囲を担当し（コンシステントハッシュで割り当て）、すべてのインスタンスが同じデータをスキャンすることを回避。

4. タイムゾーンの問題

Shopee の事業は東南アジアの複数の国をカバーしており、各国のタイムゾーンが異なる。cron 式 0 0 * * *（毎日零時）はどの零時なのか？最終方針：すべての内部時刻は UTC で保存・計算し、Cron ルールは明示的にタイムゾーンを指定する必要がある CRON_TZ=Asia/Singapore 0 0 * * *。SDK は cron 式にタイムゾーン情報が含まれているかチェックし、含まれていなければ拒否する。

5. 冪等性の普及の難しさ

技術的には最もシンプルだが普及が最も難しい問題。「少なくとも 1 回の配信」は業務エグゼキュータが冪等でなければならないことを要求するが、多くの業務開発者は理解していないか重視していなかった。本番稼働後に「二重課金」「二重クーポン発行」の事故が何度か発生したが、すべてエグゼキュータが冪等性を実装していなかったため。最終的に SDK レベルで idempotency_key の入力を必須とし、Redis ベースの重複排除ヘルパーを提供して、ツールレベルでエラーの確率を下げた。

振り返りと反省

このシステムは 2019 年半ばに設計を開始し、2020 年初めに最初の事業が接続し、2021 年のダブルイレブンで 50K QPS に耐えた。3 年間の感想：

  • ストレージ選定がシステムの上限を決める：Kafka + MySQL の組み合わせは華やかではないが、成熟度と信頼性が勝る。ミドルウェアシステムにおいて「データを失わない」ことは「低遅延」よりはるかに重要。Kafka の永続化と at-least-once セマンティクスがこのシステムの礎石であり、いかなる代替案もまずこの関門を通過しなければならない。
  • SDK の使いやすさがシステムの普及速度を決める：どんなに良いシステムでも、SDK が使いにくければ事業チームは使わない。SDK 設計に費やした時間は決して無駄にならない。
  • Kubernetes Operator はミドルウェアチームのレバレッジ：運用操作を CRD + Controller としてコード化し、事業チームがセルフサービスで接続でき、プラットフォームチームがボトルネックでなくなる。このパターンは後に他のミドルウェア（設定センター、メッセージキュー）でも再利用された。
  • 監視は後付けではなく、初日から必要：システム稼働初日から完全な Prometheus + Grafana ダッシュボードがあった。初期の多くの問題はメトリクスによって事前に発見され、事業からの障害報告を待つのではなかった。
  • 冪等性は社会的問題であり、技術的問題ではない：ツールは提供できるが、最終的には業務開発者が理解し重視する必要がある。ドキュメント、トレーニング、コードレビュー、事故の振り返り、どれも欠かせない。

もう一度やるとしたら

最大の教訓は：まずユースケースを明確にモデリングし、それから技術選定を行う。

当時、「非同期タスク」を統一的な概念としてシステムを設計したが、実際には異なるタスクは重要な属性において差異が大きい。もう一度やるなら、十分な時間をかけて事業チームと一緒に各タスクの核心属性を整理する：

属性 取りうる値 影響する設計上の決定
遅延タイプ 即時 / 短遅延 / 長遅延 / Cron ストレージ選定（Kafka vs MySQL）、スケジューリング戦略
優先度 高（注文キャンセル）/ 中 / 低（レポート） キュー分離戦略、リソース割り当て
一意性 重複許可 / 同一時刻で一意 / グローバルに一意 重複排除メカニズム、冪等性要件
配信セマンティクス 少なくとも 1 回 / 多くとも 1 回 / 正確に 1 回 確認メカニズム、リトライ戦略、メッセージ永続化
実行時間 ミリ秒レベル / 秒レベル / 分レベル以上 タイムアウト設定、ハートビート検出、リソース分離
失敗処理 リトライ / 破棄 / 手動介入 リトライ戦略、デッドレターキュー、アラートルール
依存関係 独立 / 前提依存あり / DAG タスクオーケストレーション、ステートマシン

これらの属性の異なる組み合わせは、全く異なる設計上のトレードオフを意味する。いくつかの例：

  • 注文タイムアウトキャンセル（短遅延 + 高優先度 + グローバルに一意 + 少なくとも 1 回）：独立した高優先度キューが必要で、重複排除メカニズムが信頼できなければならず、メッセージは絶対に失われてはならない——Kafka が正しい選択で、Redis は不可。
  • 統計レポート生成（Cron + 低優先度 + 重複許可 + 多くとも 1 回）：2 回実行しても問題ないが、高優先度タスクをブロックしてはならない——キュー分離が必要で、注文タスクとの共有は不可。
  • データエクスポート（即時 + 実行時間が長い + 前提依存あり）：タイムアウトと誤判定されないようハートビート検出が必要で、手動直列ではなく DAG オーケストレーションが必要。

当時犯した過ちは、これらの差異の大きいタスクを同一のモデルに押し込めたこと。結果として：高優先度タスクが低優先度タスクにブロックされ（キュー分離なし）、一意性が必要なタスクが重複実行され（重複排除メカニズムなし）、オーケストレーションが必要なタスクが業務側の手動直列に依存した（DAG サポートなし）。

その他の具体的な改善：

  • デッドレターキューの導入：リトライ回数を使い果たしたタスクは failed とマークするだけでなく、デッドレターキューに入れるべき。当時このメカニズムがなく、一部の失敗タスクが忘れ去られた。
  • 優先度によるキュー分離：高優先度タスクに独立した Kafka topic と consumer group を使用し、低優先度タスクの蓄積による速度低下を回避。これは後から補ったが、最初からユースケースモデリング段階で識別していれば、実装はもっとクリーンだっただろう。
  • タスク一意性制約：宣言的な一意性設定（例：unique_key = "order:{order_id}"）を提供し、同じ key のタスクが重複実行されないことをシステムレベルで保証する。業務側の冪等性に完全に依存するのではなく。
  • DAG 式タスクオーケストレーション：業務では頻繁に「タスク A 完了後にタスク B をトリガー」が必要。当時は業務側がエグゼキュータ内で手動で次のタスクを送信しており、エラーが起きやすかった。宣言的な DAG オーケストレーション機能を提供すべき。

一言でまとめると：技術選定はユースケースに奉仕するものであり、その逆ではない。まず時間をかけてタスクの属性マトリクスを明確にし、各組み合わせがどんなトレードオフに対応するかを考え抜いてから、コードを書き始める。「Kafka を使うか Redis を使うか」で悩むことから始めるより、はるかに価値がある。
