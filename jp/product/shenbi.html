神筆：ブラウザ上で子ども向けプログラミング言語を作る

2026-03-08
この記事の価値：Scratch は強力だが小さな子どもには複雑すぎ、Apple Swift Playgrounds は美しいが実際のプログラミング言語との接続がない。この記事では別のアプローチを共有する——ブラウザ上でゼロから中英バイリンガルの Python ライクなプログラミング言語を実装する。完全な Lexer → Parser → Compiler → VM パイプライン、ブロック/コードの双方向同期、そして同じ言語で異なるタイプのゲームを駆動できる World 抽象を含む。「言語の作り方」や「子ども向けプログラミングプロダクトの設計」に興味があれば、ここに完全なエンジニアリングの詳細がある。

なぜ自作するのか

市場にある子ども向けプログラミングツールは大きく二種類に分かれる：

  • Scratch 系：純粋なブロックのドラッグ＆ドロップで、機能は非常に強力（イベントシステム、並行処理、メッセージブロードキャスト）だが、5-8 歳の子どもには概念が多すぎる。「キャラクターをゴールまで歩かせる」というタスクで、Scratch では座標系、イベントループ、ブロードキャストメッセージの理解が必要になる——これらはプログラミングの本質ではなく、Scratch 自体の複雑さである
  • Apple Swift Playgrounds 系：美しい 3D インターフェース、ガイド付き学習、体験は良い。しかし教えているのは Swift の構文であり、子どもが将来使う Python/JavaScript とは何の関係もない。Playgrounds を終えても、他の言語に移る時はゼロからやり直しになる

私が求めていたのは：

  1. 十分にシンプル：5 歳の子どもがブロックを理解でき、8 歳の子どもがコードを読める
  2. 実際の言語と接続：構文が Python と高度に一致し、学び終えたらそのまま本物の Python に移行できる
  3. 中英バイリンガル：如果 と if は同じもので、子どもは中国語で入門し、徐々に英語に切り替えられる
  4. ブロックとコードの双方向切替：「ブロックがコードを生成する」一方向の関係ではなく、ブロックとコードがいつでも相互変換できる

これらの条件を満たすツールが見つからなかったので、自分で作った。神筆（Shenbi）と名付けた——コードを書くのは絵を描くように、描いたものが動き出す。

下から上に見ると：World はゲームの状態、VM はコードを実行して World を操作し、Canvas は World の状態をレンダリングし、ブロック/コードエディタ は MiniPython コードを編集する二つの方法である。

第一部：言語を作る

Lexer：中英バイリンガルはここで実現

中英バイリンガルのコア設計は非常にシンプルで、Lexer 層で中国語キーワードと英語キーワードを同じ Token タイプにマッピングする。Parser 以降のすべてのステージは、ユーザーが中国語で書いたか英語で書いたかをまったく知らない。

const KEYWORDS: Record<string, TokenType> = {
  // 中英文映射到相同的 Token
  '如果': 'IF',      'if': 'IF',
  '否则如果': 'ELIF', 'elif': 'ELIF',
  '否则': 'ELSE',    'else': 'ELSE',
  '重复': 'REPEAT',  'repeat': 'REPEAT',
  '次': 'TIMES',     'times': 'TIMES',
  '当': 'WHILE',     'while': 'WHILE',
  '对于': 'FOR',     'for': 'FOR',
  '在': 'IN',        'in': 'IN',
  '定义': 'DEF',     'def': 'DEF',
  '返回': 'RETURN',  'return': 'RETURN',
  '真': 'TRUE',      'True': 'TRUE',
  '假': 'FALSE',     'False': 'FALSE',
  '和': 'AND',       'and': 'AND',
  '或': 'OR',        'or': 'OR',
  '不': 'NOT',       'not': 'NOT',
  '类': 'CLASS',     'class': 'CLASS',
};

識別子も中国語文字（Unicode CJK 範囲）をサポートしているので、変数名を中国語にできる：

// この二つのコードは Lexer 以降完全に等価
重复 3 次:
    前进()
    左转()

repeat 3 times:
    forward()
    turnLeft()

Lexer は Python スタイルのインデントも処理する必要がある——コードブロックの開始と終了を波括弧ではなく INDENT/DEDENT トークンで表現する。これが Lexer 全体で最も複雑な部分である。

Parser → Compiler → IR

Parser は手書きの再帰下降で、AST を生成する。Compiler は複数パスのコンパイルを行う：まず関数とクラスの定義を収集し、次にメインコードをコンパイルし、最後に関数本体とクラスメソッドをコンパイルする。出力は線形のスタック式命令セットである：

// IR 指令集（~35 个 opcode）
type OpCode =
  // 栈操作
  | 'PUSH' | 'POP' | 'DUP'
  // 变量
  | 'LOAD' | 'STORE'
  // 算术
  | 'ADD' | 'SUB' | 'MUL' | 'DIV' | 'MOD' | 'NEG'
  // 比较
  | 'EQ' | 'NEQ' | 'LT' | 'GT' | 'LTE' | 'GTE'
  // 控制流
  | 'JUMP' | 'JUMP_IF' | 'JUMP_IF_NOT'
  // 函数
  | 'CALL' | 'CALL_USER' | 'RETURN'
  // 数据结构
  | 'ARRAY_CREATE' | 'ARRAY_GET' | 'ARRAY_SET'
  | 'OBJECT_CREATE' | 'OBJECT_GET' | 'OBJECT_SET'
  // 类
  | 'CLASS_DEF' | 'NEW' | 'METHOD_CALL'
  | 'HALT' | 'NOP';

シンプルな例のコンパイル結果：

// MiniPython 源码
重复 3 次:
    前进()

// 编译后的 IR
PUSH 0          // 循环计数器 = 0
STORE $i        // 保存到变量
PUSH 3          // 循环上限
STORE $limit
LOAD $i         // ← 循环开始
LOAD $limit
LT              // i < limit ?
JUMP_IF_NOT end // 不满足则跳出
CALL forward:0  // 调用 forward()，0 个参数
POP             // 丢弃返回值
LOAD $i
PUSH 1
ADD
STORE $i        // i++
JUMP loop       // 跳回循环开始
HALT            // ← end

VM：一時停止できる仮想マシン

VM はスタック式仮想マシンで、プログラムカウンタ（PC）、オペランドスタック、グローバル変数テーブル、コールスタックを持つ。ただし、すべての命令を一気に実行するのではなく、毎回一ステップだけ実行してから呼び出し元に制御を返す。

これは子ども向けプログラミングのコア要件である：子どもが 前進() と書いたら、キャラクターが一マス進むアニメーションを見せてから次の命令を実行する必要がある。VM が一気に実行してしまうと、ユーザーは最終状態しか見られない。

// VM.step() 返回执行结果，而不是跑到底
step(): StepResult {
  const inst = this.program[this.pc];

  switch (inst.opcode) {
    case 'CALL': {
      const { name, argCount } = parseCallArg(inst.arg);
      const args = this.popN(argCount);

      // 游戏命令：_forward, _turnLeft, ...
      if (this.commandHandlers.has(name)) {
        this.commandHandlers.get(name)!(args);
        this.stack.push(null);
        this.pc++;
        return { action: name, actionArgs: args };
        //       ↑ 返回给调用方："刚执行了 forward"
        //         调用方可以播放动画，然后再调 step()
      }

      // 传感器：frontBlocked, atGoal, ...
      if (this.sensorHandlers.has(name)) {
        const result = this.sensorHandlers.get(name)!(args);
        this.stack.push(result);
        this.pc++;
        return { sensorQuery: name };
      }

      // 内置函数：len, print, random, ...
      // ...
    }
    // ... 其他 opcode
  }
}

この設計はステップ実行デバッグとブレークポイントも自然にサポートする：呼び出し元がいつ step() を呼ぶかを制御し、各ステップの間で変数を検査したり現在の行をハイライトしたりできる。

さらに、VM は巻き戻しデバッグもサポートしている——各ステップの実行前に VM の完全な状態スナップショットを保存し、ユーザーがプログラムの実行過程を「逆再生」できる。子どもがコードの流れを理解するのに特に役立つ。

第二部：ブロック ↔ コードの双方向同期

多くのブロックプログラミングプラットフォームは一方向的で、ブロックがコードを生成するが、コードの変更はブロックに反映されない。つまり子どもがコードモードに切り替えて何か変更すると、二度とブロックモードに戻れなくなる。

神筆は双方向同期を実現した：

ブロック → コード

各ブロックタイプ（command、repeat、while、if、関数定義……）に対応するコード生成ルールがあり、同時に LineToBlockMap を維持する——生成された各行がどのブロック ID に対応するかを記録し、実行時に対応するブロックをハイライトするために使用する：

function generateBlockCode(block: Block, indent: number,
                           lineMap: LineToBlockMap) {
  const spaces = '    '.repeat(indent);

  switch (block.type) {
    case 'command':
      lines.push(`${spaces}${commandName}(${args})`);
      lineMap.set(currentLine, block.id);
      break;

    case 'repeat':
      lines.push(`${spaces}repeat ${block.repeatCount} times:`);
      lineMap.set(currentLine, block.id);
      // 子ブロックを再帰的に処理
      for (const child of block.children) {
        generateBlockCode(child, indent + 1, lineMap);
      }
      break;

    case 'if':
      lines.push(`${spaces}if ${conditionToCode(block)}:`);
      // ...
  }
}

コード → ブロック

こちらがより興味深い方向である。コードはまず本物の Lexer/Parser でコンパイルされて AST になり、その後 AST ノードを逆変換してブロックにする：

function parseCodeToBlocks(code: string): Block[] | null {
  try {
    const ast = compile(code);  // 本物のコンパイラで解析
    return ast.body
      .map(stmt => statementToBlock(stmt))
      .filter(b => b !== null);
  } catch {
    return null;  // 構文エラー、変換不可
  }
}

ここでは MiniPython のコンパイラを再利用している——簡略化した別のパーサーを書くのではなく、本当にコードをコンパイルし、AST を取得してからブロックに変換する。これによりコード→ブロックの変換とブロック→コードの変換が完全に対称であることが保証される。

難しい点は、中国語の関数名、英語の関数名、レガシーな命名のマッピングを処理する必要があることである。例えば forward、前進、move はすべて同じ command タイプのブロックにマッピングされる：

const COMMAND_MAP: Record<string, CommandId> = {
  move: 'move', turn: 'turn',
  移动: 'move', 转向: 'turn',
  forward: 'move', backward: 'move',  // legacy
  前进: 'move', 后退: 'move',
  左转: 'turn', 右转: 'turn',
  setColor: 'setColor', 设置颜色: 'setColor',
};

結果：子どもがブロックモードでロジックを組み立て、コードモードに切り替えると対応する Python コードが表示される。コードモードで数行変更し、ブロックモードに戻ると対応するブロックの変化が表示される。二つのモードは常に同期している。

第三部：World モデル

これが最もうまく設計できた部分だと思う。神筆は迷路ゲームだけでなく、タートルグラフィックス（Turtle Graphics）もあり、今後さらに多くのゲームタイプに拡張できる。問題は：同じ言語と VM でまったく異なるタイプのゲームをどう駆動するか？

答えは World 抽象である：

架构模式：
- World：数据模型（状态 + 命令 + 传感器）
- VM：代码执行器（把代码映射到 World 的方法）
- Canvas：渲染器（显示 World 的状态）

各ゲームタイプはそれぞれの World、VM バインディング、Canvas を実装するが、MiniPython 言語と VM コアは共有する。

MazeWorld

class MazeWorld {
  // 状态
  grid: Cell[][];
  x: number; y: number;
  direction: 'up' | 'down' | 'left' | 'right';
  collected: number;
  steps: number;

  // 命令
  moveForward(): boolean;
  turnLeft(): void;
  turnRight(): void;
  collect(): boolean;

  // 传感器
  isFrontBlocked(): boolean;
  isAtGoal(): boolean;
  hasStarHere(): boolean;

  // 序列化（给 VM 用）
  toSharedState(): SharedMazeState;
  syncFromSharedState(state: SharedMazeState): void;
}

TurtleWorld

class TurtleWorld {
  // 状态
  x: number; y: number;
  angle: number;
  penDown: boolean;
  penColor: string;
  lines: LineSegment[];

  // 命令
  forward(distance: number): void;
  turnLeft(degrees: number): void;
  penUp(): void;
  setColor(color: string): void;

  // 序列化
  toSharedState(): SharedTurtleState;
  syncFromSharedState(state: SharedTurtleState): void;
}

重要な設計：SharedState ブリッジ

World と VM は SharedState で接続される。VM 起動時に World の状態を通常の JavaScript オブジェクトにシリアライズし、MiniPython のグローバルスコープに world として注入する：

// MazeVM.loadWithSource()
loadWithSource(userCode: string): void {
  const fullCode = MAZE_STDLIB + '\n' + userCode;
  const program = compileToIR(compile(fullCode));
  this.sharedState = this.world.toSharedState();
  this.vm.load(program);
  this.vm.setGlobal('world', this.sharedState);
  //                  ↑ MiniPython コード内で直接 world.x, world.y, ... を読み書きできる
}

そして MiniPython の標準ライブラリコードがこの world オブジェクトを直接操作する：

# MiniPython 标准库（迷宫）
def forward():
    offset = DIRECTION_OFFSETS[world.direction]
    newX = world.x + offset[0]
    newY = world.y + offset[1]
    if getCell(newX, newY) == 'wall':
        return False
    _forward()    # ← 唯一の native 呼び出し、Canvas にアニメーション再生を通知
    return True

def frontBlocked():
    offset = DIRECTION_OFFSETS[world.direction]
    newX = world.x + offset[0]
    newY = world.y + offset[1]
    return getCell(newX, newY) == 'wall'

注目すべきは、forward() の衝突検出や座標計算はすべて MiniPython 自身が行っていることである。_forward() だけが native コマンドで、Canvas に「キャラクターが一マス移動した」と通知し、Canvas がアニメーションを再生する。標準ライブラリは自分自身の言語で書かれている。

タートルグラフィックスの標準ライブラリも同じパターンである：

# MiniPython 标准库（海龟）
def forward(distance):
    radians = world.angle * 3.14159265359 / 180
    dx = _cos(radians) * distance * world.stepSize
    dy = _sin(radians) * distance * world.stepSize * -1
    oldX = world.x
    oldY = world.y
    world.x = world.x + dx
    world.y = world.y + dy
    if world.penDown:
        _drawLine(oldX, oldY, world.x, world.y,
                  world.penColor, world.penWidth)

_cos、_sin、_drawLine だけが native である——三角関数と描画は MiniPython ではできないためだ。残りの座標計算はすべて MiniPython 内で完結している。

新しいゲームの追加に必要なもの

World モデルにより、新しいゲームの拡張は非常に明確になる：

  1. 新しい World クラスを定義する（状態 + コマンド + センサー + toSharedState）
  2. 対応する VM バインディングクラスを書き、少数の native コマンドを登録する
  3. MiniPython で標準ライブラリを書く——ゲームロジックの大部分はここに置く
  4. World の状態をレンダリングする Canvas コンポーネントを書く

言語、コンパイラ、VM コア、ブロックエディタ——すべて再利用、変更ゼロ。

ステージシステム：条件も式である

各ステージは JSON ファイルで、ASCII アートでマップを定義する：

{
  "id": "maze-01",
  "name": "First Steps",
  "grid": [
    "########",
    "#>...*G#",    // > = 玩家朝右, * = 星星, G = 终点
    "########"
  ],
  "availableBlocks": ["command"],
  "winCondition": "collectedCount() >= 1 and atGoal()",
  "failCondition": "stepCount() > 10",
  "hints": [
    "Use the Forward block to move the robot",
    "Keep going until you reach the flag!"
  ]
}

winCondition と failCondition はハードコードされた列挙ではなく、MiniPython の式であり、VM が評価する。つまりステージ設計者（教師）はコードを変更せずに任意の条件を組み合わせられる：

// 各种 win condition
"atGoal()"                                // 到达终点
"collectedCount() >= 3 and atGoal()"      // 收集3颗星 + 到达终点
"stepCount() <= 8 and atGoal()"           // 8步内到达终点
"collectedCount() == getRemainingStars()" // 收集所有星星

まとめ

最初の目標に立ち戻ると：

  • 十分にシンプル → ブロックモードで 5 歳の子どもが使え、コードモードの構文は Python と一致する
  • 実際の言語と接続 → MiniPython の構文は Python のサブセットそのもので、学び終えたらそのまま Python が書ける
  • 中英バイリンガル → Lexer 層でキーワードマッピングを実現し、後続のパイプラインは完全に無関知
  • ブロックとコードの双方向切替 → コンパイラの AST を再利用してコード→ブロックの逆変換を実現

エンジニアリング上最も満足している三つの設計：

  1. 中英バイリンガルは Lexer 層だけで処理：一つのキーワードマッピングテーブルで解決し、Parser/Compiler/VM は言語をまったく気にしない。この決定によりシステム全体の複雑さが「バイリンガル」のために倍増しなかった
  2. World 抽象：ゲームワールドは純粋なデータモデルで、VM は SharedState でブリッジし、標準ライブラリは自身の言語で書かれている。新しいゲームを追加しても言語と VM を変更する必要がない
  3. VM の step() 設計：毎回一ステップだけ実行して最後まで走らないので、アニメーション再生、ステップ実行デバッグ、ブレークポイント、巻き戻しデバッグがすべて自然に実現できる
