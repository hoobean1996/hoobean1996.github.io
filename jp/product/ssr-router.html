OCaml で URL Router を書く：面白いオーバーエンジニアリング

2026-03-08
この記事の価値：これは「オーバーエンジニアリング」の実験である——OCaml の ocamllex + menhir（コンパイラツールチェーン）を使って URL ルーティングマッチングを実装し、js_of_ocaml で JavaScript にコンパイルして Node.js で動かす。実用的な価値は限られるが、コンパイラフロントエンド技術（字句解析、構文解析、AST）が日常的な問題にどう適用できるかに興味があれば、面白いケーススタディである。

きっかけ

きっかけは単純だった。Next.js 風の SSR フレームワークの実験をしていて、ルーティングマッチャーが必要になった。普通なら正規表現か path-to-regexp のようなライブラリを使って 10 行で済ませるところだ。しかしちょうど OCaml を学んでいたので、「ocamllex + menhir でまともなルーティングパーサーが書けないか？」と思い立った。

結果が ssrrouter——コンパイラフロントエンド技術スタックで URL を解析するルーティングライブラリである。独自の字句解析器、構文解析器、AST、型システム（int と string しかないが）を持ち、最終的に js_of_ocaml で JavaScript にコンパイルされ、Node.js の SSR サーバーで動作する。

どう動くのか

システム全体の処理フロー：

路由定义字符串
    ↓ Lexer (ocamllex)
Token 流
    ↓ Parser (menhir)
AST (route_pattern)
    ↓ Matcher
匹配结果 + 参数绑定

第一歩：AST の定義

まずルーティングの型システムを定義する。ルーティングパターンは複数のセグメントで構成され、各セグメントは静的パス、スラッシュ、または型付きパラメータのいずれかである：

(* ast.ml *)
type param = IntParam of string | StringParam of string
type value = StringValue of string | IntValue of int

type segment = Static of string | Slash | Param of param

type route_pattern = {
  path : string;
  segments : segment list;
  entrypoint : string;
}

同時にルーティングインスタンス（実際の URL）も定義する。こちらのセグメントはパターンではなく具体的な値である：

type segment_instance = IStatic of string | ISlash | IValue of value

type route_instance = {
  path : string;
  segments : segment_instance list;
  query_params : query_param list;
}

ここでの重要な設計は、ルーティング定義とルーティングインスタンスが異なる型を使うことである。定義には Param (IntParam "id") があり、インスタンスには IValue (IntValue 42) がある。これにより型システムがマッチングロジックの正確性をチェックしてくれる。

第二歩：字句解析

ocamllex で文字列をトークンに分割する：

(* lexer.mll *)
rule token = parse
    | '/'       { SLASH }
    | ':'       { COLON }
    | '('       { LPAREN }
    | ')'       { RPAREN }
    | '?'       { QUESTION_MARK }
    | '='       { EQUALS }
    | '&'       { AMPERSAND }
    | "route"   { ROUTE }
    | "int"     { INT_TYPE }
    | "string"  { STRING_TYPE }
    | ['0'-'9']+ as n { NUMBER (int_of_string n) }
    | ['A'-'Z']['a'-'z''A'-'Z''0'-'9']*"Entrypoint" as e { ENTRYPOINT e }
    | ['a'-'z']['a'-'z''A'-'Z''0'-'9''-']* as id { ID id }
    | [' ' '\t' '\n']     { token lexbuf }
    | eof       { EOF }
    | _         { raise (LexError ("Unexpected char: " ^ Lexing.lexeme lexbuf)) }

ENTRYPOINT のマッチングルールに注目：大文字で始まり "Entrypoint" で終わる必要がある。これは規約であり、ルーティング定義は次のような形になる：

route /admin/:id(int) AdminDetailEntrypoint

第三歩：構文解析

menhir（OCaml のパーサージェネレータ）で文法を定義する：

(* parser.mly *)
route:
  | ROUTE segments = route_path entrypoint = ENTRYPOINT EOF  {
    {
      path = build_path_from_segments segments;
      segments = segments;
      entrypoint = entrypoint;
    }
  }

route_segments:
  | p = ID rest = route_segments_tail     { Static p :: rest }
  | param = param_segment rest = route_segments_tail  { param :: rest }
  |                                    { [] }

param_segment:
  | COLON id = ID LPAREN INT_TYPE RPAREN  { Param (IntParam id) }
  | COLON id = ID LPAREN STRING_TYPE RPAREN { Param (StringParam id) }

Parser はルーティング定義とルーティングインスタンス（クエリストリング付き）の両方を処理し、二つの生成規則が同じ lexer を共有する。

第四歩：マッチング

マッチャーのコアロジックは驚くほどシンプルで、セグメントごとに比較し、型が一致する必要がある：

(* matcher.ml *)
let match_segment (pattern : segment) (instance : segment_instance) =
  match (pattern, instance) with
  | Static s1, IStatic s2 -> s1 = s2
  | Static s1, IValue (StringValue s2) -> s1 = s2
  | Slash, ISlash -> true
  | Param (IntParam _), IValue (IntValue _) -> true
  | Param (StringParam _), IValue (StringValue _) -> true
  | _ -> false

let match_route (pattern : route_pattern) instance =
  List.length pattern.segments = List.length instance.segments
  && List.for_all2 match_segment pattern.segments instance.segments

マッチング成功後、Router はパラメータバインディングとクエリパラメータを抽出する：

(* router.ml *)
let extract_bindings pattern_segments instance_segments =
  List.fold_left2
    (fun acc pattern_seg instance_seg ->
      match (pattern_seg, instance_seg) with
      | Param (StringParam name), IValue value -> { name; value } :: acc
      | Param (IntParam name), IValue value -> { name; value } :: acc
      | _, _ -> acc)
    [] pattern_segments instance_segments

第五歩：JavaScript へのコンパイル

最後のステップは js_of_ocaml で JS インターフェースを公開すること：

(* jsmain.ml *)
Js.export "ssrrouter"
  (object%js
     method createRouter routes =
       let routes_list =
         routes |> Js.to_array |> Array.to_list |> List.map Js.to_string
       in
       let r = Router.create routes_list in
       current_router := Some r

     method lookup path =
       match !current_router with
       | None -> Js.null
       | Some router ->
           match Router.lookup router (Js.to_string path) with
           | None -> Js.null
           | Some result -> (* JS オブジェクトを構築して返す *)
  end)

Node.js での使用方法は次の通り：

const ssrrouter = require("./ssrrouter");

ssrrouter.createRouter([
  "route /      AppEntrypoint",
  "route /admin AdminEntrypoint",
  "route /admin/:id(int) AppEntrypoint",
]);

const result = ssrrouter.lookup("/admin/42?sort=name");
// { entrypoint: "AppEntrypoint", bindings: [{name:"id", value:42}], queryParams: [{key:"sort", value:"name"}] }

良かった点

  • 型安全なパラメータ：ルーティング定義段階で int と string を区別し、/users/42 は :id(int) にマッチし、/users/john は :name(string) にマッチする。実行時の手動変換やバリデーションが不要になる。
  • コンパイラフロントエンドの完全な体験：Lexer → Parser → AST → 意味処理、小さいながらも一通り揃っている。コンパイラフロントエンドを学びたいなら、URL router を書く方が電卓を書くよりずっと面白い。
  • 親切なエラーメッセージ：完全な lexer/parser があるため、構文エラーを行番号と列番号で正確に特定でき、大雑把な "invalid route" ではなくなる。
  • OCaml の pattern matching はこのシーンに最適：マッチャーのコアロジックは match 式で非常に明快に書け、ルールのほぼ直訳になる。
  • クロス言語での再利用：OCaml で一度書き、js_of_ocaml で JS にコンパイルすれば、サーバーサイドとクライアントサイドで同じルーティングロジックを使える。

良くなかった点

  • オーバーエンジニアリングの典型：コンパイラ技術スタックで URL を解析するのは、ロケットで出前を届けるようなものだ。path-to-regexp は正規表現で同じことをコード量一桁少なく実現し、実戦で長く使われている。
  • ビルドチェーンの複雑さ：OCaml ツールチェーン（opam、dune）+ js_of_ocaml が必要で、CI を動かすだけでも苦労する。チームに OCaml を書ける人がいなければ、保守コストは極めて高い。
  • マッチングアルゴリズムが素朴すぎる：List.find_opt ですべてのルーティングをリニアスキャンしている。本格的なルーター（Go の httprouter など）は radix tree を使い、O(パス長) でマッチングできる。ルーティングが増えると遅くなる。
  • ワイルドカードとネストルーティング未対応：/api/* や /files/** のような glob マッチングも、ルーティングのネスト（layout routes）もない。フレームワークレベルのルーターとしては機能が限定的すぎる。
  • 型システムが単純すぎる：int と string しかない。実際のプロジェクトでは UUID、日付、列挙型などのパラメータ型も必要である。
  • js_of_ocaml の JS 連携が辛い：jsmain.ml のコードを見ればわかる——手動で JS オブジェクトを構築し、手動で型変換を行い、型安全性は皆無。この部分のコードは純粋な OCaml 部分よりもはるかに醜い。

共有に値する経験

  • コンパイラ技術はコンパイラを書くためだけのものではない：Lexer/Parser の仕組みは構造を持つ任意の入力に適用できる——設定ファイル、DSL、クエリ言語、ルーティングルール。一度マスターすれば、多くの場面で活用できることに気づくだろう。
  • OCaml の代数的データ型 + pattern matching はツリー構造処理の最良のツール：AST の定義とマッチングコードはほぼ仕様の直訳であり、コンパイラがすべてのケースを処理しているかチェックしてくれる（exhaustiveness check）。
  • 「正しい」アプローチが必ずしも「適切な」アプローチではない：計算機科学の観点からは、formal grammar でルーティング構文を定義するのは「正しい」。しかしエンジニアリングの観点からは、正規表現で十分である。いつ何を使うべきかを知ることは、すべてを使えることより重要だ。
  • js_of_ocaml のポジショニング：既存の OCaml ライブラリをブラウザにコンパイルして使う場合（例えば OCaml の各種形式検証ツール）には適しているが、JS との大量の相互運用が必要なプロジェクトをゼロから書くには適さない。コードの半分が Js.Unsafe.obj の構築なら、TypeScript で直接書いた方が良い。
  • 小さなプロジェクトは最良の学習方法：このルーターは「実用的でない」が、書き終えた後に lexer generator、parser generator、AST 設計、クロス言語コンパイルについて直感的な理解が得られた。これらの知識は、後に他のコンパイラのコードを読む際に繰り返し役立った。

まとめ

ssrrouter は典型的な「学習のためのオーバーエンジニアリング」プロジェクトである。シンプルな問題に対して最もヘビーなアプローチを使ったが、その過程でコンパイラフロントエンドのコア概念をカバーした：字句解析、構文解析、AST 設計、型マッチング、クロス言語コンパイル。

もしコンパイラや OCaml を学んでいるなら、同様の小さな問題を見つけて練習することをお勧めする。「プログラミング言語を作る」からは始めないこと——大きすぎる。URL router、SQL サブセットパーサー、Markdown レンダラーから始めよう。問題が十分に小さいからこそ、各工程をしっかり理解できる。
