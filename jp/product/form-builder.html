Schema 駆動フォーム（上）：データモデルとレンダラー

2026-03-08
この記事の価値：B 向けプロダクトのフォーム要件は尽きることがない——登録、承認、アンケート、設定、シナリオごとに異なる。各フォームにコンポーネントを書くのではなく、JSON Schema でフォーム構造を記述し、フロントエンドが自動レンダリングする方が良い。この記事では本番検証済みのデータモデルを共有する：FormSpec → Question → QuestionResponse の 3 層構造で、15 以上のフィールドタイプ、ページ分割、バリデーションをサポートし、非開発者も JSON を編集してフォームを作成できる。

問題の出発点

ほぼすべての B 向けプロダクトが同じ問題に直面する：フォームが多すぎる。

登録フォーム、承認フォーム、アンケート、設定ページ、データ入力……各ビジネスシナリオにフォームが必要で、各フォームのフィールド、バリデーションルール、レイアウトが異なる。各フォームを手書きで React コンポーネントにすると、コード量が爆発的に増え、フィールドを 1 つ変えるだけでリリースが必要になる。

さらに悪いことに、多くのフォームの作成者は開発者ではない——プロダクトマネージャー、運用担当者、HR である。彼らが必要としているのは「ドラッグ&ドロップでフォームを作れる」ツールであり、開発スケジュールを待つことではない。

これが Schema 駆動フォームの出番：JSON でフォームの構造を記述し、フロントエンドがその JSON に基づいて完全なフォームを自動レンダリングする。フィールド変更？JSON を変更。バリデーション追加？JSON を変更。レイアウト変更？やはり JSON を変更。コードの変更もリリースも不要。

核心的アイデア

Schema 駆動フォームの核心はシンプルな分離：

フォーム定義（What） →  JSON Schema  →  データベースに保存
フォームレンダリング（How） →  汎用レンダラー →  1 セットのコードですべてのフォームに対応
フォームデータ（Data） →  JSON レスポンス →  構造化された保存とクエリ

データモデル設計

完全な Schema 駆動フォームシステムには 3 層のデータモデルが必要：

第 1 層：FormSpec（フォーム定義）

// FormSpec —— 1 つのフォームのメタデータ
type FormSpec struct {
    ID          int
    Name        string       // "入社情報収集"
    Description string       // フォーム説明
    Cover       string       // カバー画像 URL
    Enabled     bool         // 有効かどうか
    CreatedBy   int          // 作成者
    Groups      []QuestionGroup  // 質問グループ
}

// QuestionGroup —— 質問グループ（フォームのページ分けまたはセクション分けに使用）
type QuestionGroup struct {
    ID        int
    Label     string       // "基本情報" / "職歴"
    Order     int          // ソート順
    Questions []Question
}

第 2 層：Question（質問定義）

type QuestionType string

const (
    ShortText   QuestionType = "short_text"
    Paragraph   QuestionType = "paragraph"
    MultiChoice QuestionType = "multi_choice"
    Checkboxes  QuestionType = "checkboxes"
    DropDown    QuestionType = "drop_down"
    FileUpload  QuestionType = "file"
    LinearScale QuestionType = "linear_scale"
    Date        QuestionType = "date"
    Time        QuestionType = "time"
)

type Question struct {
    ID       int
    Label    string         // "あなたの氏名"
    Type     QuestionType   // "short_text"
    Required bool           // 必須かどうか
    Options  []string       // 選択肢の選択肢
    Config   QuestionConfig // 追加設定
}

type QuestionConfig struct {
    Placeholder  string  // 入力ヒント
    MinLength    *int    // 最小長
    MaxLength    *int    // 最大長
    MinValue     *int    // linear_scale 最小値
    MaxValue     *int    // linear_scale 最大値
    FileTypes    []string // 許可されるファイルタイプ
}

第 3 層：QuestionResponse（ユーザーの回答）

type QuestionResponse struct {
    ID             int
    QuestionID     int      // Question に関連付け
    FormInstanceID int      // 特定の記入に関連付け
    Label          string   // 質問ラベルの冗長保存（クエリの便宜のため）
    Value          string   // ユーザーの回答（統一的に string で保存）
}

type FormInstance struct {
    ID         int
    FormSpecID int          // FormSpec に関連付け
    SubmittedBy int         // 記入者
    SubmittedAt time.Time
    Responses  []QuestionResponse
}

ここに重要な設計上の決定がある：QuestionResponse.Value を統一的に string で保存する。日付、数値、複数選択のいずれもすべて文字列にシリアライズする。この利点は schema がシンプルで、各タイプに異なるカラムを作る必要がないこと。欠点はクエリ時にデシリアライズが必要だが、フォームのシナリオでは、ほとんどのクエリは「特定の記入のすべての回答をエクスポート」であり、「年齢が 25 歳以上のすべての回答を見つける」ではないため、このトレードオフは価値がある。

フロントエンドレンダラー

レンダラーの核心は switch 文——Question の type フィールドに基づいて異なる UI コンポーネントをレンダリングする：

// Question.tsx
function Question({ question, value, onChange }) {
  switch (question.type) {
    case "short_text":
      return (
        <div className="form-field">
          <label>{question.label}{question.required && " *"}</label>
          <input
            type="text"
            value={value}
            onChange={e => onChange(e.target.value)}
            placeholder={question.config?.placeholder}
            required={question.required}
          />
        </div>
      );

    case "paragraph":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <textarea
            value={value}
            onChange={e => onChange(e.target.value)}
            rows={4}
          />
        </div>
      );

    case "multi_choice":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          {question.options.map(opt => (
            <label key={opt} className="radio-option">
              <input
                type="radio"
                name={`q-${question.id}`}
                value={opt}
                checked={value === opt}
                onChange={() => onChange(opt)}
              />
              {opt}
            </label>
          ))}
        </div>
      );

    case "checkboxes":
      const selected = value ? JSON.parse(value) : [];
      return (
        <div className="form-field">
          <label>{question.label}</label>
          {question.options.map(opt => (
            <label key={opt} className="checkbox-option">
              <input
                type="checkbox"
                checked={selected.includes(opt)}
                onChange={() => {
                  const next = selected.includes(opt)
                    ? selected.filter(s => s !== opt)
                    : [...selected, opt];
                  onChange(JSON.stringify(next));
                }}
              />
              {opt}
            </label>
          ))}
        </div>
      );

    case "drop_down":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <select value={value} onChange={e => onChange(e.target.value)}>
            <option value="">選択してください</option>
            {question.options.map(opt => (
              <option key={opt} value={opt}>{opt}</option>
            ))}
          </select>
        </div>
      );

    case "date":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <input type="date" value={value} onChange={e => onChange(e.target.value)} />
        </div>
      );

    case "file":
      return (
        <div className="form-field">
          <label>{question.label}</label>
          <input
            type="file"
            accept={question.config?.fileTypes?.join(",")}
            onChange={e => {
              const file = e.target.files?.[0];
              if (file) uploadAndGetUrl(file).then(onChange);
            }}
          />
        </div>
      );

    default:
      return <div>Unsupported question type: {question.type}</div>;
  }
}

この switch 文は一見「愚直」に見えるが、これこそが利点：各タイプのレンダリングロジックが完全に独立し、互いに干渉しない。新しいタイプを追加？case を追加。あるタイプの UI を変更？その case だけを変更。「ラジオボタンの修正がチェックボックスに影響する」という問題は存在しない。

完全なフォームレンダリング

Question コンポーネントを組み合わせて、フォーム全体をレンダリングする：

function FormRenderer({ formSpec }) {
  const [responses, setResponses] = useState({});

  const handleChange = (questionId, value) => {
    setResponses(prev => ({ ...prev, [questionId]: value }));
  };

  const handleSubmit = async () => {
    // 必須項目のバリデーション
    for (const group of formSpec.groups) {
      for (const q of group.questions) {
        if (q.required && !responses[q.id]) {
          alert(`入力してください：${q.label}`);
          return;
        }
      }
    }

    // 送信データの構築
    const payload = Object.entries(responses).map(([qId, value]) => ({
      questionId: parseInt(qId),
      value: String(value),
    }));

    await submitFormResponse(formSpec.id, payload);
  };

  return (
    <form onSubmit={e => { e.preventDefault(); handleSubmit(); }}>
      <h2>{formSpec.name}</h2>
      <p>{formSpec.description}</p>

      {formSpec.groups.map(group => (
        <fieldset key={group.id}>
          <legend>{group.label}</legend>
          {group.questions.map(q => (
            <Question
              key={q.id}
              question={q}
              value={responses[q.id] || ""}
              onChange={v => handleChange(q.id, v)}
            />
          ))}
        </fieldset>
      ))}

      <button type="submit">送信</button>
    </form>
  );
}

バックエンド API 設計

GraphQL で API を設計するのは非常に自然で、フォームのデータ構造自体がネストされているため：

type FormSpec {
  id: ID!
  name: String!
  description: String
  enabled: Boolean!
  groups: [QuestionGroup!]!
}

type QuestionGroup {
  id: ID!
  label: String!
  order: Int!
  questions: [Question!]!
}

type Question {
  id: ID!
  label: String!
  type: QuestionType!
  required: Boolean!
  options: [String!]
  config: QuestionConfig
}

enum QuestionType {
  SHORT_TEXT
  PARAGRAPH
  MULTI_CHOICE
  CHECKBOXES
  DROP_DOWN
  FILE
  LINEAR_SCALE
  DATE
  TIME
}

# クエリ
type Query {
  formSpec(id: ID!): FormSpec
  formInstances(formSpecId: ID!): [FormInstance!]!
}

# 送信
input QuestionResponseInput {
  questionId: ID!
  value: String!
}

type Mutation {
  submitForm(formSpecId: ID!, responses: [QuestionResponseInput!]!): FormInstance!
}

GraphQL のネストクエリにより、フロントエンドは 1 回のリクエストで完全なフォーム定義（FormSpec → Groups → Questions）を取得でき、複数回のラウンドトリップが不要。Relay や Apollo の fragment メカニズムと組み合わせると、Question コンポーネントが必要なデータを宣言し、フレームワークが自動的にクエリを統合する。

ORM コード生成でバックエンドを簡素化

Go + Ent ORM を使えば、schema 定義から直接 CRUD コードと GraphQL リゾルバーを生成できる：

// ent/schema/formspec.go
func (FormSpec) Fields() []ent.Field {
    return []ent.Field{
        field.String("name").NotEmpty(),
        field.String("description").Optional(),
        field.String("cover").Optional(),
        field.Bool("enabled").Default(true),
        field.Time("created_at").Default(time.Now),
        field.Int("created_by"),
    }
}

func (FormSpec) Edges() []ent.Edge {
    return []ent.Edge{
        edge.To("question_groups", QuestionGroup.Type),
    }
}

// ent/schema/question.go
func (Question) Fields() []ent.Field {
    return []ent.Field{
        field.String("label").NotEmpty(),
        field.Enum("type").Values(
            "short_text", "paragraph", "multi_choice",
            "checkboxes", "drop_down", "file",
            "linear_scale", "date", "time",
        ),
        field.Bool("required").Default(false),
        field.JSON("options", []string{}).Optional(),
    }
}

Ent はこれらの schema 定義から自動生成する：データベースマイグレーション、Go 型定義、CRUD メソッド、GraphQL type とリゾルバー（entgql と組み合わせ）。schema から使用可能な API まで、ほとんどグルーコードを書く必要がない。

バリデーション戦略

フォームバリデーションは 2 層に分かれる：

フロントエンドバリデーション（即時フィードバック）：

function validate(question, value) {
  if (question.required && !value) {
    return `${question.label} は必須項目です`;
  }

  const config = question.config;
  if (!config) return null;

  switch (question.type) {
    case "short_text":
    case "paragraph":
      if (config.minLength && value.length < config.minLength)
        return `最低 ${config.minLength} 文字入力してください`;
      if (config.maxLength && value.length > config.maxLength)
        return `最大 ${config.maxLength} 文字まで入力できます`;
      break;

    case "file":
      if (config.fileTypes && !config.fileTypes.some(t => value.endsWith(t)))
        return `${config.fileTypes.join(", ")} 形式のみ対応しています`;
      break;
  }
  return null;
}

バックエンドバリデーション（安全保証）：

func ValidateResponse(spec *ent.FormSpec, responses []ResponseInput) error {
    questionMap := buildQuestionMap(spec)

    for _, resp := range responses {
        q, ok := questionMap[resp.QuestionID]
        if !ok {
            return fmt.Errorf("unknown question ID: %d", resp.QuestionID)
        }
        if q.Required && resp.Value == "" {
            return fmt.Errorf("question %q is required", q.Label)
        }
        if err := validateType(q.Type, resp.Value); err != nil {
            return fmt.Errorf("question %q: %w", q.Label, err)
        }
    }

    // すべての必須項目に回答があるかチェック
    for id, q := range questionMap {
        if q.Required && !hasResponse(responses, id) {
            return fmt.Errorf("missing required question: %q", q.Label)
        }
    }
    return nil
}

フロントエンドバリデーションはユーザー体験のため（サーバーへのラウンドトリップなしでエラーを確認できる）、バックエンドバリデーションはデータ整合性のため（フロントエンドバリデーションはバイパスできる）。どちらも省略できない。

実際に遭遇する問題

  • Schema のバージョン管理：フォーム公開後に Schema を変更した場合（例えば質問の削除）、既に送信された Response はどうなるか？一般的なアプローチは元バージョンを修正するのではなく新バージョンを作成するか、質問を soft delete して過去の回答は保持する。
  • 条件ロジック：現実のフォームには「A を選んだ場合、質問 X を表示」というロジックが頻繁にある。Schema に visibility_rules フィールドを追加し、レンダラーが現在の回答に基づいて質問を動的に表示/非表示にする必要がある。これが複雑さの主な源。
  • パフォーマンス：大きなフォーム（100 以上の質問）をレンダリングする際、各 Question コンポーネントの再レンダリングがボトルネックになる。解決策：React.memo + onChange コールバックを useCallback でラップし、値が変化した Question のみが再レンダリングされるようにする。
  • ファイルアップロード：ファイルタイプの質問には特別な処理が必要——まず OSS にアップロードして URL を取得し、その URL を value として保存する。つまりファイルアップロードとフォーム送信は 2 つの独立したフローである。
  • エクスポートと分析：フォームデータは最終的に Excel へのエクスポートや統計分析が必要。Value が string 型のため、エクスポート時に Question Type に基づいてフォーマットする必要がある（日付を Date 形式に変換、複数選択をカンマ区切りに変換など）。

市場のソリューションとの比較

ソリューション Schema 保存 レンダリング 適用シナリオ
Google Forms プロプライエタリ形式 Google ホスティング シンプルなアンケート
JSON Schema + react-jsonschema-form 標準 JSON Schema 自動生成 開発者ツール、設定フォーム
Formily / XRender カスタム DSL ビジュアルエディタ + レンダラー エンタープライズ級の複雑なフォーム
自作 FormSpec モデル データベース + ORM カスタムレンダラー 深いカスタマイズが必要なプロダクト

react-jsonschema-form は標準 JSON Schema（Draft 7）を使用しており、エコシステムが良くバリデーションライブラリ（ajv）が既にある。しかし標準 JSON Schema はデータ構造の記述のために設計されたもので、フォーム UI の記述のためではない——「グループ」「プレースホルダー」「条件表示」といった概念がなく、uiSchema で補完する必要がある。

自作 FormSpec モデルの利点はデータ構造を完全にコントロールでき、ビジネス要件に合わせた深いカスタマイズが可能なこと（承認フローの追加、権限制御の追加、テンプレートマーケットの追加など）。コストはレンダラーとバリデーションロジックを自分で書く必要があること。

アーキテクチャ上の考察

  • Schema が契約：FormSpec がフロントエンドとバックエンドの唯一の契約。バックエンドはフロントエンドのレンダリング方法を知る必要がなく、フロントエンドはバックエンドの保存方法を知る必要がない。Schema のフォーマットが変わらなければ、両側が独立して進化できる。
  • コード生成 vs 手書き：Ent + gqlgen のようなコード生成ツールを使えば、Schema 定義から API エンドポイントまでほぼ全自動。手書きの CRUD コードには何の価値もない——マシンに生成させ、ビジネスロジックにエネルギーを注ぐ。
  • Switch 文の美学：レンダラーの大きな switch は「原始的」に見えるが、正しい抽象レベルである。「レジストリパターン」や「プラグインシステム」に過度に抽象化してはいけない——サードパーティ開発者に質問タイプを拡張させる必要がない限り。ほとんどの場合、9 種類で十分。
  • Value の string 保存について：これは古典的な「柔軟性 vs クエリ能力」のトレードオフ。回答に対する複雑なクエリが必要な場合（「A オプションを選択したすべてのユーザーを見つける」）、PostgreSQL の JSONB 型の使用や、送信時に分析用のワイドテーブルに追加で書き込むことを検討できる。

まとめ

Schema 駆動フォームの本質は、古くからのソフトウェアエンジニアリングの思想の具体的な応用：データと振る舞いの分離。フォームの構造（データ）は Schema で記述し、フォームのレンダリングとバリデーション（振る舞い）は汎用コードで処理する。これにより非開発者がフォームを作成・変更でき、開発者は 1 セットのコードですべてのフォームシナリオに対応できる。

このパターンはフォームに限らない。設定ページ、承認フロー、データダッシュボード——「構造は変わるが振る舞いは固定」のあらゆるシナリオに Schema 駆動のアプローチが使える。核心は常に：良い Schema を定義し、良いレンダラーを書くこと。

しかし現時点では、我々の Schema はフラットなフォームフィールドしか記述できない。条件レンダリング、動的表現、レイアウトのネスト、インタラクションアクションが必要な場合は？次の記事で Schema の能力を UI ツリー全体に拡張する。
