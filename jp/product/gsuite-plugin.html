Google Workspace プラグイン向け Webpack スキャフォールドの構築

2026-03-08
この記事の価値：Google Workspace プラグインの開発体験は非常に原始的——npm なし、ビルドツールなし、すべてのリソースを単一の HTML ファイルにインラインで含める必要がある。この記事では、React + TypeScript で GSuite プラグインを開発できる Webpack スキャフォールドを共有する。新しいエントリの自動検出、JS/CSS の自動インライン化、1 回のビルドですべての GSuite プロダクトをカバー。GSuite プラグインで複雑な UI インタラクションが必要な場合、このスキャフォールドで大量の反復作業を省ける。

Google Workspace プラグインとは

Google Workspace（旧 GSuite）では、Docs、Sheets、Slides、Forms などのプロダクト向けにプラグインを開発できる。プラグインの UI 形態は 2 つ：Sidebar（サイドバー、常駐）と Dialog（ダイアログ、一時的）。

プラグインのアーキテクチャは 2 層に分かれる：

┌──────────────────────────────────────────┐
│           Google Docs / Sheets / ...      │
│                                           │
│   ┌─────────────┐    ┌───────────────┐   │
│   │   Sidebar    │    │    Dialog     │   │
│   │  (HTML/JS)   │    │  (HTML/JS)   │   │
│   │  ← iframe →  │    │  ← iframe → │   │
│   └──────┬───────┘    └──────┬───────┘   │
│          │                    │           │
│          │  google.script.run │           │
│          ▼                    ▼           │
│   ┌──────────────────────────────────┐   │
│   │        AppScript (サーバーサイド)  │   │
│   │  - Drive / Gmail / Sheets にアクセス │   │
│   │  - ドキュメント内容の操作            │   │
│   │  - メール送信                      │   │
│   └──────────────────────────────────┘   │
└──────────────────────────────────────────┘
  • フロントエンド：iframe サンドボックス内で動作する HTML ページ、AppScript の HtmlService がホスティング
  • バックエンド：AppScript コード、Google のサーバー上で動作し、Google API（Drive、Gmail、Sheets など）を呼び出せる
  • 通信：フロントエンドは google.script.run 経由でバックエンド関数を非同期呼び出し

このモデルには 3 つの必須制約があり、ビルド方針を直接決定する：

  1. CSP 制限：iframe サンドボックスは外部スクリプトとスタイルシートの読み込みを禁止。すべての JS と CSS を HTML ファイル内にインラインで含める必要がある
  2. 単一ファイル配信：HtmlService.createHtmlOutputFromFile("sidebar") は HTML ファイル名を 1 つだけ受け付ける。複数ファイルに分割できない
  3. クロスプロダクトデプロイ：同じプラグインを Docs、Sheets、Slides、Forms にそれぞれデプロイする必要があり、各プロダクトは独立した AppScript プロジェクト

公式が推奨する開発方法は、Apps Script エディタ内で直接 HTML + JavaScript を書くこと。シンプルなプラグインにはこれで十分だが、UI に状態管理、マルチステップフォーム、条件レンダリングなどの要件があると、コンポーネント化とビルドツールなしの開発体験は苦痛になる。

以下に、これらの問題を解決するために Webpack で構築したスキャフォールドを紹介する。

スキャフォールドの目標

このスキャフォールドが達成すべきこと：

  1. React + TypeScript 開発：モダンフロントエンドの方法で GSuite プラグイン UI を書く
  2. 自己完結型 HTML 出力：ビルド成果物は単一の HTML ファイルで、JS/CSS はすべてインライン化され、GSuite の CSP 制限を直接満たす
  3. 自動検出：新しい dialog を追加するには、規約のディレクトリにフォルダを作るだけで、ビルド設定の変更は不要
  4. マルチプロダクトマトリクスビルド：1 回のビルドで、各 GSuite プロダクト向けに独立したデプロイディレクトリを生成
  5. ローカル開発：GSuite にデプロイしなくてもブラウザで UI をデバッグ可能

プロジェクト構造の規約

project/
├── src/
│   ├── sidebar/           # メイン sidebar（必須）
│   │   ├── App.tsx
│   │   └── index.tsx
│   └── dialogs/           # dialog ディレクトリ（自動検出）
│       ├── about/
│       │   ├── App.tsx
│       │   └── index.tsx
│       └── settings/
│           ├── App.tsx
│           └── index.tsx
├── appscript/
│   └── shared.gs          # AppScript サーバーサイドコード
├── clasp/
│   ├── docs/
│   │   ├── .clasp.json    # Docs の AppScript プロジェクト ID
│   │   └── appsscript.json

│   └── ...
├── webpack.config.js

この構造のコア規約：src/sidebar/ は固定エントリで、src/dialogs/ 以下の index.tsx を含むサブディレクトリが自動的に dialog エントリになる。

第 1 ステップ：Entry Points の自動検出

Webpack の entry は通常手動で設定するが、dialog を追加する際に設定を変更したくない。そこで Webpack 起動時にファイルシステムをスキャンする：

const path = require("path");
const fs = require("fs");

function discoverDialogs() {
  const dialogsPath = path.resolve(__dirname, "src/dialogs");
  if (!fs.existsSync(dialogsPath)) return [];

  return fs.readdirSync(dialogsPath, { withFileTypes: true })
    .filter(item => item.isDirectory())
    .filter(dir => {
      return fs.existsSync(
        path.join(dialogsPath, dir.name, "index.tsx")
      );
    })
    .map(dir => dir.name);
}

// entry points を動的生成
const dialogs = discoverDialogs();
const entry = { sidebar: "./src/sidebar/index.tsx" };
dialogs.forEach(name => {
  entry[name] = `./src/dialogs/${name}/index.tsx`;
});

こうすると entry は { sidebar: "...", about: "...", settings: "..." } になる。新しい dialog を追加したい場合は、src/dialogs/ にフォルダを作り index.tsx を置くだけで、次のビルドで自動的に有効になる。

第 2 ステップ：JS と CSS のインライン化

通常の Webpack ビルド成果物は独立した .js と .css ファイルで、HTML が <script src> と <link rel="stylesheet"> で参照する。GSuite ではこれが機能しない——iframe サンドボックスがこれらの外部リソースの読み込みをブロックする。

解決策は 2 つのプラグインの組み合わせ：

const HtmlWebpackPlugin = require("html-webpack-plugin");
const HtmlInlineScriptPlugin =
  require("html-inline-script-webpack-plugin");

// JS インライン：HtmlInlineScriptPlugin が Webpack 出力の JS バンドルを
//          直接 HTML の <script> タグに埋め込む
plugins: [
  new HtmlWebpackPlugin({
    template: "./public/index.html",
    chunks: ["sidebar"],
    inject: "body",
  }),
  new HtmlInlineScriptPlugin(),
]

// CSS インライン：mini-css-extract-plugin の代わりに style-loader を使用
module: {
  rules: [
    {
      test: /\.css$/,
      use: ["style-loader", "css-loader"],
      // style-loader はランタイムに CSS を <style> タグに注入
      // 独立した .css ファイルを生成しない
    },
  ],
}

ビルド後、各 HTML ファイルは完全に自己完結型——外部依存ゼロ。HtmlService.createHtmlOutputFromFile("sidebar") でそのまま使用可能。

第 3 ステップ：マルチプロダクトマトリクス生成

同じプラグインを Docs、Sheets、Slides、Forms の 4 つのプロダクトにデプロイする必要がある。各プロダクトは独立した AppScript プロジェクトで、独立したデプロイディレクトリが必要。Webpack 設定ですべての HtmlWebpackPlugin インスタンスを動的生成する：

const GSUITE_PRODUCTS = ["docs", "slides", "sheets", "forms"];
const htmlPlugins = [];

GSUITE_PRODUCTS.forEach(product => {
  // sidebar
  htmlPlugins.push(new HtmlWebpackPlugin({
    template: "./public/index.html",
    filename: `${product}/sidebar.html`,
    chunks: ["sidebar"],
    inject: "body",
  }));

  // 自動検出された各 dialog
  dialogs.forEach(name => {
    htmlPlugins.push(new HtmlWebpackPlugin({
      template: "./public/index.html",
      filename: `${product}/${name}.html`,
      chunks: [name],
      inject: "body",
    }));
  });
});

ビルド成果物の構造：

dist/
├── docs/
│   ├── sidebar.html       # 自己完結型 HTML
│   ├── about.html
│   ├── settings.html
│   └── appscript/
│       └── shared.gs      # コピーされた AppScript コード
├── sheets/
│   ├── sidebar.html
│   ├── about.html
│   └── settings.html
├── slides/
│   └── ...
└── forms/
    └── ...

4 プロダクト × 3 エントリ = 12 の HTML ファイルがすべて自動生成される。JS コードは 1 回だけコンパイルされ、chunks 設定で対応する HTML に配布される。

第 4 ステップ：AppScript ファイルのコピー

AppScript の .gs ファイルはコンパイル不要だが、各プロダクトの出力ディレクトリにコピーする必要がある。CopyWebpackPlugin を使用：

const CopyWebpackPlugin = require("copy-webpack-plugin");

GSUITE_PRODUCTS.forEach(product => {
  plugins.push(new CopyWebpackPlugin({
    patterns: [{
      from: "appscript/",
      to: `${product}/appscript/`,
    }],
  }));
});

これにより clasp push 時、各プロダクトディレクトリに HTML と .gs の両方があり、完全な AppScript プロジェクトになる。

第 5 ステップ：ローカル開発サポート

google.script.run は GSuite 環境内でのみ存在する。ローカルブラウザで開発する際は undefined。解決方針はシンプル：環境を検出し、存在しない場合はモックデータを使用。

// AppScript 呼び出しのラッパー
function callAppScript<T>(
  methodName: string,
  ...args: any[]
): Promise<T> {
  return new Promise((resolve, reject) => {
    if (typeof google !== "undefined" && google.script) {
      const runner = google.script.run
        .withSuccessHandler(resolve)
        .withFailureHandler(reject);
      runner[methodName](...args);
    } else {
      // ローカル開発：モックデータを返す
      resolve(getMockData(methodName, args) as T);
    }
  });
}

// 使用
const sheets = await callAppScript<Sheet[]>("getGoogleSheets");

これには 2 つの利点がある：

  1. ローカル開発が GSuite 環境に依存しない：webpack-dev-server でローカルサービスを起動し、ブラウザで UI とインタラクションロジックをデバッグ
  2. コールバックスタイルの google.script.run を Promise にラップ：React コンポーネント内で async/await が使え、コードがよりクリーンに

第 6 ステップ：TypeScript 型宣言

google.script.run には公式の型定義がない。手動で宣言し、TypeScript にどの AppScript 関数が呼び出し可能か伝える必要がある：

// types/google.d.ts
interface AppScriptRunner {
  getGoogleSheets: () => void;
  getWorksheets: (spreadsheetId: string) => void;
  getSelectedSheetData: (
    spreadsheetId: string,
    worksheetName: string
  ) => void;
  getDocumentHtml: (docId: string) => void;
  sendMailMerge: (
    spreadsheetId: string,
    worksheetName: string,
    docId: string,
    subject: string,
    replyTo: string,
    testMode: boolean
  ) => void;
}

declare const google: {
  script: {
    run: {
      withSuccessHandler: (cb: (result: any) => void) => {
        withFailureHandler: (
          cb: (error: any) => void
        ) => AppScriptRunner;
      };
    };
  };
};

新しい AppScript 関数を追加するたびに、ここに 1 行の宣言を追加。戻り値の型はまだ any（AppScript は動的型付け）だが、少なくとも呼び出しパラメータの型チェックがある。

第 7 ステップ：デプロイパイプライン

Google の clasp CLI でビルド成果物を AppScript プロジェクトにプッシュする。各 GSuite プロダクトは独立した AppScript プロジェクトで、独立した .clasp.json を持つ：

# clasp/docs/.clasp.json
{
  "scriptId": "1a2b3c...",        # Docs の AppScript プロジェクト ID
  "rootDir": "../../dist/docs"    # ビルド成果物を指す
}

# デプロイフロー
npm run build                     # すべてのプロダクトの HTML をビルド
cd clasp/docs && clasp push       # Docs の AppScript プロジェクトにプッシュ
cd clasp/sheets && clasp push     # Sheets の AppScript プロジェクトにプッシュ
# ... 各プロダクトを個別にプッシュ

clasp push は rootDir 以下のすべての .html と .gs ファイルを Google Apps Script にアップロードする。すべてのプロダクトのプッシュをまとめるスクリプトを書ける：

# deploy.sh
npm run build
for product in docs sheets slides forms; do
  echo "Deploying to $product..."
  (cd clasp/$product && clasp push)
done

GSuite AppScript 側のパターン

スキャフォールドはフロントエンドのビルド問題を解決するが、AppScript 側にも記録に値するパターンがある。

クロスプロダクト適応：同じプラグインが Docs/Sheets/Slides/Forms で動作するが、AppScript には「現在どのプロダクトか」を示す API がない。try/catch で検出するしかない：

function getUi() {
  try { return DocumentApp.getUi(); } catch(e) {}
  try { return SpreadsheetApp.getUi(); } catch(e) {}
  try { return SlidesApp.getUi(); } catch(e) {}
  try { return FormApp.getUi(); } catch(e) {}
  return null;
}

function showSidebar() {
  var ui = getUi();
  if (!ui) return;
  var html = HtmlService.createHtmlOutputFromFile("sidebar")
    .setTitle("My Plugin").setWidth(320);
  ui.showSidebar(html);
}

google.script.run の仕組み：フロントエンドがバックエンド関数を呼び出し、コールバックで結果を取得。AppScript のすべてのグローバル関数がフロントエンドから呼び出し可能：

// AppScript 側——グローバル関数を定義
function getGoogleSheets() {
  var files = DriveApp.getFilesByType(MimeType.GOOGLE_SHEETS);
  var sheets = [];
  while (files.hasNext()) {
    var file = files.next();
    sheets.push({
      id: file.getId(),
      name: file.getName(),
      url: file.getUrl(),
    });
  }
  return { success: true, sheets: sheets };
}

// React 側——呼び出し
google.script.run
  .withSuccessHandler((result) => {
    setSheets(result.sheets);
  })
  .withFailureHandler((error) => {
    console.error(error);
  })
  .getGoogleSheets();

実用 API：AppScript は Google Docs の HTML エクスポート、Sheets データの読み取り、Gmail の送信などを呼び出せる。これらが GSuite プラグインのコア機能：

// Sheets データの読み取り
function getSheetData(spreadsheetId, worksheetName) {
  var spreadsheet = SpreadsheetApp.openById(spreadsheetId);
  var sheet = spreadsheet.getSheetByName(worksheetName);
  var values = sheet.getDataRange().getValues();
  return { headers: values[0], rows: values.slice(1) };
}

// Docs コンテンツの読み取り（HTML としてエクスポート、フォーマット保持）
function getDocumentHtml(docId) {
  var url = "https://docs.google.com/feeds/download/"
    + "documents/export/Export?id=" + docId
    + "&exportFormat=html";
  var response = UrlFetchApp.fetch(url, {
    headers: {
      Authorization: "Bearer " + ScriptApp.getOAuthToken()
    }
  });
  return response.getContentText();
}

// メール送信
function sendEmail(to, subject, htmlBody) {
  GmailApp.sendEmail(to, subject, "", {
    htmlBody: htmlBody,
  });
}

実際のケース：Mail Merge

このスキャフォールドで Mail Merge 機能を作った：Google Sheets から受信者リストを読み取り、Google Docs をテンプレートとして使い、パーソナライズされたメールを一括送信。

ユーザー操作（sidebar 内の 4 ステップウィザード）：

Step 1: データソース選択 → Drive 内の Sheets を一覧 → sheet タブを選択 → ヘッダーをプレビュー
Step 2: テンプレート選択 → Drive 内の Docs を一覧 → HTML コンテンツをプレビュー
Step 3: 設定       → メール件名（{{Name}} マージフィールド対応）→ reply-to アドレス
Step 4: 実行       → サマリー確認 → テストモード → 送信して結果を表示

フロントエンドは標準的な React マルチステップフォーム（state で step を制御）で、各ステップのデータは callAppScript 経由でバックエンドから取得。バックエンドがフィールド置換とメール送信を行う：

function sendMailMerge(spreadsheetId, worksheetName,
                       docId, subject, replyTo, testMode) {
  var data = getSheetData(spreadsheetId, worksheetName);
  var template = getDocumentHtml(docId);
  var results = [];

  for (var i = 0; i < data.rows.length; i++) {
    var row = {};
    data.headers.forEach(function(h, j) { row[h] = data.rows[i][j]; });

    var body = template.replace(/\{\{([^}]+)\}\}/g,
      function(match, field) {
        return row[field.trim()] || match;
      });
    var emailSubject = subject.replace(/\{\{([^}]+)\}\}/g,
      function(match, field) {
        return row[field.trim()] || match;
      });

    var recipient = testMode
      ? Session.getActiveUser().getEmail()
      : row["Email"];

    try {
      GmailApp.sendEmail(recipient, emailSubject, "", {
        htmlBody: body,
        replyTo: replyTo || undefined,
      });
      results.push({ row: i + 1, status: "sent" });
    } catch (e) {
      results.push({ row: i + 1, status: "error", error: e.message });
    }

    Utilities.sleep(100); // 送信速度の制御
  }

  return { success: true, results: results };
}

この Mail Merge はスキャフォールドの価値を示している：フロントエンドは複雑なマルチステップインタラクション（ファイル選択、データプレビュー、マージフィールドの設定、送信結果の表示）で、ネイティブ HTML で書くのは苦痛。React のコンポーネント化と状態管理がここで明確な優位性を持つ。

開発体験のペインポイント

このアプローチは完璧ではない。回避できない問題がいくつかある：

  • ホットリロードなし：AppScript コードを変更したら clasp push + GSuite ページのリフレッシュが必要。フロントエンド部分は webpack-dev-server でローカルデバッグ可能だが、AppScript インタラクションに関わるロジックは GSuite 内でしかテストできない
  • バンドルサイズ：React を HTML にインライン化すると、各ファイルは約 1MB。4 プロダクト × N エントリ = 総成果物がかなり大きくなる可能性。Preact で React を置き換えるか、TerserPlugin で圧縮可能
  • AppScript の 6 分実行制限：長いタスク（例えば 1000 通のメール送信）はタイムアウトする可能性。ループ内で速度を制御するか、バッチ処理が必要
  • デバッグの困難さ：iframe 内の console.log が見えない場合がある。AppScript 側は Logger.log() を使い、Apps Script エディタの実行ログで確認

まとめ

このスキャフォールドの核心的アプローチ：Webpack で「モダンフロントエンド開発」と「GSuite サンドボックス制約」をブリッジする。

  • HtmlInlineScriptPlugin + style-loader が CSP 制限を解決——自己完結型 HTML を生成
  • ファイルシステムスキャンで dialog の自動検出を実現——ゼロ設定で新しいエントリを追加
  • プロダクト × エントリのマトリクスループで HtmlWebpackPlugin を生成——1 回のビルドですべてのプロダクトをカバー
  • clasp + rootDir 設定でプロダクト別デプロイを実現——各プロダクトを独立してプッシュ
  • 環境検出 + モックデータでローカル開発を実現——GSuite 環境に依存せず UI をデバッグ

GSuite プラグインで複雑な UI インタラクションが必要な場合、このスキャフォールドで多くの反復作業を省ける。ボタン 1-2 個のシンプルなプラグインなら、ネイティブ HTML を直接書けば十分——このツールは不要。
