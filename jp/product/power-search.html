PowerSearch：コンポーザブルなフロントエンド高度検索バーの構築

2026-03-08
この記事の価値：ほぼすべての B 向けプロダクトに高度検索が必要だが、ほとんどの実装は入力ボックスを並べた粗末なものか、組み合わせやネストができない半端なものである。ここでは再帰的な条件ツリーに基づくデータモデルを共有し、検索バーが AND/OR の組み合わせと無限のネストを自然にサポートできるようにする——データ構造が正しければ、UI インタラクションもバックエンドマッピングもシンプルになる。

問題

B 向けプロダクトのリストページにはほぼすべて検索とフィルタリングが必要。最もシンプルなやり方は入力ボックスやドロップダウンを並べることだが、フィールドが増えると見た目が悪くなり、複雑なロジックを表現できない——例えば「ステータスが完了済み かつ（作成者が田中 または 作成者が佐藤）」。

Jira、Linear、Notion といったプロダクトの検索バーは、ユーザーが動的にフィルター条件を追加し、フィールド、演算子、値を選択し、さらに AND/OR で条件を組み合わせ、グループをネストできる。このインタラクションを PowerSearch（または Advanced Filter、Structured Search）と呼ぶ。

複雑に見えるが、データ構造は実は非常に明確——本質的に条件式のツリーである。

データモデル

システム全体の型定義はわずかいくつか：

// フィールド設定——検索可能なフィールドを定義
type FieldConfig = {
  id: string;          // "status"
  label: string;       // "ステータス"
  type: "text" | "number" | "date" | "select" | "boolean";
  operators: OperatorOption[];     // そのフィールドがサポートする演算子
  options?: { value: string; label: string }[];  // select タイプの選択肢
};

// 演算子
type OperatorOption = {
  id: string;      // "eq"
  label: string;   // "等しい"
  symbol: string;  // "="
};

// 単一のフィルター条件
type FilterCondition = {
  id: string;
  type: "condition";
  fieldId: string;     // どのフィールド
  operatorId: string;  // どの演算子
  value: string;       // 比較する値
};

// 条件グループ（再帰構造、ネスト対応）
type ConditionGroup = {
  id: string;
  type: "group";
  operator: "AND" | "OR";
  conditions: (FilterCondition | ConditionGroup)[];  // ネスト可能
};

鍵は最後の型：ConditionGroup の conditions 配列が FilterCondition（リーフノード）も ConditionGroup（サブツリー）も含められること。これが再帰的な式ツリーである。

実際のクエリはこのような形：

// "ステータス = 完了済み AND (作成者 = 田中 OR 作成者 = 佐藤)"
{
  id: "root",
  type: "group",
  operator: "AND",
  conditions: [
    {
      id: "c1", type: "condition",
      fieldId: "status", operatorId: "eq", value: "done"
    },
    {
      id: "g1", type: "group",
      operator: "OR",
      conditions: [
        {
          id: "c2", type: "condition",
          fieldId: "creator", operatorId: "eq", value: "tanaka"
        },
        {
          id: "c3", type: "condition",
          fieldId: "creator", operatorId: "eq", value: "sato"
        }
      ]
    }
  ]
}

インタラクションフロー

フィルター条件の追加は 3 ステップのガイドフロー：

1. フィールド選択 → フィールドリストをポップアップ（Field Dropdown）
                ┌─────────────────┐
                │ Select Field    │
                ├─────────────────┤
                │ 名前      text  │
                │ ステータス select│
                │ 作成日    date  │
                │ 優先度  number  │
                └─────────────────┘

2. 演算子選択 → フィールドタイプに応じた演算子を表示
                ┌─────────────────┐
                │ Select Operator │
                ├─────────────────┤
                │ 等しい        = │
                │ 等しくない    ≠ │
                └─────────────────┘

3. 値入力 → フィールドタイプに応じた入力方法を表示
   text   → テキスト入力ボックス
   number → 数値入力ボックス
   date   → 日付ピッカー
   select → 選択肢リスト
   boolean → true/false 選択

各ステップ完了後に自動的に次のステップへ進み、3 ステップ完了で自動的に条件が送信される。この段階的なガイドは、3 つのドロップダウンを横に並べて直接表示するより体験が良い——ユーザーは毎回 1 つの判断だけすればよい。

状態管理

コンポーネント内部には 2 つのコア状態がある：

// 1. 条件ツリー（完全なクエリ構造）
const [rootGroup, setRootGroup] = useState<ConditionGroup>({
  id: "root",
  type: "group",
  operator: defaultOperator,
  conditions: [],
});

// 2. 現在編集中の条件（3 ステップガイドの中間状態）
const [editingCondition, setEditingCondition] = useState<{
  condition: Partial<FilterCondition>;  // フィールドのみ選択済みで演算子未選択かもしれない
  groupId: string;   // どのグループに属するか
  index?: number;    // 既存条件の編集時のインデックス
} | null>(null);

条件ツリーの変更はすべてディープコピーで実装（immutable update）し、React が変更を正しく検出できるようにする：

// 条件ツリーのディープコピー
const deepCloneGroup = (group: ConditionGroup): ConditionGroup => ({
  ...group,
  conditions: group.conditions.map((c) =>
    c.type === "group" ? deepCloneGroup(c) : { ...c }
  ),
});

// ネストされたツリー内でターゲットグループを見つける
const findGroupById = (
  group: ConditionGroup, id: string
): ConditionGroup | null => {
  if (group.id === id) return group;
  for (const c of group.conditions) {
    if (c.type === "group") {
      const found = findGroupById(c, id);
      if (found) return found;
    }
  }
  return null;
};

条件を変更するたびに：ツリーをディープコピー → ターゲットグループを見つける → 変更 → 新しい状態を設定 → onSearch コールバックをトリガー。

値入力のタイプ適応

第 3 ステップ「値入力」はフィールドタイプに応じて異なる入力コンポーネントをレンダリングする。これは switch の分岐：

const renderValueInput = () => {
  const field = getField(editingCondition.condition.fieldId);

  switch (field.type) {
    case "select":
      // 選択肢リストをレンダリング、クリックで直接送信
      return <OptionsList options={field.options} onSelect={handleSubmit} />;

    case "boolean":
      // true/false の 2 つの選択肢のみ
      return <BooleanPicker onSelect={handleSubmit} />;

    case "date":
      // 日付ピッカー
      return <input type="date" onChange={handleValueChange} />;

    case "number":
      // 数値入力ボックス
      return <input type="number" onChange={handleValueChange} />;

    default:  // text
      // テキスト入力ボックス、Enter で送信
      return <input type="text" onChange={handleValueChange}
                     onKeyDown={e => e.key === "Enter" && handleSubmit()} />;
  }
};

select と boolean タイプの特別な処理に注目：ユーザーが選択肢をクリックすると直接条件が送信される。追加の「確認」ボタンが不要。クリック 1 回分が減り、インタラクションがよりスムーズになる。

AND/OR 切り替えと条件グループ化

条件間の論理関係は ConditionGroup.operator で制御する。UI 上ではクリック可能な小さなラベルで、クリックするたびに AND と OR を切り替える：

const handleToggleOperator = (groupId: string) => {
  setRootGroup((prev) => {
    const updated = deepCloneGroup(prev);
    const group = findGroupById(updated, groupId);
    if (group) {
      group.operator = group.operator === "AND" ? "OR" : "AND";
    }
    onSearch(updated);
    return updated;
  });
};

ネストされたグループによりユーザーはより複雑なロジックを表現できる。例えば「(A AND B) OR (C AND D)」——トップレベルが OR で、2 つのサブグループがそれぞれ AND。実装上、最大ネスト深度を 3 層に制限し、条件ツリーが複雑になりすぎるのを防ぐ：

// 深度 < 3 の場合のみ「グループ追加」ボタンを表示
{depth < 3 && (
  <button onClick={() => handleAddGroup(group.id)}>
    Add Group
  </button>
)}

条件のレンダリング

条件ツリーのレンダリングも再帰的。各 ConditionGroup が自身の operator ラベルと conditions リストをレンダリングし、サブグループに出会ったら自身を再帰呼び出しする：

const renderGroup = (group: ConditionGroup, depth = 0) => (
  <div className={depth > 0 ? "nested-group" : ""}>
    {/* AND/OR 切り替えボタン */}
    {group.conditions.length > 0 && (
      <button onClick={() => handleToggleOperator(group.id)}>
        {group.operator}
      </button>
    )}

    {/* 各条件を順にレンダリング */}
    {group.conditions.map((condition, index) => {
      if (condition.type === "group") {
        // サブグループを再帰的にレンダリング
        return renderGroup(condition, depth + 1);
      } else {
        // 単一の条件ピルをレンダリング
        const field = getField(condition.fieldId);
        const operator = getOperator(condition.fieldId, condition.operatorId);
        return (
          <div className="condition-pill">
            <span className="field">{field?.label}</span>
            <span className="operator">{operator?.symbol}</span>
            <span className="value">{condition.value}</span>
            <button onClick={() => handleRemove(group.id, index)}>×</button>
          </div>
        );
      }
    })}

    {/* 条件/グループ追加ボタン */}
    <button onClick={() => handleAddCondition(group.id)}>+ Add Filter</button>
    {depth < 3 && <button onClick={() => handleAddGroup(group.id)}>() Add Group</button>}
  </div>
);

ネストされたサブグループは左ボーダーとインデント（border-left + padding-left）で表現し、視覚的に階層関係を明確に伝える。

バックエンドとの連携

onSearch コールバックが返すのは ConditionGroup ツリー。バックエンドは通常これをデータベースクエリに変換する必要がある。この変換は再帰的：

// SQL WHERE 句に変換（擬似コード）
function toSQL(group: ConditionGroup): string {
  const parts = group.conditions.map((c) => {
    if (c.type === "group") {
      return `(${toSQL(c)})`;
    }
    const op = operatorToSQL(c.operatorId);  // "eq" → "=", "contains" → "LIKE"
    const val = escapeSQL(c.value);
    return `${c.fieldId} ${op} ${val}`;
  });
  return parts.join(` ${group.operator} `);
}

// 出力: status = 'done' AND (creator = 'tanaka' OR creator = 'sato')

GraphQL を使う場合、ツリー構造をネストされた where input に直接マッピングできる：

// GraphQL where input
{
  and: [
    { status: { eq: "done" } },
    {
      or: [
        { creator: { eq: "tanaka" } },
        { creator: { eq: "sato" } }
      ]
    }
  ]
}

データ構造は同型——フロントエンドの ConditionGroup と GraphQL の where input は同じツリーの異なる表現に過ぎない。変換はほぼ一対一のマッピング。

設計上の考察

  • なぜフラット配列ではなくツリーか：フラットな { field, op, value }[] 配列は全 AND または全 OR の関係しか表現できない。混合ロジック（「A AND (B OR C)」）が必要になった時点でツリーが必要。ツリーは最も汎用的な構造であり、フラット配列はツリーの退化した特殊ケースに過ぎない。
  • なぜ FieldConfig に operators を含めるか：異なるタイプのフィールドは異なる演算子をサポートする——テキストフィールドは「含む」、数値フィールドは「より大きい/より小さい」、ブールフィールドは「等しい」のみ。operators をフィールドに紐付けることで（グローバル定義ではなく）、「日付フィールドで含むを選択」という無意味な組み合わせを防ぐ。
  • なぜ 3 つのドロップダウンを横に並べるのではなく 3 ステップに分けるか：段階的ガイドが認知的負荷を減らす。ユーザーは毎回 1 つの選択に集中するだけで、3 つのドロップダウンに同時に面してどこから始めるか迷わない。
  • ネスト深度の制限：理論上、再帰構造は無限にネストできるが、実際には 3 層を超えるネストの検索条件はほぼ保守不能。深度の制限はユーザーへの保護。
  • 条件はデータそのもの：検索状態全体がシリアライズ可能な JSON ツリー。つまり URL query string に保存（検索結果の共有）、ユーザー設定に永続化（よく使うフィルターの保存）、「ビュー」として保存（Notion の database views のように）できる。

拡張の方向性

  • ビューとして保存：ConditionGroup JSON をデータベースに保存し、ユーザーが次回アクセスした際に直接ロード。Notion のフィルタービューのように。
  • URL シリアライズ：条件ツリーを URL にエンコード（base64 またはカスタム構文）し、検索結果ページを他の人と共有可能に。
  • フィールド間の依存：「国 = 日本」を選択した後、「都市」フィールドの選択肢が自動的に日本の都市にフィルタリングされる。FieldConfig に dependsOn と動的な options ローディングを追加する必要がある。
  • カスタム演算子：例えば「リスト内」（IN）、「空」（IS NULL）、「範囲内」（BETWEEN）。FieldConfig.operators に新しい OperatorOption を追加するだけで、レンダラーが自動適応。
  • キーボードナビゲーション：矢印キーで条件間を移動、Tab でステップ切り替え、Delete で条件削除。高頻度使用ユーザーにとってキーボード操作はマウスより遥かに速い。

まとめ

PowerSearch のコアは 2 つだけ：再帰的な条件ツリーデータ構造と、3 ステップの段階的な条件編集インタラクション。データ構造がどれだけ複雑なクエリを表現できるかを決め（任意の AND/OR ネスト）、インタラクション設計が複雑なクエリを構築しやすいかを決める（フィールド → 演算子 → 値のガイドフロー）。

エンジニアリングの観点から、このコンポーネントのコアコード量は多くない——型定義 30 行、状態管理 50 行、再帰レンダリング 50 行。大部分のコードは UI の詳細（ドロップダウンのスタイル、ホバーエフェクト、キーボードイベント）にある。これはフロントエンドコンポーネントの常態：コアロジックはシンプルだが、良いインタラクション体験にするには大量のディテールの作り込みが必要。
