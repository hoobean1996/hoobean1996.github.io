SSR Server を自作する：GraphQL アプリのためのカスタムサーバーサイドレンダリング

2026-03-08
この記事の価値：Next.js は優れているが、技術スタックが Relay + GraphQL の場合、その SSR モデルは Relay の EntryPoint データプリロード哲学と衝突する。この記事ではゼロから SSR Server を自作した過程を記録する：ルーティングマッチング、データプリロード、ストリーミングレンダリング、Hydration、そして Relay の EntryPoint パターンをサーバーサイドで正しく動作させる方法。SSR 方案を深くカスタマイズする必要があるチームの参考になるだろう。

背景

Next.js は素晴らしいが、Relay（Meta の GraphQL クライアント）との統合はずっとスムーズとは言えない。Relay には独自のデータプリロード哲学——EntryPoint パターンがあり、ルーティングマッチング時にデータの fetch を開始し、コンポーネントのレンダリング時まで待たないことを望む。これは Next.js の getServerSideProps / Server Components のモデルと微妙に衝突する。

そこで自分で SSR server を書くことにした。Relay + Fluent UI の技術スタック向けに専用にカスタマイズした。この記事ではその過程の重要なステップと踏んだ落とし穴を記録する。

全体アーキテクチャ

浏览器请求 GET /pages/admin/42?sort=name
    ↓
Express Server
    ↓
路由匹配 (ssrrouter.lookup)
    ↓ 返回 entrypoint 名称 + 参数绑定
EntryPoint 解析
    ↓ 根据名称找到 EntryPoint 配置
GraphQL 数据预加载 (Relay loadQuery)
    ↓
React renderToPipeableStream (流式 SSR)
    ↓
HTML 流式响应 → 浏览器
    ↓
hydrateRoot (客户端接管)

第一歩：EntryPoint インターフェースの定義

EntryPoint はアーキテクチャ全体のコア抽象である。各「ページ」は React コンポーネントではなく EntryPoint オブジェクトであり、何のコンポーネントをレンダリングするか、どのパラメータが必要か、どのデータをプリロードするかを記述する。

// EntryPoint.tsx
interface EntryPoint<TParams> {
  root: ComponentType<PreloadedProps>;
  getParams: (path: string) => TParams;
  getPreloadedProps: (params: TParams) => PreloadedProps;
}

interface PreloadedProps {
  queries: {
    [key: string]: {
      query: GraphQLTaggedNode;
      variables: Variables;
    };
  };
}

具体的な EntryPoint はこのようになる：

// App.entrypoint.tsx
const AppEntryPoint: EntryPoint<Params> = {
  root: App,
  getParams: (path: string): Params => {
    return { limit: 10 };
  },
  getPreloadedProps: (params: Params): PreloadedProps => {
    return {
      queries: {
        preloadedQuery: {
          query: AppQuery,        // Relay GraphQL query
          variables: {},
        },
      },
    };
  },
};

この設計は Relay チームの EntryPoint pattern に由来する。核心的な考え方は、データ依存をコンポーネント層ではなくルーティング層で宣言すること。ルーティングがマッチした瞬間にどのデータを fetch する必要があるかが分かり、React のレンダリングツリーが構築されるのを待つ必要がない。

第二歩：ルーティングマッチング

ルーティング設定は URL パターンを EntryPoint にマッピングする：

// routes.ts
export const ssrRouterConfig = [
  "route /      AppEntrypoint",
  "route /admin AdminEntrypoint",
  "route /admin/:id(int) AppEntrypoint",
];

export const ssrEntrypointConfig = new Map<string, EntryPoint<any>>([
  ["AppEntrypoint", AppEntryPoint],
  ["AdminEntrypoint", AdminEntryPoint],
]);

リクエストが来たら、まずルーティングマッチングを行い、対応する EntryPoint を検索する：

const path = req.path.substring(6);  // /pages プレフィックスを除去
const entrypointRoute = ssrrouter.lookup(path);
const entrypoint =
  ssrEntrypointConfig.get(entrypointRoute?.entrypoint ?? "") ??
  FallbackEntryPoint;

第三歩：GraphQL データプリロード

これが SSR で最も重要なステップである。EntryPoint を取得したら、getPreloadedProps を呼んで必要なすべての GraphQL query を取得し、Relay の loadQuery でサーバーサイドからリクエストを発行する：

const { root: Root, getPreloadedProps, getParams } = entrypoint;
const params = getParams(path);
const { queries } = getPreloadedProps(params);

const preloadedData = Object.keys(queries).reduce((acc, key) => {
  acc[key] = loadQuery(
    RelayEnvironment,    // サーバーサイドの Relay Environment
    queries[key].query,
    queries[key].variables
  );
  return acc;
}, {});

loadQuery は即座に fetch を開始し、参照を返す。React がこのデータを必要とするコンポーネントをレンダリングする時、データが既に到着していればそのまま使い、まだなら suspend する（Suspense と連携）。

第四歩：ストリーミング SSR レンダリング

React 18 は renderToPipeableStream を導入し、ストリーミング HTML 出力をサポートした。従来の renderToString と比べて、二つのコアな利点がある：

  • Time to First Byte が短い：shell の準備ができたら送信開始、すべてのデータが揃うのを待たない
  • Suspense をサポート：データが準備できていない部分はまず fallback を送り、データが到着したらストリーミングで補完する
const { pipe, abort } = renderToPipeableStream(
  <RelayEnvironmentProvider environment={RelayEnvironment}>
    <Suspense fallback={<div>...</div>}>
      <RendererProvider renderer={renderer}>
        <SSRProvider>
          <FluentProvider theme={webLightTheme}>
            <Root queries={preloadedData} />
          </FluentProvider>
        </SSRProvider>
      </RendererProvider>
    </Suspense>
  </RelayEnvironmentProvider>,
  {
    bootstrapScripts: ["/entrypoint.client.js"],
    onShellReady() {
      res.setHeader("content-type", "text/html");
      res.write(`<!DOCTYPE html>
<html>
<head>...styles...</head>
<body>
  <div id="root">`);
      pipe(res);        // React のレンダリングストリームを HTTP レスポンスに接続
      res.write(`</div>
</body>
</html>`);
    },
    onShellError(error) {
      res.statusCode = 500;
      res.send("Something went wrong");
    },
  }
);

onShellReady コールバックに注目——React が同期部分（shell）のレンダリングを完了した時に発火する。まず HTML のヘッダーを書き出し、pipe(res) でレンダリングストリームを HTTP レスポンスストリームに接続する。Suspense boundary 内の非同期データは、後続の <script> タグの形でストリーミング注入される。

第五歩：Fluent UI の SSR スタイル処理

Fluent UI（Microsoft の React コンポーネントライブラリ）は CSS-in-JS を使用しており、SSR 時に特別な処理が必要で、さもなければページがちらつく（FOUC）：

const renderer = createDOMRenderer();
const style = renderToStaticMarkup(
  <>{renderToStyleElements(renderer)}</>
);

まず DOM renderer を作成し、レンダリング時に使用されたすべてのスタイルを収集し、スタイルを HTML の <head> にインラインで挿入する。クライアントサイドの hydrate 時にこれらのスタイルが再利用され、重複注入を防ぐ。

第六歩：クライアントサイド Hydration

クライアントサイドではサーバーサイドとまったく同じルーティングマッチングとデータプリロードを行うが、一つ重要な違いがある——fetchPolicy: "store-only"：

// Entrypoint.client.tsx
ssrrouter.createRouter(ssrRouterConfig);  // 同じルーティング設定

const path = window.location.pathname.substring(6);
const entrypointRoute = ssrrouter.lookup(path);
const entrypoint = ssrEntrypointConfig.get(entrypointRoute?.entrypoint ?? "");
const { root: Root, getPreloadedProps, getParams } = entrypoint;
const params = getParams(path);
const { queries } = getPreloadedProps(params);

const preloadedData = Object.keys(queries).reduce((acc, key) => {
  acc[key] = loadQuery(
    RelayEnvironment,
    queries[key].query,
    queries[key].variables,
    { fetchPolicy: "store-only" }  // store からのみ読み取り、再 fetch しない
  );
  return acc;
}, {});

hydrateRoot(
  document.getElementById("root"),
  <RelayEnvironmentProvider environment={RelayEnvironment}>
    <Root queries={preloadedData} />
  </RelayEnvironmentProvider>
);

store-only はクライアントサイドが GraphQL リクエストを再送しないことを意味し、サーバーサイドで既に取得され HTML を通じて渡されたデータを使用する。これにより hydration 時のリクエスト重複を防ぐ。

主な課題

1. Relay Environment のサーバー/クライアント差異

Relay の Environment はネットワーク層をカプセル化する。サーバーサイドは GraphQL server を直接呼び出せる（関数の直接呼び出しも可能）が、クライアントサイドは HTTP を経由する必要がある。両側の Environment 設定はまったく異なるが、使う query は同じである。サーバーサイドで fetch したデータをクライアントサイドにシームレスに渡す方法が最大の課題だった。

解決策：サーバーサイドでデータを fetch した後、Relay の store が HTML にシリアライズされる（React 18 のストリーミング注入メカニズムがこれを処理する）。クライアントサイドの hydrate 時に store-only で読み取る。

2. ストリーミングレンダリングと Suspense のタイミング問題

renderToPipeableStream は非同期である。onShellReady が発火した時、同期部分のみがレンダリング完了している。すべてのデータが Suspense boundary の後ろにある場合、shell はほぼ空になる可能性がある。どの内容を Suspense の外に置くか（即座にレンダリング）、どの内容を Suspense 内に置くか（遅延可能）を慎重に設計する必要がある。

3. CSS-in-JS とストリーミングレンダリングの衝突

Fluent UI のスタイルはレンダリング過程で動的に生成される。しかしストリーミングレンダリングでは <head> をレンダリング開始前に送信する必要がある。解決策は先に「プリレンダリング」でスタイルを収集し、次に本番のストリーミングレンダリングを行うこと。つまりコンポーネントツリーは実際には二回レンダリングされる——一回はスタイルのため、一回は HTML のため。

4. ルーティングロジックの同型性

サーバーサイドとクライアントサイドはまったく同じルーティングマッチングロジックを使わなければならず、さもなければ hydration が失敗する（クライアントサイドのレンダリング結果がサーバーサイドと一致しない）。これがルーターを JS にコンパイルした理由——同じコードが両側で実行されることを保証する。

5. リクエスト中断の処理

ユーザーがページの読み込み完了前に離れる可能性がある。ストリーミングレンダリング時にはリクエストのクローズイベントを監視し、タイムリーにレンダリングを abort する必要がある：

req.on("close", () => {
  abort();
});

処理しないと、サーバーサイドは誰も必要としないページのレンダリングを続け、リソースを浪費する。

振り返って

このプロジェクトで SSR のあらゆる工程を深く理解できたが、正直に言って本番環境でこのアプローチは推奨しない。理由：

  • 保守コストが極めて高い：React SSR の内部挙動はマイナーバージョンアップデートでも変わる可能性があり、継続的にフォローする必要がある。Next.js チームは React チームと緊密に連携してこれらの詳細の正確性を保証している——一人では追いつけない。
  • 欠けている機能が多すぎる：コード分割、プリフェッチ、ISR、静的生成、ミドルウェア、エラーバウンダリ……本番レベルの SSR フレームワークに必要なものは「コンポーネントを HTML にレンダリングする」だけではない。
  • Relay の SSR サポート自体が不完全：Relay チームは Meta 内部に独自の SSR インフラを持っており、OSS 版の SSR ドキュメントやサンプルは非常に少ない。ソースコードの中で答えを探すのに大量の時間を費やすことになる。

しかし学習プロジェクトとしての価値は大きい：

  • renderToPipeableStream が何をしているか、単に「renderToString より速い」以上に本当に理解できる
  • hydration がなぜ mismatch するのか理解できる。両側のレンダリングロジックを自分の手で書いたからだ
  • EntryPoint パターンがなぜ getServerSideProps より大規模アプリに適しているか理解できる——データ依存をルーティング層に引き上げ、データ取得とコンポーネントレンダリングを分離する
  • CSS-in-JS の SSR シーンにおける本当のペインポイントを理解できる

もし試してみたいなら

推奨するステップ：

  1. まず renderToString + Express で最もシンプルな SSR を組む（GraphQL なし、純粋な静的コンポーネント）
  2. renderToPipeableStream に切り替え、ストリーミングレンダリングと onShellReady を理解する
  3. Suspense を追加し、一部のデータを非同期でロードさせる
  4. GraphQL クライアントを接続する（Apollo は Relay より入門しやすい）、サーバーサイドのデータプリロードを処理する
  5. クライアントサイドの hydration を実装し、両側のレンダリング結果が一致することを確認する
  6. CSS を処理する（まず vanilla CSS から始め、次に CSS-in-JS を試す）

各ステップで SSR の理解が一段深まる。一度にすべてを実現しようとしないこと——hydration mismatch のエラーメッセージの中で迷子になるだけだ。
