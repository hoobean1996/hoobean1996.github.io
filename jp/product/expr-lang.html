OCaml で埋め込み式表現言語を実装する

2026-03-08
この記事の価値：Schema 駆動の UI システムにおいて、条件レンダリング、フィールドバリデーション、動的計算にはすべて表現エンジンが必要である。文字列で JavaScript を組み立てて eval するのは安全でも制御可能でもない。この記事では、OCaml でゼロから安全な埋め込み式表現言語を実装した完全なプロセスを共有する——Lexer、Parser、AST、Pattern Matching、型推論、高階関数——最終的に js_of_ocaml で JavaScript にコンパイルし、フロントエンド UI フレームワークに埋め込む。プロダクトに安全で制御可能な計算エンジンを埋め込む必要があれば、このアプローチを参考にできる。

なぜ独自の表現言語が必要か

Schema 駆動 UI の記事では、JSON でインターフェース全体を記述した。しかし JSON は静的で——「@age > 18 のときにこのフィールドを表示」「@price * @quantity で合計を計算」といった動的ロジックを記述できない。

最も簡単な方法は JavaScript の eval() や new Function() を使うことだが、2 つの致命的な問題がある：

  • セキュリティ：eval() は任意のコードを実行できる。表現がユーザー入力（例えば運用担当者がフォームルールを設定する場合）から来る場合、巨大なセキュリティホールになる
  • 制御性：表現ができることを制限できない。DOM にアクセスしたり、ネットワークリクエストを送ったり、グローバル状態を変更したりできる。「設定駆動」のシステムでは、サンドボックス化された計算のみを行うエンジンが必要

そこで独自の表現言語を実装することにした。OCaml で書き、js_of_ocaml で JavaScript にコンパイルして、フロントエンドフレームワークに埋め込む。

言語の見た目

まずいくつかの例を見て、この言語の能力を感じてほしい：

// 基本的な算術と比較
1 + 2                         // → 3
@price * @quantity            // → フォームフィールドを参照して計算
@age >= 18 and @country = "CN"  // → 条件判定

// LET バインディング
let tax = @price * 0.1 in
let total = @price + tax in
total                         // → 変数バインディングとスコープ

// 配列とレコード
let items = [1, 2, 3] in
items[0]                      // → 1
let user = {name: "Alice", age: 30} in
user.name                     // → "Alice"

// Record Spread
let base = {x: 1, y: 2} in
{...base, z: 3}              // → {x: 1, y: 2, z: 3}

// 条件式
if @score >= 90 then "A"
else "B"                      // → OCaml スタイルの if-then-else

// Pattern Matching
match @status with
| "active" -> "オンライン"
| "idle" when @lastSeen > 60 -> "離席"
| "idle" -> "待機"
| _ -> "オフライン"            // → ガード付きパターンマッチング

// Lambda と Pipe
[1, 2, 3, 4]
|> FILTER(fun x -> x > 2)
|> MAP(fun x -> x * 10)      // → [30, 40]

// 型定義
type User = {name: string, age: int} in
let u: User = {name: "Bob", age: 25} in
u.age                         // → 25

これはおもちゃではない——LET バインディング、ガード付き Pattern Matching、Lambda、カリー化、Pipe 演算子、Record Spread、型システムを備えている。しかし汎用プログラミング言語でもない——ループなし、副作用なし、IO なし。純粋な計算エンジンである。

古典的なコンパイラフロントエンドアーキテクチャ。以下、各層を展開する。

Lexer：ocamllex

Lexer は文字列を token ストリームに分割する。OCaml の ocamllex ツールで生成：

(* lexer.mll *)
let keyword_table = [
  ("if", IF); ("then", THEN); ("else", ELSE);
  ("let", LET); ("in", IN);
  ("match", MATCH); ("with", WITH); ("when", WHEN);
  ("fun", FUN); ("type", TYPE);
  ("and", AND); ("or", OR);
  ("true", TRUE); ("false", FALSE); ("null", NULL);
]

rule token = parse
  | [' ' '\t' '\n']  { token lexbuf }       (* 空白をスキップ *)
  | "//" [^'\n']* '\n'  { token lexbuf }     (* 単一行コメント *)
  | "/*"              { comment lexbuf }      (* 複数行コメント *)
  | '-'? ['0'-'9']+ as i  { NUM (int_of_string i) }
  | '"'              { read_string (Buffer.create 16) lexbuf }
  | ['a'-'z''A'-'Z']['a'-'z''A'-'Z''0'-'9''_']* as id {
      match keyword_of_string id with
      | Some token -> token
      | None -> IDENT id
    }
  | '@' ['a'-'z''A'-'Z']+ as id {
      CONTEXT_VAR (String.sub id 1 (String.length id - 1))
    }
  | "->"  { ARROW }
  | "|>"  { PIPE }
  | "??"  { NULLCOALESCE }
  | "..." { DOTDOTDOT }
  | ...

注目すべきは @ プレフィックス変数（CONTEXT_VAR）。表現内で @price と書くと、Lexer はコンテキスト変数として認識する——評価時に外部から渡された context オブジェクトから値を検索する。これが表現言語とホスト環境のインターフェース。

文字列処理はエスケープ文字と文字列補間（"Hello ${name}"）をサポートし、Lexer は ${ パターンを検出すると INTERP_STRING とマークし、評価段階で補間表現を再帰的に解析する。

AST：データすなわちプログラム

AST は言語全体のコアデータ構造。OCaml の代数的データ型（ADT）はこれを表現するのに最適：

(* ast.ml *)
type pattern =
  | PInt of int          (* 整数リテラルにマッチ *)
  | PBool of bool        (* ブール値にマッチ *)
  | PString of string    (* 文字列にマッチ *)
  | PVar of string       (* 変数をバインド *)
  | PArray of pattern list    (* 配列を分解 [x, y, z] *)
  | PRecord of (string * pattern) list  (* レコードを分解 {name, age} *)
  | PAny                 (* ワイルドカード _ *)

type expr =
  | Num of int | Float of float | Bool of bool
  | String of string | InterpString of string | Null
  | Var of string
  (* データ構造 *)
  | Array of expr list
  | ArrayAccess of expr * expr
  | Record of (string * expr) list
  | RecordAccess of expr * string
  | RecordSpread of expr * (string * expr) list
  (* 算術と論理 *)
  | Add of expr * expr | Sub of expr * expr
  | Mul of expr * expr | Div of expr * expr
  | Gt of expr * expr | Lt of expr * expr | ...
  | And of expr * expr | Or of expr * expr | Eq of expr * expr
  | NullCoalesce of expr * expr
  (* 制御フロー *)
  | If of expr * expr * expr
  | Let of string * type_expr option * expr * expr
  | Match of expr * (guarded_pattern * expr) list
  (* 関数 *)
  | FuncDef of string * string list * expr
  | Lambda of string list * expr
  | App of expr * expr      (* カリー化適用 f x *)
  | Pipe of expr * expr     (* x |> f *)
  (* 型システム *)
  | TypeDef of type_decl * expr
  | Seq of expr * expr

and guarded_pattern = pattern * expr option

いくつかの設計上の決定について説明する：

  • Let にオプショナルな型注釈：Let of string * type_expr option * expr * expr——let x: int = 5 in ... でも let x = 5 in ... でも合法
  • Match の case は guarded_pattern * expr：guarded_pattern = pattern * expr option で、expr option がオプショナルな when ガード
  • App は二項演算：f x y は App(App(f, x), y) として解析される——標準的なカリー化表現

Parser：menhir

Parser は menhir（OCaml の LR パーサージェネレータ）で定義。優先順位は低から高：

(* parser.mly *)
%nonassoc IN
%nonassoc ARROW
%left SEMICOLON
%left PIPE
%nonassoc ELSE THEN
%left BAR
%left NULLCOALESCE
%left AND OR
%left EQ
%left LT GT LTE GTE
%left PLUS MINUS
%left TIMES DIV
%nonassoc APP           (* カリー化関数適用 *)
%left DOT
%nonassoc LBRAKET       (* 配列アクセス *)

Pattern Matching の構文規則：

match_cases:
  | guarded_pattern ARROW expr                     { [($1, $3)] }
  | BAR guarded_pattern ARROW expr                 { [($2, $4)] }
  | guarded_pattern ARROW expr BAR match_cases     { ($1, $3) :: $5 }
  | BAR guarded_pattern ARROW expr BAR match_cases { ($2, $4) :: $6 }

guarded_pattern:
  | pattern                  { ($1, None) }
  | pattern WHEN expr        { ($1, Some $3) }

pattern:
  | NUM { PInt $1 }
  | UNDERSCORE { PAny }
  | bool_literal { PBool $1 }
  | STRING { PString $1 }
  | IDENT { PVar $1 }
  | LBRAKET array_patterns RBRAKET { PArray $2 }
  | LBRACE record_pattern_fields RBRACE { PRecord $2 }

最初の BAR（|）はオプショナルなので、以下のどちらのスタイルでも書ける：

// どちらのスタイルもOK
match x with
  1 -> "one"
| 2 -> "two"
| _ -> "other"

match x with
| 1 -> "one"
| 2 -> "two"
| _ -> "other"

Record pattern はショートハンドをサポート：{name} は {name: name} と等価で、JavaScript の分割代入構文と同じ。

評価器：Pattern Matching の実装

評価器は再帰関数 eval_expr : env -> expr -> value * env。ほとんどのノードの評価は直接的（加算は加算、比較は比較）だが、最も興味深いのは Pattern Matching。

核心は pattern_matches 関数——値とパターンを受け取り、マッチ成功時の変数バインディング（Some env）または失敗（None）を返す：

let rec pattern_matches value = function
  | PInt i ->
      (match value with
       | VInt i' when i' = i -> Some Env.empty
       | _ -> None)
  | PString s ->
      (match value with
       | VString s' when s' = s -> Some Env.empty
       | _ -> None)
  | PVar name ->
      (* 変数パターン：常にマッチし、変数名をバインド *)
      Some (Env.singleton name value)
  | PArray patterns ->
      (match value with
       | VArray elements ->
           if List.length patterns <> List.length elements then None
           else
             (* 各要素を再帰的にマッチ *)
             let matches = List.mapi
               (fun i pat -> pattern_matches (List.nth elements i) pat)
               patterns in
             if List.exists (fun x -> x = None) matches then None
             else Some (merge_all_envs matches)
       | _ -> None)
  | PRecord pattern_fields ->
      (match value with
       | VRecord value_fields ->
           (* 各パターンフィールドについて、値のレコードから検索し再帰的にマッチ *)
           let matches = List.map (fun (key, pat) ->
             match List.assoc_opt key value_fields with
             | Some field_value -> pattern_matches field_value pat
             | None -> None
           ) pattern_fields in
           if List.exists (fun x -> x = None) matches then None
           else Some (merge_all_envs matches)
       | _ -> None)
  | PAny -> Some Env.empty

Match 式の評価ロジック——上から順に各 case を試し、マッチ成功かつガード通過時に body を評価する：

| Match (e1, cases) ->
    let v1, _ = eval_expr env e1 in
    let rec try_match v = function
      | [] -> failwith "Match error: no pattern matches"
      | ((pattern, guard_opt), body) :: rest ->
          match pattern_matches v pattern with
          | Some pattern_env ->
              let merged_env = Env.union ... pattern_env env in
              (* ガードをチェック *)
              let guard_passes = match guard_opt with
                | None -> true
                | Some guard_expr ->
                    match eval_expr merged_env guard_expr with
                    | VBool b, _ -> b
                    | _ -> failwith "Guard must be boolean"
              in
              if guard_passes then eval_expr merged_env body
              else try_match v rest    (* ガード失敗、次の case へ *)
          | None -> try_match v rest   (* パターン不一致、次へ *)
    in
    try_match v1 cases

ポイント：pattern_env と現在の env をマージしてガードと body に渡す。つまりガードはパターンでバインドされた変数を参照できる：

match user with
| {name, age} when age > 18 -> CONCAT(name, " is adult")
//  ↑ name と age はガード内で使用可能
| {name} -> CONCAT(name, " is minor")

カリー化と Pipe

関数適用はカリー化をサポート——引数が足りない場合、新しいクロージャを返す：

| App (func_expr, arg_expr) ->
    (* すべての引数を収集 *)
    let func_base, collected_args = collect_args func_expr [arg_val] in
    match find_closure func_base with
    | VClosure (params, body, closure_env) ->
        let n_params = List.length params in
        let n_args = List.length collected_args in
        if n_args < n_params then
          (* 部分適用：新しいクロージャを返す *)
          let applied = take n_args params in
          let remaining = drop n_args params in
          let env' = bind_params closure_env applied collected_args in
          (VClosure (remaining, body, env'), env)
        else if n_args = n_params then
          (* 完全適用：body を評価 *)
          let env' = bind_params closure_env params collected_args in
          eval_expr env' body
        else
          failwith "Too many arguments"

Pipe 演算子（|>）は左側の値を右側の関数の第一引数として渡す：

// この 2 つは等価
[1, 2, 3] |> MAP(fun x -> x * 2)
MAP([1, 2, 3], fun x -> x * 2)

// チェーン呼び出し可能
[1, 2, 3, 4, 5]
|> FILTER(fun x -> x > 2)     // → [3, 4, 5]
|> MAP(fun x -> x * 10)       // → [30, 40, 50]
|> SUM                         // → 120

実装上、Pipe は右側の型に応じて異なる処理を行う：関数呼び出し FUNC(args) の場合は左側の値を第一引数として挿入し、変数名の場合は単一引数関数として呼び出し、Lambda の場合は直接適用する。

型システム

この言語は完全な型推論システムを持つ。型の表現：

type typ =
  | TInt | TFloat | TBool | TString | TDate | TNull
  | TArray of typ
  | TRecord of (string * typ) list
  | TFunction of typ list * typ
  | TUnion of typ list        (* int | string *)
  | TAny                      (* 動的型 *)
  | TName of string           (* ユーザー定義型の参照 *)

型推論は双方向：

// 値から型を推論
let x = 42 in ...              // x : int
let arr = [1, 2, 3] in ...     // arr : int[]
let u = {name: "Alice"} in ... // u : {name: string}

// 型注釈からチェック
let x: int = 42 in ...         // OK
type User = {name: string, age: int} in
let u: User = {name: "Bob", age: 25} in ...  // OK

型システムは構造的部分型（structural subtyping）をサポート——Record 型の部分型関係はフィールドの包含に基づき、名前の一致ではない：

let is_subtype t1 t2 =
  match (t1, t2) with
  | _, TAny -> true              (* すべての型は Any の部分型 *)
  | TInt, TFloat -> true         (* Int は Float の位置で使用可能 *)
  | TNull, _ -> true             (* Null は任意の型 *)
  | TRecord f1, TRecord f2 ->
      (* 構造的部分型：f1 が f2 のすべてのフィールドを含む *)
      List.for_all (fun (k2, v2) ->
        match List.assoc_opt k2 f1 with
        | Some v1 -> is_subtype v1 v2
        | None -> false
      ) f2
  | ...

If-then-else の型推論は unify を使用——2 つのブランチの型の和集合を取り、互換性がなければ Union 型を返す：

| If (_, then_expr, else_expr) ->
    let t1 = infer_type env then_expr in
    let t2 = infer_type env else_expr in
    simplify_type (unify t1 t2)
    // if ... then 1 else 2.0  → float（Int が Float に昇格）
    // if ... then 1 else "a"  → int | string（Union 型）

組み込み関数ライブラリ

言語自体の機能に加えて、カテゴリ別に整理された豊富な組み込み関数ライブラリがある：

カテゴリ関数例
テキストUPPER, LOWER, TRIM, CONCAT, LEFT, RIGHT, MID, FIND, SUBSTITUTEUPPER("hello") → "HELLO"
数学ABS, ROUND, FLOOR, CEIL, POW, SQRT, SUM, MAX, MIN, AVERAGEROUND(3.7) → 4
配列LEN, CONTAINS, INDEX, SLICE, SORT, REVERSE, UNIQUE, JOINUNIQUE([1,2,2,3]) → [1,2,3]
高階MAP, FILTER, REDUCE, EVERY, SOME, FLATMAPFILTER([1,2,3], fun x -> x > 1) → [2,3]
条件IFS, IFNAIFS(@x > 0, "正", @x < 0, "負", true, "零")
日付TODAY, YEAR, MONTH, DAY, DATEADD, DATEDIFFYEAR(TODAY()) → 2026

フォーマットFORMAT_NUMBER, FORMAT_PERCENTFORMAT_PERCENT(0.85) → "85%"

高階関数の実装には特別な処理が必要——組み込み関数がユーザーから渡された Lambda を呼び出す必要がある。ここでは循環依存を解消するために ref を使用した：

(* builtin.ml *)
(* eval_expr への参照、driver.ml で初期化される *)
let eval_closure_ref : (value Env.t -> expr -> value) ref =
  ref (fun _ _ -> failwith "eval_closure not initialized")

let apply_closure closure args =
  match closure with
  | VClosure (params, body, closure_env) ->
      let env' = bind_params closure_env params args in
      !eval_closure_ref env' body
  | _ -> failwith "Expected a function"

(* driver.ml で初期化 *)
let () =
  Builtin.eval_closure_ref := fun env expr ->
    let result, _ = eval_expr env expr in
    result

これは OCaml でモジュール間の循環依存を処理する古典的パターン——ref で遅延バインディングを行う。

JavaScript へのコンパイル

OCaml コード全体を js_of_ocaml で JavaScript ファイルにコンパイルし、npm パッケージ @gigaboo/expr として公開：

// JavaScript 側での使用
import { libexpr } from '@gigaboo/expr';

// 単純な評価
libexpr.eval('1 + 2');                    // → 3
libexpr.eval('LET x = 10 IN x * 2');     // → 20

// コンテキスト付き評価（@ 変数はここから取得）
libexpr.evalInContext('@price * @qty', {
  price: 99.9,
  qty: 3
});                                       // → 299.7

// 型推論
libexpr.inferType('LET x = [1, 2] IN x'); // → "int[]"

真理値表生成：業務担当者が自分でルールを検証

Schema 駆動フォームを作る中で一つの問題に気づいた：業務担当者（プロダクトマネージャー、運用）が条件ルールを設定する際、自分が書いたロジックが正しいか不確かなことが多い。例えば承認フロールール isManager and (amount > 1000 or isUrgent) の場合、すべての状況を頭の中で網羅するのは難しい。

しかし彼らは Excel を使う。多くの業務担当者がロジックを検証する方法は、Excel で手動で真理値表を作ること——すべての変数の組み合わせを列挙し、各行の結果が期待通りか確認する。

表現評価器があるので、この作業は自動化できる。ブール式を与えると、完全な真理値表を自動生成する：

libexpr.analyze('a and (b or c)');

// 出力：
// -------------------
// | a | b | c | result |
// -------------------
// | T | T | T | T      |
// | T | T | F | T      |
// | T | F | T | T      |
// | T | F | F | F      |
// | F | T | T | F      |
// | F | T | F | F      |
// | F | F | T | F      |
// | F | F | F | F      |
// -------------------

実装方法：表現 AST からすべての変数名を再帰的に収集し、2^n 個のブール組み合わせを生成し、それぞれを評価する。

let rec collect_vars = function
  | Var name -> [name]
  | And (e1, e2) | Or (e1, e2) -> collect_vars e1 @ collect_vars e2
  | _ -> []

let rec gen_combinations = function
  | [] -> [[]]
  | v :: vs ->
      let rest = gen_combinations vs in
      List.map (fun l -> (v, true) :: l) rest
      @ List.map (fun l -> (v, false) :: l) rest

let analyze_expr expr =
  let vars = List.sort_uniq String.compare (collect_vars expr) in
  let combinations = gen_combinations vars in
  List.map (fun env ->
    let env_map = (* [(name, bool)] を評価環境に変換 *) in
    let result = eval_expr env_map expr in
    (env, result)
  ) combinations

プロダクトでは、この機能がフォームルールの設定画面に統合されている：業務担当者が条件式を書き終えた後、「検証」をクリックすると真理値表が表示され、各状況での動作が期待通りか直感的に確認できる。本番デプロイしてからテストする必要も、開発者にロジックのチェックを依頼する必要もない。

なぜ OCaml を使うか

OCaml で表現エンジンを書くのは、格好いいからではなく、いくつかの特性がこの種の作業を非常にスムーズにするから：

  • 代数的データ型 + Pattern Matching：AST 定義と再帰評価を ADT で書くと極めて自然。match expr with Add(e1, e2) -> ... と書く方が、大量の if-else や visitor pattern よりはるかにすっきりしている
  • ocamllex + menhir：産業グレードの Lexer/Parser ジェネレーターで、構文定義から直接効率的な解析コードを生成。手書きの再帰下降は不要
  • js_of_ocaml：JavaScript へのコンパイルの成熟したソリューション。生成された JS ファイルは npm パッケージとしてそのまま公開でき、呼び出し側に対して完全に透過的
  • 型安全性：OCaml の型システムが評価器の網羅性を保証——AST に新しいノードを追加しても処理を忘れると、コンパイラがエラーを報告する

まとめ

この表現言語の設計目標は：設定駆動の UI システムにおいて、安全で制御可能な、十分な表現力を持つ計算エンジンを提供すること。

その能力の境界は明確：

  • できること：算術、比較、論理演算、条件分岐、パターンマッチング、配列/レコード操作、高階関数、型チェック
  • できないこと：ループ、副作用、IO、DOM アクセス、ネットワークリクエスト

この「できないこと」こそが価値——運用担当者がフォームルールを設定できるシステムでは、表現エンジンが計算のみを行い、危険なことは一切できないことが望ましい。

完全な実装は約 1000 行の OCaml コード（lexer + parser + AST + evaluator + type system + builtins）。プロダクトに表現エンジンを埋め込む必要があれば、OCaml + js_of_ocaml は検討に値するアプローチである。
