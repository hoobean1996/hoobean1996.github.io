CPU から GPU へ：LLM 推論エンジンの CUDA 最適化の道

2026-03-18
Series: LLM Inference Engine
Part 1: ゼロから生成まで — 純粋な C++ で推論エンジンを一から構築
Part 2: CPU から GPU へ — CUDA 最適化の道 (本稿)

はじめに

前回の記事では、純粋な C++ でゼロから LLM 推論エンジンを構築し、Qwen2.5-0.5B を動かした。当時 CPU（OpenBLAS + INT8 量子化）で 33 tok/s を達成していた。

この記事では続く最適化の過程を記録する：エンジンを GPU に移し、33 tok/s から 504 tok/s まで一歩ずつ引き上げた。各最適化ステップには明確なプロファイリングデータに基づく裏付けがあり、推測ではない。

最適化パスの全体像：

フェーズ Decode スループット 向上
CPU BLAS + INT8 33 tok/s baseline
CUDA + cuBLAS + INT8 110 tok/s 3.3×
+ FP16 inference 153 tok/s 4.6×
+ Flash Attention 161 tok/s 4.9×
+ GPU-resident sampling 161 tok/s -
+ CUDA caching allocator 504 tok/s 15.3×

最後のステップ（caching allocator）が最大の向上をもたらしたが、nsys プロファイリングが本当のボトルネックを明らかにしなければ、メモリ割り当ての最適化を思いつくことはなかっただろう。これがこの記事で最も伝えたいポイントである：先にプロファイル、次に最適化。

Step 1: CUDA バックエンド — 計算を GPU に移す

前回の Backend 抽象がここで効果を発揮する。新しい BackendCUDA サブクラスを作成し、同じインターフェースを実装し、内部は CUDA kernel に置き換える：

class BackendCUDA : public BackendCPU {
public:
    void linear(float* out, const float* inp, const float* weight,
                const float* bias, std::size_t M, std::size_t N,
                std::size_t K) override;
    void rms_norm(...) override;
    void rope(...) override;
    void causal_attention(...) override;
    // ... すべての演算子に GPU 実装
};

コア設計：

  • 重みキャッシュ：重みは初回使用時に GPU にアップロードし（cudaMemcpy H2D）、以降は device pointer を再利用。unordered_map<host_ptr, device_ptr> で完了。
  • cuBLAS GEMM：線形投影に cublasGemmEx（FP16 入力 × FP16 重み → FP32 出力）を使用し、Tensor Core で加速。
  • GPU-resident KV Cache：KV キャッシュを GPU 上で直接割り当てて増加させ、attention の読み書きは PCIe を通らない。各 decode ステップで 96 回の memcpy を節約（2 tensor × 2 KV × 24 層）。
  • INT8 → FP16 dequant：INT8 重みを GPU にアップロード後、カーネルで FP16 に逆量子化し、FP16 GEMM を実行。

効果：CPU BLAS の 33 tok/s から CUDA の 110 tok/s へ、3.3 倍。

Step 2: FP16 全パイプライン推論

第 1 版の CUDA バックエンドのデータフローは：FP32 中間結果 → FP16 に変換 → cuBLAS GEMM → FP32 出力。途中に大量の FP32↔FP16 変換がある。

最適化の方針：すべての中間 Tensor を FP16 にする。embedding 出力を FP16 に変換した後、パイプライン全体（RMSNorm、RoPE、Attention、MLP）を FP16 で実行し、最終的な logits 出力時のみ FP32 に変換する。

各カーネルに FP16 版を作成した。RMSNorm を例にすると、内部では FP32 で累積して精度損失を防ぐ：

__global__ void rms_norm_fp16_kernel(half* out, const half* x,
                                      const half* weight, size_t n,
                                      float eps) {
    // FP32 で累積し、精度損失を防ぐ
    float local_sum = 0.0f;
    for (size_t i = tid; i < n; i += stride) {
        float v = __half2float(x[i]);
        local_sum += v * v;
    }
    // ... reduction ...
    float rms = sqrtf(sdata[0] / n + eps);

    for (size_t i = tid; i < n; i += stride) {
        out[i] = __float2half(
            __half2float(x[i]) / rms * __half2float(weight[i]));
    }
}

ここに一つ罠がある：Sampler も logits のデータ型を知る必要がある。logits を FP16 から FP32 に変換してから sampler に渡すが、sampler 内部に GPU サンプリングカーネルがどの精度で読み取るかを示す fp16 フラグがある。logits は既に FP32 なので、このフラグは常に 0 であるべきだが——コードが誤って backend.is_fp16() に基づいて設定していた。

結果：FP16 モードで出力がすべて文字化けになった。INT8 FP32 は正常、FP16 は文字化け——サンプリングカーネルが FP32 データを FP16 として読んだため、各バイトペアが half float として誤って解釈された。

1 行のコード修正：

// Before (bug):
int fp16 = backend_.is_fp16() ? 1 : 0;
return cuda_sample(d_logits_, logits.size(), ..., fp16);

// After (fix):
// Logits are always FP32 (converted in transformer_forward)
return cuda_sample(d_logits_, logits.size(), ..., /*fp16=*/0);

教訓：データ型はパイプラインのすべての段階で整合させる必要がある。ある場所で FP32 に変換したのに、下流がまだ FP16 として読んでいれば、破滅的な結果になる。

FP16 全パイプラインの実現後：110 tok/s → 153 tok/s、+39%。帯域幅半減 + Tensor Core 利用率向上。

Step 3: Flash Attention

従来の attention カーネルは、完全な scores[attend_len] 配列を shared memory に保存していた。これには 2 つの問題がある：

  1. Shared memory のサイズがシーケンス長に比例して増加し、約 12K token で上限に達する（48KB 制限）
  2. すべての QKT スコアを計算し終えてからでないと softmax ができず、softmax してからでないと V の加重ができない——3 ステップが直列

Flash Attention の核心的洞察：softmax はインクリメンタルに計算できる。毎回小さなブロック（128 個の KV position）のみを処理し、2 つのスカラー running_max と running_sum で過去の累積値を補正する：

// メインループ：毎回 128 個の KV position を処理
for (tile_start = 0; tile_start < attend_len; tile_start += 128) {

    // 1. この 128 個のスコアのみ計算 → shared memory (512 bytes)
    for (t in tile)
        scores[t] = dot(q, k[t]) * scale;

    // 2. このタイルの max を求める
    tile_max = reduce_max(scores);

    // 3. 過去の値を補正（Flash Attention の真髄）
    new_max = max(running_max, tile_max);
    correction = exp(running_max - new_max);
    running_sum *= correction;
    out_acc *= correction;       // 過去のすべてのタイルの累積値を補正

    // 4. 現在のタイルを累積
    for (t in tile) {
        e = exp(scores[t] - new_max);
        running_sum += e;
        out_acc += e * v[t];     // 加重 V の累積
    }
    running_max = new_max;
}

// 最終的な正規化
out = out_acc / running_sum;

なぜ correction で正しく補正できるのか？数学的に：

exp(score - old_max) × exp(old_max - new_max) = exp(score - new_max)

したがって、すべての過去の累積値に correction を掛けることは、new_max で再計算したのと等価である。

Shared memory が (attend_len + 256) × 4 から定数 1536 バイトになり、シーケンスの長さに関係なく一定。

attend_len 旧カーネル smem/head Flash Attention smem/head
541.2 KB1.5 KB
4482.8 KB1.5 KB
409617.0 KB1.5 KB
1228849.0 KB (OOM)1.5 KB

短いシーケンスではパフォーマンスはほぼ同等（ボトルネックが attention にない）だが、長いコンテキスト能力が解放された。Decode スループットは 50～750 token の attend_len 範囲で約 161 tok/s を安定して維持。

Step 4: GPU-Resident Sampling

Flash Attention の修正後、nsys でデータフローを確認したところ、不要な CPU ラウンドトリップが見つかった：

GPU: FP16 logits → fp16_to_fp32 → FP32 logits
                                          ↓ cudaMemcpy D2H (593KB)
CPU: std::vector<float> logits
                                          ↓ cudaMemcpy H2D (593KB)
GPU: sample_kernel → token ID

Qwen2.5-0.5B の vocab_size=151936 で、各 decode ごとに 1.2MB を PCIe 経由で往復転送。

修正方針：forward() が std::vector<float> ではなく Tensor（GPU-resident）を返すようにする。Sampler は GPU Tensor から直接サンプリングする：

// Before: forward() が CPU にダウンロードし、sampler が再度 GPU にアップロード
std::vector<float> forward(model, token_ids, pos, kv_cache);
sampler->sample(logits, step);  // 内部: H2D upload + sample kernel

// After: logits は GPU 上に留まり、sampler が直接 device pointer を使用
Tensor forward(model, token_ids, pos, kv_cache);
sampler->sample(logits, step);  // 直接: sample kernel on device ptr

この変更自体は decode スループットへの影響は大きくない（PCIe 転送はカーネル計算時間に比べて短い）が、コードがよりクリーンになり——次の最適化への道を開いた。

Step 5: nsys Profile → 本当のボトルネックを発見 → CUDA Caching Allocator

これがこの記事全体で最も重要なステップである。アルゴリズムの最適化ではなく、プロファイリングツールによって全く予想外のボトルネックが発見された。

nsys profile を実行：

nsys profile --trace=cuda,osrt --stats=true \
  bazel-bin/bin/qwen/qwen \
    --model models/Qwen2.5-0.5B \
    --prompt "Once upon a time" \
    --max_tokens 50 \
    --backend cuda --quantize int8 --fp16

CUDA API 時間分布：

 Time (%)  Total Time (ns)  Num Calls   Name
 --------  ---------------  ---------   ----
     36.0%       216913486       1341   cudaFree           ← 36%!!
     33.6%       202353991       2963   cudaMemcpy
     17.3%       104017852      17773   cudaLaunchKernel   ← 実際の計算
      8.2%        49292731       1341   cudaMalloc         ← 8%

このデータを見て驚いた：cudaMalloc + cudaFree が CUDA API 時間の 44% を占め（266ms）、実際の GPU カーネル計算（104ms）より 2.5 倍も多い！

なぜか？cudaMalloc と cudaFree はどちらも同期操作だからである——CPU スレッドをブロックし、GPU 上のすべての pending カーネルの実行完了を待つ。つまり、割り当て/解放のたびに GPU パイプラインが強制同期される。

1341 回の malloc + 1341 回の free。どこから？毎回の transformer_forward() が約 12 個の一時 Tensor を作成する：

Tensor x({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor norm_out({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor q({seq_len * num_heads * head_dim}, DType::Float16, Device::CUDA);
Tensor k_new({seq_len * kv_dim}, DType::Float16, Device::CUDA);
Tensor v_new({seq_len * kv_dim}, DType::Float16, Device::CUDA);
Tensor attn_out({seq_len * hidden}, DType::Float16, Device::CUDA);
Tensor gate_up({seq_len * 2 * intermediate}, DType::Float16, Device::CUDA);
Tensor gate({seq_len * intermediate}, DType::Float16, Device::CUDA);
Tensor final_out({hidden}, DType::Float16, Device::CUDA);
Tensor logits({vocab_size}, DType::Float16, Device::CUDA);
// ... plus logits_fp32, x_fp32 for conversions

関数が返るとき、これらの Tensor のデストラクタが cudaFree を呼ぶ。54 回の forward × 約 25 個の alloc/free ≈ 1350 回。nsys レポートの 1341 と完全に一致する。

重要な洞察：decode フェーズでは、毎回の forward の Tensor shape は全く同じ（seq_len=1）。毎回同じサイズのバッファを malloc し、使い終わったら free し、次にまた同じサイズを malloc する——これは純粋な無駄である。

修正：CUDA caching allocator

核心的アイデアは極めてシンプル：解放された GPU バッファを本当には解放せず、free list（サイズでインデックス化）に入れる。次に同じサイズのバッファを要求した際、free list から直接取得し、O(1) 検索で GPU 同期ゼロ：

// Free list: byte_size → [ptr, ptr, ...] (exact size matching)
static std::multimap<std::size_t, void*> cuda_free_list_;

// Live allocations: ptr → byte_size (for returning to correct bucket)
static std::unordered_map<void*, std::size_t> cuda_alloc_sizes_;

static void* cuda_pool_alloc(std::size_t bytes) {
    auto it = cuda_free_list_.find(bytes);
    if (it != cuda_free_list_.end()) {
        void* ptr = it->second;
        cuda_free_list_.erase(it);
        cuda_alloc_sizes_.emplace(ptr, bytes);
        return ptr;    // ← ヒット！GPU 同期ゼロ
    }
    void* ptr = cuda_malloc(bytes);  // miss: 実際に割り当て
    cuda_alloc_sizes_.emplace(ptr, bytes);
    return ptr;
}

static void cuda_pool_free(void* ptr) {
    auto it = cuda_alloc_sizes_.find(ptr);
    if (it != cuda_alloc_sizes_.end()) {
        cuda_free_list_.emplace(it->second, ptr);  // free list に回収
        cuda_alloc_sizes_.erase(it);
    } else {
        cuda_free(ptr);  // pool 管理外のバッファは実際に解放
    }
}

30 行のコード。device_allocate() と device_deallocate() に挿入するだけで、上位層に対して完全に透過的。

最初の forward では、すべての Tensor がミス（コールドスタート）し、cudaMalloc を通る。関数が返ると、バッファが free list に入る。2 回目の forward 以降、同じサイズのすべての Tensor がヒットし、直接再利用——cudaMalloc ゼロ、cudaFree ゼロ、GPU 同期ゼロ。

修正後に再度 nsys を実行：

 Time (%)  Total Time (ns)  Num Calls   Name
 --------  ---------------  ---------   ----
      7.5%        45833654        751   cudaMalloc    ← -44% calls
      7.3%        44586600        729   cudaFree      ← -79% time!
     16.6%       101269184      17773   cudaLaunchKernel  ← 変化なし

cudaFree が 217ms から 45ms に減少（-79%）。612 回の割り当てが pool にヒットし、GPU 同期を完全に回避した。

パフォーマンス結果：

指標 Before (pool なし) After (caching) 向上
Decode スループット 153 tok/s 504 tok/s +3.3×
Decode 平均レイテンシ 6.5 ms 2.0 ms -69%
Decode P50 レイテンシ 6.0 ms 1.6 ms -73%
cudaFree time 217 ms 45 ms -79%

3.3 倍の高速化が、30 行のメモリプールコードから。これがプロファイリングの価値——ボトルネックが演算子にあるとか、attention にあるとか、GEMM にあるとか推測するのではなく、データに語らせる：最大のボトルネックはなんと cudaFree だった。

まとめ

CPU の 33 tok/s から GPU の 504 tok/s へ、15 倍の高速化。プロセス全体の重要な経験：

1. Profile first, optimize second. nsys を実行しなければ、CUDA カーネルの occupancy や shared memory レイアウトの最適化に数日を費やしていたかもしれない——しかし実際のボトルネックは cudaFree の GPU 同期だった。データは嘘をつかない。

2. 最大の向上は最もシンプルな修正から来ることが多い。Flash Attention の実装に数時間かかり、効果は +5%。caching allocator は 30 行で、効果は +230%。複雑さと効果は比例しない。

3. エンドツーエンドの思考。FP16 sampler のバグはエンドツーエンドの型整合の問題。GPU-resident sampling はデータ転送の問題。これらはカーネルレベルの最適化ではなく、システムレベルの最適化。パイプライン全体を見る必要があり、単一の演算子だけを見つめていてはいけない。

4. 抽象層の価値はコード再利用だけではない。Backend 抽象により CPU/BLAS/CUDA の切り替えがゼロ侵襲。Tensor の device_allocate/device_deallocate 抽象により caching allocator を透過的に挿入でき、上位コードの変更は不要。良い抽象は最適化を局所化する。

プロジェクト全体は現在 6200 行の C++/CUDA で、Qwen2.5-0.5B を RTX 4060 上で 504 tok/s の decode を達成している。まだやれることは多い——INT4 量子化、prefill Q tiling、マルチリクエスト batching——しかしプロファイリング駆動の最適化ループは既に確立された。

"Premature optimization is the root of all evil. But so is premature pessimism — profile, then decide."
