PostgreSQL ストレージエンジンの内部構造（下）：WAL、Checkpoint と永続性

2026-03-08
PostgreSQL ストレージエンジンの内部構造
上：Heap Page と Buffer Pool · 中：インデックススキャンと MVCC · 下：WAL、Checkpoint と永続性
この記事の価値：データベースは「トランザクションがコミットされたらデータは失われない」と約束する。この約束はどう実現されるのか？答えは WAL（Write-Ahead Log）――データページを変更する前に、まず変更の記録をログに書き込む。この記事では INSERT 文が WAL レコードを生成し、WAL Writer がディスクにフラッシュし、Checkpoint がリカバリポイントを作成するまでの完全なパスを追跡し、この仕組みがクラッシュリカバリとストリーミングレプリケーションを同時にどう支えるかを説明する。

WAL の核心思想

Write-Ahead Logging のルールはただ一つ：データページを変更する前に、その変更を記述するログレコードを永続ストレージに書き込まなければならない。

なぜか？変更されたデータページをディスクに書き込むコストは高い（ランダム I/O、一度に 8KB）が、ログレコードをファイル末尾に追加するのは安価（シーケンシャル I/O、通常数十〜数百バイト）だからだ。WAL はランダム書き込みをシーケンシャル書き込みに変換しつつ、永続性を保証する。

  WAL がない世界（データページを直接書き込み）：

  トランザクションコミット → すべての変更 Page をディスクにフラッシュ → ランダム I/O → 遅い
  クラッシュ → 一部の Page が中途半端に書かれた → データ破損 → リカバリ不能

  WAL がある世界：

  トランザクションコミット → WAL レコードのみディスクにフラッシュ → シーケンシャル I/O → 速い
  データページは後でゆっくり書ける（非同期）
  クラッシュ → WAL ログをリプレイ → データが一貫した状態にリカバリ

XLogRecord：WAL レコードの構造

各データベース書き込み操作（INSERT、UPDATE、DELETE、および各種内部操作）は 1 つまたは複数の XLogRecord を生成する：

typedef struct XLogRecord {
    uint32      xl_tot_len;   // レコード全体の長さ
    TransactionId xl_xid;     // このレコードを生成したトランザクション ID
    XLogRecPtr  xl_prev;      // 前の WAL レコードの位置（リンクリストを形成）
    uint8       xl_info;      // 操作タイプ（例：XLOG_HEAP_INSERT）
    RmgrId      xl_rmid;      // リソースマネージャ ID（どのサブシステムが生成したか）
    pg_crc32c   xl_crc;       // CRC チェック
} XLogRecord;

INSERT が生成する WAL レコードはおおよそこのようになる：

XLogRecord {
    xl_tot_len = 156,              // レコード全体で 156 バイト
    xl_xid = 12345,                // トランザクション ID
    xl_prev = 0/1A2B3C4D,          // 前のレコードの LSN
    xl_info = XLOG_HEAP_INSERT,    // ヒープ挿入操作
    xl_rmid = RM_HEAP_ID,          // ヒープテーブルリソースマネージャ
    xl_crc = 0x1A2B3C4D            // CRC チェック
}
// 後続に：変更された Page の block reference + 実際に挿入されたデータ

LSN：WAL のタイムライン

LSN（Log Sequence Number）は WAL 内の各レコードのグローバルに一意な位置識別子で、フォーマットは セグメント番号/セグメント内オフセット（例：0/1A2B3C4D）。これは PostgreSQL における「時間」のもう一つの表現である：

  • 各 Page の pd_lsn は、その Page を最後に変更した WAL レコードの LSN を記録する
  • リカバリ時、Page の pd_lsn が WAL レコードの LSN 以上であれば、その WAL はすでに適用済みなのでスキップ
  • レプリケーション時、standby は primary に「LSN X まで再生済み」と通知し、primary はどの WAL が回収可能かを判断する

リソースマネージャ：このログを生成したのは誰か

異なるサブシステムが異なるタイプの WAL レコードを生成し、各サブシステムが「リソースマネージャ」（Resource Manager）を登録して独自の redo およびデコード関数を提供する：

RmgrIdサブシステム操作タイプ例
RM_HEAP_IDヒープテーブルINSERT、DELETE、UPDATE、HOT_UPDATE
RM_BTREE_IDB-Tree インデックスインデックスページ分割、インデックスエントリ挿入
RM_XACT_IDトランザクション管理COMMIT、ABORT
RM_CLOG_IDトランザクションコミットログCLOG ページ更新
RM_XLOG_IDWAL 自体Checkpoint レコード

リカバリ時、各 WAL レコードに対して対応するリソースマネージャの rm_redo 関数が呼ばれる。ロジカルレプリケーション時は rm_decode 関数で物理的な変更を論理的な変更に変換する（例：DecodeInsert が XLOG_HEAP_INSERT を理解可能な行変更イベントに解析する）。

WAL の書き込みパス：メモリからディスクへ

WAL バッファ：共有メモリ内のログステージングエリア

  │                                                    │    │

  │  │ Page 0 │ Page 1 │ Page 2 │ Page 3 │  ...   │   │    │
  │  │ 8KB    │ 8KB    │ 8KB    │ 8KB    │        │   │    │

  │                                                    │    │
  │  制御情報：                                         │    │
  │  - LogwrtRqst: 書き込み要求位置                     │    │
  │  - LogwrtResult: 書き込み完了位置                   │    │
  │  - xlblocks[]: 各ページの終了 LSN                   │    │
  │  - walwriter_latch: WAL Writer を起動               │    │
  └─────────────────────────────────┬─────────────────┘    │
                                    │                       │
                                    │  XLogWrite()          │
                                    ▼                       │
                          ┌──────────────────┐             │

                          │  各 16MB           │  XLogBackgroundFlush()
                          └──────────────────┘

2 つの重要なプロセス間通信メカニズム：

  • 共有メモリ：Backend が WAL レコードを共有バッファに書き込み、WAL Writer が共有バッファから読み取ってディスクに書き込む
  • Latch：Backend が SetLatch() で WAL Writer を起動し、WAL Writer が WaitLatch() で起動シグナルを待つ。Latch は PG の軽量同期プリミティブで、セマフォよりオーバーヘッドが低い

Write vs Flush：2 段階書き込み

WAL のディスク書き込みは 2 つの段階に分かれており、この違いを理解することは極めて重要である：

段階操作データ到達先性能永続性
Writeメモリ → OS ファイルキャッシュOS カーネルバッファ高速（メモリコピー）停電で消失
FlushOS ファイルキャッシュ → ディスク永続ストレージ低速（実際の I/O）停電でも保持

トランザクション COMMIT 時には WAL Flush の完了を確認しなければならない――これが永続性保証の鍵である。synchronous_commit パラメータでこの動作を制御する：

  • on（デフォルト）：COMMIT は WAL Flush 完了まで待ってから返却
  • off：COMMIT は Flush を待たない。直近数百ミリ秒のトランザクションを失う可能性がある。性能は良いが永続性を犠牲にする

XLogWrite：実際のディスク書き込み

XLogWrite は最下層の書き込み関数である。そのコアロジック：

  • クリティカルセクションに入る（START_CRIT_SECTION）――書き込み中は割り込み不可、途中でキャンセルされて WAL ファイルが破損することを防止
  • すべての進行中の WAL 挿入の完了を待機（WaitXLogInsertionsToFinish）――複数の Backend が同時に WAL バッファに書き込んでいる可能性がある
  • 書き込む Page 範囲を計算し、pg_pwrite でバッチ書き込み
  • WAL Segment をまたぐ場合の処理――現在のファイルが 16MB 書き終わったら次のファイルに切り替え
// XLogWrite のコア書き込みループ（簡略化）
do {
    written = pg_pwrite(openLogFile, from, nleft, startoffset);
    if (written <= 0) {
        if (errno == EINTR) continue;  // シグナル割り込み、リトライ
        ereport(PANIC, ...);           // 書き込み失敗 → PANIC、データベース停止
    }
    nleft -= written;
    from += written;
    startoffset += written;
} while (nleft > 0);

書き込み失敗時は PANIC であることに注意――WAL 書き込みの失敗は永続性保証が破壊されたことを意味し、データベースは即座に停止して人手の介入を待たなければならない。

WAL ファイルの構成

  $PGDATA/pg_wal/

  ┌──────────────────────────────────────────────────────────┐
  │ 000000010000000000000001  (16MB)                          │
  │ ┌──────────┬──────────┬──────────┬─────┬──────────┐      │
  │ │ WAL Page │ WAL Page │ WAL Page │ ... │ WAL Page │      │
  │ │  0 (8KB) │  1 (8KB) │  2 (8KB) │     │ 2047     │      │
  │ │ ┌──────┐ │          │          │     │          │      │
  │ │ │Record│ │          │          │     │          │      │
  │ │ │Record│ │          │          │     │          │      │
  │ │ │Record│ │          │          │     │          │      │
  │ │ └──────┘ │          │          │     │          │      │
  │ └──────────┴──────────┴──────────┴─────┴──────────┘      │
  ├──────────────────────────────────────────────────────────┤
  │ 000000010000000000000002  (16MB)                          │
  ├──────────────────────────────────────────────────────────┤
  │ 000000010000000000000003  (16MB)                          │
  └──────────────────────────────────────────────────────────┘

  2048 個の 8KB Page = 16MB = 1 つの WAL Segment

  ファイル名エンコーディング: {Timeline}{SegmentHigh}{SegmentLow}
  例: 00000001 00000000 00000001 = Timeline 1, Segment 1

  ライフサイクル:
  1. 新しいファイルを作成
  2. 16MB 書き終わったら → 次のファイルに切り替え
  3. Checkpoint 後 → 古いファイルは回収可能
  4. 回収 = リネームして再利用（create/delete のオーバーヘッドを回避）

Checkpoint：一貫性リカバリポイント

Checkpoint がなければ、クラッシュリカバリは最初の WAL レコードからリプレイを開始する必要がある――運用時間が伸びるにつれて、リカバリ時間が無限に増大する。Checkpoint はこの問題を解決する：定期的にすべてのダーティページをディスクにフラッシュし、「一貫性ポイント」を作成する。リカバリは直近の Checkpoint からのリプレイのみで済む。

  時間 ──────────────────────────────────────────────────────►

  WAL: ═══════════════════════════════════════════════════════
       │              │                    │          │
    Checkpoint 1   Checkpoint 2         Checkpoint 3  クラッシュ!
       │              │                    │          │
       │              │                    │◄────────►│
       │              │                    リカバリはこの
       │              │                    区間の WAL のみ
       │              │                    リプレイが必要
       │              │
       │              Checkpoint 2 以降
       │              Checkpoint 1 以前の WAL は回収可能

Checkpoint のエントリは CheckpointerMain（B_CHECKPOINTER プロセス）で、トリガータイミング：

  • タイムアウトトリガー：checkpoint_timeout パラメータ（デフォルト 5 分）
  • WAL 量トリガー：WAL 生成量が max_wal_size（デフォルト 1GB）を超えた場合
  • 手動トリガー：CHECKPOINT コマンドの実行
  • シャットダウントリガー：データベースの正常シャットダウン時

Checkpoint の実行ステップ

  CreateCheckPoint:

  ┌─────────────────────────────────┐
  │ 1. 現在の WAL 挿入位置を取得     │
  │    (redo LSN)                    │
  │    「このポイント以降の WAL は    │
  │     リプレイが必要」              │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ 2. すべてのダーティバッファを     │
  │    ディスクにフラッシュ           │
  │    Buffer Pool 内の各 dirty     │
  │    buffer を走査し fsync を呼出  │
  │    （最も時間がかかるステップ）   │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ 3. WAL をディスクにフラッシュ     │
  │    すべての WAL レコードが        │
  │    永続化されたことを確認         │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ 4. 制御ファイルを更新             │
  │    $PGDATA/global/pg_control    │
  │    Checkpoint 位置情報を書き込み  │
  └──────────────┬──────────────────┘
                 ▼
  ┌─────────────────────────────────┐
  │ 5. 古い WAL ファイルを回収        │
  │    Checkpoint 以前の WAL は      │
  │    リカバリに不要 → リネームして  │
  │    再利用                         │
  └─────────────────────────────────┘

重要な制約：Checkpoint 中のダーティページフラッシュは漸進的に行われ（checkpoint_completion_target パラメータ）、瞬間的な大量 I/O が通常のクエリに影響することを回避する。デフォルト設定では、2 回の Checkpoint 間隔の 90% の時間をかけて均等にフラッシュを完了する。

クラッシュリカバリ：WAL のリプレイ

データベースがクラッシュ後に再起動すると、StartupProcessMain（B_STARTUP プロセス）がリカバリを担当する：

  • pg_control ファイルを読み取り、直近の Checkpoint 位置を特定
  • Checkpoint の redo LSN から WAL の読み取りを開始
  • 各 WAL レコードについて、対応する Page の pd_lsn を確認：pd_lsn >= record_lsn であれば、その変更はすでに永続化されているのでスキップ；そうでなければ rm_redo を呼んでリプレイ
  • WAL 末尾まで再生したらリカバリ完了、データベースは通常サービスを開始
// クラッシュリカバリのコアロジック（疑似コード）
checkpoint = ReadControlFile();       // pg_control を読み取り
lsn = checkpoint.redo;                // リカバリ開始ポイント

while (record = ReadWALRecord(lsn)) {
    page = GetPage(record.target);    // ターゲット Page を取得

    if (page.pd_lsn >= record.lsn)
        continue;                     // すでに適用済み、スキップ

    rmgr = GetRmgr(record.rmid);
    rmgr.rm_redo(record);             // この WAL をリプレイ
    lsn = NextRecordLSN(record);
}
// リカバリ完了

ストリーミングレプリケーション：WAL のもう一つの消費者

WAL はクラッシュリカバリだけに使われるのではない――ストリーミングレプリケーションの基盤でもある。B_WAL_SENDER プロセスが WAL を standby にリアルタイムで送信する：

  Primary                                    Standby
  ┌──────────────────────┐                  ┌──────────────────────┐
  │                      │                  │                      │
  │  Backend Process     │                  │  WAL Receiver        │
  │  INSERT/UPDATE/...   │                  │  (B_WAL_RECEIVER)    │
  │       │              │                  │       │              │
  │       ▼              │                  │       ▼              │
  │  WAL Buffer          │ ストリーミング    │  WAL ファイルに書込  │
  │       │              │ レプリケーション   │       │              │
  │       ▼              │ プロトコル        │       ▼              │
  │  WAL Sender          │ ◄──────────────► │  Startup Process     │
  │  (B_WAL_SENDER)      │                  │  WAL を継続的にリプレイ│
  │                      │                  │       │              │
  │  XLogSendPhysical()  │ ─── WAL データ ──►│       ▼              │
  │  または               │                  │  データが Primary と │
  │  XLogSendLogical()   │                  │  一致                │
  └──────────────────────┘                  └──────────────────────┘

物理レプリケーション vs ロジカルレプリケーション

物理レプリケーションロジカルレプリケーション
送信内容生の WAL バイトストリームデコードされた論理的変更（INSERT row X）
送信関数XLogSendPhysicalXLogSendLogical
Standby 要件完全に同一の PG バージョンとプラットフォーム異なるバージョンや異なるデータベースも可
粒度クラスタ全体テーブルレベル、選択的にレプリケーション可能
用途高可用性、読み取りレプリカデータ統合、バージョン間マイグレーション

ロジカルレプリケーションのデコードプロセス：XLogSendLogical → XLogReadRecord で WAL レコードを読み取り → LogicalDecodingProcessRecord が対応するリソースマネージャの rm_decode 関数を呼び出す。例えば DecodeInsert が XLOG_HEAP_INSERT タイプの WAL レコードをロジカルレプリケーションが理解できる行変更イベントに解析する。

Replication Slot と LSN

Replication Slot は各サブスクライバーの消費進捗を追跡する：

  • restart_lsn：この slot が WAL を保持する必要のある開始位置――これより古い WAL は回収できない
  • confirmed_flush_lsn：サブスクライバーが受信し処理済みの位置

standby が長時間切断されると、restart_lsn が WAL 回収を阻止し、primary のディスクが満杯になる。これは本番環境でよくある障害シナリオである。

Timeline：データベースのブランチ履歴

Timeline は PG がデータベースの「ブランチ」を処理するメカニズムである。各 Timeline は一意の数字 ID を持ち、以下のシナリオで新しい Timeline が作成される：

  • PITR（Point-in-Time Recovery）：ある過去の時点からのリカバリ
  • Standby Promotion：standby がマスターに昇格
  • バックアップからのリストア
  Timeline 1: ════════════════════════╗
                                      ║ Standby 昇格
  Timeline 2:                         ╚═══════════════════
                                            │
                                    00000002.history ファイル
                                    TL2 が TL1 のどの LSN から分岐したかを記録

Timeline 履歴ファイル（例：00000002.history）は分岐ポイントを記録し、リカバリプロセスが正しい WAL ファイルを選択する助けとなる。

PG プロセスの全体像

最後に、本シリーズで登場したすべてのバックグラウンドプロセスをまとめる：

  postmaster (メインプロセス)
  │
  ├── Backend (B_BACKEND)           クライアントの SQL クエリを処理
  │   └── WAL Sender (B_WAL_SENDER) レプリケーション接続昇格後に変化
  │
  ├── Background Writer (B_BG_WRITER)  定期的にダーティページをフラッシュし Checkpoint の負荷を軽減
  ├── Checkpointer (B_CHECKPOINTER)    定期的に Checkpoint を実行
  ├── WAL Writer (B_WAL_WRITER)        定期的に WAL バッファをディスクにフラッシュ
  ├── Autovacuum Launcher              VACUUM 作業をスケジューリング
  │   └── Autovacuum Worker            具体的な VACUUM を実行
  ├── WAL Receiver (B_WAL_RECEIVER)    Standby 上で WAL を受信
  ├── WAL Summarizer                   WAL サマリ（増分バックアップ用）
  ├── Startup (B_STARTUP)              クラッシュリカバリ / Standby の継続的リプレイ
  ├── Archiver (B_ARCHIVER)            古い WAL ファイルをアーカイブ
  ├── IO Worker (B_IO_WORKER)          非同期 I/O ワーカースレッド
  └── Syslogger (B_LOGGER)            ログ収集

PG はマルチプロセスアーキテクチャで、各バックグラウンドプロセスが独立して動作し、共有メモリと Latch で通信する。利点は分離性が良いこと（1 つのプロセスがクラッシュしても他のプロセスを巻き込まない）とデバッグのしやすさ。代償はプロセス間通信がスレッド間通信より重いこと。

まとめ

覚えておくべき重要な直感：

  • WAL = シーケンシャルログ書き込み + データページの遅延書き込み：高コストなランダム I/O を安価なシーケンシャル I/O に変換する。トランザクションコミットは WAL フラッシュのみを確認すればよく、データページは非同期で書ける
  • Write ≠ Flush：Write は OS キャッシュまで、Flush はディスクまで。COMMIT は永続性を保証するために Flush が必要
  • WAL Writer と Backend は共有メモリ + Latch で通信：複数の Backend が WAL バッファに並行書き込みし、WAL Writer がフラッシュを担当する。書き込み失敗 → PANIC
  • Checkpoint = ダーティページ全フラッシュ + リカバリポイント作成：クラッシュリカバリは直近の Checkpoint からのリプレイのみで、最初から始める必要はない。Checkpoint 以前の WAL は回収可能
  • WAL はリカバリとレプリケーションの両方に対応：同じ WAL レコードが、クラッシュリカバリ時には Startup プロセスによりリプレイされ、ストリーミングレプリケーション時には WAL Sender により Standby に送信される
  • 物理 vs 論理：物理レプリケーションは生のバイトストリームを送信（シンプルで効率的）、ロジカルレプリケーションはリソースマネージャを通じて行変更イベントにデコード（柔軟だがより複雑）

これで PostgreSQL ストレージエンジン三部作は完了である。Heap Page の 8KB 内部レイアウトから、Buffer Pool のページキャッシュ管理、インデックススキャンと MVCC の可視性判定、WAL の永続性保証と Checkpoint のリカバリ機構まで――これらの多層に入れ子になった抽象が、リレーショナルデータベースの最も核心的な骨格を構成している。
