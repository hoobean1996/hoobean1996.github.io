ステートフルサービスのスケーリング：方法論から Redis、Memcached、MySQL の実践まで

2026-03-08
この記事の価値：ステートレスサービスのスケーリングは簡単だ――マシンを追加してロードバランサーの後ろに配置すればいい。しかしステートフルサービス（データベース、キャッシュ、メッセージキュー）のスケーリングは分散システムにおける最も難しい問題の一つだ：データはどのマシンにあるのか？どう移動するのか？移動中のリクエストはどうするのか？この記事ではまず再利用可能な 5 ステップの方法論を提示し、次に Redis Cluster、Memcached、MySQL の 3 つの実際のシステムで各ステップがどう実装されているかを説明する。

ステートフルサービスのスケーリングが難しい理由

ステートレスサービスのスケーリングモデル：

  リクエスト ──► Load Balancer ──┬──► Server A
                           ├──► Server B    どのサーバーでも処理可能
                           └──► Server C    マシン追加で拡張

ステートフルサービスのスケーリングモデル：

  リクエスト (user_id=42)
    │
    │  このリクエストは user_id=42 のデータを持つマシンにしか送れない
    │  任意に送ることはできない
    ▼
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │ Node A   │  │ Node B   │  │ Node C   │
  │ id: 1-100│  │id:101-200│  │id:201-300│
  └──────────┘  └──────────┘  └──────────┘

  Node D を追加する場合：
  - 誰のデータを移動するのか？
  - 移動中の読み書きはどうルーティングするのか？
  - 移動の途中でクラッシュしたらどうするのか？

本質的な問題はデータに所有権があることだ。ステートフルサービスのスケーリング = サービスを停止せずにデータの所有権を再分配すること。

方法論：5 ステップスケーリングモデル

ステップ 1：シャーディング戦略――Key から Node へのマッピング

最も直感的な方法は hash(key) % N で N 台のマシンに直接マッピングすることだ。しかしマシンを 1 台追加すると N が N+1 になり、ほぼすべての key のマッピングが変わる――つまり全データの移行が必要になる。

解法は論理スロット（Virtual Slot）という中間層を追加すること：

  悪い設計：Key → Node（直接マッピング）
  hash(key) % 3 = Node A
  hash(key) % 4 = Node B  ← マシン 1 台追加で、ほぼ全 key の移動が必要

  良い設計：Key → Slot → Node（2 層マッピング）
  hash(key) % 16384 = Slot 5742    ← この層は永久に変わらない
  Slot 5742 → Node A               ← Slot の所属を移動するだけ

  ┌─────────────────────────────────────────────────────────┐
  │  Key Space                                               │
  │  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┬─────┐     │
  │  │Slot │Slot │Slot │Slot │Slot │Slot │Slot │ ... │     │
  │  │  0  │  1  │  2  │  3  │  4  │  5  │  6  │     │     │
  │  └──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴──┬──┴─────┘     │
  │     │     │     │     │     │     │                      │
  │     ▼     ▼     ▼     ▼     ▼     ▼                      │
  │   Node A        Node B        Node C                     │
  │  Slot 0,1,2    Slot 3,4      Slot 5,6                   │
  └─────────────────────────────────────────────────────────┘

  拡張：Slot 2 を Node A から Node D に移動
  - Key から Slot へのマッピングは変わらない
  - Slot から Node へのマッピングのみ変更
  - Slot 2 のデータのみ移動

Slot 数の選択：通常はマシン数の 100 倍以上。少なすぎると Slot の粒度が粗く均等に分配できない；多すぎるとメタデータのオーバーヘッドが増大する。Redis Cluster は 16384 を選択しており、実践で検証されたバランスポイントである。

ステップ 2：コントロールプレーン――メタデータの保存場所

「Slot X は Node A 上にある」というルーティングテーブルはどこかに保存する必要があり、すべての参加者が一貫したバージョンを参照できなければならない。

方式原理利点欠点代表的システム
Gossipノード間で相互に伝播、結果整合性単一障害点なし、自己修復収束が遅い、スプリットブレインのリスク、複雑なスケジューリングが困難Redis Cluster
中央集権型ストレージetcd/ZooKeeper に権威あるメタデータを格納強一貫性、オーケストレーションが容易追加コンポーネントの運用が必要TiDB PD、Vitess
設定サービスMcrouter が設定ファイルからルーティングテーブルを読み取り極めてシンプル、ランタイム依存なし変更にはプッシュが必要、動的スケーリングに不向きMemcached + Mcrouter

2026 年のトレンド：中央集権型コントロールプレーン + K8s Operator が主流。etcd が強一貫性のメタデータストレージを提供し、Operator がスケーリング、フェイルオーバー、データ移行などの操作を自動化する。Gossip プロトコルは障害検出（ノードダウンの迅速な発見）には引き続き使われるが、ルーティング決定は担わなくなっている。

大規模本番環境では Proxy モデルが好まれる。数十種類のプログラミング言語のクライアントがある場合、各言語に複雑なスマートクライアントを維持することは不可能だ。Proxy がルーティングロジックを集中管理し、クライアントはプロトコルを話せるだけでよい。

ステップ 4：レプリカと可用性――データは 1 つだけでは不十分

シャーディングは容量の問題を解決するが、各シャードは依然として単一障害点だ。レプリカが可用性を解決する：

  Slot 0-5461              Slot 5462-10922          Slot 10923-16383
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ Master A     │         │ Master B     │         │ Master C     │
  │ (読み書き)   │         │ (読み書き)   │         │ (読み書き)   │
  └──────┬───────┘         └──────┬───────┘         └──────┬───────┘
         │ 非同期レプリケーション  │ 非同期レプリケーション  │ 非同期レプリケーション
         ▼                        ▼                        ▼
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ Replica A'   │         │ Replica B'   │         │ Replica C'   │
  │ (読み取り専用)│         │ (読み取り専用)│         │ (読み取り専用)│
  └──────────────┘         └──────────────┘         └──────────────┘

フェイルオーバーの標準フロー：

  1. Master A のダウンを検出（ハートビートタイムアウト / Gossip 投票 / コントロールプレーン監視）
  2. レプリカから新しい Master を選出（レプリケーション進捗が最も近いものを選択）
  3. 重要なステップ：コントロールプレーンのメタデータを更新――Slot 0-5461 の Owner を A から A' に変更
  4. ルーティング層に通知（Proxy がルーティングテーブルを更新 / Client が Slot Map をリフレッシュ）
  5. 新 Master A' が読み書きの受け入れを開始

レプリケーションモードの選択：

非同期レプリケーションデータ損失の可能性あり最低最高Redis、Memcached
半同期レプリケーション少なくとも 1 レプリカが確認中程度高MySQL Semi-sync
Raft/Paxos多数派確認、強一貫性やや高い中程度TiKV、CockroachDB

ステップ 5：データ移行――飛行中にエンジンを交換する

これが最も難しいステップだ。新しいマシンを追加した後、一部の Slot を旧ノードから新ノードに移動する必要があるが、同時にオンラインサービスを中断してはならない。

  Slot 3 を Node A → Node D に移行：

  フェーズ 1: 移行状態のマーク
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Slot 3:  │                    │ Slot 3:  │
  │ MIGRATING│                    │ IMPORTING│
  └──────────┘                    └──────────┘

  フェーズ 2: 全量スナップショット転送
  ┌──────────┐  ══ スナップショットデータ ══►   ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ サービス  │                    │ データ    │
  │ 継続      │                    │ 受信中    │
  └──────────┘                    └──────────┘

  フェーズ 3: 増分追い付き
  ┌──────────┐  ── 増分ログ ──►   ┌──────────┐
  │ Node A   │  (移行中の新規書込) │ Node D   │
  │ サービス  │                    │ 追い付き中│
  │ 継続      │                    │          │
  └──────────┘                    └──────────┘

  フェーズ 4: アトミック切り替え (重要な瞬間)
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Slot 3 の│  ← 短時間の書込停止 │ 追い付き  │
  │ 書込を停止│  (通常数ミリ秒)     │ 完了!    │
  └──────────┘                    └──────────┘
       │
       │  コントロールプレーン更新: Slot 3 → Node D
       │  ルーティング層リフレッシュ
       ▼
  ┌──────────┐                    ┌──────────┐
  │ Node A   │                    │ Node D   │
  │ Slot 3   │                    │ Slot 3:  │
  │ データ削除│                    │ 通常サービス│
  └──────────┘                    └──────────┘

フェーズ 4 がプロセス全体で最も危険な瞬間である。増分ログが追い付けない場合（書き込みが速すぎる）、ブロック時間が延び、オンラインサービスに影響する。2 つの対処戦略：

  • 書き込み制限：追い付きフェーズで該当 Slot への書き込みを制限し、書き込み速度を下げて増分の追い付きを可能にする
  • デュアルライトモード：切り替え期間中に両方のノードに同時書き込み、ブロックしないがロジックが複雑になり、競合処理が必要

実践：Redis Cluster のスケーリング

ステップRedis Cluster の実装
シャーディング16384 Slot、CRC16(key) % 16384
コントロールプレーン分散型 Gossip。各ノードが完全な Slot Map を維持し、Gossip プロトコルで同期
ルーティングSmart Client。クライアントが Slot Map をキャッシュし、ターゲットノードに直接接続。間違えた場合、ノードが MOVED リダイレクトを返却
レプリカ各 Master に 1-N 個の Replica を配置、非同期レプリケーション。フェイルオーバーは Gossip 投票で新 Master を選出
移行CLUSTER SETSLOT + MIGRATE コマンド、Key ごとに移行

Redis 移行の詳細

# 1. 移行状態のマーク
CLUSTER SETSLOT 3 MIGRATING target-node-id   # ソースノードでマーク
CLUSTER SETSLOT 3 IMPORTING source-node-id   # ターゲットノードでマーク

# 2. Key ごとに移行（Redis にはスナップショット移行がなく、1 つずつ移動）
CLUSTER GETKEYSINSLOT 3 100                  # Slot 3 の 100 個の Key を取得
MIGRATE target-host 6379 "" 0 5000 KEYS key1 key2 key3 ...

# 3. 切り替え完了
CLUSTER SETSLOT 3 NODE target-node-id        # すべてのノードで実行

Redis 移行の特徴と制限：

  • Key ごとの移行であり、スナップショットではない。大きな Key（百万要素の Hash など）の移行時にソースノードがブロックされる
  • 移行中、ソースノードは MIGRATING Slot に対して：Key がまだローカルにある → 通常処理；Key がすでに移動済み → ASK リダイレクトをターゲットノードに返却
  • Gossip の収束が遅い――大規模クラスタ（200+ ノード）では、1 回の Slot 変更がすべてのノードに伝播するのに数秒かかる可能性がある
  • 増分ログがない：Key の移行はアトミック（MIGRATE は DEL + RESTORE）で、「移行中に Key が変更される」問題は存在しないが、代償として大きな Key の移行がブロックする

Redis 拡張の実際の操作

  拡張前: 3 Master、各 ~5461 Slots

  Master A: Slot 0-5460
  Master B: Slot 5461-10922
  Master C: Slot 10923-16383

  Master D を追加、A/B/C から各々一部の Slot を移動：

  Master A: Slot 0-4095          (1366 Slot を移出)
  Master B: Slot 5461-9556       (1366 Slot を移出)
  Master C: Slot 10923-15017     (1366 Slot を移出)
  Master D: Slot 4096-5460,      (4098 Slot を受入)
            9557-10922,
            15018-16383

  redis-cli --cluster reshard が上記プロセスを自動完了

実践：Memcached のスケーリング

ステップMemcached の実装
シャーディングクライアント側コンシステントハッシュ（Consistent Hashing）。Slot の概念はなく、ハッシュリング上の仮想ノード
コントロールプレーンなし。Memcached サーバー間は完全に通信せず、互いの存在を知らない
ルーティング純粋なクライアントルーティング。Mcrouter（Proxy モード）または libmemcached（Smart Client モード）
レプリカデフォルトではなし。Mcrouter で shadow/replica pool を設定して読み取りレプリカを構成可能
移行移行しない。キャッシュ miss 後にオリジンから再構築

Memcached のスケーリング哲学は Redis とまったく異なる――自身を純粋なキャッシュとして位置づけており、データストレージではない：

コンシステントハッシュ：マシン追加時に隣接ノードのみ影響

  ハッシュリング (0 ~ 2^32)

        Node A                    Node A
       ╱      ╲                  ╱      ╲
      ╱   key1  ╲               ╱   key1  ╲
  Node D ─── Node B    →    Node D ─── Node B
      ╲        ╱               ╲    ↑   ╱
       ╲      ╱                 ╲ Node E╱    ← 新規参加
        Node C                   Node C

  Node E 参加：
  - Node B と Node E の間の key のみ再マッピングが必要
  - 他の key はまったく影響を受けない
  - データ移行は不要――キャッシュ miss 後に自然にオリジンから Node E に書き込まれる

Mcrouter：Memcached の「頭脳」

Memcached 自体はシンプルで、すべてのインテリジェンスは Mcrouter（Meta がオープンソースにした Proxy）にある：

  • コンシステントハッシュルーティング：key がどの Memcached に行くべきかを計算
  • 障害処理：ノードダウン時にリングから自動除外し、key が次のノードに自動再マッピング
  • 段階的切り替え：設定ファイルでルーティングルールを更新し、段階的にトラフィックを新クラスタに切り替え可能
  • コネクションプール：フロントエンドの数千のクライアント接続を、Memcached への少数の長時間接続に多重化
  • コールドスタートウォームアップ：新クラスタ起動時、読み取り miss を自動的に旧クラスタに問い合わせ、同時に新クラスタに書き込み
// Mcrouter ルーティング設定例
{
  "pools": {
    "A": { "servers": ["mc1:11211", "mc2:11211", "mc3:11211"] },
    "B": { "servers": ["mc4:11211", "mc5:11211", "mc6:11211"] }
  },
  "route": {
    "type": "WarmUpRoute",          // コールドスタートウォームアップ
    "cold": "PoolRoute|B",          // 新クラスタ
    "warm": "PoolRoute|A",          // 旧クラスタ（miss 時にフォールバック）
    "exptime": 180
  }
}

Memcached 拡張の核心思想：データを移動せず、キャッシュの自己修復に頼る。マシン追加 → Mcrouter 設定を更新 → 新しい key は自動的に新ノードにルーティング → 古い key は miss 後にオリジンから再構築。代償は拡張直後の短期間のキャッシュヒット率低下（コールドスタート）だが、キャッシュシナリオでは許容できる。

実践：MySQL のスケーリング

ステップMySQL の実装
シャーディングアプリケーション層シャーディング（ShardingSphere）またはミドルウェアシャーディング（Vitess）。user_id や order_id でモジュロ
コントロールプレーンVitess: etcd にトポロジーを格納。ShardingSphere: ZooKeeper/etcd
ルーティングVitess: VTGate (Proxy)。ShardingSphere: Proxy または JDBC Driver (Smart Client) を選択可
レプリカInnoDB 半同期レプリケーション / Group Replication / Raft (TiDB)
移行Binlog ベースの増分レプリケーション + アトミック切り替え

MySQL のスケーリングは Redis や Memcached よりはるかに複雑だ。トランザクション、JOIN、外部キーなどのリレーショナルセマンティクスが関わるからだ。

MySQL シャード移行：Binlog ベース

MySQL のデータ移行は本質的に PG のストリーミングレプリケーションと同じパターンだ――全量スナップショット + 増分ログ追い付き：

  1. 全量スナップショット：ソース Shard に対して mysqldump または Xtrabackup 物理スナップショットを取得し、新ノードに転送
  2. 増分追い付き：スナップショット時の Binlog ポジションから Binlog の消費を継続し、新ノードに適用。このステップは PG の WAL replay と同様
  3. 検証：ソースとターゲットのデータ一貫性を比較（Vitess の VDiff）
  4. アトミック切り替え：短時間の書き込み停止 → Binlog の追い付き完了を待機 → トポロジー更新 → 書き込み再開

Vitess の Reshard コマンドがこのフローを自動化する。鍵は VReplication エンジン――単純に Binlog をリプレイするのではなく、シャーディングルールでフィルタリングする：ターゲット Shard に属する行変更のみを適用する。

MySQL スケーリングの特殊な課題

  • クロスシャード JOIN：SELECT * FROM orders JOIN users ON orders.user_id = users.id、orders と users が異なる Shard にある場合、Proxy が個別にクエリして結果をマージする必要がある。性能が急激に低下
  • クロスシャードトランザクション：2 フェーズコミット（2PC）が必要で、Vitess は VTGate で調整するが、性能と信頼性は単一シャードトランザクションに及ばない
  • シャードキーの選択：一度決定したら（例：user_id）、変更はほぼ不可能。間違った選択（例：order_id でシャーディングしたが user_id で頻繁にクエリ）は大量のクロスシャードクエリを引き起こす
  • ホットスポット問題：一人の大口顧客（user_id=1）のデータ量が他のユーザーをはるかに超える場合、その Shard がホットスポットになる。解法：その Shard をさらに分割するか、アプリケーション層で特別な処理を行う

まとめ

ステートフルサービスを拡張する 5 ステップモデル：

  1. シャーディング：Key → Slot → Node の 2 層マッピング、Slot 数 = マシン数 × 100+
  2. コントロールプレーン：ルーティングテーブルの保存場所、権威ある情報源は誰か。トレンドは etcd + Operator
  3. ルーティング：Smart Client（低レイテンシ）vs Proxy（運用しやすい）。多言語環境では Proxy を選択
  4. レプリカ：非同期（高速だが損失の可能性）vs 半同期（バランス）vs Raft（強一貫性だが低速）
  5. 移行：全量スナップショット + 増分ログ + アトミック切り替え。鍵は切り替えの瞬間の処理戦略

3 つの異なるレベルの実践：

  • Memcached：最もシンプル――データを移動せず、キャッシュ miss の自己修復に頼る。純粋なキャッシュシナリオに適合
  • Redis：中程度の複雑さ――Key ごとに移行、Gossip 協調。キャッシュと軽量ストレージの両方に適合
  • MySQL：最も複雑――スナップショット + Binlog 増分、さらにクロスシャード JOIN とトランザクションの処理が必要。永続化リレーショナルデータに適合

具体的なシステムがどう変わっても、5 ステップモデルは普遍的だ。次に新しいステートフルシステムを拡張する必要がある時は、まずこの 5 つの質問をしよう：どうシャーディングするか、メタデータをどこに保存するか、リクエストをどうルーティングするか、レプリカをどうするか、データをどう移動するか。
