リクエストの旅：データセンターネットワークの全体像

2026-03-08
この記事の価値：ほとんどのバックエンドエンジニアは、リクエストがブラウザから発信され、サーバーで処理され、レスポンスが返ることを知っている。しかしその間に何を通るのか？PoP ノード、バックボーン光ファイバー、Clos スイッチファブリック、eBPF L4 ロードバランサー、VXLAN コンテナネットワーク――これらのインフラがあなたのシステムで何ができて何ができないかを決める。この記事では、ユーザーのスマートフォンからデータセンターのサーバーまでの完全な旅を追跡し、各レイヤーの重要な技術を解説する。

全体像：3 層アーキテクチャ

大規模インターネット企業のネットワークインフラはすべて 3 層構造である：

層役割例
Edge / PoPユーザートラフィックの入口、TLS 終端、地理的ルーティング東京 PoP が日本のユーザーのリクエストを受信
Backboneデータセンター間の高速プライベート光ファイバー、SR/MPLS を運用シンガポール DC ↔ 米国 DC、~120ms RTT
DC InternalClos スイッチファブリックが DC 内のすべてのサーバーを接続Leaf-Spine-Core 3 層スイッチング

東京のユーザーがあるアプリを開く：スマートフォンの DNS が最寄りの東京 PoP IP に解決 → PoP が TLS を終端し、静的コンテンツは CDN から直接返却 → 動的リクエストはバックボーン（SR/MPLS）を経由してシンガポール DC へ → DC に入ると Katran（L4 LB）→ Proxygen（L7 LB）→ アプリケーションサーバー → 米国 DC のデータが必要なら、もう一度大洋横断バックボーンを経由 → レスポンスは DSR でユーザーに直接返却し、L4 LB をバイパスする。

このパスに沿って、各レイヤーを分解していこう。

DC 内部：Clos アーキテクチャ

一台の大型スイッチでは数十万台のサーバーにスケールできない。Clos トポロジーは大量の安価な小型スイッチを組み合わせて、ノンブロッキングで水平スケール可能なネットワークを構成する。

トラフィックパスのホップ数：

  • 同一ラック：Leaf ローカルスイッチング、0 ホップ
  • 同一 Pod 別ラック：Leaf → Spine → Leaf、2 ホップ
  • Pod 間：Leaf → Spine → Core → Spine → Leaf、4 ホップ

Pod：繰り返し可能な構成単位

Pod は一連の Leaf + 対応する Spine のセットで、DC スケーリングの基本単位である。拡張するには？Pod を一つ追加して Core に接続するだけ。

64 ポートスイッチで規模を計算してみよう：

Leaf：32 ポートが下向きでサーバーに接続 → 各ラック 32 台のサーバー
Pod： 32 個の Leaf              → 32 × 32 = 1,024 台のサーバー
DC：  32 個の Pod               → 32 × 1,024 = 32,768 台のサーバー

もっと必要なら？もう一層追加（5-stage Clos / Super Spine）すれば、数十万台のサーバーを支えられる。大手インターネット企業の F16 Fabric はこのマルチステージ Clos 設計である。

ECMP：等コストマルチパス

Clos アーキテクチャの核心設計原則。Leaf 1 から Leaf 4 へパケットを送る時、異なる Spine を経由する複数の等コストパスがある。スイッチが五元組（送信元 IP、宛先 IP、送信元ポート、宛先ポート、プロトコル）をハッシュし、すべてのパスに均等に分配する。

  Leaf 1 ──パケット送信──►  Leaf 4 へ、4 本の等コストパス：

  Leaf 1 ─── Spine A ─── Leaf 4    ← hash(五元組) = 0
  Leaf 1 ─── Spine B ─── Leaf 4    ← hash(五元組) = 1
  Leaf 1 ─── Spine C ─── Leaf 4    ← hash(五元組) = 2
  Leaf 1 ─── Spine D ─── Leaf 4    ← hash(五元組) = 3

  トラフィックが均等に分散 → ボトルネックなし → Spine 1 台がダウンしても自動再分配

ECMP の意義：

  • ボトルネックなし：トラフィックが均等に分散され、単一リンクの過負荷がない
  • フォールトトレランス：Spine 1 台がダウンしても、トラフィックは残りの Spine に自動再分配される
  • リニアスケーリング：帯域幅がもっと必要？Spine を追加するだけ

Oversubscription：オーバーサブスクリプション比率

Leaf スイッチの下り帯域幅（サーバー向き）と上り帯域幅（Spine 向き）の比率。

比率意味シナリオ
1:1ノンブロッキング、すべてのサーバーが同時にフル速度で通信可能レイテンシ重視（DB クラスタ、ML トレーニング）
3:1サーバー帯域幅の 1/3 のみ同時に上り可能汎用コンピューティング（統計的に十分）
4:1+より高いオーバーサブスクリプション、安価だがリスクありコールドストレージ、バッチ処理

実際の影響：サービス設計時、頻繁に通信するサービスは同一ラックまたは Pod 内に配置し、オーバーサブスクリプションのボトルネックを避ける。これが HDFS がレプリカの一つを同一ラックに、もう一つを別ラックに配置する理由である。

バックボーン：Segment Routing

データセンター間はプライベートバックボーンで接続される。従来の MPLS は各中間ルーターが転送状態を維持する必要がある（LDP/RSVP 経由）。Segment Routing（SR）はその進化形：完全なパスをパケットヘッダにエンコードし、中間ルーターは次の命令を読むだけでよい。

MPLS の例え：タクシーに乗って、交差点ごとに道を聞く
SR の例え：  出発前に完全な GPS ナビを受け取り、交差点ごとに次の指示だけ見る
  MPLS：各ルーターが転送状態を維持する必要あり
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ A │───►│ B │───►│ C │───►│ D │     B,C が各自 LDP テーブルを参照して次ホップを決定
  └───┘    └───┘    └───┘    └───┘

  SR：パスがパケットヘッダにエンコードされ、中間ルーターは命令を実行するだけ
  ┌───┐    ┌───┐    ┌───┐    ┌───┐
  │ A │───►│ B │───►│ C │───►│ D │
  └───┘    └───┘    └───┘    └───┘
  ヘッダ: [B, C, D]   [C, D]    [D]       各ホップで SID を一つポップ

2 種類の Segment ID：

  • Node SID：最短パスであるルーターへ――「ルーター D へ行け」
  • Adjacency SID：特定の物理リンクを通る――「B→C のリンクを使え」

SRv6 は各大手ハイパースケールデータセンターが推進している方向である。128 ビットの SID で宛先と操作（例えば「デカプセレーション」「VPN テーブル参照」）を同時にエンコードでき、SR-MPLS の 20 ビットラベルよりはるかに柔軟である。

ロードバランシング：L4 + L7 の 2 層

データセンターには本質的に異なる 2 種類の「ロードバランシング」が協調して動作している。多くの人がこれらを混同するが、解決する問題はまったく異なる：

アプリケーション認識なし完全に認識
所在レイヤーL3/L4（スイッチ）L4/L7（ソフトウェア）

3 つの重要な設計判断：

  • XDP/eBPF：パケットは NIC ドライバ層で処理され、カーネルネットワークスタックを通過しない。単一マシンで数百万 PPS を達成可能
  • コンシステントハッシュ：同じ TCP コネクションは常に同じバックエンドにマッピングされる。バックエンドが変更されても、再マッピングが必要なのはごく一部のコネクションのみ
  • DSR（Direct Server Return）：レスポンスはバックエンドサーバーからユーザーに直接返却され、Katran をバイパスする。レスポンスはリクエストよりはるかに大きいため（HTTP リクエスト vs 画像を含むページを考えよう）、DSR は LB が帯域幅のボトルネックになることを防ぐ

オーバーレイネットワーク：コンテナはどう通信するか

あなたのコードはコンテナで動いており、コンテナには仮想 IP がある。物理ネットワークはこれらの IP を認識しない。オーバーレイネットワークがこのギャップを埋める。

アンダーレイ vs オーバーレイ

層内容IPデバイス
UnderlayClos 物理スイッチファブリック192.168.x.x（ホスト IP）Leaf, Spine, Core
Overlayコンテナ仮想ネットワーク10.244.x.x（Pod IP）veth, bridge, VXLAN

Linux 仮想ネットワークデバイス

  • veth pair：2 つのネットワーク名前空間を接続する仮想ネットワークケーブル。Docker/K8s がペアを作成：一端がコンテナに入り（eth0）、もう一端がホストに残り bridge に接続
  • bridge：ホスト上の仮想 L2 スイッチ。同一ホストのコンテナはこれを通じて通信し、物理ネットワークを経由しない
  • VXLAN device (VTEP)：コンテナが別ホスト上のコンテナにアクセスする場合、VXLAN デバイスが元のフレームを UDP パケット（ポート 4789）にカプセル化し、ホスト IP を外側アドレスとして使用する。物理ネットワークは通常通り ECMP でルーティングし、宛先ホストの VXLAN デバイスがデカプセル化してターゲットコンテナに配送する

なぜ VLAN ではなく VXLAN なのか？VLAN は 4,096 個の ID のみサポート（12 ビット）。VXLAN は 24 ビット VNI を使い、1600 万個の仮想ネットワークをサポートする――マルチテナント DC 環境に必須である。

ハイパースケールデータセンターはおそらく Calico BGP に似たダイレクトルーティング方式を使用している――物理スイッチがコンテナサブネットのルートを認識し、カプセル化が不要で性能が最良だが、ネットワークインフラとの深い統合が必要である。

DC 内部 vs 公共インターネット：2 つのまったく異なる世界

これが最も直感に反する部分である。教科書の TCP/IP は公共インターネット向けに設計されている――高レイテンシ、高パケットロス、不明な帯域幅。DC 内部はまったく異なる環境であり、各レイヤーで異なる最適化が必要である。

カーネルバイパス技術

カーネルネットワークスタックはコンテキストスイッチとプロトコル処理によりレイテンシを増加させる。DC ワークロードはこれを直接バイパスする：

技術原理用途
DPDKユーザー空間パケット処理、カーネルを完全にバイパス高性能 LB、NFV
RDMA/RoCENIC がリモートメモリを直接読み書き、CPU 不介入GPU トレーニング、分散ストレージ
XDP/eBPFNIC ドライバ層でパケットをインターセプト（カーネル最初のフックポイント）Katran LB、Cilium
io_uring非同期 I/O、システムコールのオーバーヘッドを削減高性能ストレージエンジン

Incast：DC 特有の輻輳パターン

1 つのリクエストが N 台のサーバーにファンアウトし（例：分散クエリ）、N 台が同時にレスポンスを返すと、リクエスト元の Leaf スイッチのバッファを瞬時に圧倒する。結果：パケットロス、TCP 再送、レイテンシの急上昇。

解決策：

  • ECN：スイッチバッファが閾値（例：30%）に達するとパケットにマークし、送信元がパケットロス発生前に速度を下げる
  • DCTCP：ECN マーク付きパケットの割合に応じて比例的に速度を下げ、二値的な「半減」ではない
  • アプリケーション層のずらし：レスポンス時間にジッターを加え、同期的なバーストを回避

ユーザーがリクエストを発信してからレスポンスを受け取るまでに、DNS、CDN、PoP、バックボーン、ボーダールーター、ECMP、eBPF L4 LB、L7 LB、コンテナオーバーレイネットワークを経由する――各レイヤーが特定の問題を解決しており、各レイヤーの設計は物理法則（光速、帯域幅、バッファサイズ）に制約されている。

まとめ

覚えておくべき重要な直感：

  • Clos + ECMP：安価な小型スイッチを組み合わせて大規模ネットワークを構成し、ハッシュでトラフィックを均等に分配する。DC ネットワークの物理的基盤
  • Katran + DSR：eBPF が NIC 層でロードバランシングを行い、レスポンスは LB をバイパスして直接ユーザーに返却する。従来の LB の帯域幅ボトルネックと単一障害点を解決
  • DC の RTT は公共インターネットより 1000 倍低い：このオーダーの差がすべての前提を変える。公共インターネットの TCP 輻輳制御、接続管理、セキュリティモデルは DC 内部では適用されず、専用の方式（DCTCP、RDMA、Zero Trust）が必要
  • オーバーレイネットワークは接着剤：コンテナの仮想 IP は VXLAN/BGP で物理ネットワークにマッピングされる。このカプセル化/デカプセル化のレイヤーを理解すれば、クロスホストコンテナ通信が同一ホスト内より遅い理由がわかる
  • オーバーサブスクリプションがサービスデプロイ戦略を決める：頻繁に通信するサービスを同一ラックまたは Pod に配置するのは「最適化」ではなく、オーバーサブスクリプションのボトルネックを避けるための必要な措置である
