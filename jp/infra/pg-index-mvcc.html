PostgreSQL ストレージエンジンの内部構造（中）：インデックススキャンと MVCC

2026-03-08
PostgreSQL ストレージエンジンの内部構造
上：Heap Page と Buffer Pool · 中：インデックススキャンと MVCC · 下：WAL、Checkpoint と永続性
この記事の価値：前回は Heap Page の内部レイアウトと Buffer Pool がページキャッシュをどう管理するかを見た。しかし重要な問題がまだ答えられていない：複数のトランザクションが同時に同じテーブルを読み書きする時、各トランザクションが見るデータはどう決まるのか？この記事は 2 つのパスを追跡する――インデックススキャンが B-Tree のリーフノードから Heap Page にジャンプしてデータを取得する仕組みと、MVCC スナップショットが行データが「あなたに可視か不可視か」をどう決定するか。これらを理解して初めて、トランザクション分離レベルが抽象的な概念ではなく、具体的なデータ構造であることが真に理解できる。

インデックススキャン：B-Tree から Heap へ

前回はシーケンシャルスキャン――Heap 全体をページごとに走査する方法を説明した。インデックススキャンはもう一つのパスである：まずインデックスで TID を特定し、次に TID で Heap からデータを取得する。

インデックスもテーブルである

PostgreSQL では、インデックスも通常のテーブルと同様に Relation であり、独自のファイル、独自の Page 構造を持つ。違いはインデックス Page にはユーザーデータではなく、インデックスキー + TID のマッピングが格納されていることだ。

  CREATE INDEX users_age_idx ON users(age);

  インデックススキャンは 2 つの Relation に関与する：

  indexRelation (users_age_idx)          heapRelation (users)
  ┌─────────────────────────┐           ┌──────────────────────────┐
  │  B-Tree Page            │           │  Heap Page               │
  │                         │           │                          │
  │  ┌───────────────────┐  │           │  ┌────────────────────┐  │
  │  │ age=20 → TID(1,3) │  │     ┌────►  │ ItemId[3] → Tuple  │  │
  │  │ age=25 → TID(5,1) │──┼─────┘    │  │  (id=42, name=     │  │
  │  │ age=25 → TID(8,7) │  │          │  │   'Alice', age=25) │  │
  │  │ age=30 → TID(2,5) │  │          │  └────────────────────┘  │
  │  └───────────────────┘  │           │                          │
  └─────────────────────────┘           └──────────────────────────┘

  IndexScanDesc は両方の Relation への参照を保持する

IndexScanDesc はインデックススキャンの記述子であり、主要フィールドは：

  • heapRelation：インデックス対象のテーブル
  • indexRelation：インデックス自体
  • xs_heaptid：現在見つかった TID、次のヒープアクセスに使用

Runtime Key は重要な概念である：クエリプランニング時に確定できない値のこと。例えば WHERE id = $1 のパラメータ、WHERE id = (SELECT max_id FROM others) のサブクエリ結果。これらの値は executor 起動時に初めて計算され、インデックススキャンに渡される。

B-Tree 内部：btgettuple

B-Tree は PostgreSQL で最もよく使われるインデックスタイプである。btgettuple のロジックは直截的だ：

  • 初回呼び出し：_bt_first() が B-Tree 内で開始位置を特定する――ルートノードからリーフノードまで下降し、スキャン条件を満たす最初のインデックスエントリを見つける
  • 以降の呼び出し：_bt_next() がリーフノードのリンクリスト上で前方/後方に移動し、次のエントリを取得
  • 位置を見つけた後、_bt_returnitem() が現在位置から TID を抽出する：scan->xs_heaptid = currItem->heapTid
  B-Tree 構造（簡略化）：

          ┌──────────────┐
          │  Root Page   │
          │  [20] [50]   │
          └──┬──────┬────┘
             │      │
      ┌──────▼──┐  ┌▼──────────┐
      │ Internal │  │ Internal   │
      │ [10][20] │  │ [30][50]   │
      └──┬───┬──┘  └──┬────┬───┘
         │   │        │    │
    ┌────▼┐ ┌▼────┐ ┌─▼──┐ ┌▼────┐
    │Leaf │↔│Leaf │↔│Leaf│↔│Leaf │    ← リーフノード双方向リンクリスト
    │5,8  │ │12,18│ │25  │ │35,42│
    │TIDs │ │TIDs │ │TIDs│ │TIDs │
    └─────┘ └─────┘ └────┘ └─────┘

  WHERE age = 25:
  1. _bt_first: Root → Internal → 第 3 リーフを特定
  2. リーフ内で age=25 のエントリを見つけ、TID を抽出
  3. _bt_next: リーフリンクリスト上でスキャンを継続、age ≠ 25 まで

ヒープアクセス：TID から Tuple へ

TID を取得した後、index_fetch_heap が Heap からデータを取得する。ヒープテーブルの場合、実際には heapam_index_fetch_tuple が呼ばれ、その核心は heap_hot_search_buffer である：

// インデックスヒープアクセスの完全なパス（簡略化）

// 1. インデックスが TID = (5, 1) を提供
scan->xs_heaptid = {block: 5, offset: 1};

// 2. Heap Page 5 を Buffer Pool に読み込み
buffer = ReadBuffer(heapRelation, 5);

// 3. Page 5 上で HOT チェーンを検索
heap_hot_search_buffer(
    tid,        // 開始 TID (5, 1)
    relation,   // users テーブル
    buffer,     // Page 5 の buffer
    snapshot,   // 現在のトランザクションのスナップショット  ← MVCC はここで機能
    &found
);

// 4. HOT チェーンをたどる：
//    ItemId[1] → Tuple A (t_ctid が ItemId[2] を指す)
//    ItemId[2] → Tuple B (t_ctid が ItemId[3] を指す)
//    ItemId[3] → Tuple C (t_ctid が自分自身を指す = チェーン末端)
//
//    各バージョンで HeapTupleSatisfiesVisibility(snapshot) を呼び出し
//    現在のスナップショットに可視なバージョンを返却

ここで snapshot パラメータが登場する――これが MVCC の入口である。インデックス自体にはバージョン情報が格納されておらず、すべてのバージョン判定はヒープアクセス後に行われる。つまりインデックススキャンでインデックスから TID を取得しても、ヒープアクセス後にその行が現在のトランザクションに不可視だと判明する場合があり、その場合は次の TID の取得に進む。

Lossy インデックスと Recheck

一部のインデックス（GIN、GiST など）は「有損」である――インデックスは「この Tuple はマッチする可能性がある」と教えてくれるが、確実なマッチは保証しない。ヒープアクセス後に元の条件で再チェックが必要である：

// IndexNext の recheck ロジック
while (index_getnext_slot(scandesc, direction, slot)) {
    if (scandesc->xs_recheck) {
        // インデックスが lossy なので、元の条件で再検証が必要
        if (!ExecQualAndReset(node->indexqualorig, econtext))
            continue;  // マッチしない、スキップ
    }
    return slot;  // マッチ、返却
}

MVCC：多バージョン同時実行制御

ここからこの記事の核心に入る。MVCC の目標は読み取りが書き込みをブロックせず、書き込みが読み取りをブロックしないことだ。PG は各行データの複数バージョンを同時に存在させることでこれを実現する――各トランザクションが見るのはある時点の「スナップショット」である。

トランザクション ID：論理クロック

PG はトランザクション ID（TransactionId、略称 XID）を「論理クロック」として使用する。各書き込みトランザクション開始時に単調増加する XID が割り当てられる。可視性判定の核心は XID の比較である：

行ヘッダフィールド意味可視性判定
t_xminこのバージョンを作成したトランザクションxmin のトランザクションがコミット済みでスナップショット前 → このバージョンは「生まれた」
t_xmaxこのバージョンを削除/更新したトランザクションxmax = 0 または xmax がコミット未済 → このバージョンは「まだ生きている」

Tuple があなたに可視な条件：すでに生まれている（xmin がコミット済み）かつ、まだ死んでいない（xmax が無効またはコミット未済）。

  トランザクションタイムライン：

  XID:  100      101      102      103      104
         │        │        │        │        │
         │  INSERT │        │ UPDATE │        │
         │  xmin=101       │ 新バージョン│      │
         │        │        │ xmin=103│        │
         │        │        │ 旧バージョン│      │
         │        │        │ xmax=103│        │

  トランザクション 104 のスナップショットはここで撮影 ──────►│

  トランザクション 104 にとって：
  旧バージョン：xmin=101(コミット済み✓) xmax=103(コミット済み✓) → 死亡 → 不可視
  新バージョン：xmin=103(コミット済み✓) xmax=0(まだ生存✓)   → 可視 ✓

SnapshotData：スナップショットのデータ構造

スナップショットは「時点」ではなく、具体的なデータ構造である：

typedef struct SnapshotData {
    TransactionId xmin;    // XID < xmin のすべてのトランザクションはコミット済み（確実に可視）
    TransactionId xmax;    // XID >= xmax のすべてのトランザクションは未開始（確実に不可視）
    TransactionId *xip;    // xmin と xmax の間で実行中のトランザクションリスト
    uint32 xcnt;           // xip 配列の長さ
} SnapshotData;
  スナップショットの 3 つの区間：

  XID:  95  96  97  98  99  100  101  102  103  104  105
        ────────────┬───────────────────────┬────────────
                    │                       │
               snapshot.xmin           snapshot.xmax

  ◄── 確実に可視 ──►│◄── xip 配列の確認要 ──►│◄── 確実に不可視 ──►

  xip[] = {99, 101}  ← スナップショット撮影時にまだ実行中のトランザクション

  判定ロジック：
  XID 96: < xmin → コミット済み → 可視 ✓
  XID 99: xip 内にある → まだ実行中 → 不可視 ✗
  XID 100: xip 内にない → コミット済み → 可視 ✓
  XID 101: xip 内にある → まだ実行中 → 不可視 ✗
  XID 104: >= xmax → 未開始 → 不可視 ✗

この設計は精巧だ：xmin と xmax で大部分のトランザクションを高速に除外し、中間の区間に入るものだけ xip 配列を確認する必要がある。

分離レベル：いつスナップショットを撮るか

トランザクション分離レベルの本質的な違いは「何を見るか」ではなく、いつスナップショットを撮るかにある：

分離レベルスナップショット戦略効果
Read Committed各ステートメント開始時に新しいスナップショットを撮影同一トランザクション内で、2 回の SELECT が異なるデータを見る可能性がある
Repeatable Readトランザクション開始時に 1 回スナップショットを撮影し、トランザクション全体で再利用同一トランザクション内で、複数回の SELECT が一貫したデータを見る
SerializableRepeatable Read と同様 + 追加の競合検出シリアル実行をシミュレートし、競合検出時にロールバック
 
-- Read Committed：各ステートメントごとに新しいスナップショット
BEGIN;
SELECT count(*) FROM users;  -- スナップショット S1 → 100 行を参照
-- この時、別のトランザクションが 1 行挿入してコミット
SELECT count(*) FROM users;  -- スナップショット S2（新しい！）→ 101 行を参照
COMMIT;

-- Repeatable Read：トランザクション全体で 1 つのスナップショットを再利用
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM users;  -- スナップショット S1 → 100 行を参照
-- この時、別のトランザクションが 1 行挿入してコミット
SELECT count(*) FROM users;  -- まだスナップショット S1 → 依然として 100 行を参照
COMMIT;

コードレベルでは、GetTransactionSnapshot() がこの判断を行う：

  • Read Committed：呼び出しごとに新しいスナップショットを作成
  • Repeatable Read / Serializable：初回呼び出しでスナップショットを作成してキャッシュし、以降はキャッシュされたスナップショットを返却

いつスナップショットが必要か

すべての SQL がスナップショットを必要とするわけではない：

ステートメントタイプスナップショット要否理由
SELECT / INSERT / UPDATE / DELETE要テーブルデータにアクセスする必要があり、可視性判定が必須
CREATE TABLE / DROP TABLE / ALTER否DDL 操作はメタデータに対するもので、行レベルの可視性は関与しない
EXPLAIN場合によるEXPLAIN ANALYZE の場合は実際に実行するため必要

PG は analyze_requires_snapshot 関数で現在のステートメントにスナップショットが必要かどうかを判断し、不要な操作のためにスナップショットを作成するオーバーヘッドを回避する。

可視性判定の完全なルール

実際の可視性判定は「xmin がコミット済みで xmax が無効」よりはるかに複雑である。HeapTupleSatisfiesMVCC は以下のケースを処理する必要がある：

シナリオxmin の状態xmax の状態結果
挿入直後、トランザクション未コミット実行中—挿入者自身のみに可視
挿入されコミット済みコミット済み、スナップショット前無効 (0)可視
削除済み、削除トランザクション未コミットコミット済み実行中可視（削除はまだ有効でない）
削除されコミット済みコミット済みコミット済み、スナップショット前不可視（死亡）
更新済み、新バージョンが存在コミット済みコミット済み不可視（新バージョンを参照）
挿入トランザクションがロールバック済みロールバック済み—不可視（存在しなかった）

PG にはもう一つの最適化がある：t_infomask の hint bits。初回に行の xmin/xmax の状態を確認する際に CLOG（トランザクションコミットログ）を参照する必要があるが、確認後に t_infomask に「コミット済み」または「ロールバック済み」のマークを設定し、以降のチェックではマークを直接読み取って CLOG 参照をスキップする。

EPQ：同時更新の競合処理

スキャン中に Tuple が別のトランザクションによって修正されていることを検出すると、EvalPlanQual（EPQ）メカニズムがトリガーされる：

  • 通常のスキャン中にターゲット Tuple が同時修正されていることを検出
  • 修正トランザクションの完了（コミットまたはロールバック）を待機
  • 修正後の新バージョン Tuple を取得
  • 現在のクエリの WHERE 条件で新バージョンを再評価
  • 新バージョンが条件を満たす → 新バージョンを返却；そうでなければスキップ

これは Read Committed 分離レベルでの UPDATE と DELETE に必要なメカニズムである――修正中の行をスキップすることはできないし、旧バージョンに基づいて判断を下すこともできない。

スキャン方式の全体像

PG は複数のスキャン方式を提供し、それぞれ異なる accessMtd 関数ポインタを持つ：

スキャン方式accessMtd原理適用シナリオ
Sequential ScanSeqNextHeap 全体をページごとに走査フルテーブルスキャン、インデックスなし
Index ScanIndexNextB-Tree で特定 → ヒープアクセスで Tuple 取得高選択性クエリ
Index Only ScanIndexOnlyNextインデックスのみ読み取り、ヒープアクセスなしクエリ列がすべてインデックスに含まれる
Bitmap Heap ScanBitmapHeapNextまず全 TID を収集しソート、次に一括ヒープアクセス中程度の選択性、ランダム I/O 削減
TID ScanTidNextTID で直接特定CURRENT OF カーソル

すべてのスキャン方式は最終的に ExecScanExtended で統一的にディスパッチされる：

// 統一スキャンフレームワーク
ExecScanExtended(
    ScanState *node,
    ExecScanAccessMtd accessMtd,    // SeqNext / IndexNext / BitmapHeapNext / ...
    ExecScanRecheckMtd recheckMtd,  // 対応する recheck 関数
    EPQState *epqstate,             // 同時実行制御
    ExprState *qual,                // WHERE 条件
    ProjectionInfo *projInfo        // SELECT プロジェクション
);

この設計により、ストレージアクセスメソッドは完全にプラグ可能になっている――heapam_methods はデフォルトのヒープテーブル実装だが、PG の Table Access Method API によりストレージエンジン全体を置き換えることができる：

// heapam_methods：ヒープテーブルの関数テーブル
static const TableAmRoutine heapam_methods = {
    .scan_begin          = heap_beginscan,
    .scan_getnextslot    = heap_getnextslot,
    .index_fetch_tuple   = heapam_index_fetch_tuple,
    .tuple_insert        = heapam_tuple_insert,
    .tuple_delete        = heapam_tuple_delete,
    .tuple_update        = heapam_tuple_update,
    // ... 50+ の関数ポインタ
};

つまり理論的にはカラムナストレージエンジン、LSM-Tree エンジンなどを実装し、PG の実行フレームワークに組み込むことができる。Heap はデフォルトの選択であり、唯一の選択ではない。

インデックスタイプの全体像

PG は複数のインデックスタイプをサポートしており、それぞれ独自の IndexAmRoutine を持つ：

インデックスタイプamgettupleデータ構造適用シナリオ
B-Treebtgettuple平衡探索木、リーフノードリンクリスト等値クエリ、範囲クエリ、ソート（最も汎用的）
Hashhashgettupleハッシュバケット等値クエリ（B-Tree より高速だが範囲非対応）
GINgingettuple転置インデックス全文検索、配列包含、JSONB
GiSTgistgettuple汎用探索木幾何データ、範囲型、全文検索
BRIN—ブロック範囲インデックス時系列データなど物理的に順序付けされた大規模テーブル

これらは同一の index_getnext_slot 呼び出しフレームワークを共有し、rd_indam->amgettuple 関数ポインタで具体的な実装にディスパッチされる。

  │                                                           │
  │  ExecIndexScan                                            │
  │      │                                                    │

まとめ

覚えておくべき重要な直感：

  • インデックスにはバージョン情報が格納されない：インデックスは key → TID マッピングのみを格納し、すべての MVCC 判定はヒープアクセス後に行われる。つまりインデックススキャンで TID を取得しても、ヒープアクセス後に不可視だと判明して次の TID を取得する場合がある。Index Only Scan は唯一の例外――Visibility Map でヒープアクセスを回避する
  • スナップショットはデータ構造であり、時点ではない：SnapshotData の xmin/xmax で大部分のトランザクションを高速に除外し、xip 配列で中間区間を正確に判定する。分離レベルの違いは、いつスナップショットを撮るかだけ
  • 可視性 = xmin コミット済み + xmax 無効：これが最も核心的な直感。すべての複雑なケース（ロールバック、実行中、HOT チェーン）はこのルールの変種である
  • hint bits が CLOG の重複参照を回避：初回の可視性チェック後に行ヘッダにマークを設定し、以降のチェックでは直接マークを読み取る
  • ストレージエンジンはプラグ可能：TableAmRoutine が 50+ の関数ポインタを定義し、heapam はデフォルト実装に過ぎない。IndexAmRoutine も同様――B-Tree、Hash、GIN、GiST は同一の呼び出しフレームワークを共有する

次回は WAL と Checkpoint の世界に入る：一度の書き込みがどう WAL レコードになるか、WAL Writer がメモリ内のログをどうディスクにフラッシュするか、Checkpoint がどう一貫性回復ポイントを作成するか、そしてこれらすべてがレプリケーションと障害回復をどう支えるか。
